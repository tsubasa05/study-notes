Exceptions
例外

このHTMLは、Chapter 10「Exceptions」の9項目を、Java/Spring Bootで実装するときに使える判断基準として再構成したものです。原文の丸写しではなく、主張・設計意図・実装上の使い方を日本語で整理し、短い本書由来の例と学習用の追加コードを含めています。

結論： 例外処理は「エラーが起きたらtry-catchを書く技術」ではありません。APIの契約、失敗時の状態、呼び出し側に回復を強制するか、下位層の詳細を漏らさないか、デバッグ情報を残せるか、という設計問題です。

中心テーマ

例外は通常処理ではなく、異常系・契約違反・回復可能な失敗を表現するために使う。

API設計

呼び出し側がどう扱うべきかを、例外型・throws・Javadoc・メッセージで伝える。

実務接続

Spring Bootでは、ドメイン例外、アプリケーション例外、HTTPレスポンス変換を分ける。

Item 69

例外は「例外的状況」にだけ使う

この項目の主張は単純ですが、例外処理設計の土台です。例外を通常の制御フローに使ってはいけません。例外は「通常の分岐」ではなく、「通常の前提が崩れたこと」を表すための仕組みです。

本書の代表例：配列の終端を例外で検出する

本書では、配列を走査するときに ArrayIndexOutOfBoundsException を発生させ、それを捕まえてループ終了とみなす悪い例が示されます。これは「境界チェックを省略できて速そう」に見えますが、実際には読みづらく、遅く、バグを隠しやすいコードです。

// 悪い例: 例外で通常のループ終了を表現している
try {
    int i = 0;
    while (true) {
        range[i++].climb();
    }
} catch (ArrayIndexOutOfBoundsException e) {
    // 配列の終端に達した、という意味で握りつぶしている
}

文法レベルの解説

コード要素意味問題点
try { ... }このブロック内で発生した例外を、後続の catch で捕捉できるようにする。本来は異常時の処理を書くための構文なのに、ここでは通常のループ終了に使っている。
while (true)条件式が常に true なので、自力では終了しない無限ループ。ループ終了条件がコード上に見えない。読み手は「どこで終わるのか」を catch まで追う必要がある。
range[i++]i の現在値で配列要素にアクセスし、その後 i を1増やす。後置インクリメント。i が配列長以上になると ArrayIndexOutOfBoundsException が発生する。その失敗を終了条件として利用している。
catch (ArrayIndexOutOfBoundsException e)配列の範囲外アクセス例外を捕捉する。配列終端だけでなく、ループ内部のバグで同じ例外が出ても握りつぶしてしまう。

なぜ悪いコードなのか

  • 終了条件が隠れている：通常のループなら i < range.length や hasNext() のように終了条件が見えるべきです。このコードは「例外が出たら終わる」という暗黙ルールに依存しています。
  • 例外の意味を歪めている：ArrayIndexOutOfBoundsException は「配列アクセスの契約違反」を表す例外です。「配列の最後まで処理した」という正常な事実を表すものではありません。
  • バグを隠す：range[i++].climb() の内部で別の配列アクセスミスが発生しても、同じ catch に吸い込まれ、正常終了したように見える可能性があります。
  • 性能上も不利になりやすい：例外生成にはスタックトレース構築などのコストがあり、JVMの最適化も通常のループほど期待しにくくなります。

良いコード：通常の制御フローは通常の構文で書く

// 良い例: 拡張for文で「全要素を処理する」という意図を直接表現する
for (Mountain mountain : range) {
    mountain.climb();
}

このコードでは、range の各要素を順番に mountain に代入し、要素数分だけ処理します。配列や Iterable の走査という通常処理を、Javaの通常のループ構文で表現しています。

  • for (Mountain mountain : range) は、range に含まれる要素を1つずつ取り出すという意味です。
  • 終了条件はJavaの拡張for文が自然に扱います。配列の範囲外アクセスを意図的に起こす必要がありません。
  • climb() 内部で本当のバグが起きた場合、例外は握りつぶされず、異常として表面化します。

Iteratorの例：状態検査メソッドを用意する

本書は、API設計側にも責任があると説明します。利用者に「例外が出るまで呼び続ける」使い方を強制するAPIは悪いAPIです。Iterator はこの点でよくできています。

// 良いAPI利用: hasNext() で事前に状態を確認できる
Iterator<Foo> iterator = collection.iterator();
while (iterator.hasNext()) {
    Foo foo = iterator.next();
    process(foo);
}
メソッド役割設計上の意味
hasNext()次の要素が存在するかを true/false で返す。通常の分岐に使うための状態検査メソッド。
next()次の要素を返し、イテレータを次へ進める。次の要素があるという前提で呼ぶ状態依存メソッド。
NoSuchElementException次の要素がないのに next() を呼んだ場合の例外。通常終了ではなく、APIの使い方を誤ったことを表す。
 
// 悪いAPI利用: next() が失敗するまで回す
try {
    Iterator<Foo> iterator = collection.iterator();
    while (true) {
        Foo foo = iterator.next();
        process(foo);
    }
} catch (NoSuchElementException ignored) {
    // 終端を例外で表現している
}

状態検査メソッド vs Optional / null

本書は、状態依存メソッドに対して次の選択肢を示します。ここは Item 71 ともつながります。

選択肢向いている場面注意点
状態検査メソッドhasNext() のように、状態確認と実行の間に状態が変わらない場面並行実行や外部要因で状態が変わる場合はTOCTOU問題になる
Optional結果がないことが自然で、呼び出し側に明示的に扱わせたい場面失敗理由の詳細を持たせにくい
特別値 / null性能上 Optional を避けたい低レイヤー処理などチェック漏れがバグになりやすい
Spring Bootでの接続：Repositoryの findById が Optional<T> を返すのは、存在しないことを通常ケースとして扱えるからです。一方、認可違反、不正な状態遷移、API契約違反は例外として扱う価値があります。

Item 70

checked例外は回復可能な状況、runtime例外はプログラミングエラーに使う

Javaの例外設計で最も迷いやすいのが、checked exception と unchecked exception/runtime exception の使い分けです。本書の判断軸は明確です。

判断基準：呼び出し側に「その失敗を想定し、具体的な回復処理を書く責任」を負わせるべきなら checked 例外。API契約違反・事前条件違反・プログラミングミスなら runtime 例外。

分類

種類代表クラス意味呼び出し側の扱い
Checked exceptionIOException, 独自の InsufficientFundsException呼び出し側が回復可能な失敗catch するか throws で伝播させる必要がある
Runtime exceptionIllegalArgumentException, IllegalStateExceptionAPI契約違反、事前条件違反、プログラミングミス基本的に捕まえず、コードまたは呼び出し条件を修正すべき
ErrorOutOfMemoryError, StackOverflowErrorJVMや実行環境レベルの重大問題通常はアプリケーションで回復しようとしない

checked例外が向く例：ただし「回復」の意味を誤解しない

本書では、ギフトカード残高不足のように、呼び出し側が不足額を知ることで購入者に不足額を伝えられるケースが checked 例外の例として示されます。ただし、ここは誤解しやすいです。

補足：「別の支払い手段を促せる」とは、必ずしもプログラムが自動で別決済を選ぶという意味ではありません。呼び出し側のプログラム、たとえばアプリケーションサービスやUI層が、例外から不足額などの構造化情報を取得し、ユーザに再選択を促せる、という意味です。
// 学習用に再構成した例: 回復に必要な情報を持つ checked exception
public class InsufficientBalanceException extends Exception {
    private final int shortageYen;

    public InsufficientBalanceException(int shortageYen) {
        super("Insufficient balance. shortageYen=" + shortageYen);
        this.shortageYen = shortageYen;
    }

    public int shortageYen() {
        return shortageYen;
    }
}

public void purchase(GiftCard card, int priceYen)
        throws InsufficientBalanceException {
    int shortage = priceYen - card.balanceYen();
    if (shortage > 0) {
        throw new InsufficientBalanceException(shortage);
    }
    card.debit(priceYen);
}

支払い例をどう判断すべきか

あなたの疑問は正しいです。「ユーザが別の支払い手段を選ぶだけで、プログラム内で解決しないなら checked 例外として不適切では？」という観点はかなり重要です。結論は、API境界によって変わります。

状況checked例外が妥当か理由
決済ライブラリのような低〜中レイヤーAPIで、呼び出し側が不足額を使って画面表示・再入力・別手段選択フローへ進める 妥当になり得る 呼び出し側に「この失敗は想定済みなので、処理方針を決めろ」と強制する価値がある。例外に shortageYen() のようなアクセサを持たせる意味もある。
Web APIのService内部で、最終的にはHTTP 400/409などに変換してフロントへ返すだけ runtime例外の方が実務上自然なことが多い Javaの呼び出し側がその場で業務処理を継続するというより、リクエストを失敗レスポンスに変換するだけだから。Springでは @ControllerAdvice で扱う設計と相性がよい。
呼び出し側ができることがログ出力、エラー終了、上位へ丸投げだけ 不適切 checked例外にしても有用な回復処理がない。Item 71の「不要なchecked例外」に該当しやすい。

つまり、「ユーザに選ばせるから checked 例外として絶対に不適切」とまでは言い切れません。プログラムの回復処理には、自動的に代替処理を実行するだけでなく、ユーザに判断材料を提示して正常な次アクションへ誘導することも含められます。ただし、Spring Bootの一般的なWeb APIでは、checked例外よりもドメイン上のruntime例外として表し、HTTPレスポンスや画面表示に変換する方が扱いやすい場面が多いです。

runtime例外が向く例：API契約違反

public void repeat(int times) {
    if (times < 0) {
        throw new IllegalArgumentException("times must be >= 0: " + times);
    }
    for (int i = 0; i < times; i++) {
        runOnce();
    }
}

この例では、times に負数を渡すこと自体がメソッド契約違反です。呼び出し側が catch して回復するというより、そもそも負数を渡さないように呼び出しコードを修正すべきです。したがって runtime例外が適切です。

実務上の注意

  • Error のサブクラスを独自に作るべきではありません。通常、独自に作る unchecked throwable は RuntimeException のサブクラスにします。
  • checked例外には、回復に使えるアクセサを持たせる価値があります。例外メッセージの文字列をパースさせる設計は脆いです。
  • 判断に迷うなら unchecked 例外寄りにする、というのが本書の実践的な立場です。理由は Item 71 で扱います。
Spring Bootでの接続：Web APIでは、入力不正・存在しないID・不正な状態遷移を runtime 例外として投げ、@ControllerAdvice でHTTP 400/404/409などに変換する設計が多いです。Java APIとして呼び出し側に回復処理を強制したい明確な理由がないなら、checked例外は慎重に使うべきです。

Item 71

checked例外の不要な使用を避ける

checked例外は、正しく使えば信頼性を高めます。しかし乱用すると、API利用者に try-catch や throws を強制し、コードを扱いにくくします。Java 8以降では、checked例外を投げるメソッドがStream APIと相性が悪い点も重くなっています。

checked例外を使う条件

checked例外が正当化されるのは、適切にAPIを使っても防げない失敗であり、かつ呼び出し側が有用な回復処理を実行できる場合です。

呼び出し側が結局こう書くしかないなら、checked例外にする意味は弱いです。

// 悪い兆候1: 起こらない前提でAssertionErrorにするだけ
try {
    obj.action();
} catch (TheCheckedException e) {
    throw new AssertionError(e); // Can't happen
}
// 悪い兆候2: 出力して終了するしかない
try {
    obj.action();
} catch (TheCheckedException e) {
    e.printStackTrace();
    System.exit(1);
}

代替案1：Optionalを返すメリット

Optional は、「結果が存在しない」ことを例外ではなく戻り値の型で表現します。これは、失敗というより「見つからないことも普通にあり得る」ケースに向いています。

// checked例外を使わず、見つからないことを通常ケースとして表現する
public Optional<User> findUser(UserId id) {
    return userRepository.findById(id);
}

findUser(id).ifPresentOrElse(
    user -> use(user),
    () -> handleNotFound(id)
);
メリット意味
例外制御にならない「存在しない」ことを通常の戻り値として扱えるため、try-catch が不要になる。
呼び出し側に不在ケースを意識させる戻り値が Optional<User> なので、利用者は「ない場合」を考えざるを得ない。
メソッドの意味が明確になるfindUser は「見つからない可能性がある検索」、getRequiredUser は「存在しなければ異常」といった区別を型と名前で表しやすい。
Streamや関数型APIと組み合わせやすいchecked例外を投げるメソッドより、ラムダ式やStream処理に載せやすい。
注意：Optional が向くのは「失敗理由の詳細が不要」または「不在そのものが主要な情報」の場合です。残高不足額、失敗した制約、リトライ可能時刻など、回復に必要な詳細情報が多いなら、Optionalだけでは情報不足になります。

代替案2：状態検査メソッドに分けるメリット

状態検査メソッドとは、「実行できるか」を事前に確認するメソッドです。代表例は Iterator.hasNext() と Iterator.next() の組み合わせです。

// checked例外版: confirm() を呼ぶまで失敗するか分からない
try {
    reservation.confirm();
} catch (CannotConfirmReservationException e) {
    handle(e);
}

// 状態検査メソッド版: 通常のif文で分岐できる
if (reservation.canConfirm()) {
    reservation.confirm();
} else {
    handleCannotConfirm(reservation);
}
メリット意味
通常の分岐として読めるif (reservation.canConfirm()) は、業務ルール上の分岐として読みやすい。例外を読まないと通常ケースが分からない、という状態を避けられる。
APIの使い方を明示できるcanConfirm() があることで、「confirm前に確認できる状態がある」と利用者に伝わる。
例外を本当の異常に残せるcanConfirm() が false の状態で confirm() を呼んだ場合は、プログラム側の呼び出しミスとして IllegalStateException を投げられる。
UI制御に使いやすいボタンの活性/非活性、確認メッセージ表示など、実行前に状態を見たい場面で扱いやすい。

Optionalと状態検査メソッドの使い分け

観点Optionalが向く状態検査メソッドが向く
代表例findById(id), tryParse(text), poll()hasNext() + next(), canConfirm() + confirm()
結果がない場合戻り値として Optional.empty() を返す。canXxx() が false を返し、呼び出し側が xxx() を呼ばない。
並行実行への強さ比較的強い。確認と取得が1メソッド内で完結するため、確認後に状態が変わる問題を避けやすい。弱い場合がある。canXxx() と xxx() の間に状態が変わると危険。
読みやすさ「ないかもしれない値」を型で表せる。業務上の可能/不可能を if で明示できる。
失敗理由の表現弱い。empty だけでは理由を持てない。理由を別メソッドや状態オブジェクトで表現すれば強くできるが、設計はやや増える。

ただし状態検査メソッドには限界がある

  • 並行実行で、canConfirm() と confirm() の間に状態が変わるなら危険。
  • 確認処理と実行処理で同じ重い計算が必要なら、性能上不利。
  • 呼び出し側が常に回復処理を書くべきケースなら、checked例外のままでもよい。

より実務的な代替：結果型を返す

Java標準の話からは少し外れますが、業務アプリでは Optional だけでは理由が足りず、checked例外は重すぎることがあります。その場合、成功/失敗と理由を持つ結果型を返す設計もあります。

public sealed interface ConfirmResult {
    record Success(Reservation reservation) implements ConfirmResult {}
    record Failure(String reasonCode, String message) implements ConfirmResult {}
}

public ConfirmResult tryConfirm(Reservation reservation) {
    if (!reservation.canConfirm()) {
        return new ConfirmResult.Failure("CANNOT_CONFIRM", "予約を確定できない状態です");
    }
    reservation.confirm();
    return new ConfirmResult.Success(reservation);
}

これは「例外にするほど異常ではないが、Optional.empty() だけでは情報が足りない」場合の現実的な逃げ道です。

厳しめに言うと：「なんとなく業務例外だからchecked」という設計は弱いです。呼び出し側が何を回復できるのか説明できないchecked例外は、API利用者への負債です。

Item 72

標準例外を優先して使う

独自例外を作る前に、Java標準の例外で意味を表せないかを検討します。標準例外を再利用すると、API利用者が意味を理解しやすく、コードも読みやすくなります。

よく使う標準例外

例外使う場面例
IllegalArgumentExceptionnull以外の引数値が不適切回数に負数が渡された
IllegalStateExceptionオブジェクト状態がメソッド呼び出しに不適切初期化前に実行された、既に確定済みの予約を変更しようとした
NullPointerExceptionnull禁止の引数にnullが渡されたObjects.requireNonNull(name)
IndexOutOfBoundsExceptionインデックス値が範囲外list.get(-1)
ConcurrentModificationException禁止された並行変更を検出イテレーション中にコレクションが変更された
UnsupportedOperationExceptionそのオブジェクトが操作をサポートしない読み取り専用Listに add する

Objects.requireNonNull() の意味

Objects.requireNonNull(x, "x") は、引数 x が null ではないことをチェックするための標準メソッドです。null なら NullPointerException を投げ、null でなければその値をそのまま返します。

this.name = Objects.requireNonNull(name, "name");

上のコードは、概念的には次のコードとほぼ同じです。

if (name == null) {
    throw new NullPointerException("name");
}
this.name = name;

requireNonNull の利点は、nullチェックと代入を1行で書けること、そして「この引数はnull禁止である」という意図が標準的な形で伝わることです。

書き方動作
Objects.requireNonNull(id)id が null なら NullPointerException。メッセージは簡素。
Objects.requireNonNull(id, "id")id が null なら、メッセージ付きの NullPointerException。
this.id = Objects.requireNonNull(id, "id")nullチェック後、問題なければ戻り値をそのままフィールドに代入する。

コード例

public User findRequired(UserId id) {
    Objects.requireNonNull(id, "id"); // idがnullならNullPointerException

    return userRepository.findById(id)
        .orElseThrow(() -> new IllegalArgumentException("unknown userId: " + id));
}

public void ship(Order order) {
    Objects.requireNonNull(order, "order"); // orderがnullならNullPointerException
    if (!order.isPaid()) {
        throw new IllegalStateException("order is not paid: orderId=" + order.id());
    }
    order.ship();
}

IllegalArgumentException とは何か

IllegalArgumentException は、メソッドに渡された引数の値が不正であることを表す runtime例外です。ポイントは、オブジェクトの状態ではなく、呼び出し時に渡した値がメソッド契約に反していることです。

public void setRetryCount(int retryCount) {
    if (retryCount < 0) {
        throw new IllegalArgumentException(
            "retryCount must be >= 0: " + retryCount);
    }
    this.retryCount = retryCount;
}

この例では、retryCount は0以上であるべきです。負数を渡すのは呼び出し側の契約違反なので、IllegalArgumentException が自然です。

IllegalStateException とは何か

IllegalStateException は、オブジェクトの現在状態が、そのメソッド呼び出しを許さないことを表す runtime例外です。引数そのものが悪いというより、「今この状態でその操作を呼んではいけない」という意味です。

public void ship(Order order) {
    Objects.requireNonNull(order, "order");
    if (!order.isPaid()) {
        throw new IllegalStateException(
            "Cannot ship unpaid order: orderId=" + order.id());
    }
    order.ship();
}

この例では、order という引数がnullではない限り、引数の型自体は正しいです。しかし注文が未払いという状態では出荷できません。そのため IllegalStateException が自然です。

IllegalArgumentException と IllegalStateException の境界

本書は、カードデッキから手札を配る例で境界の難しさを説明します。要求された手札サイズが残カード数より大きい場合、引数が悪いとも、デッキ状態が悪いとも言えます。判断基準はこうです。

どんな引数でも成功しない状態なら IllegalStateException。別の引数なら成功したなら IllegalArgumentException。
public List<Card> deal(int handSize) {
    if (handSize < 0) {
        throw new IllegalArgumentException("handSize must be >= 0: " + handSize);
    }
    if (deck.isEmpty()) {
        throw new IllegalStateException("deck is empty");
    }
    if (handSize > deck.size()) {
        throw new IllegalArgumentException(
            "handSize exceeds remaining cards: handSize=" + handSize
                + ", remaining=" + deck.size());
    }
    return draw(handSize);
}
ケース選ぶ例外理由
handSize == -1IllegalArgumentException負の枚数を配るという引数値が不正。
デッキが空IllegalStateExceptionどんな正の枚数を指定しても配れない。デッキの状態が不正。
残り3枚なのに5枚要求IllegalArgumentException または設計次第2枚なら成功するため、引数が現在状態に対して大きすぎると見るなら IllegalArgumentException。API仕様で「残枚数不足」を状態問題として扱うなら専用例外や IllegalStateException もあり得る。

避けるべき例外

Exception, RuntimeException, Throwable, Error を直接投げる設計は避けます。抽象的すぎて、呼び出し側が意味に応じた処理を書けません。

// 悪い例: 何が悪いのか型から分からない
throw new RuntimeException("bad request");

// 良い例: 引数が不正だと型から分かる
throw new IllegalArgumentException("userId must be positive: " + userId);

Item 73

抽象化に適した例外を投げる

上位層のメソッドが、下位層の実装詳細に依存した例外をそのまま投げると、APIが汚染されます。利用者から見て「なぜこの処理でその例外が出るのか」が分からない状態は、抽象化の破綻です。

例外翻訳

下位層の例外を捕まえ、上位層の抽象度で説明できる例外に変換します。

// 例外翻訳の基本形
try {
    lowerLevelOperation();
} catch (LowerLevelException e) {
    throw new HigherLevelException("failed to perform business operation", e);
}

本書の代表例：List#get の抽象に合わせる

本書では、List の get(index) が、内部のイテレータ由来の NoSuchElementException をそのまま漏らすのではなく、IndexOutOfBoundsException に変換する例が示されます。

public E get(int index) {
    ListIterator<E> iterator = listIterator(index);
    try {
        return iterator.next();
    } catch (NoSuchElementException e) {
        throw new IndexOutOfBoundsException("Index: " + index);
    }
}

例外チェイニング

下位層の原因がデバッグに役立つ場合は、原因例外を cause として保持します。これにより、上位層には適切な例外を見せつつ、障害調査では根本原因を追えます。

public class OrderImportException extends RuntimeException {
    public OrderImportException(String message, Throwable cause) {
        super(message, cause);
    }
}

public void importOrders(Path csv) {
    try {
        csvReader.read(csv);
    } catch (IOException e) {
        throw new OrderImportException("failed to import orders: file=" + csv, e);
    }
}

Spring Bootでの層別例外設計

// Infrastructure層: JDBCや外部APIの低レベル例外を扱う
class UserRepositoryAdapter implements UserRepository {
    public User findById(UserId id) {
        try {
            return jdbcTemplate.queryForObject(...);
        } catch (DataAccessException e) {
            throw new UserRepositoryException("failed to load user: id=" + id, e);
        }
    }
}

// Application層: ユースケースとして意味のある例外に変換する
class GetUserUseCase {
    public UserDto execute(UserId id) {
        try {
            return repository.findById(id).toDto();
        } catch (UserRepositoryException e) {
            throw new UserQueryFailedException("failed to get user: id=" + id, e);
        }
    }
}
注意：例外翻訳は万能ではありません。そもそも下位層の例外を避けられるなら、先に入力検証や事前条件チェックをすべきです。翻訳しすぎると、逆に原因が追いにくくなります。

Item 74

各メソッドが投げる例外を文書化する

例外はAPI契約の一部です。どの条件でどの例外が出るのかが文書化されていないAPIは、呼び出し側にとって安全に使えません。

checked例外は個別に宣言し、Javadocにも書く

/**
 * 指定したファイルから設定を読み込む。
 *
 * @param path 設定ファイルのパス
 * @return 読み込んだ設定
 * @throws IOException ファイルを開けない、または読み取りに失敗した場合
 * @throws ConfigFormatException 設定ファイルの形式が不正な場合
 */
public Config load(Path path) throws IOException, ConfigFormatException {
    ...
}

悪い例：広すぎるthrows

// 悪い例: 何が起こるのか利用者に伝わらない
public Config load(Path path) throws Exception {
    ...
}

unchecked例外もJavadocに書く。ただしthrows句には書かない

unchecked例外は、呼び出し側がcatchを強制されるものではありません。しかし、多くの場合は「メソッドの事前条件」を表します。つまり、Javadocに書く価値があります。

/**
 * 注文を出荷済みにする。
 *
 * @param orderId 注文ID
 * @throws NullPointerException orderId が null の場合
 * @throws IllegalStateException 注文が支払い済みでない場合、または既にキャンセル済みの場合
 */
public void ship(OrderId orderId) {
    ...
}

クラス全体で共通する例外はクラスコメントにまとめてもよい

/**
 * ユーザー管理サービス。
 *
 * <p>このクラスの全publicメソッドは、null禁止の引数にnullが渡された場合、
 * NullPointerExceptionを投げる。</p>
 */
public class UserService {
    ...
}
Spring Bootでの接続：Controllerのメソッド単位ではなく、ServiceやDomainのpublicメソッドに「どの前提条件を破るとどの例外が出るか」を書く方が設計資料として価値があります。ControllerではそれをHTTPエラー仕様に接続します。

Item 75

詳細メッセージには失敗を捕捉する情報を含める

本番障害では、開発者が再現環境を持てないことがあります。そのとき頼れるのは、ログとスタックトレースです。例外メッセージに失敗時の値が入っていないと、調査コストが一気に上がります。

入れるべき情報

  • 失敗に関与したパラメータ値
  • 不変条件を破ったフィールド値
  • 期待された範囲・実際の値
  • 業務識別子：userId, orderId, requestId など

本書の例の考え方：IndexOutOfBoundsException

範囲外インデックスなら、単に「範囲外」ではなく、下限・上限・実際のindexがあると原因を絞れます。

public class DetailedIndexOutOfBoundsException extends IndexOutOfBoundsException {
    private final int lowerBound;
    private final int upperBound;
    private final int index;

    public DetailedIndexOutOfBoundsException(int lowerBound, int upperBound, int index) {
        super("lowerBound=" + lowerBound + ", upperBound=" + upperBound + ", index=" + index);
        this.lowerBound = lowerBound;
        this.upperBound = upperBound;
        this.index = index;
    }

    public int lowerBound() { return lowerBound; }
    public int upperBound() { return upperBound; }
    public int index() { return index; }
}

悪い例と良い例

// 悪い例: 情報が足りない
throw new IllegalArgumentException("invalid amount");

// 良い例: 調査に必要な値がある
throw new IllegalArgumentException(
    "amount must be positive: amount=" + amount + ", orderId=" + orderId);

// さらに良い例: 例外型にフィールドとして保持する
throw new InvalidPaymentAmountException(orderId, amount);

セキュリティ上の注意

パスワード、アクセストークン、秘密鍵、個人情報などを例外メッセージに含めてはいけません。スタックトレースはログ、監視、SRE、サポートなど多くの人が見る可能性があります。

// 悪い例: tokenをログやスタックトレースに出してしまう
throw new AuthenticationException("invalid token: " + accessToken);

// 良い例: 識別に必要な安全な情報だけ出す
throw new AuthenticationException("invalid token: tokenId=" + tokenId + ", userId=" + userId);
実務ではここが効きます：例外メッセージが雑なコードは、障害対応で確実に時間を失います。「何がダメだったか」ではなく「どの値が、どの期待条件を、どう外れたか」を残してください。

Item 76

失敗原子性を保つ

失敗原子性とは、メソッドが例外を投げたとき、オブジェクトが呼び出し前と同じ状態に保たれる性質です。特にchecked例外では、呼び出し側が回復する前提なので、失敗後もオブジェクトが使えることが重要です。

本書の代表例：Stack.pop

空スタックをpopする場合、先にサイズを検査すれば、状態を壊さずに適切な例外を投げられます。

public Object pop() {
    if (size == 0) {
        throw new EmptyStackException();
    }
    Object result = elements[--size];
    elements[size] = null; // 古い参照を消す
    return result;
}

もし先頭の検査なしに --size してから配列アクセスし、そこで例外が出ると、size が負数になるなど、オブジェクトが壊れた状態になり得ます。

失敗原子性を実現する方法

方法説明例
不変オブジェクトにする既存状態を変更しないため、失敗しても既存インスタンスは壊れないrecord, value object
変更前に検証するパラメータや状態の検査を先に済ませるif (size == 0) throw ...
失敗し得る処理を先に行うオブジェクト変更前に、型チェック・計算・外部検証を済ませるTreeMap に追加前に比較で失敗する
一時コピーで処理するコピー上で処理し、成功後に本体へ反映するソート、バッチ更新
ロールバックする途中失敗を捕捉して変更前に戻す永続データ構造、トランザクション

悪い例：途中まで変更してから失敗する

// 悪い例: 途中で例外が出るとtagsが一部だけ変更される
public void replaceTags(List<String> newTags) {
    tags.clear();
    for (String tag : newTags) {
        if (tag.isBlank()) {
            throw new IllegalArgumentException("blank tag");
        }
        tags.add(tag);
    }
}

良い例：検証してから変更する

public void replaceTags(List<String> newTags) {
    Objects.requireNonNull(newTags, "newTags");
    List<String> copy = new ArrayList<>(newTags);
    for (String tag : copy) {
        if (tag == null || tag.isBlank()) {
            throw new IllegalArgumentException("tag must not be blank: " + tag);
        }
    }

    tags.clear();
    tags.addAll(copy);
}
Spring Bootでの接続：DB更新ではトランザクションが失敗原子性を担います。ただし、メモリ上のドメインオブジェクトや集約の状態変更は、トランザクション以前に壊れる可能性があります。ドメインメソッド内でも「検証してから変更」を徹底してください。

Item 77

例外を無視しない

空の catch ブロックは、例外処理の目的を破壊します。API設計者が例外を宣言しているのは、呼び出し側に何かを伝えたいからです。

悪い例：空catch

// 悪い例: 例外を完全に握りつぶしている
try {
    sendNotification(user);
} catch (NotificationException e) {
}

最低限やるべきこと

  • 回復できるなら回復処理を書く。
  • 回復できないなら伝播させる。
  • 無視するなら、なぜ無視できるのかコメントを書く。
  • 変数名を ignored にして、意図的であることを示す。
  • 必要ならログに残す。

本書の代表例：タイムアウトしたらデフォルト値を使う

本書では、計算結果が望ましいが必須ではない場合に、タイムアウトや実行失敗を意図的に無視し、デフォルト値を使う例が示されます。

Future<Integer> future = exec.submit(planarMap::chromaticNumber);
int numColors = 4; // デフォルト値
try {
    numColors = future.get(1L, TimeUnit.SECONDS);
} catch (TimeoutException | ExecutionException ignored) {
    // デフォルト値を使う。最小色数は望ましいが必須ではない。
}

ログに残す例

try {
    cache.refresh();
} catch (CacheRefreshException e) {
    logger.warn("cache refresh failed; continue with stale cache", e);
}
率直に：空catchは、ほぼバグです。レビューで見つけたら止めるべきです。例外を無視してよいケースはありますが、その場合こそ理由をコードに残す必要があります。

Spring BootでChapter 10を実装に落とすなら

本書はJava一般のAPI設計の本なので、Spring Boot特有の @ControllerAdvice やHTTPステータス設計を直接深く扱うわけではありません。ですが、Chapter 10の原則はSpringアプリの例外設計にもそのまま効きます。

推奨レイヤー構成

Controller原則として例外を直接作り込まず、入力バリデーションとレスポンス変換を担当@Valid, @ControllerAdvice

ControllerAdviceの例

@RestControllerAdvice
public class ApiExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    ResponseEntity<ErrorResponse> handleIllegalArgument(IllegalArgumentException e) {
        return ResponseEntity.badRequest()
            .body(new ErrorResponse("BAD_REQUEST", e.getMessage()));
    }

    @ExceptionHandler(IllegalStateException.class)
    ResponseEntity<ErrorResponse> handleIllegalState(IllegalStateException e) {
        return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(new ErrorResponse("CONFLICT", e.getMessage()));
    }

    @ExceptionHandler(UserNotFoundException.class)
    ResponseEntity<ErrorResponse> handleUserNotFound(UserNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse("USER_NOT_FOUND", e.getMessage()));
    }
}

まとめ

Chapter 10は、例外処理を「try-catch構文」ではなく、失敗をどうAPI契約として表現するかの章として読むべきです。

  1. 通常フローに例外を使わない。
  2. checked例外は、呼び出し側が本当に回復できる場合に限定する。
  3. 標準例外を優先する。
  4. 下位層の例外を上位層の抽象度に翻訳する。
  5. 例外は文書化し、メッセージには調査に必要な情報を入れる。
  6. 失敗時にも状態を壊さない。
  7. 例外を握りつぶさない。
あなたの現在の学習に対する優先順位：まず Item 70, 72, 73, 75, 76 を重点的に読むべきです。Spring Bootの実装では、特に「runtime例外で業務エラーを表し、ControllerAdviceでHTTPレスポンスに変換する」「Repositoryなど下位層の例外をService/UseCaseの抽象に翻訳する」感覚を身につけるのが効果的です。
