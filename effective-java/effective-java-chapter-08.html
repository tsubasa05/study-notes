メソッド

第8章は、Javaのメソッドを「単に動く処理」ではなく、他者が安全に使えるAPIとして設計するための章です。 テーマは、パラメータ検証、防御的コピー、シグネチャ設計、オーバーロード、可変長引数、戻り値設計、Optional、Javadocです。

対象: Item 49〜56 目的: APIの使いやすさ・堅牢性・保守性 Java/Springの実装学習向け補足あり コード例つき

第8章の全体像

この章の本質は、メソッド境界をどう設計するかです。メソッド境界とは、呼び出し側と実装側の契約です。 境界設計が悪いと、バグは実装内部ではなく、APIの使われ方として広がります。

入力側の設計

  • 受け取れる値の制約を明示し、早く検出する。
  • mutableな引数を内部に保持するならコピーする。
  • パラメータ数・型・名前を、呼び出し側が間違えにくい形にする。

出力側の設計

  • 空を表すために null を返さない。
  • 値がない可能性を呼び出し側に意識させたい場合だけ Optional を使う。
  • 戻り値の意味・例外・副作用をJavadocで契約として残す。
厳しめに言うと: 「内部実装を頑張ればよい」という考えは弱いです。Javaで中級以上を目指すなら、メソッドの引数・戻り値・例外・ドキュメントを含めて設計できないと、API利用者に負債を押し付けるコードになります。

Item 49: Check parameters for validity
パラメータの妥当性を検査する

メソッドやコンストラクタには、ほぼ必ず「受け取ってよい値」の制約があります。たとえば、インデックスは0以上、参照はnull不可、期間の開始日は終了日以前、などです。 本Itemの主張は単純で、制約はJavadocで明示し、メソッド冒頭で検証するというものです。

狙い: 失敗をできるだけ早く、原因に近い場所で、適切な例外として発生させる。これにより、バグの調査範囲が狭くなり、failure atomicity、つまり失敗時にオブジェクトを壊れた状態にしない性質も守りやすくなります。

本書掲載コード: BigInteger.mod のパラメータ検証

/**
 * Returns a BigInteger whose value is (this mod m). This method
 * differs from the remainder method in that it always returns a
 * non-negative BigInteger.
 *
 * @param m the modulus, which must be positive
 * @return this mod m
 * @throws ArithmeticException if m is less than or equal to 0
 */
public BigInteger mod(BigInteger m) {
    if (m.signum() <= 0)
        throw new ArithmeticException("Modulus <= 0: " + m);
    ... // Do the computation
}

この例では、m は正でなければならない、という制約を @param と @throws で記述し、メソッド冒頭で検証しています。 ここで重要なのは、計算を始める前に制約違反を止めることです。途中まで処理してから落ちると、例外の意味が不明確になり、場合によってはオブジェクトの状態が中途半端に変更されます。

null検査は Objects.requireNonNull を使う

// Inline use of Java's null-checking facility
this.strategy = Objects.requireNonNull(strategy, "strategy");

Objects.requireNonNull は、引数が null なら NullPointerException を投げ、非nullならその値を返します。 そのため、フィールド代入とnull検査を同時に書けます。SpringのServiceやComponentでも、コンストラクタで依存オブジェクトを受け取る場合に有効です。

注意: requireNonNull は「nullでない」ことしか保証しません。数値範囲、文字列の空、業務上の整合性は別途チェックが必要です。

非公開メソッドではassertを使える

// Private helper function for a recursive sort
private static void sort(long a[], int offset, int length) {
    assert a != null;
    assert offset >= 0 && offset <= a.length;
    assert length >= 0 && length <= a.length - offset;
    ... // Do the computation
}

private や package-private のメソッドは、呼び出し元を自分たちのパッケージ内で制御できます。そのため「この条件は必ず満たして呼ぶ」という内部契約を assert で表現できます。 ただし、assertは通常実行時には無効です。-ea を付けた時だけ有効になるため、公開APIの入力検証をassertに任せてはいけません。

補足例: Spring/業務アプリでの考え方

public User findUser(UserId id) {
    // public APIなら、責務として明示的に検証する
    Objects.requireNonNull(id, "id");
    return repository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
}

private int sliceLength(List<?> list, int from, int to) {
    // 非公開メソッドなら、呼び出し側を自分で制御できるため assert で内部契約を表現できる
    assert list != null;
    assert 0 <= from && from <= to && to <= list.size();
    return to - from;
}

このItemの判断基準

  • 公開APIなら、制約をJavadocやAPI仕様に書き、明示的に検証する。
  • コンストラクタでは特に厳密に検証する。壊れたオブジェクトを生成させない。
  • 後で使うために保存する引数は、保存前に検証する。後から失敗すると原因追跡が難しくなる。
  • 計算中に自然に適切な例外が出る場合は、事前検証を省けることもある。ただし、例外種別が仕様と違うなら例外変換を検討する。

Item 50: Make defensive copies when needed
必要な場合は防御的コピーを行う

Javaはメモリ安全な言語ですが、それだけではクラスの不変条件は守れません。mutableなオブジェクトの参照を外部と共有すると、外部コードから内部状態を壊されます。 本Itemは、mutableな引数や内部状態を外部と共有しないための防御的コピーを扱います。

本書掲載コード: 壊れたimmutableクラス

// Broken "immutable" time period class
public final class Period {
    private final Date start;
    private final Date end;

    /**
     * @param start the beginning of the period
     * @param end the end of the period; must not precede start
     * @throws IllegalArgumentException if start is after end
     * @throws NullPointerException if start or end is null
     */
    public Period(Date start, Date end) {
        if (start.compareTo(end) > 0)
            throw new IllegalArgumentException(start + " after " + end);
        this.start = start;
        this.end = end;
    }

    public Date start() {
        return start;
    }

    public Date end() {
        return end;
    }
}

一見すると final フィールドだけで構成された不変クラスに見えます。しかし Date はmutableです。 final が保証するのは「参照先を差し替えられない」ことであって、「参照先オブジェクトの中身が変わらない」ことではありません。

攻撃例1: コンストラクタに渡したDateを後から変更する

// Attack the internals of a Period instance
Date start = new Date();
Date end = new Date();
Period p = new Period(start, end);
end.setYear(78); // Modifies internals of p!

Period は end への参照をそのまま保持しているため、呼び出し側が end.setYear(...) を実行すると、Period の内部状態も変わります。 これはカプセル化の破綻です。

改善: コンストラクタで防御的コピーを作る

// Repaired constructor - makes defensive copies of parameters
public Period(Date start, Date end) {
    this.start = new Date(start.getTime());
    this.end = new Date(end.getTime());

    if (this.start.compareTo(this.end) > 0)
        throw new IllegalArgumentException(this.start + " after " + this.end);
}
重要: コピーしてから検証します。検証してからコピーすると、別スレッドが「検証後・コピー前」の隙間で元オブジェクトを書き換える可能性があります。この種の問題はTOCTOU、time-of-check/time-of-use攻撃と呼ばれます。

また、本書では Date の clone を使ってコピーしない点も強調されます。Date はfinalではないため、悪意あるサブクラスの clone が返る可能性を排除できません。

攻撃例2: アクセサが内部Dateをそのまま返す

// Second attack on the internals of a Period instance
Date start = new Date();
Date end = new Date();
Period p = new Period(start, end);
p.end().setYear(78); // Modifies internals of p!

コンストラクタでコピーしても、getterが内部のmutableオブジェクトをそのまま返すと、外部から内部状態を変更されます。

改善: アクセサでも防御的コピーを返す

// Repaired accessors - make defensive copies of internal fields
public Date start() {
    return new Date(start.getTime());
}

public Date end() {
    return new Date(end.getTime());
}

現代Javaでの第一選択

新規コードでは、Date ではなく Instant、LocalDateTime、ZonedDateTime などの不変な java.time 型を優先します。 mutableな型をAPIに出さない設計にすれば、防御的コピーの必要性自体を減らせます。

補足例: Listを内部保持する場合

public final class Receipt {
    private final List<LineItem> items;

    public Receipt(List<LineItem> items) {
        // List自体がmutableなので、外部のList参照を保持しない
        this.items = List.copyOf(Objects.requireNonNull(items, "items"));
    }

    public List<LineItem> items() {
        // List.copyOfで不変化しているため、そのまま返しても安全
        return items;
    }
}

このItemの判断基準

状況対応
mutableな引数を内部に保持するコンストラクタ/メソッドでコピーしてから保持する
mutableな内部状態を返すコピーまたは不変ビューを返す
配列を返す配列は常にmutableなので、原則コピーする
呼び出し側から所有権を受け取るコピー省略もあり得るが、ドキュメントで明確に所有権移転を説明する
コストが大きすぎる信頼境界と性能測定を踏まえて、コピー省略を文書化する

Item 51: Design method signatures carefully
メソッドシグネチャを慎重に設計する

メソッド名、引数の数、引数の型、戻り値型は、APIの使いやすさを決めます。ここでの設計ミスは、実装内部ではなく呼び出し側全体に波及します。

1. メソッド名は慎重に選ぶ

名前は標準命名規則に従い、同じパッケージ内で一貫しているべきです。長すぎる名前も避けます。ただし短さを優先して意味が曖昧になるなら失敗です。 迷ったらJDK標準APIの命名を参考にします。

2. 便利メソッドを増やしすぎない

便利メソッドは一見親切ですが、増やしすぎるとAPIの学習・テスト・保守・ドキュメント化の負担が増えます。 特にinterfaceでは、実装クラス側の負担も増えます。本当に頻繁に使われる省略形だけを提供するのが基本です。

3. パラメータリストは短くする

本書は目安として、4個以下を推奨します。長い引数リスト、特に同じ型が連続する引数は危険です。順番を間違えてもコンパイルが通るからです。 

// 悪い例: 同じ型が長く並び、呼び出し側が順番を間違えてもコンパイルが通る
reserve(String userId, String roomId, LocalDate start, LocalDate end,
        boolean breakfast, boolean smoking, boolean lateCheckout);

// 改善例: 意味のあるまとまりを型にする
record ReservationPeriod(LocalDate start, LocalDate end) {}
record ReservationOptions(boolean breakfast, boolean smoking, boolean lateCheckout) {}

reserve(UserId userId, RoomId roomId, ReservationPeriod period, ReservationOptions options);

// さらに任意項目が増えるなら、メソッド呼び出しにもBuilderを使う
ReservationRequest request = ReservationRequest.builder(userId, roomId, period)
        .breakfast(true)
        .lateCheckout(true)
        .build();
reservationService.reserve(request);

長い引数リストを短くする3つの方法

  1. 複数メソッドに分割する: List.subList と indexOf のように、直交性のあるメソッドの組み合わせで表現する。
  2. ヘルパークラスを作る: rank/suitを Card にするように、頻出する引数のまとまりを型にする。
  3. Builderをメソッド呼び出しに応用する: 任意パラメータが多い処理では、リクエストオブジェクトに値を設定してからexecute/reserveする。

4. 引数型は具象クラスよりinterfaceを優先する

HashMap を受け取るのではなく Map を受け取ります。これにより、TreeMap、ConcurrentHashMap、将来のMap実装も受け取れます。 具象型を要求すると、呼び出し側に不要な変換やコピーを強制します。

5. boolean引数より2値enumを検討する

public enum TemperatureScale { FAHRENHEIT, CELSIUS }

Thermometer.newInstance(TemperatureScale.CELSIUS);
// Thermometer.newInstance(true) より意味が明確で、将来 KELVIN を追加しやすい

true や false は呼び出し側だけ見ると意味が分かりません。メソッド名から意味が明白でない限り、enumの方が読みやすく、将来の選択肢追加にも強いです。

Item 52: Use overloading judiciously
オーバーロードを慎重に使う

このItemはかなり重要です。Javaでは、オーバーロードはコンパイル時の型で選ばれる一方、オーバーライドは実行時の型で選ばれるからです。 この違いを曖昧にしたままAPIを設計すると、直感に反するバグを作ります。

本書掲載コード: 壊れたCollectionClassifier

// Broken! - What does this program print?
public class CollectionClassifier {
    public static String classify(Set<?> s) {
        return "Set";
    }

    public static String classify(List<?> lst) {
        return "List";
    }

    public static String classify(Collection<?> c) {
        return "Unknown Collection";
    }

    public static void main(String[] args) {
        Collection<?>[] collections = {
            new HashSet<String>(),
            new ArrayList<BigInteger>(),
            new HashMap<String, String>().values()
        };

        for (Collection<?> c : collections)
            System.out.println(classify(c));
    }
}
このプログラムは、多くの人が Set、List、Unknown Collection と表示されると予想します。しかし実際には Unknown Collection が3回表示されます。

理由は、ループ変数 c のコンパイル時型が常に Collection<?> だからです。実体が HashSet や ArrayList でも、オーバーロード解決には使われません。

改善: overloadではなく明示的な型判定にする

public static String classify(Collection<?> c) {
    return c instanceof Set  ? "Set" :
           c instanceof List ? "List" : "Unknown Collection";
}

対比: overrideは実行時型で選ばれる

class Wine {
    String name() { return "wine"; }
}

class SparklingWine extends Wine {
    @Override String name() { return "sparkling wine"; }
}

class Champagne extends SparklingWine {
    @Override String name() { return "champagne"; }
}

public class Overriding {
    public static void main(String[] args) {
        List<Wine> wineList = List.of(
            new Wine(), new SparklingWine(), new Champagne());
        for (Wine wine : wineList)
            System.out.println(wine.name());
    }
}

この例では、リストの要素のコンパイル時型はすべて Wine ですが、実行時の実体が Wine、SparklingWine、Champagne なので、それぞれの name() が呼ばれます。

仕組み選択タイミング何で決まるか
オーバーロードコンパイル時引数のコンパイル時型
オーバーライド実行時レシーバオブジェクトの実行時型

安全な方針: 同じ引数個数のoverloadを公開しない

本書は保守的な方針として、同じ数の引数を取るオーバーロードを公開しないことを勧めます。 メソッド名を変えられるなら、writeBoolean、writeInt、writeLong のように名前を分けた方が明確です。

本書掲載コード: List.remove と autoboxing の罠

public class SetList {
    public static void main(String[] args) {
        Set<Integer> set = new TreeSet<>();
        List<Integer> list = new ArrayList<>();

        for (int i = -3; i < 3; i++) {
            set.add(i);
            list.add(i);
        }

        for (int i = 0; i < 3; i++) {
            set.remove(i);
            list.remove(i);
        }

        System.out.println(set + " " + list);
    }
}

set.remove(i) は Integer 要素の削除として動きますが、list.remove(i) は int インデックス削除として解釈されます。 そのため、同じ remove(i) でもSetとListで違う意味になります。 

for (int i = 0; i < 3; i++) {
    set.remove(i);
    list.remove((Integer) i); // or remove(Integer.valueOf(i))
}

Java 8以降: ラムダ・メソッド参照でもoverloadは危険

new Thread(System.out::println).start();

ExecutorService exec = Executors.newCachedThreadPool();
exec.submit(System.out::println); // Runnable と Callable<T> のoverloadが絡み、直感に反する

異なる関数型interfaceを同じ引数位置に取るオーバーロードは、ラムダやメソッド参照の型推論と衝突しやすくなります。 Runnable と Callable<T> のように見た目が似ている型は、呼び出し側にとって混乱の原因になります。

やむを得ずoverloadする場合

既存APIの進化などで避けられない場合は、同じ引数でどのoverloadが選ばれても振る舞いが同じになるようにします。本書では、より具体的なoverloadから一般的なoverloadへ委譲する例が示されています。 

// Ensuring that 2 methods have identical behavior by forwarding
public boolean contentEquals(StringBuffer sb) {
    return contentEquals((CharSequence) sb);
}

Item 53: Use varargs judiciously
可変長引数を慎重に使う

可変長引数、varargsは、0個以上の引数を受け取るメソッドを定義できます。便利ですが、内部的には呼び出しごとに配列が作られる点と、「最低1個必要」という制約を表現しにくい点に注意します。

本書掲載コード: 単純なvarargs

// Simple use of varargs
static int sum(int... args) {
    int sum = 0;
    for (int arg : args)
        sum += arg;
    return sum;
}

sum(1, 2, 3) は6、sum() は0になります。0個引数でも意味が自然なら、varargsは適しています。

悪い例: 最低1個必要なのにvarargsだけで表す

// The WRONG way to use varargs to pass one or more arguments!
static int min(int... args) {
    if (args.length == 0)
        throw new IllegalArgumentException("Too few arguments");

    int min = args[0];
    for (int i = 1; i < args.length; i++)
        if (args[i] < min)
            min = args[i];
    return min;
}

この実装は min() をコンパイル時に禁止できません。実行時に IllegalArgumentException で落ちます。 API設計としては、呼び出せてはいけない形を呼び出せるようにしている点が弱いです。

良い例: 必須引数 + varargs

// The right way to use varargs to pass one or more arguments
static int min(int firstArg, int... remainingArgs) {
    int min = firstArg;
    for (int arg : remainingArgs)
        if (arg < min)
            min = arg;
    return min;
}

最初の引数を通常引数にすることで、最低1個必要という制約をコンパイル時に表現できます。ループも自然になります。

性能上の注意: varargsは配列を作る

varargsメソッドを呼ぶたびに配列の生成と初期化が発生します。通常は気にしなくてよいですが、性能が重要なホットパスでは測定が必要です。 本書では、ほとんどの呼び出しが少数引数なら、少数引数用の通常overloadと、多数引数用のvarargsを組み合わせるパターンが示されています。 

public void foo() { }
public void foo(int a1) { }
public void foo(int a1, int a2) { }
public void foo(int a1, int a2, int a3) { }
public void foo(int a1, int a2, int a3, int... rest) { }
実務感覚: 業務アプリのService層やController層でvarargsが性能問題になることは稀です。ただし、ログ・文字列処理・数値計算・ライブラリ内部など、極端に多く呼ばれる場所では無視できない可能性があります。

Item 54: Return empty collections or arrays, not nulls
nullではなく空コレクション・空配列を返す

「該当なし」を表すために null を返すAPIは、呼び出し側に余計な分岐を強制します。本Itemの結論は明確です。 コレクションや配列を返すメソッドでは、要素がないなら空を返す。

本書掲載コード: nullを返す悪い例

// Returns null to indicate an empty collection. Don't do this!
private final List<Cheese> cheesesInStock = ...;

/**
 * @return a list containing all of the cheeses in the shop,
 * or null if no cheeses are available for purchase.
 */
public List<Cheese> getCheeses() {
    return cheesesInStock.isEmpty() ? null
        : new ArrayList<>(cheesesInStock);
}

このAPIを使う呼び出し側は、毎回nullチェックを強制されます。 

List<Cheese> cheeses = shop.getCheeses();
if (cheeses != null && cheeses.contains(Cheese.STILTON))
    System.out.println("Jolly good, just the thing.");

呼び出し側がnullチェックを忘れると、実際にバグが起きる場所はAPI呼び出し箇所ではなく、後続処理のどこかになります。これは調査しにくいバグです。

良い例: 空でもListを返す

// The right way to return a possibly empty collection
public List<Cheese> getCheeses() {
    return new ArrayList<>(cheesesInStock);
}

cheesesInStock が空なら、空の ArrayList が返ります。呼び出し側は for や contains を自然に使えます。

最適化が必要な場合だけ空の不変コレクションを共有する

// Optimization - avoids allocating empty collections
public List<Cheese> getCheeses() {
    return cheesesInStock.isEmpty() ? Collections.emptyList()
        : new ArrayList<>(cheesesInStock);
}

ただし、これは最適化です。まずは通常の空コレクションを返す設計で十分です。性能問題が測定で確認された場合だけ検討します。

配列の場合

// The right way to return a possibly empty array
public Cheese[] getCheeses() {
    return cheesesInStock.toArray(new Cheese[0]);
}
// Optimization - avoids allocating empty arrays
private static final Cheese[] EMPTY_CHEESE_ARRAY = new Cheese[0];

public Cheese[] getCheeses() {
    return cheesesInStock.toArray(EMPTY_CHEESE_ARRAY);
}

本書では、toArray(new Cheese[0]) が通常の正しい書き方として示されます。 空配列の生成が問題になる場合は、static finalな空配列を使って共有できます。

避けるべき配列生成

// Don't do this - preallocating the array harms performance!
return cheesesInStock.toArray(new Cheese[cheesesInStock.size()]);

サイズぴったりの配列を事前生成する方が速そうに見えますが、本書では逆効果とされています。直感ではなく測定と実績に従うべき箇所です。

Item 55: Return optionals judiciously
Optionalを慎重に返す

Java 8以前は、「値を返せない可能性」を表す方法は主に例外かnullでした。Optional<T> は第三の選択肢です。 ただし、Optionalは万能ではありません。主に戻り値として、値がない可能性を呼び出し側に意識させたい場合に使うものです。

本書掲載コード: 空なら例外を投げるmax

// Returns maximum value in collection - throws exception if empty
public static <E extends Comparable<E>> E max(Collection<E> c) {
    if (c.isEmpty())
        throw new IllegalArgumentException("Empty collection");

    E result = null;
    for (E e : c)
        if (result == null || e.compareTo(result) > 0)
            result = Objects.requireNonNull(e);
    return result;
}

空コレクションで最大値が存在しない場合、例外を投げています。しかし、空が通常起こり得るケースなら、例外は重すぎます。

改善: Optionalを返す

// Returns maximum value in collection as an Optional<E>
public static <E extends Comparable<E>> Optional<E> max(Collection<E> c) {
    if (c.isEmpty())
        return Optional.empty();

    E result = null;
    for (E e : c)
        if (result == null || e.compareTo(result) > 0)
            result = Objects.requireNonNull(e);
    return Optional.of(result);
}

Optional.empty() は値がないことを表し、Optional.of(result) は非null値を持つOptionalを返します。 Optionalを返すメソッドでnullを返してはいけません。 それはOptionalの目的を壊します。

Streamを使う場合

// Returns max val in collection as Optional<E> - uses stream
public static <E extends Comparable<E>> Optional<E> max(Collection<E> c) {
    return c.stream().max(Comparator.naturalOrder());
}

呼び出し側の選択肢

// Using an optional to provide a chosen default value
String lastWordInLexicon = max(words).orElse("No words...");

// Using an optional to throw a chosen exception
Toy myToy = max(toys).orElseThrow(TemperTantrumException::new);

// Using optional when you know there's a return value
Element lastNobleGas = max(Elements.NOBLE_GASES).get();

Optionalの良さは、値がない場合に呼び出し側が方針を選べる点です。デフォルト値を使う、例外に変換する、存在が保証される場合だけ get() する、といった選択ができます。

isPresent() より map や orElse を優先する

Optional<ProcessHandle> parentProcess = ph.parent();
System.out.println("Parent PID: " + (parentProcess.isPresent() ?
    String.valueOf(parentProcess.get().pid()) : "N/A"));

// mapを使うと短く、Optionalらしい書き方になる
System.out.println("Parent PID: " +
    ph.parent().map(h -> String.valueOf(h.pid())).orElse("N/A"));

isPresent() は安全弁ですが、乱用するとnullチェックと同じような命令的コードになります。多くの場合、map、orElse、orElseThrow、orElseGet の方が短く明確です。

Stream<Optional<T>> を Stream<T> にする

// Java 8
streamOfOptionals
    .filter(Optional::isPresent)
    .map(Optional::get);

// Java 9+
streamOfOptionals
    .flatMap(Optional::stream);

Optionalを使うべきでない戻り値

// 悪い例: コンテナをOptionalで包む必要はない
Optional<List<Order>> findOrders(UserId userId);

// 良い例: 0件なら空Listを返す
List<Order> findOrders(UserId userId);

// 悪い例: OptionalをMapの値にする。キー不在と空Optionalで「ない」を二重表現してしまう
Map<UserId, Optional<UserProfile>> profiles;
  • Optional<List<T>> より空の List<T> を返す。
  • Optional<Map<K,V>> より空の Map<K,V> を返す。
  • Optional<int> は存在しない。Optional<Integer> ではなく OptionalInt を検討する。
  • OptionalをMapの値、Listの要素、配列要素にするのは原則避ける。
  • Optionalをフィールドに持つのは多くの場合bad smell。ただし、任意項目が多い値オブジェクトでは例外的に妥当な場合がある。
判断基準: 「値がないことが通常ケースとしてあり得て、呼び出し側にその分岐を明示的に考えてほしい」ならOptionalを返す候補です。性能が極めて重要な箇所では、Optionalのオブジェクト生成・参照間接化も測定対象にします。

Item 56: Write doc comments for all exposed API elements
公開API要素すべてにdoc commentを書く

APIは、コードだけでは完成しません。公開クラス、interface、コンストラクタ、メソッド、フィールドには、利用者が正しく使えるだけのドキュメントが必要です。 Javadocは、ソースコード中のdoc commentからAPIドキュメントを生成します。

メソッドのdoc commentに書くべき契約

  • 何をするメソッドか。原則として「どう実装しているか」ではなく「何を保証するか」を書く。
  • precondition: 呼び出す前に満たすべき条件。
  • postcondition: 正常終了後に成り立つこと。
  • 副作用: 状態変更、バックグラウンドスレッド開始など、戻り値だけでは分からない影響。
  • @param: 全パラメータ。
  • @return: void以外の戻り値。
  • @throws: checked/uncheckedを問わず、発生し得る例外。

本書掲載コード: List.get のJavadoc例

/**
 * Returns the element at the specified position in this list.
 *
 * <p>This method is <i>not</i> guaranteed to run in constant
 * time. In some implementations it may run in time proportional
 * to the element position.
 *
 * @param index index of element to return; must be
 *        non-negative and less than the size of this list
 * @return the element at the specified position in this list
 * @throws IndexOutOfBoundsException if the index is out of range
 *         ({@code index < 0 || index >= this.size()})
 */
E get(int index);

{@code ...} はコード片をコードフォントで表示し、HTMLタグとして解釈されるのを防ぎます。 複数行コード例では <pre>{@code ...}</pre> を使います。

@implSpec: 継承用に実装上の自己利用パターンを書く

/**
 * Returns true if this collection is empty.
 *
 * @implSpec
 * This implementation returns {@code this.size() == 0}.
 *
 * @return true if this collection is empty
 */
public boolean isEmpty() { ... }

通常のdoc commentは「メソッドとクライアントの契約」を書きます。一方、@implSpec は「メソッドとサブクラスの契約」を書きます。 継承を許すクラスでは、サブクラスが安全にoverrideできるよう、自己利用パターンを明記する必要があります。

HTML特殊文字には {@literal} を使う

* A geometric series converges if {@literal |r| < 1}.

<、>、& はHTMLの特殊文字です。Javadoc中でそのまま使うと表示崩れや解釈ミスが起こるため、{@literal} を使います。

summary descriptionの落とし穴

/**
 * A college degree, such as B.S., {@literal M.S.} or Ph.D.
 */
public class Degree { ... }

Javadocでは、最初の「文」がsummaryとして使われます。略語中のピリオドでsummaryが途中終了することがあるため、{@literal} で囲む工夫が必要です。

Java 9の {@index}

* This method complies with the {@index IEEE 754} standard.

ジェネリクス、enum、annotationのドキュメント

ジェネリクスでは型パラメータも @param <T> で説明します。

/**
 * An object that maps keys to values. A map cannot contain
 * duplicate keys; each key can map to at most one value.
 *
 * @param <K> the type of keys maintained by this map
 * @param <V> the type of mapped values
 */
public interface Map<K, V> { ... }

enumでは、型だけでなく各定数にもコメントを書きます。

/**
 * An instrument section of a symphony orchestra.
 */
public enum OrchestraSection {
    /** Woodwinds, such as flute, clarinet, and oboe. */
    WOODWIND,
    /** Brass instruments, such as french horn and trumpet. */
    BRASS,
    /** Percussion instruments, such as timpani and cymbals. */
    PERCUSSION,
    /** Stringed instruments, such as violin and cello. */
    STRING;
}

annotationでは、annotation型自体と各メンバーを説明します。

/**
 * Indicates that the annotated method is a test method that
 * must throw the designated exception to pass.
 */
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface ExceptionTest {
    /**
     * The exception that the annotated test method must throw
     * in order to pass. (The test is permitted to throw any
     * subtype of the type described by this class object.)
     */
    Class<? extends Throwable> value();
}

その他の重要点

  • package単位のコメントは package-info.java に書く。
  • module単位のコメントは module-info.java に書く。
  • thread-safetyはItem 82の観点で明記する。
  • Serializableなクラスはserialized formも説明する。
  • {@inheritDoc} でsupertypeのdoc commentを継承できるが、万能ではない。
  • Javadoc生成後のHTMLを実際に読む。テストと同じで、生成物を見るとコメントの欠陥が見つかる。
