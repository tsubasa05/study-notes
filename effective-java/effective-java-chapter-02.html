オブジェクトの生成と破棄

章全体の見取り図

この章のテーマは「オブジェクトのライフサイクルを、クラス設計・API設計・リソース管理として制御する」ことです。 JavaにはGCがあるためメモリ解放の多くは自動化されますが、生成の仕方と参照・外部リソースの手放し方は依然として設計者の責任です。

Item 中心テーマ 避けたい失敗 実務での使いどころ
1 コンストラクタだけでなくstatic factory methodを検討する 生成APIが読みにくい、実装クラスを露出する、毎回newする 値オブジェクト、コレクション生成、インスタンス制御

この章の核心： 「オブジェクト生成」は単なる文法ではなく、APIの読みやすさ、拡張性、テスト容易性、メモリ効率、リソース安全性を左右する設計判断です。
Item 1

コンストラクタの代わりに static factory method を検討する

クラスのインスタンスを作る一般的な手段はpublicコンストラクタです。しかし本書は、クラスが public static なファクトリメソッドを提供する設計を、コンストラクタと同列に検討すべきだと述べます。 ここでいうstatic factory methodは、GoFのFactory Methodパターンとは別物です。

本書の代表例：Boolean.valueOf

public static Boolean valueOf(boolean b) {
    return b ? Boolean.TRUE : Boolean.FALSE;
}

この例が示す重要点は、new Boolean(...) のように毎回新しいオブジェクトを作る必要がないということです。 true と false は取り得る値が2つしかないため、あらかじめ用意されたインスタンスを返せば十分です。

利点1：名前を付けられる

コンストラクタはクラス名と同じ名前しか使えません。引数だけでは意味が分からない場合、static factory methodの名前がAPIの説明になります。 

// コンストラクタだけだと、何を生成しているか分かりづらい
BigInteger p = new BigInteger(1024, 100, random);

// 名前があると意図が明確になる
BigInteger p = BigInteger.probablePrime(1024, random);

これは実務でもかなり重要です。たとえば UserId のような値オブジェクトで、 DB由来なのか、外部システム由来なのか、ランダム生成なのかをメソッド名で表せます。 

public final class UserId {
    private final String value;

    private UserId(String value) {
        if (value == null || value.isBlank()) {
            throw new IllegalArgumentException("userId must not be blank");
        }
        this.value = value;
    }

    public static UserId of(String value) {
        return new UserId(value);
    }

    public static UserId fromExternalSystem(String externalId) {
        return new UserId("EXT-" + externalId);
    }

    public static UserId generate() {
        return new UserId(UUID.randomUUID().toString());
    }
}

利点2：毎回新しいオブジェクトを作らなくてよい

コンストラクタは呼ばれるたびに新しいインスタンスを作ります。一方、static factory methodは、既存インスタンスを返しても構いません。 これは不変クラス、値の種類が少ないクラス、キャッシュ可能なクラスで有効です。 

public final class Priority {
    public static final Priority HIGH = new Priority("HIGH");
    public static final Priority NORMAL = new Priority("NORMAL");
    public static final Priority LOW = new Priority("LOW");

    private final String value;

    private Priority(String value) {
        this.value = value;
    }

    public static Priority of(String value) {
        return switch (value) {
            case "HIGH" -> HIGH;
            case "NORMAL" -> NORMAL;
            case "LOW" -> LOW;
            default -> throw new IllegalArgumentException("unknown priority: " + value);
        };
    }
}

これにより、インスタンス数を制御できます。本書ではこのようなクラスを instance-controlled class と呼び、Singletonや不変値クラスの設計に関係すると説明しています。

利点3：戻り値型の任意のサブタイプを返せる

コンストラクタは、そのクラス自身のインスタンスしか返せません。しかしstatic factory methodは、宣言された戻り値型のサブタイプを返せます。 これにより、実装クラスをAPI利用者から隠せます。 

public interface NotificationSender {
    void send(String message);

    static NotificationSender email(String smtpHost) {
        return new EmailNotificationSender(smtpHost); // 実装クラスは非公開でもよい
    }

    static NotificationSender noop() {
        return message -> { }; // ラムダや別実装も返せる
    }
}

利用側は EmailNotificationSender という実装に依存せず、NotificationSender にだけ依存します。 これは「実装ではなくインターフェースに依存する」という設計にもつながります。

利点4：入力値によって返す実装クラスを変えられる

本書では EnumSet の例が出ます。要素数が少ないenumなら long ベースの実装、多いenumなら long[] ベースの実装を返す、というように、利用者に見せず内部実装を切り替えられます。 

// 利用者は具象クラスを知らなくてよい
EnumSet<DayOfWeek> weekend = EnumSet.of(SATURDAY, SUNDAY);

// 内部では enum の要素数などに応じて異なる実装が返され得る

利点5：メソッド作成時に戻り値の具象クラスが存在しなくてもよい

これはサービスプロバイダフレームワークの考え方です。JDBCのように、利用側は DriverManager.getConnection(...) を呼びますが、 実際にどのDBドライバ実装が使われるかは後から登録されたプロバイダに委ねられます。 

Connection conn = DriverManager.getConnection(url, user, password);

利用側は「Connectionが欲しい」とだけ言えばよく、PostgreSQLなのかMySQLなのかという具象ドライバの生成詳細から切り離されます。

欠点1：public/protectedコンストラクタがないと継承できない

static factory methodだけを提供し、コンストラクタをprivateにすると、利用者はそのクラスを継承できません。 ただし本書は、これは悪いことばかりではなく、継承よりコンポジションを促すため、むしろ利点にもなり得ると見ています。

欠点2：APIドキュメント上で見つけにくい

コンストラクタはJavadocで目立ちますが、static factory methodは通常メソッドに混ざります。 そのため、一般的な命名規則に従うことが重要です。

名前意味例
from単一引数からの型変換Date.from(instant)
of複数値から集約して生成EnumSet.of(JACK, QUEEN, KING)
valueOffrom/of より冗長な代替BigInteger.valueOf(...)
getInstance条件に合うインスタンスを返す。新規とは限らないStackWalker.getInstance(...)
newInstance毎回新しいインスタンスを返す意図Array.newInstance(...)
getType別クラス上にあるfactoryFiles.getFileStore(path)
newType別クラス上にあり、新規生成を示すFiles.newBufferedReader(path)
判断基準： 単純なデータクラスで引数も少なく意味が明確ならコンストラクタでよいです。 ただし「名前で意図を表したい」「生成方法が複数ある」「キャッシュしたい」「実装クラスを隠したい」ならstatic factory methodを先に検討すべきです。
Item 2

多数のコンストラクタ引数があるなら Builder を検討する

static factory methodもコンストラクタも、オプション引数が多いクラスでは破綻します。 本書は、必須パラメータが少数あり、任意パラメータが多数ある例として NutritionFacts を使います。

悪い方向1：テレスコーピングコンストラクタ

引数の数に応じてコンストラクタを何個も作る方法です。動きはしますが、読みづらく、引数順のミスが起きやすくなります。 

public class NutritionFacts {
    private final int servingSize;   // 必須
    private final int servings;      // 必須
    private final int calories;      // 任意
    private final int fat;           // 任意
    private final int sodium;        // 任意
    private final int carbohydrate;  // 任意

    public NutritionFacts(int servingSize, int servings) {
        this(servingSize, servings, 0);
    }

    public NutritionFacts(int servingSize, int servings, int calories) {
        this(servingSize, servings, calories, 0);
    }

    public NutritionFacts(int servingSize, int servings,
                          int calories, int fat, int sodium, int carbohydrate) {
        this.servingSize = servingSize;
        this.servings = servings;
        this.calories = calories;
        this.fat = fat;
        this.sodium = sodium;
        this.carbohydrate = carbohydrate;
    }
}
// 240, 8, 100, 0, 35, 27 が何を意味するか、呼び出し側だけでは分かりにくい
NutritionFacts cocaCola = new NutritionFacts(240, 8, 100, 0, 35, 27);
問題点： 同じ int が並ぶと、sodium と carbohydrate を逆にしてもコンパイルは通ります。 つまり、型システムが守ってくれないバグになります。

悪い方向2：JavaBeansパターン

引数なしコンストラクタで作ってからsetterで値を入れる方法です。呼び出し側の読みやすさは改善しますが、 オブジェクトが生成途中の不整合状態を持つ問題があります。 

public class NutritionFacts {
    private int servingSize = -1; // 必須だが初期値なし
    private int servings = -1;
    private int calories = 0;
    private int sodium = 0;

    public NutritionFacts() { }

    public void setServingSize(int val) { servingSize = val; }
    public void setServings(int val)    { servings = val; }
    public void setCalories(int val)    { calories = val; }
    public void setSodium(int val)      { sodium = val; }
}
NutritionFacts cocaCola = new NutritionFacts();
cocaCola.setServingSize(240);
cocaCola.setServings(8);
cocaCola.setCalories(100);
cocaCola.setSodium(35);

このコードは読みやすい一方、setServings を呼ぶ前のオブジェクトも存在してしまいます。 さらにsetterを公開するため、クラスを不変にしづらく、スレッドセーフ性も自分で考える必要があります。

改善：Builderパターン

Builderでは、必須項目をBuilderのコンストラクタで受け取り、任意項目はメソッドチェーンで設定し、最後に build() で完成した不変オブジェクトを作ります。 

public class NutritionFacts {
    private final int servingSize;
    private final int servings;
    private final int calories;
    private final int fat;
    private final int sodium;
    private final int carbohydrate;

    public static class Builder {
        // 必須
        private final int servingSize;
        private final int servings;

        // 任意：デフォルト値をここに集約
        private int calories = 0;
        private int fat = 0;
        private int sodium = 0;
        private int carbohydrate = 0;

        public Builder(int servingSize, int servings) {
            if (servingSize <= 0 || servings <= 0) {
                throw new IllegalArgumentException("servingSize and servings must be positive");
            }
            this.servingSize = servingSize;
            this.servings = servings;
        }

        public Builder calories(int val) {
            if (val < 0) throw new IllegalArgumentException("calories must not be negative");
            calories = val;
            return this;
        }

        public Builder fat(int val) {
            if (val < 0) throw new IllegalArgumentException("fat must not be negative");
            fat = val;
            return this;
        }

        public Builder sodium(int val) {
            if (val < 0) throw new IllegalArgumentException("sodium must not be negative");
            sodium = val;
            return this;
        }

        public Builder carbohydrate(int val) {
            if (val < 0) throw new IllegalArgumentException("carbohydrate must not be negative");
            carbohydrate = val;
            return this;
        }

        public NutritionFacts build() {
            return new NutritionFacts(this);
        }
    }

    private NutritionFacts(Builder builder) {
        servingSize = builder.servingSize;
        servings = builder.servings;
        calories = builder.calories;
        fat = builder.fat;
        sodium = builder.sodium;
        carbohydrate = builder.carbohydrate;
    }
}
NutritionFacts cocaCola = new NutritionFacts.Builder(240, 8)
        .calories(100)
        .sodium(35)
        .carbohydrate(27)
        .build();

これにより、呼び出しコードは「名前付き任意引数」に近い読みやすさを持ちます。 しかも完成物である NutritionFacts は final フィールドだけを持つ不変オブジェクトにできます。

バリデーションはどこで行うべきか

単一パラメータの妥当性はBuilderのコンストラクタや各setter風メソッドで検査します。 複数パラメータにまたがる不変条件は、build() が呼ぶ本体側コンストラクタで検査するのが自然です。 

private NutritionFacts(Builder builder) {
    servingSize = builder.servingSize;
    servings = builder.servings;
    calories = builder.calories;
    fat = builder.fat;
    sodium = builder.sodium;
    carbohydrate = builder.carbohydrate;

    if (fat > 0 && calories == 0) {
        throw new IllegalArgumentException("fat is set but calories is zero");
    }
}

クラス階層でのBuilder

本書ではPizzaの例を使い、抽象クラスと具象サブクラスに対応するBuilder階層を紹介しています。 ポイントは、親Builderのメソッドチェーンが子Builderでも正しく続くように、再帰的ジェネリクスと self() を使うことです。 

public abstract class Pizza {
    public enum Topping { HAM, MUSHROOM, ONION, PEPPER, SAUSAGE }

    final Set<Topping> toppings;

    abstract static class Builder<T extends Builder<T>> {
        EnumSet<Topping> toppings = EnumSet.noneOf(Topping.class);

        public T addTopping(Topping topping) {
            toppings.add(Objects.requireNonNull(topping));
            return self();
        }

        abstract Pizza build();
        protected abstract T self();
    }

    Pizza(Builder<?> builder) {
        toppings = builder.toppings.clone();
    }
}
public class NyPizza extends Pizza {
    public enum Size { SMALL, MEDIUM, LARGE }
    private final Size size;

    public static class Builder extends Pizza.Builder<Builder> {
        private final Size size;

        public Builder(Size size) {
            this.size = Objects.requireNonNull(size);
        }

        @Override public NyPizza build() {
            return new NyPizza(this);
        }

        @Override protected Builder self() {
            return this;
        }
    }

    private NyPizza(Builder builder) {
        super(builder);
        size = builder.size;
    }
}
NyPizza pizza = new NyPizza.Builder(NyPizza.Size.SMALL)
        .addTopping(Pizza.Topping.SAUSAGE)
        .addTopping(Pizza.Topping.ONION)
        .build();
使いどころの線引き： Builderは便利ですが、常に使うものではありません。引数が2〜3個で意味が明確なら過剰です。 一方、任意項目が多い、同じ型の引数が並ぶ、今後項目が増える見込みがあるなら、最初からBuilderにした方がAPIの劣化を防げます。
Item 3

Singleton性は private constructor または enum で保証する

Singletonとは、インスタンスが厳密に1つだけ存在するクラスです。 典型例は、状態を持たない関数的オブジェクトや、システム上本質的に一意なコンポーネントです。 ただし、Singletonはテストしづらくなりやすいので、多用すべきではありません。

方法1：public static final field

public class Elvis {
    public static final Elvis INSTANCE = new Elvis();

    private Elvis() {
        // 外部からnewできない
    }

    public void leaveTheBuilding() {
        // ...
    }
}

API上、Elvis.INSTANCE しか取得口がありません。シンプルで、Singletonであることも読み取りやすい方法です。

方法2：static factory method

public class Elvis {
    private static final Elvis INSTANCE = new Elvis();

    private Elvis() {
    }

    public static Elvis getInstance() {
        return INSTANCE;
    }

    public void leaveTheBuilding() {
        // ...
    }
}

この方法の利点は、APIを変えずに後から実装方針を変えられることです。 例えば将来「スレッドごとに別インスタンスを返す」などに変更する余地があります。

シリアライズ時の注意

上の2方式で Serializable にすると、デシリアライズ時に別インスタンスが作られ、Singleton性が壊れる可能性があります。 本書では readResolve を使って唯一のインスタンスを返す方法が紹介されています。 

private Object readResolve() {
    return INSTANCE;
}

推奨：単一要素のenum

public enum Elvis {
    INSTANCE;

    public void leaveTheBuilding() {
        // ...
    }
}

enum Singletonは簡潔で、シリアライズに強く、リフレクションによる多重生成にも強いです。 そのため、本書は多くの場合で単一要素enumを最善のSingleton実装と見なしています。

実務上の注意： Singletonは「どこからでもアクセスできるグローバル変数」に近い使われ方をしがちです。 依存関係が見えづらくなり、テストで差し替えにくくなります。 Springを使う場合でも、「Beanのスコープがsingleton」であることと、「アプリケーション設計としてSingletonに依存すること」は区別してください。
Item 4

インスタンス化できないクラスは private constructor で明示する

java.lang.Math や java.util.Arrays のように、staticメソッドやstaticフィールドだけをまとめたクラスがあります。 このようなUtility classは、インスタンス化されても意味がありません。

間違い：abstractにする

インスタンス化を防ぐ目的で abstract にするのは不適切です。サブクラスを作ればインスタンス化できてしまううえ、 「継承して使うクラスなのか」と誤解させます。

正しいイディオム

public final class StringUtils {
    private StringUtils() {
        throw new AssertionError("No instances");
    }

    public static boolean isBlank(String s) {
        return s == null || s.isBlank();
    }
}

明示的なprivateコンストラクタがあると、コンパイラはpublicなデフォルトコンストラクタを生成しません。 AssertionError は必須ではありませんが、クラス内部から誤って呼ばれた場合にも即座に失敗させられます。

副作用： privateコンストラクタしかないため、サブクラスのコンストラクタから親コンストラクタを呼べません。 つまり、このイディオムは継承も防ぎます。
Item 5

リソースを直接固定せず、Dependency Injectionを選ぶ

多くのクラスは、内部で使うリソースに依存します。たとえばスペルチェッカーは辞書に依存します。 本書は、そのリソースをクラス内部で固定する実装を避けるべきだと述べます。

悪い例：static utilityにリソースを固定する

public class SpellChecker {
    private static final Lexicon dictionary = new EnglishLexicon();

    private SpellChecker() {
    }

    public static boolean isValid(String word) {
        return dictionary.contains(word);
    }
}

これだと、英語辞書以外を使いたい場合、テスト用辞書を使いたい場合、専門用語辞書を使いたい場合に困ります。 クラスの振る舞いがリソースに依存するのに、そのリソースが固定されているからです。

悪い例：Singletonにリソースを固定する

public class SpellChecker {
    public static final SpellChecker INSTANCE =
            new SpellChecker(new EnglishLexicon());

    private final Lexicon dictionary;

    private SpellChecker(Lexicon dictionary) {
        this.dictionary = dictionary;
    }

    public boolean isValid(String word) {
        return dictionary.contains(word);
    }
}

これも「辞書は1種類だけ」という仮定をクラスに埋め込んでいます。将来の拡張やテストを妨げます。

改善：コンストラクタで依存を注入する

public class SpellChecker {
    private final Lexicon dictionary;

    public SpellChecker(Lexicon dictionary) {
        this.dictionary = Objects.requireNonNull(dictionary);
    }

    public boolean isValid(String word) {
        return dictionary.contains(word);
    }

    public List<String> suggestions(String typo) {
        return dictionary.suggest(typo);
    }
}

これなら、呼び出し側が用途に応じて辞書を選べます。 

SpellChecker englishChecker = new SpellChecker(new EnglishLexicon());
SpellChecker testChecker = new SpellChecker(new InMemoryLexicon(List.of("test", "java")));

Factoryを注入するバリエーション

リソースそのものではなく、リソースを作るfactoryを渡すこともあります。 Java 8以降なら Supplier<T> が使えます。 

public class ReportService {
    private final Supplier<? extends ReportFormatter> formatterFactory;

    public ReportService(Supplier<? extends ReportFormatter> formatterFactory) {
        this.formatterFactory = Objects.requireNonNull(formatterFactory);
    }

    public String createReport(Report report) {
        ReportFormatter formatter = formatterFactory.get();
        return formatter.format(report);
    }
}

Springとの接続

Springでは、手動で new SpellChecker(...) しなくても、コンストラクタインジェクションで依存を渡せます。 これは本書のDependency Injectionの考え方と同じ方向です。 

@Service
public class SpellChecker {
    private final Lexicon dictionary;

    public SpellChecker(Lexicon dictionary) {
        this.dictionary = Objects.requireNonNull(dictionary);
    }
}
判断基準： クラスの振る舞いが、DB、HTTPクライアント、Clock、設定値、辞書、Repositoryなどの外部リソースに依存するなら、 その依存をクラス内部で直接作らず、外から渡す設計を基本にしてください。
Item 6

不要なオブジェクト生成を避ける

このItemは「オブジェクト生成は絶対に悪い」と言っているわけではありません。 小さなオブジェクト生成は現代JVMでは安いことが多いです。 重要なのは、同じ意味のオブジェクトを無駄に繰り返し作らないことです。

極端な悪い例：new String

String s = new String("bikini"); // やってはいけない

文字列リテラル "bikini" 自体がすでに String インスタンスです。 そのため、さらに new String(...) する意味はありません。 

String s = "bikini"; // これでよい

static factory methodで再利用する

不変クラスで、コンストラクタとstatic factory methodの両方がある場合、後者を選ぶことで不要な生成を避けられることがあります。 

Boolean b1 = Boolean.valueOf("true"); // 再利用され得る
// new Boolean("true") は古い書き方であり、避けるべき

高コストオブジェクトをキャッシュする：Patternの例

正規表現は、見た目以上に高コストです。String.matches は内部で Pattern を作って使い捨てます。 繰り返し呼ばれる処理では、あらかじめ Pattern をコンパイルして再利用します。 

// 悪い例：呼び出すたびにPatternが作られる
static boolean isRomanNumeral(String s) {
    return s.matches("^(?=.)M*(C[MD]|D?C{0,3})"
            + "(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");
}
// 改善：Patternをstatic finalで再利用する
public class RomanNumerals {
    private static final Pattern ROMAN = Pattern.compile(
            "^(?=.)M*(C[MD]|D?C{0,3})"
          + "(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");

    static boolean isRomanNumeral(String s) {
        return ROMAN.matcher(s).matches();
    }
}

autoboxingによる隠れた生成

本書の有名な例です。Long と long の1文字違いで、大量の不要なオブジェクト生成が発生します。 

// 非常に遅い：sumがLongなので、加算のたびにboxing/unboxingが絡む
private static long sum() {
    Long sum = 0L;
    for (long i = 0; i <= Integer.MAX_VALUE; i++) {
        sum += i;
    }
    return sum;
}
// 改善：プリミティブを使う
private static long sum() {
    long sum = 0L;
    for (long i = 0; i <= Integer.MAX_VALUE; i++) {
        sum += i;
    }
    return sum;
}
誤解してはいけない点： 不要な生成を避けるために、何でもオブジェクトプールにするのは逆効果です。 DBコネクションのように生成コストが極端に高いものはプールに値しますが、普通の軽量オブジェクトはJVMのGCに任せた方がよいことが多いです。

また、このItemはItem 50「防御的コピー」と対になります。 再利用すべきところで新規生成するのは主に性能・スタイルの問題ですが、 コピーすべきところで既存オブジェクトを再利用すると、バグやセキュリティホールになります。

Item 7

古くなったオブジェクト参照を取り除く

JavaにはGCがありますが、GCは「到達不能なオブジェクト」を回収するだけです。 プログラムが不要な参照を保持し続けていれば、GCから見るとまだ必要なオブジェクトに見えます。 これが、GC言語におけるメモリリーク、より正確には「意図しないオブジェクト保持」です。

本書のStack例：pop後も配列に参照が残る

public class Stack {
    private Object[] elements;
    private int size = 0;

    public Object pop() {
        if (size == 0) {
            throw new EmptyStackException();
        }
        return elements[--size]; // ここで返した後も、配列内に参照が残る
    }
}

size より後ろの領域はStackとしては使っていません。 しかし配列 elements には古い参照が残っているため、GCはそのオブジェクトを回収できません。

改善：不要になった参照をnullにする

public Object pop() {
    if (size == 0) {
        throw new EmptyStackException();
    }

    Object result = elements[--size];
    elements[size] = null; // 古い参照を消す
    return result;
}

ただし、すべてのローカル変数に null を代入すべきという意味ではありません。 通常は、変数のスコープを狭くし、自然にスコープアウトさせるのが最善です。

いつnull代入が必要か

本書のStackのように、クラスが自前でメモリを管理している場合です。 具体的には、配列の一部だけを有効領域として使い、残りを空き領域として扱うような実装です。

自前コレクション 内部配列、バッファ、プールなどを自分で管理する場合は、不要になった参照を消す。
キャッシュ 一度入れた参照が残り続けやすい。期限、サイズ上限、弱参照などを検討する。
リスナー / コールバック 登録解除されないと参照が蓄積する。弱参照や明示的解除APIを考える。
長寿命オブジェクト Application scopeのMapやstatic fieldは特に注意。短命オブジェクトより影響が大きい。

キャッシュの扱い

キャッシュはメモリリークの温床です。 キーへの外部参照がなくなったらキャッシュエントリも不要、という性質なら WeakHashMap が使える場合があります。 ただし、WeakHashMap は「キーの寿命」でエントリを消す仕組みであり、値の寿命を直接管理するものではありません。 

Map<Key, Value> cache = new WeakHashMap<>();

より一般的には、サイズ上限、期限切れ、LRU、バックグラウンドクリーンアップなどを設計します。 LinkedHashMap#removeEldestEntry を使う簡易LRUも選択肢です。 

class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxSize;

    LruCache(int maxSize) {
        super(16, 0.75f, true); // access-order
        this.maxSize = maxSize;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxSize;
    }
}
Item 8

finalizer と cleaner を避ける

本書は、finalizerは予測不能で危険であり、一般に不要だと強く述べます。 Java 9以降の代替としてcleanerがありますが、cleanerも予測不能・低速・多くの場合不要です。

なぜ危険か

問題内容
すぐ実行される保証がない オブジェクトが到達不能になってから、finalizer/cleanerがいつ走るか分からない。
実行される保証もない プログラム終了まで実行されないことがある。永続状態の更新やロック解放に使ってはいけない。
例外が扱いづらい finalizer内の例外は無視され、オブジェクトが壊れた状態になり得る。
性能が悪い GC最適化を妨げ、通常のcloseより大幅に遅くなり得る。
セキュリティ上の問題 finalizer attackにより、コンストラクタで失敗したはずの半端なオブジェクトが生き残る可能性がある。

代替：AutoCloseable + try-with-resources

ファイル、ソケット、DB接続、スレッドプールなど、明示的に終了処理が必要なリソースは AutoCloseable を実装し、利用側に close() させます。 

public final class ResourceHandle implements AutoCloseable {
    private boolean closed;

    public void use() {
        if (closed) {
            throw new IllegalStateException("already closed");
        }
        // リソースを使う
    }

    @Override
    public void close() {
        if (!closed) {
            // リソースを解放する
            closed = true;
        }
    }
}
try (ResourceHandle handle = new ResourceHandle()) {
    handle.use();
}

Cleanerを使うなら安全網としてだけ

本書では、Roomという例でcleanerを「close忘れに対する安全網」として使う方法を示します。 ただし、cleanerのStateが元のRoomインスタンスを参照してはいけない点が重要です。 

public class Room implements AutoCloseable {
    private static final Cleaner cleaner = Cleaner.create();

    // Roomを参照してはいけないため、static nested classにする
    private static class State implements Runnable {
        int numJunkPiles;

        State(int numJunkPiles) {
            this.numJunkPiles = numJunkPiles;
        }

        @Override public void run() {
            System.out.println("Cleaning room");
            numJunkPiles = 0;
        }
    }

    private final State state;
    private final Cleaner.Cleanable cleanable;

    public Room(int numJunkPiles) {
        state = new State(numJunkPiles);
        cleanable = cleaner.register(this, state);
    }

    @Override public void close() {
        cleanable.clean();
    }
}
重要： cleanerは「最後の保険」であり、通常のリソース解放経路にしてはいけません。 closeされなくてもいつか片付くだろう、という設計は危険です。
Item 9

try-finally より try-with-resources を使う

Javaには、明示的に close() しなければならないリソースが多くあります。 例として InputStream、OutputStream、java.sql.Connection などがあります。

古い方法：try-finally

static String firstLineOfFile(String path) throws IOException {
    BufferedReader br = new BufferedReader(new FileReader(path));
    try {
        return br.readLine();
    } finally {
        br.close();
    }
}

1リソースならまだ読めますが、複数リソースになるとネストが増えます。 

static void copy(String src, String dst) throws IOException {
    InputStream in = new FileInputStream(src);
    try {
        OutputStream out = new FileOutputStream(dst);
        try {
            byte[] buf = new byte[8192];
            int n;
            while ((n = in.read(buf)) >= 0) {
                out.write(buf, 0, n);
            }
        } finally {
            out.close();
        }
    } finally {
        in.close();
    }
}

try-finallyの本質的な問題：例外が上書きされる

try ブロック内の処理でも、finally 内の close() でも例外が発生する可能性があります。 その場合、後から発生したclose側の例外が、最初に発生した本質的な例外を覆い隠してしまうことがあります。

改善：try-with-resources

static String firstLineOfFile(String path) throws IOException {
    try (BufferedReader br = new BufferedReader(new FileReader(path))) {
        return br.readLine();
    }
}
static void copy(String src, String dst) throws IOException {
    try (InputStream in = new FileInputStream(src);
         OutputStream out = new FileOutputStream(dst)) {

        byte[] buf = new byte[8192];
        int n;
        while ((n = in.read(buf)) >= 0) {
            out.write(buf, 0, n);
        }
    }
}

try-with-resourcesを使える条件は、リソースが AutoCloseable を実装していることです。 Java標準ライブラリや多くのサードパーティライブラリのリソース型はこれに対応しています。

catchも付けられる

static String firstLineOfFile(String path, String defaultValue) {
    try (BufferedReader br = new BufferedReader(new FileReader(path))) {
        return br.readLine();
    } catch (IOException e) {
        return defaultValue;
    }
}
結論： closeが必要なリソースには、原則としてtry-with-resourcesを使ってください。 コードが短くなり、close漏れを防ぎ、例外情報も有用になります。

実務判断のまとめ：どの設計を選ぶか

状況選択肢理由
単純な値を作る。引数も少なく意味が明確 コンストラクタでもよい 過剰設計にしない。読みやすさが十分なら問題ない。
生成方法が複数ある、名前で意図を出したい static factory method of、from、valueOf などで意味を明示できる。
任意引数が多い、同じ型の引数が並ぶ Builder 引数順ミスを避け、呼び出し側の可読性と不変性を両立できる。
staticメソッドだけの補助クラス private constructor 意味のないインスタンス化と誤った継承を防ぐ。
クラスが外部リソースや別コンポーネントに依存する Dependency Injection 差し替え可能性、テスト容易性、再利用性が上がる。
正規表現、Formatter、DB接続など生成コストが高い 再利用・キャッシュ・プールを検討 ただし普通の軽量オブジェクトを無理にプールしない。
closeが必要なリソースを扱う AutoCloseable + try-with-resources finalizer/cleanerや手書きtry-finallyに頼らない。
あなたがSpring Bootでアプリを書くときの重点： まず Item 2、5、9 を優先して体に入れるべきです。 DTO・検索条件・設定系はBuilderやstatic factoryを検討し、Service/Repository/Client/Clockなどはコンストラクタインジェクションにし、 ファイル・HTTPレスポンス・DB接続などはtry-with-resourcesで確実に閉じる、という形です。

コードレビュー用チェックリスト

生成API

  • コンストラクタの引数だけで意味が伝わるか。
  • 生成方法が複数あるなら、static factory methodに名前を付けた方がよくないか。
  • new する必要がない値を毎回生成していないか。
  • 実装クラスを公開APIに露出しすぎていないか。

引数が多いクラス

  • 同じ型の引数が長く並んでいないか。
  • 任意引数のためにテレスコーピングコンストラクタを増やしていないか。
  • setterで生成途中の不整合状態を外部に見せていないか。
  • Builderにした方が呼び出し側の意図が明確にならないか。

依存関係

  • クラス内部で new Repository()、new HttpClient()、Clock.systemDefaultZone() などを固定していないか。
  • テスト時にモック・スタブ・インメモリ実装へ差し替えられるか。
  • Singletonやstatic utilityで本来差し替えるべきリソースを固定していないか。

メモリ・リソース

  • ループ内で高コストオブジェクトを毎回作っていないか。
  • 不要なautoboxingがホットパスにないか。
  • 自前配列・キャッシュ・リスナーに古い参照が残り続けていないか。
  • closeが必要なリソースにtry-with-resourcesを使っているか。
  • finalizer/cleanerに通常の終了処理を任せていないか。
最終結論： 第2章は「newとGCの章」ではなく、API設計・依存関係設計・リソース管理の章です。 ここを雑にすると、コードは動いても、読みづらく、テストしづらく、拡張しづらく、長期運用で壊れやすくなります。
