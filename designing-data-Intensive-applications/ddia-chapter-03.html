1. 第3章の中心主張

第3章は「データベースが内部でデータをどう保存し、どう探し出しているのか」を扱います。アプリ開発者がストレージエンジンを自作する必要は通常ありませんが、DBの内部動作を知らないままでは、インデックス設計、SQLチューニング、DB選定、ログ/メトリクスの解釈を誤ります。

前半: OLTP向けストレージエンジン

主に「少数レコードを高速に読み書きする」用途です。代表は hash index、SSTable/LSM-tree、B-tree です。バックエンドAPIが普段触るRDB/Key-Value Storeの多くは、この領域の理解が重要です。

後半: OLAP向けストレージエンジン

主に「大量レコードをスキャンし、少数の列だけ読み、集計する」用途です。データウェアハウス、スター・スキーマ、カラム指向ストレージ、カラム圧縮 が中心になります。

この章の実務的な読み方: 「B-treeだから良い」「LSM-treeだから速い」という雑な理解は危険です。正しくは、読み取りパターン、書き込み量、範囲検索の有無、集計クエリの比率、ディスクI/O、メモリ量、更新頻度 の組み合わせで適性が変わります。

2. 最小のDB: append-only log から始める

章の冒頭では、世界で最も単純なDBとして Bash 関数の key-value store が示されます。ポイントは「書き込み時に既存データを上書きしないで、ファイル末尾に追記する」という設計です。

原書に記載されているコード例

#!/bin/bash

db_set () {
    echo "$1,$2" >> database
}

db_get () {
    grep "^$1," database | sed -e "s/^$1,//" | tail -n 1
}

db_set は database というファイルの末尾へ key,value を追記します。db_get はファイル全体を先頭から検索し、最後に出てきた値を返します。更新が発生しても古い値を消さず、最新値を末尾に積む構造です。

$ db_set 123456 '{"name":"London","attractions":["Big Ben","London Eye"]}'
$ db_set 42 '{"name":"San Francisco","attractions":["Golden Gate Bridge"]}'
$ db_get 42
{"name":"San Francisco","attractions":["Golden Gate Bridge"]}

$ db_set 42 '{"name":"San Francisco","attractions":["Exploratorium"]}'
$ db_get 42
{"name":"San Francisco","attractions":["Exploratorium"]}

図3-1相当: 追記ログとインメモリhash map

keyoffset説明状態 1234560Londonの開始位置有効 4268最新San Francisco有効

メモリ上のhash mapは「key → ファイル内のbyte offset」を保持する。

offset 0: 123456,{"name":"London",...}
offset 40: 42,{"name":"San Francisco", "Golden Gate Bridge"}
offset 68: 42,{"name":"San Francisco", "Exploratorium"}

追記だけなら書き込みは非常に速い。一方で、インデックスがなければ読み取りは毎回フルスキャンになる。

この単純DBから見える本質

  • 書き込みは速い: ファイル末尾への追記は、ディスクにとって最も単純な書き込みの一つです。
  • 読み取りは遅い: db_get は全行を走査するため、レコード数を n とすると検索コストは O(n) です。
  • インデックスが必要: 読み取りを速くするため、主データとは別に「探すための補助構造」を持ちます。
  • インデックスは無料ではない: 読み取りを速くする代わりに、書き込み時にはインデックス更新コストが増えます。
重要なトレードオフ: インデックスはクエリを速くしますが、全ての書き込みを遅くします。何でもインデックスを張るのは設計ではなく、将来の書き込み性能とストレージ容量を削る行為です。

3. Hash Indexes

Hash index は、key-value store に対して非常に自然なインデックスです。メモリ上のhash mapに、各keyの最新値が存在するファイル位置を保持します。

仕組み

  1. 値を書き込むとき、ログファイル末尾に key,value を追記する。
  2. 同時に、メモリ上のhash mapを key → 最新レコードのoffset に更新する。
  3. 読み取り時はhash mapでoffsetを得て、その位置へseekして値を読む。

RiakのデフォルトストレージエンジンだったBitcaskは、この考え方に近い設計です。条件は、全keyがメモリに収まることです。value自体はディスク上に置けますが、keyとoffsetの対応表はRAMに置く必要があります。

ログが肥大化する問題: segment と compaction

追記だけでは古い値が永遠に残り、ディスクを使い切ります。そこでログを一定サイズの segment に分け、古いsegmentをバックグラウンドで compaction します。compactionは同じkeyの古い値を捨て、最新値だけを残す処理です。

実装上の論点

論点説明
ファイル形式CSV風の文字列より、長さ付きバイナリ形式のほうが高速で曖昧さが少ない。
削除実データをその場で消すのではなく、tombstone と呼ばれる削除マーカーを追記する。compaction時に過去値を破棄する。

並行制御追記順序を単純化するため、書き込みは1スレッドに寄せ、読み取りは不変segmentを複数スレッドで読む構成が取りやすい。

長所と限界

長所

  • 追記とsegment mergeは順次書き込みになりやすく、ランダム書き込みより速い。
  • segmentが不変なので、クラッシュ復旧や並行読み取りが比較的単純。
  • 更新頻度が高く、distinct key数がRAMに収まる場合に強い。

限界

  • key全体をメモリに置く必要がある。
  • 範囲検索に弱い。kitty00000〜kitty99999 を順に読む、といった処理には向かない。
  • ハッシュ衝突やリサイズをディスク上でうまく扱うのは難しい。

4. SSTables and LSM-Trees

Hash indexの限界を超えるために、segment内のkey-valueペアを key順にソート した形式にします。この形式が SSTable（Sorted String Table） です。SSTableはLSM-treeの基礎になります。

SSTableの重要条件

  • segment内のkey-valueペアがkey順に並んでいる。
  • compaction後のmerged segmentでは、同じkeyは1回だけ出現する。
  • segmentは書き終わったら不変であり、後からその場更新しない。

なぜSSTableは強いのか

  1. mergeが効率的: 入力segmentがソート済みなので、merge sortのように順次読み取りで結合できます。
  2. 疎なインデックスでよい: 全keyのoffsetをRAMに持つ必要がなく、数KBごとに代表keyのoffsetを持てばよい。
  3. 圧縮しやすい: key順に近いデータが並ぶため、ブロック圧縮との相性が良い。

handiwork を探す場合、handbag と handsome の間にあると分かるため、その範囲だけをスキャンする。

SSTableをどう作るか: memtable

ディスク上で常にソート済み構造を更新するのは高コストです。そこで、まずメモリ上の木構造に書き込みます。このメモリ上の構造を memtable と呼びます。

  1. 書き込みが来たら、メモリ上のbalanced tree（例: red-black tree）に追加する。
  2. memtableが一定サイズを超えたら、key順に走査してSSTableファイルとしてディスクに書き出す。
  3. 読み取り時は、まずmemtable、次に新しいSSTable、さらに古いSSTableの順で探す。
  4. バックグラウンドでSSTable同士をmerge/compactionし、古い値や削除済み値を整理する。
クラッシュ対策: memtable上の最新書き込みはまだディスクSSTable化されていないため、別途append-only logへ即時追記します。DB再起動時にはこのログからmemtableを復元し、SSTable化が終わったログは破棄できます。

LSM-treeとは何か

この「memtableに書く → SSTableへflush → SSTableをmerge/compactionする」という考え方が LSM-tree（Log-Structured Merge-Tree） です。LevelDB、RocksDB、Cassandra、HBaseなどの基礎思想として使われています。また全文検索エンジンのLuceneも、term dictionaryに似た発想を使います。

5. B-Trees

B-treeは、現在でも多くのリレーショナルDBで標準的に使われるインデックス構造です。SSTable/LSM-treeが「不変segmentを追記・mergeする」思想なのに対して、B-treeは「固定サイズのpageをその場で更新する」思想です。

基本構造

  • DBファイルを固定サイズの page/block に分割する。典型的には4KB程度です。
  • 各pageは、子pageへの参照とkey範囲の境界を持つ。
  • 検索はroot pageから始まり、keyが属する範囲のchild pageへ降りていく。
  • 最終的にleaf pageへ到達し、値そのもの、または値があるheap fileの位置を得る。

図3-6相当: B-treeによるkey検索

root page → internal page → leaf pageへ降りる。各pageのkey境界により、探索範囲が段階的に絞られる。

分岐係数と深さ

1つのpageが参照できる子page数を branching factor と呼びます。実際には数百になることもあり、そのためB-treeは非常に浅くなります。例えば4KB pageでbranching factorが500なら、4階層でも巨大な件数を扱えます。つまり、数回のpage参照で目的のkeyへ到達できます。

挿入とpage split

新しいkeyを挿入する場合、該当するleaf pageを探して追加します。pageに空きがなければ、pageを2つに分割し、親pageの境界情報を更新します。

信頼性: WAL が必要になる理由

B-treeはpageを上書きします。page splitのような処理では、複数pageを整合的に更新しなければなりません。途中でクラッシュすると、親pageと子pageの参照関係が壊れる危険があります。そこで多くの実装では WAL（write-ahead log / redo log） を使います。

WALの原則は、B-tree本体のpageを変更する前に「これから行う変更」をログへ書くことです。クラッシュ後はWALを再生し、B-treeを一貫した状態へ戻します。

6. B-tree と LSM-tree の比較

B-treeとLSM-treeの性能比較は、単純な勝敗で決まりません。ベンチマークもワークロードや設定に強く依存します。第3章では一般論として、LSM-treeは書き込みに強い傾向、B-treeは読み取りの予測可能性や成熟度に強い傾向が説明されます。

観点LSM-treeB-tree
基本思想追記、SSTable化、compaction固定pageのupdate-in-place
書き込み順次書き込みに寄せられるため高スループットを出しやすいpage更新、WAL、page splitによりランダムI/Oが増えやすい
読み取り複数SSTableを確認する場合があり、読み取りが遅くなることがあるkeyは基本的に1箇所に存在し、読み取りが予測しやすい
容量効率圧縮しやすく、断片化が少ない傾向page split後の空き領域や断片化が発生しやすい
遅延のばらつきcompactionが高パーセンタイル遅延に影響しうる比較的安定しやすいが、WALやpage splitの影響はある
トランザクション分離複数バージョン/segmentを考える必要があるkey範囲にロックを結び付けやすく、RDBの文脈で成熟している

Write amplification

書き込み1回が、内部的に何回もディスク書き込みを発生させる現象を write amplification と呼びます。B-treeはWALとpage更新で最低でも複数回書く場合があります。LSM-treeもcompactionで同じデータを繰り返し書くためwrite amplificationが発生します。

実務判断: write-heavyだから常にLSM-tree、read-heavyだから常にB-tree、という短絡は危険です。読み取りの種類がpoint lookupなのかrange scanなのか、compaction設定、Bloom filter、メモリ、SSD特性、SLOのp99要件まで見ないと判断できません。

7. その他のインデックス構造

ここまで主にprimary key的なkey-value indexを見てきました。しかし実際のDBでは、join、検索、複数条件、地理検索、全文検索など、多様なアクセスパターンがあります。

Secondary index

主キー以外の列やフィールドに張るインデックスです。RDBでは CREATE INDEX で同じテーブルに複数作れます。例えば、ユーザーIDごとの注文を探す、メールアドレスでユーザーを探す、ステータスと作成日で一覧を絞る、といった用途です。

primary keyと違い、secondary indexのkeyは一意とは限りません。実装としては、index valueを「該当行IDのリスト」にする、または同じkeyに対して複数エントリを持つ、という方法があります。

index内に何を保存するか

方式説明長所/短所
heap file参照indexには実データの場所だけを持ち、実データはheap fileに置く。複数indexでデータ重複が少ない。ただし読み取り時にindex→heapの追加アクセスが必要。
clustered indexindexのleafに行データそのものを置く。読み取りが速い場合がある。MySQL InnoDBのprimary keyが代表例。
covering indexクエリに必要な一部列をindex内に含める。indexだけでクエリを完結できる場合があるが、書き込みと容量の負担が増える。

Multi-column index

複数列を1つのkeyとして結合するインデックスです。典型例は (last_name, first_name) のような連結インデックスです。この場合、last_name だけ、または last_name + first_name には効きますが、first_name だけの検索には効きにくい点に注意が必要です。

Multi-dimensional index

地理座標など、複数軸の範囲を同時に検索する場合、単純なB-tree/LSM-treeでは苦しくなります。例えば「現在表示している地図の矩形内にあるレストラン」を探すには、緯度と経度の範囲を同時に満たす必要があります。

原書に記載されているSQL例

SELECT * FROM restaurants WHERE latitude  > 51.4946 AND latitude  < 51.5079
                          AND longitude > -0.1162 AND longitude < -0.1004;

このような用途では、空間充填曲線で2次元を1次元に変換して通常のB-treeを使う方法や、R-treeのような空間インデックスが使われます。PostGISのGiST indexは代表例です。

Full-text search と fuzzy index

これまでのindexは「完全一致」または「sort順の範囲検索」が中心でした。しかし全文検索では、類義語、語形変化、スペルミス、編集距離などを扱います。Luceneはterm dictionaryにSSTable的な構造を使いつつ、メモリ上には有限状態オートマトンなどを使い、近い単語の検索を効率化します。

8. Keeping Everything in Memory

ディスクはRAMに比べて扱いにくいものです。読み書き性能、データ構造の制約、ランダムアクセスのコストが問題になります。一方で、ディスクには「永続性」と「GBあたりの安さ」がありました。しかしRAMが安くなるにつれ、全データをメモリに置くDBも現実的になってきました。

インメモリDBのポイント

  • Memcachedのようにキャッシュ専用で、失われてもよいものがある。
  • VoltDB、MemSQL、Oracle TimesTenのように、RDB的なモデルでインメモリを売りにするものがある。
  • RedisやCouchbaseのように、非同期にディスクへ書くことで弱い永続性を持つものがある。
  • パフォーマンス上の利点は「ディスクを読まない」ことだけではなく、ディスク向けのエンコード形式へ変換しなくてよい点にもある。
OSのページキャッシュにより、ディスクベースDBでも頻繁に読むデータがメモリに乗ることはあります。それでもインメモリDBは、メモリ上のデータ構造を直接扱えるため、より単純で高速な実装にできる場合があります。

9. Transaction Processing or Analytics?

第3章後半は、ストレージエンジンの話をOLTPとOLAPの違いへ広げます。ここを理解しないと、「RDBで集計が遅い」「データウェアハウスがなぜ別にあるのか」「カラムナDBがなぜ分析に強いのか」が曖昧になります。

性質OLTP: Transaction ProcessingOLAP: Analytics

主な書き込みユーザー操作による低遅延なランダム書き込みETLやイベントストリームによる一括投入
主な利用者エンドユーザー、Webアプリ社内アナリスト、意思決定支援
データの意味現在時点の最新状態過去に起きたイベントの履歴
典型的なデータ量GB〜TBTB〜PB
誤解注意: OLTPの「transaction」は必ずACID transactionを意味するわけではありません。ここでは「少数レコードを低遅延で読み書きする対話的処理」という意味が中心です。

10. Data Warehousing と ETL

企業には、ECサイト、在庫管理、経路計画、従業員管理など複数のOLTPシステムがあります。これらは業務上重要なので、分析クエリで重くしたくありません。そこで、分析専用の別DBとして data warehouse を用意します。

図3-8相当: OLTPシステムからデータウェアハウスへのETL

ETLは Extract → Transform → Load。OLTP側からデータを取り出し、分析しやすい形に変換し、DWHへ投入する。

なぜ分けるのか

  • OLTPは業務運用に直結するため、重い分析クエリで性能劣化させたくない。
  • DWHでは分析向けにスキーマ、インデックス、ストレージレイアウトを最適化できる。
  • 複数業務システムのデータを統合・クレンジングできる。

11. Stars and Snowflakes: Schemas for Analytics

DWHでは、多くの場合リレーショナルモデルが使われます。ただしOLTPの正規化設計とは異なり、分析しやすい定型パターンとして star schema がよく使われます。

Star schema

中心に fact table があり、その周囲に dimension table が並びます。factは「発生したイベント」、dimensionは「何を、いつ、どこで、誰が、どの販促で」といった分析軸です。

図3-9相当: 小売データのstar schema

中心のfact tableから周囲のdimension tableへ外部キーが伸び、星形に見える。

Snowflake schema

dimension tableをさらに正規化し、サブdimensionへ分解したものが snowflake schema です。例えば dim_product の中にbrand/category文字列を直接持つ代わりに、dim_brand や dim_category を参照させます。より正規化されますが、分析者にとってはstar schemaの方が扱いやすいことが多いです。

12. Column-Oriented Storage

DWHのfact tableは非常に大きく、列数も多いことがあります。しかし分析クエリは全列を読むことが少なく、数列だけを読んで集計することが多いです。そこで、行単位ではなく列単位で保存します。

原書に記載されているSQL例: Example 3-1

曜日ごとに、fresh fruitとcandyの販売数量を比較する分析クエリです。

SELECT
    dim_date.weekday, dim_product.category,
    SUM(fact_sales.quantity) AS quantity_sold
FROM fact_sales
    JOIN dim_date    ON fact_sales.date_key   = dim_date.date_key
    JOIN dim_product ON fact_sales.product_sk = dim_product.product_sk
WHERE
    dim_date.year = 2013 AND
    dim_product.category IN ('Fresh fruit', 'Candy')
GROUP BY
    dim_date.weekday, dim_product.category;

このクエリで必要なのは主に date_key、product_sk、quantity です。行指向ストレージでは、各行の100列以上を読み込んでから不要列を捨てることになります。カラム指向では、必要な列ファイルだけを読めます。

図3-10相当: 行指向 vs カラム指向

Row-oriented

row1: date, product, store, qty, price...
row2: date, product, store, qty, price...
row3: date, product, store, qty, price...

1行を構成する全列が近くに保存される。1件取得・更新に強い。

Column-oriented

date_key: 140102, 140102, 140103...
product_sk: 69, 69, 31, 31...
quantity: 1, 3, 1, 3...
net_price: ...
discount_price: ...

列ごとに保存される。分析クエリで必要列だけ読むのに強い。

カラム指向というとRDBだけを想像しがちですが、Parquetのようなカラムナフォーマットはドキュメント的なデータモデルにも対応できます。

13. Column Compression

カラム指向ストレージは、必要列だけを読むだけでなく、圧縮にも強いです。同じ列には似た値が連続しやすく、値の種類も少ないことが多いからです。

Bitmap encoding

ある列に登場する distinct value ごとにbitmapを作ります。各bitmapは「その行の値がこの値なら1、違えば0」というビット列です。

bitmapが分析クエリに強い理由

原書に記載されている条件例

WHERE product_sk IN (30, 68, 69)

product_sk = 30、68、69 の3つのbitmapを読み、bitwise ORを取れば該当行を高速に求められます。

WHERE product_sk = 31 AND store_sk = 3

この場合は product_sk = 31 のbitmapと store_sk = 3 のbitmapのbitwise ANDを取ります。各列ファイルの行順が揃っているため、同じ位置のbitは同じ行を指します。

Memory bandwidth と vectorized processing

分析クエリでは、ディスクI/Oだけでなく、メモリからCPU cacheへデータを移す帯域やCPU処理もボトルネックになります。カラム指向+圧縮では、CPUのL1 cacheに圧縮データの塊を載せ、分岐の少ないtight loopで処理できます。さらにSIMD命令を使って、複数値をまとめて処理できます。これが vectorized processing です。

14. Sort Order in Column Storage

カラムストアでも、行の順序は重要です。各列を独立にソートしてはいけません。各列のk番目の値が同じ行のk番目である、という対応が壊れるからです。ソートするなら、行全体を同じ順序で並べます。

並び順が効く理由

  • クエリが日付範囲をよく指定するなら、date_key を第一ソートキーにすると直近月だけを効率よく読むことができます。
  • 第一ソートキーが同じ行では、第二ソートキーで並べると、同じ商品や店舗の行がまとまり、集計が速くなる場合があります。
  • 値が連続しやすくなるため、カラム圧縮も効きやすくなります。

複数のsort order

C-StoreやVerticaのようなシステムでは、同じデータを複数の異なるsort orderで持つ考え方があります。これは行指向DBのsecondary indexに少し似ていますが、カラムストアでは「別の場所の行を指すポインタ」ではなく、値そのものを別の並びで持つ点が異なります。

15. Writing to Column-Oriented Storage

カラム指向ストレージ、圧縮、ソートは読み取りを速くしますが、書き込みは難しくなります。圧縮済みの列ファイルの途中へ1行挿入するには、全列ファイルを整合的に書き換える必要があるからです。

ここでLSM-tree的な発想が再登場します。まず書き込みをメモリ上の構造に受け、十分たまったらソート済み・圧縮済みのカラムファイルとして一括書き出しします。クエリ実行時には、ディスク上のカラムデータとメモリ上の最近の書き込みを統合して見せます。

16. Aggregation: Data Cubes and Materialized Views

DWHでは、COUNT、SUM、AVG、MIN、MAX などの集計が多く出ます。同じ集計を毎回raw dataから計算するのは無駄です。そのため、集計済み結果を保存する materialized view が使われることがあります。

Materialized view

通常のviewはクエリのショートカットであり、実体は保存されません。一方、materialized viewはクエリ結果の実体をディスクに保存します。読み取りは速くなりますが、元データ変更時にmaterialized viewも更新する必要があり、書き込みは重くなります。

Data cube / OLAP cube

data cubeは、複数のdimensionの組み合わせごとに集計値を事前計算したものです。例えば date_key × product_sk ごとに SUM(net_price) を持てば、「特定商品の特定日の売上」や「日付ごとの合計」「商品ごとの合計」を高速に答えられます。

弱点: data cubeは事前に決めたdimensionでは速い一方、raw dataほど柔軟ではありません。例えば「価格が100ドル超の商品だけ」など、cubeに含めていない条件には弱いです。そのため、多くのDWHではraw dataを保持しつつ、一部の頻出集計だけをmaterialized viewやcubeで高速化します。
