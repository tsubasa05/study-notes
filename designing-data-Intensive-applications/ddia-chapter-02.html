1. データモデルは、単なる保存形式ではなく「考え方」を決める

ソフトウェア開発において、データモデルは極めて重要である。なぜなら、データモデルは単にデータの保存方法を決めるだけではなく、開発者が問題領域をどう捉えるか、どのようにアプリケーションを設計するかにまで影響するからである。

アプリケーションは、複数のデータモデルを層として重ねながら作られている。現実世界には人、組織、商品、取引、センサー、行動履歴などが存在する。開発者は、それらをオブジェクト、構造体、API、ドメインモデルとして表現する。さらに、それを永続化する段階では、リレーショナルテーブル、JSON/XMLドキュメント、グラフ構造など、汎用的なデータモデルに変換する。

データベース内部では、それらがメモリ・ディスク・ネットワーク上のバイト列として表現される。つまり、上位の層は下位の層の複雑さを隠蔽し、扱いやすい抽象化を提供している。

重要な見方： よいデータモデルは、開発者が問題を自然に表現できるようにする。一方で、合わないデータモデルを選ぶと、本来は単純なはずの操作が不自然になり、実装も複雑になる。

この章の中心的な問いは、「どのデータモデルが、どのようなアプリケーションに向いているのか」である。

2. リレーショナルモデルとドキュメントモデル

現在もっとも広く知られているデータモデルは、SQLで使われるリレーショナルモデルである。リレーショナルモデルでは、データはリレーション、SQLでいうテーブルとして表現され、各テーブルは行の集合として扱われる。

リレーショナルモデルは1970年にEdgar Coddによって提案された。当初は理論的な提案であり、効率的に実装できるのか疑問視されていた。しかし1980年代半ばには、RDBMSとSQLが、構造化されたデータを保存・検索するための標準的な手段になった。

もともとリレーショナルデータベースは、銀行取引、航空券予約、在庫管理、給与計算、請求処理など、企業の業務データ処理を主な対象としていた。だが、その後Webサービス、SNS、EC、ゲーム、SaaSなど、当初の想定を超えた幅広い用途にも適用され続けた。

NoSQLが登場した背景

2010年代に入ると、NoSQLという言葉が広まった。NoSQLは特定の技術を指す厳密な用語ではなく、もともとは非リレーショナルな分散データベースを扱うミートアップのハッシュタグとして使われた言葉である。その後、「Not Only SQL」と解釈されるようになった。

  • 巨大データや高い書き込みスループットへの対応が必要になった
  • 商用DBよりも、自由に使えるOSSを好む流れが強まった
  • リレーショナルモデルでは扱いづらい特殊な問い合わせが必要になった
  • 固定的なスキーマへの不満から、より柔軟なデータモデルが求められた
誤解しやすい点： NoSQLはRDBを完全に置き換えるものではない。アプリケーションごとに適したデータモデルは異なるため、リレーショナルDBと非リレーショナルDBは併存する。

3. オブジェクト指向とリレーショナルモデルのずれ

多くのアプリケーションはオブジェクト指向言語で書かれる。しかし、リレーショナルDBではデータをテーブル、行、列として扱う。このため、アプリケーション上のオブジェクト構造と、DB上のテーブル構造の間には変換が必要になる。このずれは、インピーダンスミスマッチと呼ばれる。

ORM、たとえばHibernateやActiveRecordは、この変換作業を軽減する。しかし、オブジェクトモデルとリレーショナルモデルの違いそのものを完全に消すことはできない。

LinkedInプロフィールの例

本章では、LinkedInのプロフィールのような履歴書データを例にしている。ユーザーの名前や概要のように一人につき一つだけ存在する情報は、usersテーブルの列として自然に表現できる。しかし、職歴、学歴、連絡先のように一人に対して複数存在する情報は、別テーブルに分けて外部キーで結びつける必要がある。

一方、同じプロフィールをJSONドキュメントとして表現すると、ユーザー情報、職歴、学歴、連絡先を一つのまとまった文書として保存できる。この場合、プロフィール全体を取得するときには一度の読み取りで済み、データの局所性が高い。 

{
  "user_id": 251,
  "first_name": "Bill",
  "last_name": "Gates",
  "positions": [
    { "job_title": "Co-chair", "organization": "Bill & Melinda Gates Foundation" },
    { "job_title": "Co-founder, Chairman", "organization": "Microsoft" }
  ],
  "education": [
    { "school_name": "Harvard University", "start": 1973, "end": 1975 }
  ]
}
ドキュメントモデルが自然なケース： 一つの親データに複数の子データがぶら下がり、それらをまとめて読み書きするケース。職歴や学歴はユーザーにぶら下がる一対多のツリー構造なので、JSONドキュメントの入れ子構造と相性がよい。

4. 多対一・多対多の関係が出てくると話が変わる

ドキュメントモデルが得意なのは、基本的にツリー構造である。しかし、現実のアプリケーションでは、データは次第に相互接続されていく。

たとえば、プロフィールの地域や業界を単なる文字列として保存するのではなく、標準化された地域IDや業界IDとして保存したくなる。これは、表記ゆれを防ぐ、曖昧さを避ける、名称変更に対応しやすくする、多言語表示を可能にする、検索を改善する、といった利点があるからである。

正規化の意味

人間にとって意味のある文字列を各レコードに重複して保存すると、後で変更が必要になったときにすべてのコピーを更新しなければならない。更新漏れがあれば不整合が起きる。そこで、意味のある情報は一箇所に置き、それをIDで参照する。これが正規化の基本的な考え方である。

保存方法 特徴 リスク・注意点
文字列を直接保存 単純で読みやすい。JOIN不要。 表記ゆれ、重複、変更時の更新漏れが起きやすい。
IDで参照 意味のある情報を一箇所に集約できる。正規化しやすい。 参照先を引く必要があり、JOINまたはアプリ側の結合処理が必要になる。

正規化すると、多対一や多対多の関係が増える。たとえば、多くのユーザーが一つの地域に属する。多くのユーザーが一つの業界に属する。さらに、会社や学校を単なる文字列ではなく独立したエンティティにしたり、ユーザー同士が推薦文を書けるようにしたりすると、データ同士のつながりはさらに複雑になる。

設計上の盲点： 初期段階ではドキュメントモデルにきれいに収まっていたデータでも、機能追加によって関係が増えると、徐々にドキュメントモデルだけでは扱いづらくなる。

5. ドキュメントDBは過去の歴史を繰り返しているのか

本章では、ドキュメントDBと過去の階層型データベースの類似性も論じられている。

1970年代には、IBMのIMSのような階層型データベースが使われていた。階層型モデルでは、データは親子関係を持つレコードのツリーとして表現される。これは、現代のJSONドキュメントの入れ子構造にかなり似ている。

階層型モデルは一対多の関係には向いていたが、多対多の関係やJOINには弱かった。そのため、開発者はデータを重複して持つか、参照をアプリケーション側でたどる必要があった。これは、現代のドキュメントDBで起こりがちな問題とよく似ている。

ネットワークモデルとリレーショナルモデル

この問題に対する歴史的な解決策として、ネットワークモデルとリレーショナルモデルが登場した。ネットワークモデルでは、レコードが複数の親を持つことができ、多対多を表現できた。しかし、データへアクセスするには、あらかじめ定義されたアクセスパスをアプリケーションがたどる必要があった。

これに対して、リレーショナルモデルの大きな進歩は、アクセスパスの指定をアプリケーションから切り離した点にある。開発者は「何を取得したいか」を宣言し、データベースのクエリオプティマイザが「どの順序で、どのインデックスを使って取得するか」を決める。

リレーショナルモデルの本質： 単にテーブル形式であることではない。データの物理的な辿り方をアプリケーションから隠蔽し、クエリ最適化をDB側に任せられることにある。

6. 現代におけるリレーショナルDBとドキュメントDBの比較

リレーショナルDBとドキュメントDBの優劣は、単純には決められない。どちらが優れているかではなく、データの形とアクセスパターンに合っているかが重要である。

観点 ドキュメントモデル リレーショナルモデル
得意なデータ構造 一対多のツリー構造。親子データをまとめて扱う構造。 多対一・多対多を含む相互参照の多い構造。
読み取り 関連情報を一つの文書にまとめられるため、局所性が高い。 JOINにより、必要な関係を柔軟に組み立てられる。
スキーマ 柔軟。ただし実際にはアプリケーション側が構造を期待している。 DBがスキーマを強制する。データ品質を保ちやすい。
複雑な関係 JOINが弱い場合、アプリ側で複数クエリを組み合わせる必要がある。 外部キーとJOINで自然に扱いやすい。

「スキーマレス」という表現への注意

ドキュメントDBでよく言われる「スキーマレス」という表現には注意が必要である。実際には、アプリケーションコードは何らかのデータ構造を期待している。つまり、スキーマが存在しないのではなく、DBが強制するスキーマではなく、アプリケーション側が読み取り時に解釈するスキーマになっている。

この違いは、schema-on-write と schema-on-read の違いとして理解できる。

7. クエリ言語：命令型より宣言型が強い理由

SQLは宣言型クエリ言語である。宣言型とは、「どのデータが欲しいか」を記述する形式であり、「どの順番で、どの手順でデータを取得するか」は直接指定しない。

これに対して、命令型のAPIでは、アプリケーション開発者がデータ取得の手順を細かく書く。どのレコードをたどるか、どの順番で処理するか、条件に合うかどうかをどう判定するかを、コードとして記述する必要がある。

宣言型

何が欲しいかを書く。実行方法は処理系に任せる。

命令型

どう取得するか、どの順で処理するかを書く。

利点

DBが実行計画やインデックス利用を最適化できる。

宣言型クエリの利点は、DBがクエリ最適化できる点にある。開発者がアクセスパスを固定してしまうと、インデックス追加や実行計画の改善を活かしづらい。一方、宣言型であれば、DB側が実行順序やインデックス選択を変更しても、クエリの意味は変わらない。

WebのCSSも宣言型の例として理解できる。CSSでは「この条件に合う要素にはこのスタイルを適用する」と宣言するだけでよい。JavaScriptでDOMを手続き的にたどってスタイルを変更するよりも、CSSのほうが短く、理解しやすく、ブラウザ側の最適化もしやすい。

8. MapReduceクエリ

NoSQLデータストアの一部では、MapReduceをクエリ手段として使うことがある。MapReduceは、大量のデータを複数マシンで分散処理するためのプログラミングモデルである。

MapReduceは完全な宣言型でも、完全な命令型でもない。map関数とreduce関数というコード片を開発者が書き、それを処理基盤が多数のデータに対して繰り返し適用する。

たとえば、観測データから「月ごとのサメの目撃数」を集計する場合、SQLならフィルタ、グループ化、合計を一つのクエリとして書ける。一方、MapReduceでは、各観測データから年月と個体数を取り出すmap関数と、同じ年月の個体数を合計するreduce関数を書く。

実務上の評価： MapReduceは分散処理には有用だが、使いやすさという点ではSQLのような宣言型クエリに劣る。NoSQLシステムも、使いやすさを追求するとSQLに似た宣言型機能を再発明しがちである。

9. グラフ型データモデル

データの関係がさらに複雑になり、多対多の関係が中心になると、グラフモデルが自然になる。

グラフは、頂点と辺で構成される。頂点は人、場所、組織、Webページ、道路の交差点などを表す。辺は、それらの間の関係を表す。たとえば、SNSなら人が頂点で、友人関係が辺になる。Webならページが頂点で、リンクが辺になる。道路網なら交差点が頂点で、道路が辺になる。

property graph

property graphでは、各頂点と各辺がIDを持ち、さらにkey-value形式のプロパティを持つ。辺には向きがあり、どの頂点からどの頂点へ向かう関係なのかを表す。また、辺にはラベルがあり、関係の種類を表せる。

Cypher

Cypherは、property graph向けの宣言型クエリ言語である。ノードと辺のパターンを図式的に表すような構文を持ち、「このような関係パターンに一致するデータを探す」という書き方ができる。

triple-store / SPARQL / Datalog

triple-storeは、すべての情報を「主語・述語・目的語」の三つ組で表すモデルである。RDFやSPARQLはこの系統に属する。SPARQLは、この三つ組の集合に対してパターンマッチする宣言型クエリ言語である。

Datalogは古い言語だが、ルールに基づいてデータを問い合わせる強力な表現力を持っている。一般的なアプリケーション開発者には馴染みが薄いが、複雑な推論や関係探索に向いた考え方を持っている。

10. Chapter 2の結論

この章の結論は、データモデルに絶対的な正解はない、ということに尽きる。

  • Document データが一対多のツリー構造で、親データと子データをまとめて読み書きするなら、ドキュメントモデルは自然である。
  • Relational データ同士の関係が多く、JOINや多対多の関係を頻繁に扱うなら、リレーショナルモデルが強い。
  • Graph 関係そのものが主要な関心事であり、データが密につながっているなら、グラフモデルが有力になる。

設計時に先に問うべきこと

  • データはツリー状か、相互参照が多いか
  • 一つの集約を丸ごと読むのか、部分的に多方向から検索するのか
  • 重複を許容するのか、正規化して一貫性を保つのか
  • JOINをDBに任せるのか、アプリケーション側で解決するのか
  • 将来、データ同士の関係が増える可能性は高いか
バックエンド設計への示唆： DB選定やテーブル設計を「なんとなくRDB」「なんとなくJSON」で決めると、後から機能追加で苦しくなる。Chapter 2は、データモデルを“保存形式”ではなく、“アプリケーションの拡張性・複雑性・問い合わせ能力を左右する設計判断”として捉え直す章である。
