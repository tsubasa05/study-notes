技術質問

1. REST API開発経験

ユーザー管理機能で、ユーザー登録、更新、削除、検索などのREST API設計を担当しました。

リクエストでは、ユーザーの共通属性と、事業者・一般ユーザーごとの業務属性を受け取り、裏側でKeycloak、独自DB、OpenFGAへ必要な処理を行う構成です。レスポンスは、呼び出し側が複数システムの内部構造を意識しなくてよいよう、ユーザー管理機能の業務モデルとして返す方針にしました。

API設計では、入力検証、権限による操作範囲、データ整合性、外部基盤が途中で失敗した場合の扱い、検索件数、URL長、エラーの返し方を考慮しました。

2. Spring BootのDIとは

DIはDependency Injectionの略で、クラスが必要とする依存オブジェクトを、自分で生成するのではなく外部から渡す考え方です。Springでは、DIコンテナがBeanの生成、ライフサイクル、依存関係の組み立てを管理します。

例えば、ユーザー管理のServiceがRepositoryや外部認証基盤Clientを使う場合、Serviceの中でnew UserRepository()やnew KeycloakClient()を直接書くのではなく、コンストラクタで受け取ります。

差し替えやすいとは何か

DIでいう「差し替えやすい」は、IDのような値を変えるという意味ではありません。

クラスが使う依存先を、コードの中で固定せず、外から渡せる状態にするという意味です。

例えば、次のようにServiceの中で直接newすると、UserServiceはKeycloakClientという具体クラスの生成方法まで知ってしまいます。

public class UserService {
    private final KeycloakClient keycloakClient = new KeycloakClient();
}

この場合、テストでKeycloakClientをモックにしたい、検証環境だけスタブにしたい、別実装に変えたい、となっても、UserService内部で固定されているため差し替えにくくなります。

DIを使うと、UserServiceは「Keycloak連携するものが必要」という依存だけを持ち、実際に何を渡すかは外側で決められます。

@Service
public class UserService {
    private final AuthClient authClient;

    public UserService(AuthClient authClient) {
        this.authClient = authClient;
    }
}

本番では本物のKeycloakAuthClientを渡し、単体テストではMockitoのモックを渡す、といったことができます。

AuthClient authClient = mock(AuthClient.class);
UserService service = new UserService(authClient);

つまり差し替えとは、主に次のような意味です。

  • 本番実装とテスト用モックを切り替える
  • 本番実装とスタブ実装を切り替える
  • 環境ごとにBeanを切り替える
  • インターフェースに対して複数実装を持ち、@Qualifierや@Primaryで使う実装を選ぶ

面接で言うなら、次で十分です。

DIで差し替えやすいというのは、IDを変えるというより、Serviceが依存先を自分でnewして固定しないことです。外部API ClientやRepositoryをコンストラクタで受け取る形にしておけば、本番では実実装、単体テストではモック、検証ではスタブのように、外側から渡すものを変えられます。

newしているとモックできないのか

厳密には、特殊な仕組みを使えばnewされたオブジェクトを差し替える方法もあります。ただし、一般的な設計としてはやりにくく、テストも複雑になります。

問題は、対象クラスの内部で依存先を生成してしまうと、テスト側からその依存先を渡せないことです。

public class UserService {
    public void register(User user) {
        KeycloakClient client = new KeycloakClient();
        client.createUser(user);
    }
}

この場合、テストではKeycloakClientが500エラーを返したことにしたい、タイムアウトしたことにしたい、という制御がしづらくなります。実際の外部APIに近いものを用意するか、特殊なモック機能を使う必要が出ます。

一方、DIで外から渡せるようにしておけば、テストでは依存先の振る舞いを自由に制御できます。

when(authClient.createUser(user)).thenThrow(new ExternalApiException());

これにより、外部APIが失敗したときにServiceが後続のDB保存へ進まないか、適切な例外に変換するか、ログを残すか、というService自身の判断を確認できます。

application.properties、@Configuration、DIの違い

application.propertiesやapplication.ymlは、URL、タイムアウト、接続先、認証情報のような「値」を外出しする仕組みです。

keycloak.base-url=https://example.com
keycloak.timeout=3000

@Configurationは、その設定値などを使ってBeanを作るための設定クラスです。

@Configuration
public class KeycloakConfig {
    @Bean
    public AuthClient authClient(KeycloakProperties properties) {
        return new KeycloakAuthClient(properties.baseUrl(), properties.timeout());
    }
}

DIは、こうして作られたBeanを、必要なクラスへ注入する仕組みです。

@Service
public class UserService {
    public UserService(AuthClient authClient) {
        this.authClient = authClient;
    }
}

整理すると次の通りです。

仕組み役割
application.properties環境ごとに変わる値を外出しする
@Configuration設定値などを使ってBeanの作り方を定義する
DI作られたBeanを、必要なクラスへ渡す
@Qualifier / @Primary複数Beanがある場合にどれを使うか選ぶ

面接での言い方:

application.propertiesはURLやタイムアウトのような値を外出しするものです。@Configurationは、その値を使ってClientなどのBeanを作る設定です。DIは、作られたBeanをServiceへ渡す仕組みです。

そのため、設定値の外出しとDIは関連しますが同じものではありません。設定値を変えるだけではなく、Serviceが依存先を直接生成しないことで、本番用Client、テスト用モック、スタブなどに切り替えやすくなる点がDIの重要なメリットだと理解しています。

なぜDIコンテナがBean生成と依存関係を管理するとよいのか

  1. クラス同士の結合を弱くできる
  • Serviceが具象クラスを直接生成すると、その実装に強く依存します。
  • DIを使うと、Serviceは「Repositoryを使う」「外部API Clientを使う」という依存だけを持ち、生成方法をSpring側に任せられます。
  1. 差し替えしやすい
  • 本番では実DBに接続するRepositoryを使い、テストではモックやインメモリ実装に差し替える、といったことがしやすくなります。
  • 外部API連携でも、本番用Client、スタブClient、テスト用Clientを切り替えやすくなります。
  1. 単体テストしやすい
  • Serviceのテストでは、Repositoryや外部API Clientをモック化し、Serviceの業務ロジックだけを確認できます。
  • 例えば「外部認証基盤のユーザー作成が失敗した場合に、業務DB登録を行わない」など、依存先の戻り値を制御してテストできます。
  1. 依存関係が明確になる
  • コンストラクタインジェクションにすると、そのクラスが何を必要としているかがコンストラクタを見るだけで分かります。
  • 依存が多すぎる場合は、そのクラスの責務が大きすぎるサインにもなります。
  1. 環境ごとの設定やBean構成と組み合わせやすい
  • DB接続、外部APIのURL、タイムアウト、認証情報などはapplication.propertiesへ外出しします。
  • その設定値を使って@ConfigurationでBeanを作り、DIでServiceへ渡します。
  • 開発環境、検証環境、本番環境で同じコードを使いながら、値やBean構成を変えやすくなります。

回答例

DIのメリットは、クラスが依存先の生成方法を知らなくてよくなることです。

例えばユーザー管理ServiceがRepositoryや外部認証基盤Clientを直接newすると、実装の差し替えやテストが難しくなります。SpringのDIコンテナにBean生成を任せ、Serviceはコンストラクタで依存を受け取るようにすると、本番では実DBや実外部API、テストではモックやスタブに差し替えやすくなります。

また、コンストラクタを見るだけで依存関係が分かるため、責務が大きくなりすぎていないかも確認しやすいです。循環依存が起きる場合は、DIの書き方でごまかすのではなく、責務分担を見直すべきだと考えています。

3. JUnit・モック

基本回答

JUnitを使ったテストでは、どの層の何を確認するのかを分けて考える必要があります。Service層なら業務ロジックや外部依存先が失敗したときの分岐、Controller層ならリクエスト・レスポンスやバリデーション、Repository層ならSQLやDB制約、結合寄りならSpringを起動してAPIからDBまでの流れを確認します。

JUnitはテストを実行する土台であり、実際にはMockito、MockMvc、Spring Boot Test、DataJpaTest、WireMockなどと組み合わせて、層ごとに目的を変えて使うイメージです。手動確認だけでは漏れやすい条件をコードとして残し、変更後も同じ観点を繰り返し確認できる点に価値があります。

JUnitで何をどの層に対してテストするか

JUnitは「API全体を外から叩くためだけのもの」ではありません。Spring Bootでは、次のように層を分けて使います。

テスト対象目的代表的な使い方
Service層の単体テスト業務ロジック、分岐、例外時の扱いを確認するJUnit + Mockito。Repositoryや外部API Clientはモック化する
Controller層のテストHTTPリクエスト、バリデーション、ステータスコード、レスポンス形式を確認する@WebMvcTest + MockMvc。Serviceはモック化する
Repository層のテストSQL、JPAのクエリ、DB制約、検索条件を確認する@DataJpaTest。テストDBやTestcontainersを使う
Spring起動ありの結合寄りテストController -> Service -> Repositoryの流れを確認する@SpringBootTest + MockMvcやTestRestTemplate
外部API連携のテスト外部APIの成功・4xx・5xx・タイムアウト時の扱いを確認する実APIではなくWireMockやスタブサーバを使うことが多い

面接での言い方:

JUnitはテスト実行の枠組みで、何でも同じ粒度で確認するものではありません。Service層ではRepositoryや外部API Clientをモック化して業務ロジックを確認し、Controller層ではMockMvcでステータスコードやレスポンスを確認し、Repository層ではテストDBを使ってクエリや制約を確認します。外部API連携は、本物の外部APIを毎回叩くのではなく、WireMockやスタブで成功・失敗パターンを再現するのが基本だと理解しています。

各層での具体例

##### Service層

Service層では、ControllerやHTTPの細部ではなく、業務上の判断を確認します。

例:

  • 既に同じメールアドレスのユーザーがいる場合は登録しない
  • Keycloak作成が失敗した場合はDB保存へ進まない
  • DB保存が失敗した場合に後続のOpenFGA登録へ進まない
  • OpenFGA登録に失敗した場合に、エラーにするのか、補償処理対象にするのか
  • 管理者以外が管理者向け操作をできない

Service層の単体テストでは、Repositoryや外部API Clientはモック化します。

##### Controller層

Controller層では、HTTPとして正しく受け取り、正しく返せるかを確認します。

例:

  • 必須項目がない場合に400を返す
  • 不正な形式のメールアドレスで400を返す
  • Serviceが業務エラーを返した場合に適切なステータスコードへ変換する
  • レスポンスJSONの項目名や形式が期待どおりか

ここでは、Serviceの中身までは深く見ず、Serviceはモックにすることが多いです。

##### Repository層

Repository層では、DBに対する問い合わせや制約を確認します。

例:

  • メールアドレスでユーザーを検索できる
  • 事業者IDで紐づくユーザー一覧を取得できる
  • 一意制約に違反した場合にエラーになる
  • ページングやソートが期待どおり動く

この層はモックではなく、テスト用DBを使って確認する価値があります。

##### 結合寄りテスト

ControllerからService、Repositoryまで通して確認するテストです。

例:

  • ユーザー登録APIにリクエストを投げる
  • Controllerでバリデーションされる
  • Serviceで業務ロジックが実行される
  • RepositoryでDBへ保存される
  • 期待したレスポンスが返る

外部APIは毎回本物を叩くのではなく、スタブやWireMockを使い、成功、4xx、5xx、タイムアウトを再現することが多いです。

何をテストするか

JUnitでは、実装の行数を増やすことではなく、壊れてはいけない振る舞いを確認します。

主な観点:

  • 正常系
  • 期待した入力で、期待した戻り値や状態更新になるか
  • 入力チェック
  • 必須項目がない
  • 文字数が長すぎる
  • 不正な形式
  • 許可されないロールやステータス
  • 境界値
  • 0件、1件、上限件数
  • 最大文字数
  • 日付の境界
  • トークン期限の直前・期限切れ
  • 異常系
  • DBに対象データがない
  • 既に存在する
  • 外部APIが4xx/5xxを返す
  • タイムアウトする
  • 権限
  • 一般ユーザーでは操作できない
  • 管理者のみ操作できる
  • 副作用
  • Repositoryの保存が呼ばれたか
  • 外部API呼び出しが不要な条件で呼ばれていないか
  • 失敗時に後続処理へ進んでいないか

ユーザー管理APIでのテスト観点

例えば、ユーザー登録・更新・検索のような処理では、次の観点で値を選びます。

  • ユーザー種別
  • 一般ユーザー
  • 事業者ユーザー
  • 管理者
  • 必須項目
  • メールアドレスあり・なし
  • 名前あり・なし
  • ロールあり・なし
  • 一意制約
  • 同じメールアドレスが存在しない
  • 同じメールアドレスが既に存在する
  • 外部認証基盤連携
  • Keycloak側の作成成功
  • Keycloak側の作成失敗
  • Keycloak作成成功後にDB保存で失敗
  • 認可情報連携
  • OpenFGAへのタプル登録成功
  • OpenFGA登録失敗時にどう扱うか
  • 検索
  • 0件
  • 1件
  • 複数件
  • ページング上限
  • 条件なし検索
  • 条件付き検索

面接での言い方:

ユーザー管理APIであれば、単に登録できるかだけではなく、必須項目、重複、ユーザー種別、ロール、外部認証基盤との連携失敗、認可情報更新の失敗まで考えます。

例えば、Keycloakへのユーザー作成は成功したがDB保存に失敗した場合にどう扱うか、DB保存まで成功したがOpenFGAの認可情報登録に失敗した場合に後続へ進めてよいか、といった観点です。単体テストでは依存先をモック化し、各依存先が成功・失敗したときにServiceが期待どおり分岐するかを確認します。

外部システム連携APIでのテスト観点

外部システム連携では、トークン状態と外部APIの失敗を重点的に見ます。

  • アクセストークンが有効
  • アクセストークンが期限切れ
  • 期限切れ直前なのでリフレッシュする
  • リフレッシュ成功
  • リフレッシュ失敗
  • 外部APIが成功
  • 外部APIが4xxを返す
  • 外部APIが5xxを返す
  • タイムアウトする
  • リトライしてよいエラーか
  • リトライしてはいけないエラーか
  • Bearerトークンが正しく付与されるか
  • 不要な条件で外部APIを呼んでいないか

面接での言い方:

外部システム連携APIでは、トークンの状態と外部APIの失敗パターンを重視します。アクセストークンが有効な場合、期限切れの場合、期限切れ直前でリフレッシュが必要な場合、リフレッシュが失敗した場合で、処理がどう分岐するかを確認します。

また、外部APIが4xxを返した場合と5xxやタイムアウトの場合では扱いが違うため、リトライしてよいか、ユーザーへどう返すか、ログに何を残すかも観点にします。

モックを使う理由

モックを使うのは、外部依存を切り離して、対象クラスの判断ロジックを確認するためです。

例えば、外部APIが500を返した場合、実際に外部APIを落とすのではなく、モックで500相当のレスポンスを返し、Serviceが例外を握りつぶしていないか、適切なエラーに変換しているか、後続処理に進んでいないかを確認します。

一方で、モックだけではDB制約、SQL、HTTP通信、認証ヘッダー、ネットワーク設定までは確認できません。そのため、単体テストでロジックを確認し、結合試験で実際のDBや外部API相当の環境との接続を確認する役割分担が必要だと考えています。

10-4. コードレビューで見ること

基本回答

レビューでは、まず要件と実装が一致しているかを確認します。その上で、正常系だけでなく、異常系、境界値、外部連携失敗、重複実行、権限不足時の動作を見ます。

構造面では、責務が適切な層に置かれているか、同じ処理が重複していないか、変更影響が広がりすぎないか、命名から意図を読み取れるか、テストしやすいかを確認します。

金融サービスでは、データ整合性、トランザクション境界、認可漏れ、個人情報のログ出力、監査に必要な記録も重要だと考えています。

レビューコメントでは、好みだけで断定せず、理由、影響、代替案を示します。

観点

  • 要件・仕様
  • 要件を満たしているか
  • 仕様にない副作用がないか
  • エラー時のレスポンスが決まっているか
  • API
  • 入力値のバリデーションが適切か
  • ステータスコードやエラー形式が一貫しているか
  • 後方互換性を壊していないか
  • データ
  • トランザクション境界が適切か
  • 一意制約や外部キーなどDBで守るべき制約があるか
  • 更新順序に不整合リスクがないか
  • 認証認可
  • 認証済みであることだけでなく、その操作権限があるか
  • 管理者と一般ユーザーの操作範囲が分かれているか
  • 認可漏れがないか
  • 外部連携
  • タイムアウト時の扱い
  • リトライ可否
  • 二重実行防止
  • 外部API失敗時に中途半端な状態が残らないか
  • 保守性
  • 責務が肥大化していないか
  • 同じロジックが複数箇所に散っていないか
  • 命名が意図を表しているか
  • テストしやすい構造か
  • ログ・監査
  • 問題発生時に追跡できるログがあるか
  • 個人情報やトークンをログに出していないか
  • 重要操作の監査に必要な情報が残るか

意識していること

レビューは、相手の実装を否定する場ではなく、チームとして品質を上げる場だと考えています。

そのため、指摘するときは「好み」ではなく、「この条件で不整合が起きる」「この責務がServiceに寄りすぎている」「外部API失敗時にDB更新だけ残る可能性がある」のように、理由と影響を具体的に伝えます。可能であれば、代替案や確認観点も一緒に出します。

5. リファクタリングとは

リファクタリングは、外部から見た振る舞いを変えずに、内部構造を改善することです。

目的は、重複の削減、責務の明確化、可読性、テスト容易性、変更容易性の向上です。

安全に進めるため、最初に既存挙動を守るテストを用意し、大きく一度に変えず、小さな変更ごとにテストを実行します。また、古いこと自体を問題にするのではなく、変更頻度、障害リスク、理解コスト、開発速度への影響から優先順位を決めます。

6. TDDとは

TDDは、失敗するテストを書く、テストを通す最小限の実装をする、重複や構造を改善する、という短いサイクルで開発する考え方です。

メリットは、期待する振る舞いを先に明確にできること、変更後も既存挙動を確認しやすいこと、テストしにくい設計へ早く気づけることです。

一方で、すべてを細かな単体テストだけで確認すればよいわけではありません。DB制約、外部API、メッセージング、複数サービス連携は、結合試験やE2E試験も必要です。リスクに応じてテストレベルを組み合わせることが重要だと考えています。

TDDの基本サイクル

TDDは一般的に、Red -> Green -> Refactor のサイクルで進めます。

  1. Red
  • まず、まだ実装していない振る舞いに対するテストを書く
  • 当然テストは失敗する
  • ここで「何を満たせばよいか」を明確にする
  1. Green
  • テストを通すための最小限の実装をする
  • きれいな設計を最初から完成させるより、まず期待する振る舞いを満たす
  1. Refactor
  • テストが通っている状態を保ったまま、重複、命名、責務、構造を改善する
  • 既存テストがあるため、振る舞いを壊していないか確認しながら整理できる

TDDは何のためにやるのか

TDDは、テストコードを書くこと自体が目的ではありません。

目的は、実装前に「期待する振る舞い」を小さく定義し、その振る舞いを守りながら設計を進めることです。

例えばユーザー登録処理なら、いきなり全体を作るのではなく、次のように小さな振る舞いへ分けます。

  • 必須項目がそろっていれば登録できる
  • メールアドレスが空ならエラーにする
  • 同じメールアドレスが既に存在すれば登録しない
  • Keycloak作成が失敗したらDB保存へ進まない
  • DB保存が成功したらOpenFGA登録を行う
  • OpenFGA登録が失敗した場合にどう扱うか

このように振る舞いを先にテストとして置くと、Serviceが何を責務として持つべきか、外部API ClientやRepositoryをどう分けるべきかも見えやすくなります。

TDDと通常のテストの違い

通常のテストは、実装後に「正しく動くか」を確認することが中心です。

TDDは、実装前に「どう動くべきか」をテストとして先に表現し、そのテストを満たすように設計・実装を進める点が違います。

ただし、現場では完全なTDDでなくても、重要な業務ロジックやバグ修正では、先に再現テストを書いてから修正する考え方が有効です。

どの層でTDDしやすいか

TDDは、まずService層やドメインロジックのような、外部依存を切り離しやすい部分で使いやすいです。

  • Service層
  • 業務ルール、分岐、例外時の振る舞いをテストしやすい
  • ドメインロジック
  • 状態遷移、不変条件、計算ロジックをテストしやすい
  • Controller層
  • リクエスト・レスポンスのテストはできるが、TDDの主戦場というよりAPI仕様の確認に近い
  • Repository層
  • DB依存があるため、TDDというよりDB制約やクエリの確認に近い
  • 外部API連携
  • スタブやWireMockを使えばテストできるが、ネットワークや相手先仕様まで単体テストで保証するものではない

自分の経験に合わせた回答

TDDを本格的に実務で回した経験はまだ限定的です。ただ、考え方としては、先に期待する振る舞いを小さなテストとして定義し、そのテストを通す実装を行い、最後にテストを保ったまま構造を改善する進め方だと理解しています。

特に、Service層やドメインロジックのように、外部依存をモック化しやすい部分では有効だと考えています。ユーザー登録であれば、必須項目、重複、Keycloak失敗、DB保存失敗、OpenFGA登録失敗のような振る舞いを先にテストとして置くことで、責務や例外時の扱いを明確にできます。

一方で、DB制約、実際のHTTP通信、複数サービス連携は単体テストだけでは確認できないため、TDDで書いた単体テストと、結合試験・E2E試験を役割分担して使う必要があると考えています。

7. ペアプログラミングをどう考えるか

実務経験が十分ではない場合は、その点を正直に伝えます。

ペアプログラミングは、コードを書きながら設計意図や判断を共有し、早い段階で認識のずれを見つける方法だと理解しています。特に、金融ドメインや既存コードに不慣れな状態では、レビューを後から受けるだけでなく、考え方をその場で共有できるため、立ち上がりと品質の両方に有効だと考えています。

自分も、最初は既存機能の改修や障害調査で経験者とペアになり、実装パターン、テスト観点、ドメイン上の注意点を吸収したいです。

8. DBパフォーマンスチューニング

基本回答

主担当として大規模なDBパフォーマンスチューニングを行った経験はありません。ただ、考え方としては、推測でインデックスを足すのではなく、まずどのAPIが遅いのか、その中でどのSQLが遅いのか、SQLの実行時間、発行回数、実行計画、対象データ量を確認します。

その上で、N+1、不要な全件取得、検索条件やJOIN条件、インデックス不足、ソート、ページング不足、長いトランザクション、ロック競合、コネクションプール枯渇、DBのCPU・メモリ・I/Oを見ます。

DB単体だけではなく、アプリケーション側がどのようにSQLを発行しているか、どの範囲を1トランザクションにしているかまで含めて考える必要があると理解しています。

なぜ遅いSQLが発生するのか

遅いSQLは、DBが必要以上に多くのデータを読んだり、並べ替えたり、待たされたりすると発生します。

主なパターン:

  1. インデックスが使えない
  • 検索条件に合うインデックスがない
  • 関数をかけていてインデックスが効かない
  • LIKE '%xxx' のように前方がワイルドカードになっている
  • 複合インデックスの順序と検索条件が合っていない
  1. 読む行数が多い
  • 条件が粗く、対象行が多い
  • ページングせず全件取得している
  • 必要なカラムだけでなく大きなデータまで取っている
  1. JOINが重い
  • JOIN条件にインデックスがない
  • 大きなテーブル同士を結合している
  • 結合順序や条件が悪く、中間結果が大きい
  1. ソートや集計が重い
  • ORDER BY対象にインデックスがない
  • 大量データを並び替えている
  • GROUP BYや集計が大きい
  1. N+1が起きている
  • 一覧を1回取得した後、各行ごとに詳細取得SQLを発行する
  • 100件の一覧に対して、追加で100回SQLを投げるような状態
  1. ロック待ちが発生している
  • 他のトランザクションが更新中で待たされる
  • トランザクションが長すぎる
  • 更新対象が集中している
  1. コネクションプールが詰まっている
  • DB接続数が不足し、SQLを実行する前に待つ
  • SQL自体は速くても、接続取得で遅く見える

SQL発行回数とDB内部の読み出し回数の違い

面接で混乱しやすいので分けて理解します。

  • SQL発行回数
  • アプリケーションがDBへSQLを何回投げたか
  • N+1問題ではこの回数が増える
  • 例: 一覧取得1回 + 詳細取得100回 = 101回SQLを発行
  • DB内部の読み出し量
  • 1回のSQL実行で、DBが内部的に何行・何ページを読んだか
  • SQL発行回数は1回でも、全表走査で100万行読むと遅くなる
  • 実行計画で確認する

つまり、遅さには次の2種類があります。

  • SQLを何度も発行して遅い
  • 1回のSQLが大量データを読んで遅い

実行計画で見ること

実行計画では、DBがそのSQLをどう処理するかを確認します。

  • インデックスを使っているか
  • フルテーブルスキャンになっていないか
  • JOIN順序が適切か
  • 想定行数と実際の行数が大きくずれていないか
  • ソートが発生しているか
  • 一時テーブルを使っているか

回答案

DBパフォーマンスでは、まず遅い原因を、SQL発行回数が多いのか、1回のSQLが重いのか、ロックや接続待ちなのかに分けて考えます。

例えばN+1では、一覧1回に対して詳細取得を件数分発行するため、SQL発行回数が増えて遅くなります。一方、SQL発行が1回でも、インデックスが効かずに大量行を読む場合は、その1回のSQL自体が遅くなります。

そのため、APIの処理時間、SQLログ、発行回数、実行計画、対象データ量を確認します。対策としては、N+1の解消、必要な条件へのインデックス追加、取得件数の制限、ページング、不要なJOINやカラム取得の削減、トランザクション範囲の見直しを検討します。金融サービスでは、性能だけでなく整合性も重要なので、速くするために必要なロックや整合性を壊さないよう注意します。

9. 画面共有によるAPIパフォーマンス課題の進め方

最初に確認すること

  1. 何が遅いのか
  • 平均、p95、p99のどれか
  • いつからか
  • 常に遅いか、特定条件だけか
  1. 影響範囲
  • 全ユーザーか
  • 特定API、データ、時間帯、環境だけか
  1. 直前の変更
  • アプリ、DB、インフラ、外部API、データ量
  1. 再現条件
  • 入力、件数、同時実行数

調査の順序

  1. API全体の処理時間を、アプリ、DB、外部APIに分ける
  2. ログ、メトリクス、トレースで時間を使っている箇所を特定する
  3. アプリのCPU、メモリ、GC、スレッド、コネクションプールを確認する
  4. DBのスロークエリ、実行計画、ロック、接続数を確認する
  5. 外部APIの応答時間、タイムアウト、再試行を確認する
  6. 修正前後を同じ条件の負荷試験で比較する

口頭回答例

まず、推測でインデックス追加などを行わず、遅延の発生条件と、どの区間で時間を使っているかを特定します。

APIの処理を、アプリケーション処理、DBアクセス、外部API呼出しに分け、ログ、APM、メトリクス、分散トレースがあれば確認します。

DBが原因であれば、実際に遅いSQLと実行計画を確認し、N+1、フルスキャン、インデックス、取得件数、ロック競合、長いトランザクションを見ます。アプリ側であれば、重い同期処理、不要な繰り返し、スレッドやコネクションプール枯渇、GCを確認します。外部APIであれば、応答時間、タイムアウト、再試行、サーキットブレーカー、キャッシュや非同期化の可否を検討します。

対策後は、同じデータ量と同時実行数で負荷試験を行い、平均だけでなくp95、p99、エラー率、DB負荷を比較します。金融APIでは速さだけでなく、タイムアウトや再試行によって二重処理が起きないことも確認します。

10. DDDとは

DDDは、業務で重要な概念やルールを、設計とコードの中心に置く考え方だと理解しています。単にクラスをきれいに分ける手法ではなく、業務の言葉、状態、ルール、責務を整理し、それをシステムの境界やデータの持ち方へ反映する考え方です。

銀行であれば、口座、残高、入出金、取引状態、利用者、権限などの意味と不変条件を明確にする必要があります。例えば、同じ「ユーザー」という言葉でも、認証上のユーザー、銀行口座の名義人、業務上の権限を持つ利用者は責務が違う可能性があります。ここを曖昧にすると、どのデータをどこで更新するかが崩れ、不整合や変更漏れにつながります。

自分はDDDを本格導入した経験はありません。ただ、Keycloak、OpenFGA、独自ユーザー管理機能の責務を分け、認証情報、認可ルール、業務属性を別の変更理由として整理した経験があります。この考え方は、ドメインの概念や責務境界を整理するDDDの考え方と近いと感じています。

面接での注意点

  • 「DDDを実務で本格導入しました」とは言わない
  • エンティティ、値オブジェクト、集約などの用語を並べすぎない
  • 業務の言葉、責務境界、不変条件、変更理由に接続して話す
  • 自分の経験は、認証・認可・業務ユーザー情報の責務整理として話す

11. クリーンアーキテクチャとは

クリーンアーキテクチャは、重要な業務ルールを、DB、Webフレームワーク、外部APIなどの技術詳細から分離し、依存関係を内側の業務ルールへ向ける考え方だと理解しています。

例えば、ユーザー登録というユースケースがある場合、ControllerはHTTPリクエストを受け取り、UseCaseやServiceが業務上の流れを進め、Repositoryや外部API ClientがDBやKeycloakなどの技術詳細を扱う、というように責務を分けます。

メリットは、DBや外部API、フレームワークの都合が業務ロジック全体へ漏れにくいことと、業務ロジックを単体でテストしやすいことです。

主担当として導入した経験はありませんが、外部認証基盤、認可基盤、業務ユーザー管理を分け、外部APIのトークン管理を共通処理へ閉じ込めた経験は、変更理由の異なる責務を分離するという点でつながっています。

どう活かすか

御社のような銀行バックエンドでは、DBや外部APIの詳細よりも、口座、残高、取引、本人確認、権限のような業務ルールを正しく守ることが重要です。そのため、業務ルールと技術詳細を分け、外部サービス変更やDB変更が業務ロジックへ広がりすぎない構造にすることが重要だと考えています。

10-12. KotlinとGoの特徴

Kotlin

  • JVM上で動作し、Java資産やSpringとの親和性が高い
  • null安全、data class、拡張関数、コルーチンなど、簡潔かつ安全に書きやすい
  • Javaから段階的に導入しやすい
  • 高機能な分、書き方を統一しないと複雑になり得る

Go

  • 言語仕様が比較的小さく、シンプル
  • コンパイルが速く、単一バイナリとして配布しやすい
  • goroutineとchannelにより並行処理を扱いやすい
  • マイクロサービスやネットワークサービスと相性がよい
  • 例外ではなくエラー値を明示的に扱うため、エラー処理の設計が重要

回答例

実務の中心はJava・Spring Bootで、KotlinとGoはこれから実務経験を深める領域です。

KotlinはJVMとJava資産を活かしながら、null安全や簡潔な記述で安全性と生産性を高めやすい言語だと理解しています。Javaと近い実行基盤で段階導入しやすく、Springとの相性も良い一方、高機能な分、チームで書き方を揃えることが重要だと思います。

Goは言語仕様がシンプルで、ビルドや配布がしやすく、軽量な並行処理を書きやすい言語だと理解しています。マイクロサービスやAPIサーバーに向いている一方、例外ではなくエラー値を明示的に返すため、エラー処理を丁寧に書く必要があります。

入社後は、言語の一般論だけでなく、御社がサービスごとに何を理由に選択しているかを理解し、既存コード、テスト、軽微な改修、レビューを通じてチームの実装スタイルを吸収します。

13. バッチ処理・Spring Batch

基本回答

Spring Batchを主担当として開発した経験はありません。

ただ、バッチ処理は、オンラインAPIのように1リクエストを即時処理するものではなく、一定量のデータをまとめて読み込み、加工し、DBや外部システムへ反映する処理だと理解しています。日次処理、月次処理、集計、ファイル連携、ステータス更新、リトライ処理などで使われます。

Spring Batchについては、Reader、Processor、Writerのように、読み込み、加工、書き込みを分け、Chunk単位で処理・コミットする考え方があると理解しています。

金融では、同じ処理を再実行しても二重入金や二重更新にならない冪等性、どこまで処理済みかを追跡できる監査性、失敗時に途中から再開できる設計が特に重要だと考えています。

バッチで考える観点

  • 処理対象
  • 何件処理するのか
  • 対象をどう抽出するのか
  • 処理済みと未処理をどう判定するのか
  • Chunkサイズ
  • 何件ごとにコミットするのか
  • 大きすぎると失敗時の戻しが重く、小さすぎると性能が悪くなる
  • 再実行
  • 途中で落ちた場合、最初からやり直すのか、途中から再開するのか
  • 再実行しても二重更新にならないか
  • エラーハンドリング
  • 1件エラーで全体停止するのか
  • エラー行だけスキップするのか
  • 後で再処理できるように記録するのか
  • トランザクション
  • どこまでを1つの更新単位にするか
  • 外部API呼び出しとDB更新をどう整合させるか
  • ログ・監査
  • 処理開始、終了、件数、エラー内容を記録する
  • 後から追跡できるようにする

面接での言い方

Spring Batchの実務主担当経験はありませんが、バッチ処理では、大量データをまとめて処理するため、性能だけでなく、途中失敗時の再実行、二重処理防止、トランザクション境界、処理件数やエラー内容の記録が重要だと理解しています。

特に金融では、再実行したときに二重入金や二重更新が起きないこと、どこまで処理済みかを追跡できることが重要です。Spring Batchであれば、Reader、Processor、Writer、Chunk、JobRepositoryのような考え方を使い、処理単位と再開性を設計する必要があると考えています。

14. モダン開発手法で意識していること

基本回答

モダン開発手法という言葉は広いですが、自分は、短いサイクルで変更し、その変更をテスト、レビュー、CI/CD、ログ、監視で安全に確認できる状態を作ることが重要だと考えています。

厳密なスクラムやTDDを本格運用した経験は限定的ですが、複数チームで仕様を確認し、結合試験で問題を見つけ、修正、再デプロイ、再確認を繰り返す進め方は経験しています。また、AWS環境とCI/CDを構築し、PRマージを契機にビルド・デプロイできる仕組みも担当しました。

意識していること

  • 小さく変更する
  • 大きな変更を一度に入れず、影響範囲を小さくする
  • 自動化できる確認は自動化する
  • ビルド、テスト、デプロイ、静的解析など
  • レビューしやすい単位にする
  • PRを大きくしすぎない
  • 目的と変更点を説明する
  • 早めに不確実性を潰す
  • 技術的に不明なところは小さく検証する
  • 外部連携は早めに接続確認する
  • リリース後に追えるようにする
  • ログ、メトリクス、エラー内容を残す
  • 経験していない手法は盛らない
  • TDD、ペアプロ、モブプロは理解とキャッチアップ姿勢として話す

回答案

自分が意識しているのは、速く作ることだけではなく、変更を小さくし、レビューやテストで確認できる形にすることです。

直近では、AWS環境とCI/CDを構築し、PRマージ後にビルド・デプロイできる仕組みを作りました。また、複数社の結合試験では、問題を再現し、ログで切り分け、修正、再デプロイ、再確認を短く回しました。

御社でTDDやペアプログラミングを本格的に行う場合は、経験を盛らず、まず既存の進め方、テスト観点、レビュー基準を吸収し、チームの品質を上げる形で適応したいです。

15. 気になる技術

基本回答

最近特に関心があるのは、AIエージェントによる開発プロセスの効率化と、ドメイン駆動設計、いわゆるDDDです。

AIエージェントについては、単にコードを自動生成するものというより、仕様理解、既存コード調査、影響範囲の洗い出し、テスト観点の整理、実装案の比較、PRレビュー補助まで含めて、開発プロセス全体の初速と品質を上げるものとして関心があります。

一方で、AIに任せきるのではなく、要件、ドメインルール、セキュリティ、既存設計との整合性は人が確認する必要があると考えています。特に金融サービスでは、顧客情報やソースコードの取り扱い、生成されたコードの妥当性、テストの網羅性、監査性を慎重に扱う必要があります。

DDDについては、銀行のように口座、残高、取引、入出金、借入、本人確認、権限などの概念が重要な領域で、業務上の言葉や不変条件をコードの責務へどう反映するかに関心があります。自分がこれまで担当した認証基盤、認可基盤、業務ユーザー管理の責務分担ともつながる考え方だと感じています。

AIエージェントで調べていること

AIエージェントについては、次の観点で調べています。

  • 既存コード調査
  • ある機能を変更するときに、Controller、Service、Repository、設定、テストのどこへ影響するかを洗い出す
  • 関連するクラス、API、DBテーブル、設定値、テストを横断的に確認する
  • 仕様理解・設計整理
  • 要件を小さなユースケースに分解する
  • 正常系、異常系、境界値、権限、外部連携失敗を洗い出す
  • 複数案のメリット・デメリットを整理する
  • 実装補助
  • 既存コードの書き方に合わせた実装案を出す
  • ただし、生成コードはそのまま採用せず、責務、例外処理、トランザクション、認可、ログを確認する
  • テスト補助
  • Service層、Controller層、Repository層ごとにテスト観点を出す
  • JUnit、Mockito、MockMvc、Spring Boot Testでどの粒度を確認するか整理する
  • レビュー補助
  • 変更差分から、認可漏れ、null、例外処理、ログ、個人情報出力、既存仕様への影響を確認する
  • ドキュメント化
  • 変更内容、設計判断、未決事項、リスクをPR説明や設計メモに整理する

面接での言い方:

AIエージェントは、単にコードを書かせる用途よりも、既存コード調査、影響範囲の洗い出し、テスト観点整理、レビュー観点整理のような、開発プロセスの前後工程で価値が大きいと感じています。

例えば、あるAPIを変更する場合に、Controller、Service、Repository、設定、テスト、関連するDB項目を横断的に調査し、変更影響と確認観点を整理する用途です。

ただし、金融サービスでは、AIの出力を根拠にするのではなく、最終的には公式ドキュメント、既存コード、テスト、レビューで妥当性を確認する必要があります。また、顧客情報や機密情報を入力しないなど、会社のセキュリティルールを守ることが前提だと考えています。

DDDで調べていること

DDDについては、用語を暗記するというより、業務概念とコードの責務を揃えるための考え方として調べています。

特に見ている観点は次のとおりです。

  • ユビキタス言語
  • 業務側と開発側で同じ言葉を使う
  • 例えば「ユーザー」「口座」「残高」「取引」「借入」「権限」が何を意味するかを曖昧にしない
  • 境界づけられたコンテキスト
  • 同じ言葉でも文脈によって意味が変わるため、責務の境界を分ける
  • 例えば認証上のユーザー、銀行口座の名義人、業務上の権限を持つ利用者は同じ「ユーザー」でも責務が異なる可能性がある
  • 集約・不変条件
  • どの単位で整合性を守るかを考える
  • 口座残高、取引状態、借入状態のように、更新順序や不正な状態遷移を防ぐ必要があるものを整理する
  • ドメインサービスとアプリケーションサービス
  • 業務ルールそのものと、処理の流れを進める責務を分ける
  • 技術詳細との分離
  • DB、外部API、フレームワーク都合を業務ルールへ漏らしすぎない
