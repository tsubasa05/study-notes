Amazon系 / CS基礎 S
OQ001

配列・ListとHashMapをどう使い分けますか？

面接官の意図：データ構造を、名前の暗記ではなくアクセスパターンから選べるかを見る。

回答例・補足を見る

30秒回答

配列・List・HashMapは「どのアクセスを速くしたいか」で選びます。Listは順序を持つ集合を前から処理したり、インデックスで取り出したい場合に向きます。HashMapはキーから値を高速に引く場合に向きます。LinkedHashMapや「双方向リスト＋HashMap」は、順序とキー検索を両方扱えますが、メモリ消費や実装複雑性が増えるので、Listを常に上回る万能構造ではありません。

深掘りされたときに足す観点

  • ArrayList：内部的には配列で要素を保持する。挿入順を維持し、インデックスアクセスは速いが、途中挿入・途中削除は後続要素のずらしが発生しやすい。
  • LinkedList：各要素が前後のノード参照を持つ。ノード位置が分かっていれば挿入・削除はしやすいが、インデックス指定でたどる場合は先頭/末尾から順に探索するため遅い。
  • HashMap：キーから値を引く構造。順序や重複要素の扱いより、ID・コード・キーによる検索を高速化する用途に向く。
  • LinkedHashMap：HashMapに順序付きのリンク構造を足したもの。キー検索と挿入順/アクセス順の反復を両立できる。
  • 双方向リスト＋HashMap：LRUキャッシュのように「キーで即座に探し、順序も更新し、末尾を即座に消す」用途で有効。
理解補足：JavaのListはインターフェースです。代表実装のArrayListは内部配列で順序を保持し、list.get(3)のようなインデックスアクセスが得意です。一方、HashMapはmap.get(userId)のようにキーで引く構造です。あなたの言う「リンクとHashMapを使えば順列＋キー探索ができる」は正しく、これはLinkedHashMapやLRUキャッシュの発想に近いです。ただし、キーが一意である前提が必要で、要素重複やインデックスアクセス、単純な順次処理ではListの方が自然です。
用途で見る選び方 入力順に全件処理したい → List / ArrayList 3番目・10番目を取りたい → ArrayList userId から User を引きたい → HashMap<UserId, User> キー検索 + 挿入順に走査したい → LinkedHashMap キー検索 + 最近使った順に並べたい → LinkedHashMap(accessOrder=true) / 双方向リスト + HashMap
注意点：HashMap系は平均O(1)で便利ですが、メモリを追加で使います。また、キー設計、equals/hashCode、順序要件、重複要件を誤るとバグになります。面接では「HashMapの方が速い」ではなく、「アクセスパターンに応じて選ぶ」と答えるのが正確です。

OQ002

計算量 O(n) と O(log n) の違いを説明してください。

面接官の意図：処理時間が入力サイズに対してどう増えるかを説明できるかを見る。

回答例・補足を見る

30秒回答

O(n)は入力件数に比例して処理量が増える計算量です。O(log n)は、探索範囲を半分ずつ削るような処理で、入力件数が増えても処理量の増え方が緩やかです。線形探索はO(n)、ソート済み配列に対する二分探索はO(log n)です。

深掘りされたときに足す観点

  • 1,000件ならO(n)でも問題にならないことは多いが、1,000万件や高頻度APIでは差が大きくなる。
  • O(log n)は典型的には二分探索、B-treeインデックス、平衡木などで出てくる。
  • 実務では、計算量だけでなくI/O、DBアクセス、ネットワーク、ロック待ちも支配的になる。
理解補足：コーディング面接では、まず素朴解を出してから「この実装だとO(n^2)なので、HashSetを使ってO(n)にできます」のように改善できると評価されやすいです。Webバックエンドでは、アプリ内の計算量よりDBクエリや外部API呼び出しの方が重いことも多いので、どこがボトルネックかを見る視点も重要です。
注意点：Big-Oは上限の増え方を表すもので、実測性能そのものではありません。小さい入力ではO(n)の単純実装の方が速いこともあります。
Amazon系 / Coding A
OQ003

2つの配列に共通する要素を求めるならどう実装しますか？

面接官の意図：素朴解から改善解へ進めるか、仕様確認を入れられるかを見る。

回答例・補足を見る

30秒回答

まず、共通要素を一意にするか、重複数を残すか、結果順をどうするか確認します。素朴に二重ループで比較するとO(n*m)です。改善案として、片方の配列をHashSetに入れる処理にO(n)、もう片方の配列を走査してSetに含まれるか確認する処理にO(m)かかるため、合計で平均O(n+m)になります。

深掘りされたときに足す観点

  • 「片方がすでにHashSetとして存在している」前提なら、もう片方を走査するだけなのでO(m)と説明してよい。
  • ただし、配列からHashSetを作るところも含めて実装するなら、Set構築分のO(n)が必要になる。
  • 重複を考慮するならMap<値, 出現回数>を使い、片方の出現回数を減らしながら結果に追加する。
  • メモリを追加で使いたくない場合、両方をソートして2ポインタで走査する案もある。ただしソートにO(n log n + m log m)かかる。
理解補足：O(n+m)の理由は「Setを作るコスト」と「検索するコスト」を足しているからです。例えば配列Aがn件、配列Bがm件なら、Aを全部HashSetへ入れるのにn回、Bを全部確認するのにm回処理します。HashSetのcontainsは平均O(1)なので、B側の確認はO(m)です。合計でO(n+m)です。
例：AとBの共通要素を求める 1. AをHashSetに入れる → n件処理 = O(n) 2. Bを1件ずつ見る → m件処理 = O(m) 3. set.contains(b) は平均O(1) 合計：O(n) + O(m) = O(n+m) ※ すでにAがHashSetなら、1は不要なのでO(m)
注意点：HashSetのO(1)は平均計算量です。また、結果の順序や重複をどう扱うかで実装が変わります。面接では、計算量だけでなく「仕様確認」が重要です。

OQ004

オブジェクト指向設計で、継承よりコンポジションを選ぶ理由は？

面接官の意図：変更に強い設計、責務分離、テスト容易性を理解しているかを見る。

回答例・補足を見る

30秒回答

コンポジションは、あるオブジェクトが別のオブジェクトを部品として持ち、その処理を委譲する設計です。Javaで言うと、クラスのフィールドに別クラスやインターフェース型の依存を持たせる形です。継承はis-a関係、コンポジションはhas-a関係です。継承は親クラスの変更が子クラスへ波及しやすいため、振る舞いを差し替えたい場合は、インターフェース＋実装クラスを注入するコンポジションの方が変更に強くなります。

深掘りされたときに足す観点

  • 継承：class Dog extends Animal のように「DogはAnimalである」という分類関係を表す。
  • コンポジション：OrderService が PaymentClient をフィールドとして持つように、「AはBを使う」という関係を表す。
  • インターフェース：コンポジションそのものではないが、差し替え可能な部品の契約としてよく使う。
  • Springでは、private final NotificationSender sender; のようにインターフェース型をコンストラクタ注入する形が典型。
Javaでのイメージ interface NotificationSender { void send(String message); } class EmailNotificationSender implements NotificationSender { public void send(String message) { /* メール送信 */ } } class OrderService { private final NotificationSender sender; // コンポジション：部品として持つ OrderService(NotificationSender sender) { this.sender = sender; } void completeOrder() { // 注文確定処理 sender.send("注文が完了しました"); // 振る舞いを委譲 } }
理解補足：「コンポジション＝インターフェース」ではありません。コンポジションは、あるクラスが別のオブジェクトを保持して使う構造です。インターフェースは、その保持する部品を差し替えやすくするためによく使う道具です。Java/Springでは、ServiceがRepositoryや外部APIクライアントをフィールドとして持つ設計もコンポジションです。
注意点：継承が常に悪いわけではありません。ただし「コード再利用したいから継承する」は危険です。親クラスの実装詳細に子クラスが依存しやすく、変更に弱くなるためです。

OQ005

トランザクションのACIDを説明してください。

面接官の意図：DBの基本的な信頼性モデルを理解しているかを見る。

回答例・補足を見る

30秒回答

ACIDは、Atomicity、Consistency、Isolation、Durabilityの頭文字です。Atomicityは全部成功か全部失敗、Consistencyは制約を満たす一貫した状態を保つこと、Isolationは並行実行されるトランザクション同士の干渉を制御すること、Durabilityはコミット後の結果が永続化されることです。

深掘りされたときに足す観点

  • 注文作成と在庫減算は、片方だけ成功すると不整合になるためAtomicityが重要。
  • 外部キー、一意制約、チェック制約などを満たす状態を維持するのがConsistency。
  • Isolationが弱いと、ダーティリード、ノンリピータブルリード、ファントムリードなどが問題になる。
  • Durabilityは、障害後もコミット済みデータが失われないこと。
理解補足：Web系面接では、ACIDを言えるだけでは不十分です。「どこを1トランザクションにするか」が重要です。例えば、注文作成、在庫確保、決済API呼び出しを全部DBトランザクションに入れると、外部API待ちでロック時間が伸びます。外部APIを含む業務では、DBトランザクションと業務上の補償処理を分けて考える必要があります。
注意点：外部API呼び出しはDBトランザクションではロールバックできません。決済や外部連携では、ステータス管理、リトライ、補償処理、冪等性が必要になります。

OQ006

SQLとNoSQLをどう使い分けますか？

面接官の意図：データモデル、整合性、アクセスパターンからストレージを選べるかを見る。

回答例・補足を見る

30秒回答

SQL/RDBは、テーブル構造、制約、JOIN、トランザクション、一貫性が重要なデータに向きます。NoSQLは、スキーマ変更の柔軟性、大量データ、高スループット、特定アクセスパターンに最適化したい場合に向きます。注文、請求、権限などの中核データはRDBを優先し、ログ、イベント、キャッシュ、セッション、検索用データなどはNoSQLや専用ストアを検討します。

深掘りされたときに足す観点

  • スキーマ：データの構造定義のこと。RDBではテーブル、カラム、型、制約、リレーションなどを指す。
  • RDBは、外部キー、一意制約、トランザクション、JOINに強い。
  • NoSQLは、Key-Value、Document、Wide-column、Graphなど種類によって適性が違う。
  • NoSQLは「スキーマが完全にない」ではなく、アプリケーション側が構造と整合性を管理する比率が上がることが多い。
  • アクセスパターンが明確で、JOINよりキー検索・ドキュメント単位取得が中心ならNoSQLが合う場合がある。
理解補足：スキーマは、雑に言えば「このデータはどういう形で保存されるか」という設計図です。RDBなら、usersテーブルにid、email、created_atがあり、emailは一意で、orders.user_idはusers.idを参照する、というような定義がスキーマです。NoSQLでも暗黙のデータ構造は存在するため、スキーマレスという言葉を「何も設計しなくてよい」と解釈すると危険です。
RDBスキーマの例 users - id: bigint, primary key - email: varchar, unique - name: varchar orders - id: bigint, primary key - user_id: bigint, foreign key -> users.id - status: varchar - created_at: timestamp
注意点：「NoSQLは速いから使う」は浅い回答です。RDBでも適切なインデックス設計で十分速いケースは多いです。まず整合性要求、クエリの複雑さ、データ量、更新頻度、スケール要件から選ぶべきです。

OQ007

分散システムで障害を前提に設計するとはどういうことですか？

面接官の意図：ネットワーク・依存サービス・部分障害を前提に考えられるかを見る。

回答例・補足を見る

30秒回答

分散システムでは、単一プロセス内の例外だけでなく、タイムアウト、部分障害、依存APIの遅延、リトライによる重複、メッセージ欠落、ネットワーク分断を前提にします。失敗しない設計ではなく、失敗しても影響を限定し、検知し、復旧できる設計を目指します。

深掘りされたときに足す観点

  • タイムアウトを設定しないと、依存先の遅延が呼び出し元へ波及する。
  • リトライには冪等性が必要。重複実行されても壊れない設計にする。
  • 依存先障害に対して、サーキットブレーカー、キュー、フォールバック、縮退運転を検討する。
  • 障害検知にはログ、メトリクス、トレース、アラートが必要。
理解補足：あなたのマイクロサービス経験に接続するなら、認証認可基盤や外部APIが落ちた場合にAPIをfail-closeにするか、キャッシュした権限で一時的に通すか、といった設計判断になります。セキュリティ系は基本fail-closeですが、可用性とのトレードオフがあります。
注意点：「リトライすればよい」は浅いです。リトライは依存先をさらに圧迫することがあります。バックオフ、最大回数、タイムアウト、冪等性、DLQまでセットで考える必要があります。

OQ008

LRUキャッシュを設計するとしたら、どんなデータ構造を使いますか？

面接官の意図：複数のデータ構造を組み合わせて要件を満たせるかを見る。

回答例・補足を見る

30秒回答

LRUキャッシュは、Least Recently Used、つまり「最近使われていないデータから追い出す」キャッシュです。容量がいっぱいになったとき、最も長い間アクセスされていない要素を削除します。O(1)でget/put/削除を実現するには、キー検索用のHashMapと、利用順を管理する双方向リストを組み合わせます。JavaならLinkedHashMapのaccessOrderを使うと近い実装ができます。

深掘りされたときに足す観点

  • HashMap：key → リストノードを保持し、対象要素をO(1)で見つける。
  • 双方向リスト：先頭を最近使った要素、末尾を最近使っていない要素として管理する。
  • get(key)したら、そのノードを先頭へ移動する。
  • put(key, value)で容量超過したら、末尾ノードを削除し、HashMapからも消す。
LRUのイメージ 最近使った ←----------------------→ 最近使っていない HEAD [C] <-> [A] <-> [B] <-> [D] TAIL get(B) すると B を先頭へ移動 HEAD [B] <-> [C] <-> [A] <-> [D] TAIL 容量超過時は TAIL の D を削除
理解補足：単なるListだけだと、keyから対象要素を探すのにO(n)かかります。単なるHashMapだけだと、どれが「最近使われていないか」を管理しづらいです。だから、HashMapで場所を即座に見つけ、双方向リストで利用順を管理します。あなたがOQ001で言っていた「リンク＋HashMap」は、このLRUキャッシュで典型的に使われる組み合わせです。
注意点：実務の分散キャッシュでは、単一JVM内のLRUだけでは不十分なことがあります。Redis、TTL、キャッシュ無効化、キャッシュスタンピード、複数サーバー間の整合性も論点になります。

OQ009

レースコンディションとは何ですか？どう防ぎますか？

面接官の意図：並行処理における共有状態の危険性を理解しているかを見る。

回答例・補足を見る

30秒回答

レースコンディションは、複数の処理が同じ共有データを同時に読み書きし、実行順序によって結果が変わる問題です。防ぐには、共有状態を減らす、ロックやsynchronizedを使う、Atomic型を使う、DB側でロックや一意制約を使う、処理を冪等にする、などの方法があります。

深掘りされたときに足す観点

  • 在庫数の更新、残高更新、同一ユーザーの二重申込などで起こりやすい。
  • アプリケーション内ロックだけでは、複数インスタンス構成では効かないことがある。
  • DBの一意制約や楽観ロックを使う方が堅い場合がある。
理解補足：Webアプリでは、同じAPIが同時に複数回呼ばれる前提で考える必要があります。たとえば「購入ボタンを連打」「リトライで同じリクエストが再送」「複数サーバーで同時処理」などです。アプリコードだけでなくDB制約・トランザクション・冪等キーで守る設計が現実的です。
注意点：ロックを増やせば安全になるわけではありません。ロック範囲が広いと性能低下やデッドロックが発生します。

OQ010

HTTPリクエストがブラウザからサーバーに届くまでの流れを説明してください。

面接官の意図：Webの基本的な通信経路を理解しているかを見る。

回答例・補足を見る

30秒回答

ブラウザはURLを解釈し、DNSでドメイン名をIPアドレスに解決し、TCP接続を確立します。HTTPSならTLSハンドシェイクを行い、HTTPリクエストを送信します。途中にCDN、ロードバランサ、リバースプロキシ、API Gatewayなどが入ることがあります。サーバーはリクエストを処理し、ステータスコード、ヘッダ、ボディを含むレスポンスを返します。

深掘りされたときに足す観点

  • DNS、TCP、TLS、HTTPの層を分けて説明できるとよい。
  • Webバックエンドでは、ALBやnginx、API Gateway、アプリサーバーのどこで何をしているかが重要。
  • 認証Cookie、Authorizationヘッダ、CORS、CSRFなどはHTTPレイヤと密接に関係する。
理解補足：あなたの経験に寄せるなら、ブラウザ→BFF→API Gateway→各マイクロサービス、またはALB→ECSタスクのように具体構成で説明できると強いです。単に「サーバーに飛ぶ」ではなく、名前解決、TLS、ロードバランシング、アプリ処理、レスポンスまで段階で話します。
注意点：Web面接では、HTTPステータスコードやメソッドだけでなく、ネットワーク・TLS・プロキシ・ロードバランサまで軽く聞かれることがあります。

OQ011

コーディング面接で、実装前に何を確認しますか？

面接官の意図：問題解決を協調的に進められるか、前提確認ができるかを見る。

回答例・補足を見る

30秒回答

入力の型、入力サイズ、nullや空配列、重複、ソート済みかどうか、返却形式、エラー時の扱いを確認します。その後、小さい例を作り、まず素朴解を説明し、必要なら計算量を改善します。実装中も、今何をしているか、次に何を検証するかを言葉にします。

深掘りされたときに足す観点

面接官の意図：基本アルゴリズムの境界条件を正確に扱えるかを見る。

回答例・補足を見る

30秒回答

二分探索では、配列がソート済みであること、探索範囲の境界を一貫して扱うこと、無限ループにならないことに注意します。典型的には、両端を含む[left, right]方式ならwhile (left <= right)、右端を含まない[left, right)方式ならwhile (left < right)を使います。どちらかに統一することが重要です。

深掘りされたときに足す観点

  • 両端含む方式：存在確認に向く。初期値はleft=0, right=n-1。
  • 半開区間方式：lower_bound、つまり「target以上の最初の位置」を探すときに扱いやすい。初期値はleft=0, right=n。
  • mid = left + (right - left) / 2と書くと、巨大な値でのオーバーフローを避けやすい。
  • 更新時にleft = midやright = midを誤ると、範囲が縮まず無限ループになる。
両端含む [left, right]：targetが存在するか探す left = 0, right = n - 1 while left <= right: mid = left + (right - left) / 2 if a[mid] == target: return mid if a[mid] < target: left = mid + 1 else: right = mid - 1 return -1 半開区間 [left, right)：target以上の最初の位置を探す left = 0, right = n while left < right: mid = left + (right - left) / 2 if a[mid] < target: left = mid + 1 else: right = mid return left
理解補足：二分探索で混乱しやすいのは、「rightが有効な添字なのか」「rightは範囲外の境界なのか」が曖昧になるからです。[left, right]はrightも探索対象です。[left, right)はright自体は探索対象ではなく、範囲の外側の境界です。慣れるまでは、存在確認は両端含む方式、挿入位置探索は半開区間方式、と使い分けると整理しやすいです。
注意点：配列が未ソートなら二分探索は使えません。ソートしてから二分探索する場合、ソートコストO(n log n)も含めて考える必要があります。

OQ028

レートリミッターを設計してください。

面接官の意図：API保護、アルゴリズム、分散環境での制御を理解しているかを見る。

回答例・補足を見る

30秒回答

レートリミッターは、ユーザー、IP、APIキーなどを単位に、一定時間あたりのリクエスト数を制限する仕組みです。代表方式にはFixed Window、Sliding Window、Token Bucket、Leaky Bucketがあります。単一アプリのメモリではなく、複数インスタンスで共有できるRedisやAPI Gatewayで実装することが多いです。

方式ごとの違い

  • Fixed Window：「1分あたり100回」のように固定時間枠でカウントする。簡単だが、分境界の前後で200回近く通ることがある。
  • Sliding Window Log：各リクエスト時刻を記録し、直近60秒の件数を見る。正確だがメモリを使う。
  • Sliding Window Counter：現在ウィンドウと前ウィンドウを重みづけして概算する。Fixed Windowより滑らかで、Logより軽い。
  • Token Bucket：一定速度でトークンが補充され、リクエストごとに1トークン消費する。バケット容量分の短期バーストを許容できる。
  • Leaky Bucket：リクエストをキューに入れ、一定速度で処理する。出力を平滑化しやすいが、キューが溢れると拒否する。
Token Bucket のイメージ bucket capacity = 10 refill rate = 1 token/sec リクエスト到着 ├─ tokenあり → tokenを1つ消費して許可 └─ tokenなし → 429 Too Many Requests 特徴：通常は1秒1回程度に制限しつつ、最大10回までの短期バーストは許容できる
理解補足：Fixed Windowは「13:00:00〜13:00:59に100回」のように数えるため、12:59:59に100回、13:00:00に100回のような境界バーストが起きます。Token Bucketは「普段は一定速度、ただし貯まっている分だけ一時的に多めに通す」方式なので、API利用制限でよく使われます。
注意点：アプリインスタンスごとのメモリで制限すると、10台にスケールした瞬間に実質10倍リクエストを許可してしまいます。分散環境では共有ストアかGatewayレイヤで制御する必要があります。

回答例・補足を見る

30秒回答

検索機能では、まず検索の種類を確認します。IDやメールアドレスの完全一致なのか、前方一致なのか、部分一致なのか、全文検索なのか、絞り込み・ソート・ランキングが必要なのかで設計が変わります。小規模ならRDBのインデックスやLIKEで足りますが、大量データ・日本語全文検索・ランキングが必要ならElasticsearch/OpenSearchのような検索エンジンを検討します。

検索パターンごとの考え方

  • 完全一致：where email = ?。B-treeインデックスで高速化しやすい。
  • 前方一致：where name like 'abc%'。条件によってはインデックスが効きやすい。
  • 部分一致：where name like '%abc%'。通常のB-treeインデックスが効きにくい。
  • 全文検索：文章中の単語検索、形態素解析、スコアリングが必要。検索エンジンが向く。
  • ファセット検索：カテゴリ、価格帯、ステータスなどで絞り込む検索。
  • オートコンプリート：入力途中で候補を出す。前方一致、N-gram、専用インデックスを検討する。
検索設計の選択肢 小規模・完全一致中心 → RDB + Index 管理画面の簡単な部分一致 → RDB LIKE でも可。ただしデータ量に注意 大量データ・全文検索・ランキング → OpenSearch / Elasticsearch DB更新と検索インデックス DB更新 → Event/CDC → Indexer → Search Index更新
理解補足：検索は「WHERE句を書けば終わり」ではありません。日本語検索では単語の分かち書き、表記揺れ、スコアリング、ソート、権限制御、削除反映が問題になります。特に権限付き検索では、検索結果に出してよいデータだけを返す必要があります。認可を後段でフィルタすると、ページング件数が崩れることもあります。
注意点：検索エンジンを使うと、RDBと検索インデックスの同期遅延が発生します。削除済みデータや権限変更済みデータが検索結果に残らないよう、更新イベント、再インデックス、認可フィルタを設計する必要があります。

回答例・補足を見る

30秒回答

OFFSET方式は、limitとoffsetで「何件飛ばして何件取るか」を指定するページングです。実装は簡単ですが、offsetが大きいとDBが多くの行を読み飛ばすため遅くなります。Cursor方式は、最後に取得したレコードのIDや時刻を次回リクエストに渡し、「その続き」を取得する方式です。大量データや無限スクロールではCursor方式が向きます。

深掘りされたときに足す観点

  • OFFSET：page=3&limit=20なら、40件飛ばして20件取る。ページ番号UIに向く。
  • Cursor：前回最後のcreated_atやidを基準に次の20件を取る。無限スクロールや時系列一覧に向く。
  • OFFSETは、途中でデータが追加・削除されると重複や抜けが起きやすい。
  • Cursorには安定したソートキーが必要。created_at + idのように一意性を持たせる。
  • Cursorは内部キーをそのまま出さず、Base64等でopaque tokenにすることが多い。
OFFSET方式 GET /notifications?limit=20&offset=40 SQL: order by created_at desc limit 20 offset 40 意味：上位40件を読み飛ばして、次の20件を返す 問題：offset=100000 などが遅い Cursor方式 GET /notifications?limit=20&cursor=2026-06-01T10:00:00Z_12345 SQL: where (created_at, id) < (:createdAt, :id) order by created_at desc, id desc limit 20 意味：前回最後に見たレコードより後続の20件を返す
理解補足：OFFSETは人間に分かりやすい「ページ番号」に向いていますが、DB内部では指定件数を読み飛ばす必要があります。Cursorは「最後に見た地点から次を取る」ため、大量データでも性能が安定しやすいです。ただし、任意のページ番号へジャンプする用途には向きません。
注意点：created_atだけをCursorにすると、同時刻のデータが複数ある場合に順序が不安定になります。必ずcreated_atとidのように一意に並ぶキーを使います。

面接官の意図：分散システムで可用性・性能・整合性のトレードオフを理解しているかを見る。

回答例・補足を見る

30秒回答

強い整合性は、書き込みが完了した後の読み取りで常に最新状態が見える性質です。結果整合性は、一時的には古い値や未反映の値が見える可能性があるものの、時間が経てば最終的に整合する性質です。残高、注文確定、認可判定のような誤りが重大な領域は強い整合性を重視し、通知数、検索インデックス、分析集計のように多少遅れても許容できる領域は結果整合性を検討します。

メリット・デメリット

  • 強い整合性のメリット：利用者や開発者にとって分かりやすい。二重在庫引当、残高不整合、権限ミスを防ぎやすい。
  • 強い整合性のデメリット：ロック、同期通信、合意形成が必要になり、レイテンシや可用性、スケールにコストがかかる。
  • 結果整合性のメリット：非同期処理や複製を使いやすく、可用性・スループット・スケールを上げやすい。
  • 結果整合性のデメリット：一時的に古い情報が見える。ユーザー体験、補正処理、再試行、監視が必要になる。
例 強い整合性が欲しい - 決済残高 - 注文確定 - 在庫引当 - 認可判定 結果整合性を許容しやすい - 検索インデックス反映 - 通知未読件数 - アクセス分析 - ランキング集計
理解補足：結果整合性は「不整合でよい」という意味ではありません。「短時間の遅れやズレは許容し、最終的に正しい状態へ収束させる」という意味です。例えば、DB更新後に検索インデックスへ非同期反映する場合、数秒間は検索結果に出ないことがありますが、業務上許容できるなら結果整合性でよい判断になります。
注意点：認証認可、決済、在庫、契約状態のような領域で安易に結果整合性を採用すると、セキュリティ事故や金銭事故につながります。どの不整合を何秒まで許すのか、具体的に言える必要があります。

メッセージキューで、at-least-once と exactly-once の違いは？

面接官の意図：非同期処理の重複・欠落・冪等性を理解しているかを見る。

回答例・補足を見る

30秒回答

at-least-onceは、メッセージが少なくとも1回は処理される保証です。失敗時に再配信されるため欠落しにくい一方、同じメッセージが複数回処理される可能性があります。exactly-onceは、論理的に1回だけ処理されたように見せる性質ですが、実現は難しく、ブローカーだけでなくDB更新や外部API呼び出しまで含めて考える必要があります。実務ではat-least-once前提でコンシューマを冪等にすることが多いです。

深掘りされたときに足す観点

  • at-most-once：最大1回。重複はしにくいが、失敗時に欠落する可能性がある。
  • at-least-once：少なくとも1回。欠落しにくいが、重複処理が起きる可能性がある。
  • exactly-once：論理的に1回だけ処理された状態を目指す。ただし、外部副作用を含むと難しい。
  • 冪等性：同じメッセージを2回処理しても結果が壊れないようにする。
  • 重複対策：message_id、idempotency_key、処理済みテーブル、一意制約を使う。
at-least-onceで重複が起きる例 Queue → Worker が message M を取得 Worker → DB更新成功 Worker → ack送信前にクラッシュ Queue → ack未受信なので message M を再配信 Worker → message M をもう一度処理 対策：message_id を処理済みテーブルに記録し、2回目はスキップする
理解補足：exactly-onceという言葉は誤解しやすいです。Kafkaなど一部の基盤は特定条件下でexactly-onceに近い処理をサポートしますが、メール送信、決済API、外部システム登録のような外部副作用まで完全に1回にするのは別問題です。そのため、アプリケーション側で冪等キーや状態遷移を設計する必要があります。
注意点：「exactly-onceを使えば重複しません」と言い切るのは危険です。面接では、「基盤の保証範囲」と「アプリケーション全体での副作用」を分けて説明する方が正確です。

面接官の意図：単体テスト、結合テスト、E2E、契約テストを使い分けられるかを見る。

回答例・補足を見る

30秒回答

  • 負荷・性能テスト：高トラフィックAPI、バッチ、外部連携で必要になる。

注意点：すべてをE2Eで確認しようとすると、遅く壊れやすいテストになります。逆に単体テストだけだと、DB接続、設定ミス、API契約違反、認証認可の設定漏れを見逃します。テスト種別ごとに役割を分けるべきです。

冪等性とは何ですか？API設計でなぜ重要ですか？

面接官の意図：リトライや二重送信に耐える設計を理解しているかを見る。

回答例・補足を見る

30秒回答

冪等性とは、同じ操作を1回実行しても複数回実行しても、最終的な結果が同じになる性質です。APIでは、ネットワークエラーやクライアントのリトライにより同じリクエストが複数回届くことがあります。注文作成や決済のような処理では、Idempotency Keyを使い、同じキーのリクエストは同じ結果を返すようにします。

深掘りされたときに足す観点

  • GET、PUT、DELETEは一般に冪等として設計される。POSTは通常冪等ではないが、Idempotency Keyで冪等にできる。
  • 外部API連携やキュー処理では重複実行が現実に起きる。
  • DBに処理済みキーを保存し、二重作成を防ぐ。
理解補足：冪等性は、Stripe系のIntegration面接でかなり重要な概念です。あなたの外部API連携経験にも直結します。DIPS APIラップやトークン更新処理でも、同じ処理が再実行された場合に壊れないかを考える必要があります。
注意点：「リトライします」と言うなら、必ず冪等性をセットで説明してください。冪等性がないリトライは、二重注文・二重請求・二重通知を引き起こします。

OQ046

DBスキーマ変更を安全にリリースするにはどうしますか？

面接官の意図：アプリとDBの互換性、段階リリース、ロールバックを考えられるかを見る。

回答例・補足を見る

30秒回答

DBスキーマ変更は、アプリとDBの互換性を保ちながら段階的に行います。基本は、まず後方互換な変更を入れます。例えば、nullableな新規カラム追加、アプリ側で新旧両方に対応、データ移行、読み取り切替、旧カラム削除の順に進めます。破壊的変更を一度に入れないことが重要です。

深掘りされたときに足す観点

  • Expand and Contract：追加→併用→移行→削除の段階で進める。
  • 大きなテーブルのALTERはロックや性能影響に注意。
  • データ移行はバッチ化し、途中失敗に備える。
  • ロールバック時に旧アプリが新DBでも動くか確認する。
理解補足：Web系企業では、ゼロダウンタイムデプロイが求められることがあります。アプリとDBを同時に完全同期して切り替える前提ではなく、どちらが先にリリースされても壊れにくい変更順序を考える必要があります。
注意点：カラム削除、NOT NULL追加、型変更、意味変更は破壊的になりやすいです。いきなり実施せず、移行期間を設けます。
