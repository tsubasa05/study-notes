Q001

認証と認可の違いを説明してください。

面接官の意図：あなたの主戦場。定義を曖昧にすると、認証認可経験全体が浅く見える。

回答例を見る
回答例：
認証は、操作しているユーザーが誰であるかを確認することです。認可は、その認証済みユーザーが、対象リソースに対してどの操作を実行できるかを判断することです。例えば、ログインによってユーザーIDを確認するのが認証で、そのユーザーが特定の事業者情報を参照できるか、特定APIを実行できるかを判断するのが認可です。

補足知識

  • 認証 authentication：「あなたは誰か」を確認する。ID/パスワード、多要素認証、IdPログイン、OIDCのID Tokenなどが関係する。
  • 認可 authorization：「その主体は、このリソースに、この操作をしてよいか」を判断する。subject / resource / action の組み合わせで考えると整理しやすい。
  • 面接では：「誰か」だけでなく、「どのリソースに」「何ができるか」まで言うと精度が上がる。
注意：「認可＝その人が何をしてよいか」でも大枠は正しいですが、Web系面接では「リソースに対して」という表現を入れた方が正確です。

Q002

OAuth 2.0 と OIDC の違いは何ですか？

面接官の意図：OIDC経験の本質理解を見る。

回答例を見る
回答例：
OAuth 2.0は、クライアントが保護されたリソースへアクセスするためのアクセストークンを取得する認可のフレームワークです。OIDCはOAuth 2.0の上に認証レイヤーを追加した仕様で、ID Tokenによって「ユーザーが認証されたこと」やユーザー識別子をクライアントが確認できます。つまり、OAuth 2.0は主にAPIアクセス権限、OIDCはログイン・本人確認まで扱う、という整理です。

補足知識

  • OAuth 2.0だけで「ログイン」を実装しようとすると、ユーザー認証結果の標準的な表現が不足しやすい。
  • OIDCでは ID Token、UserInfo Endpoint、scope=openid などが登場する。
  • 面接では「OAuthでログイン」は雑な表現で、厳密には「OIDCを使ったログイン」と言えるとよい。

Q003

Authorization Code Flowを使う理由は？

面接官の意図：ブラウザ/バックエンド構成の安全性と、フローの役割を理解しているか。

回答例を見る
回答例：
Authorization Code Flowは、認可サーバーがブラウザ経由で直接トークンを返すのではなく、まず短命の認可コードをクライアントへ返し、その後クライアントがトークンエンドポイントで認可コードをAccess TokenやID Tokenに交換する方式です。メリットは、トークンをURLフラグメントなどのフロントチャネルに直接露出させずに済むこと、サーバー側クライアントであればクライアント認証を組み合わせられること、PKCEによって認可コード横取りへの耐性を上げられることです。BFF構成では、この流れを使ってトークンをBFF側に保持し、ブラウザにはHttpOnly Cookieのセッションだけを持たせる設計にできます。

端的な役割

  1. ユーザーを認可サーバーへリダイレクトする。
  2. 認可サーバーでユーザー認証・同意・認可判断を行う。
  3. 認可サーバーがブラウザ経由で 認可コード をクライアントに返す。
  4. クライアントがバックチャネルでトークンエンドポイントに認可コードを提示し、Access Tokenなどを取得する。

なぜ安全性が上がるのか

  • Access Tokenそのものを認可エンドポイントからブラウザ経由で返さないため、URL、履歴、Referer、フロント側ログなどに露出するリスクを下げられる。
  • 認可コードは短命・一回限りで、盗まれてもPKCEやクライアント認証があれば悪用しにくい。
  • トークン発行をトークンエンドポイントに集約でき、クライアント認証、リダイレクトURI検証、PKCE検証などを組み合わせられる。
重要な補正：Authorization Code Flow自体が常に「ブラウザにAccess Tokenを保持させない」わけではありません。SPAでAuthorization Code Flow + PKCEを使う場合、最終的にブラウザ側がTokenを扱う構成もあります。ブラウザにAccess Tokenを持たせないのは、BFFやサーバーサイドWebアプリとして設計した場合のメリットです。

Q004

JWTのメリットとデメリットは？

面接官の意図：トークンベース認証の仕組みとトレードオフを説明できるか。

回答例を見る
回答例：
JWTは、ヘッダー・ペイロード・署名からなるトークン形式で、ペイロードにユーザー識別子、発行者、有効期限、スコープなどのClaimを入れられます。メリットは、Resource Server側で署名、有効期限、発行者、Audienceなどを検証すれば、毎回認可サーバーやセッションストアへ問い合わせずにトークンの正当性を確認できる点です。デメリットは、発行後の即時失効や権限変更の即時反映が難しいこと、漏洩時に有効期限まで悪用されること、Claimを詰め込みすぎると肥大化すること、署名付きJWTは暗号化ではないため中身を読めることです。

署名検証では何を検証しているのか

  • JWTは一般に Base64Url(header).Base64Url(payload).signature という形です。
  • 署名検証では、受け取った header.payload に対して、発行者の公開鍵、または共有秘密鍵を使って署名を検証します。
  • これにより「信頼している発行者が発行したこと」と「途中でペイロードが改ざんされていないこと」を確認します。
  • ただし署名検証だけでは不十分で、iss、aud、exp、nbf、必要な scope や role も確認します。

覚えるべきポイント

  • 署名付きJWT = 改ざん検知できる。中身を隠すものではない。暗号化したいならJWEなど別の仕組みが必要。
  • JWTはステートレスに検証しやすい。一方で、ステートレスだからこそ失効や権限変更の即時反映が難しい。
  • Access TokenとしてのJWTは、Resource ServerがAPIアクセス可否を判断する材料になる。ただし詳細な認可判断を全部JWTだけで完結させる必要はない。

Q005

認可情報をJWTに全部入れない理由は？

面接官の意図：OpenFGA利用理由とJWTの限界を語れるか。

回答例を見る
回答例：
JWTに認可情報を全部入れると、トークンが肥大化しやすく、権限変更や所属変更を即時反映しづらくなります。Access Tokenは発行された時点のClaimを持つため、有効期限が残っている間は古い権限情報がResource Serverで有効に見えてしまう可能性があります。特に、組織、リソース所有関係、業務状態に応じた細かい認可はJWTに固定的に詰め込むより、外部認可基盤やアプリ側で subject / resource / action を使って判断した方が柔軟です。

権限変更の即時反映が難しい理由

  • ユーザーが「管理者」ロールを持つAccess Tokenを受け取る。
  • その後、管理者ロールを剥奪する。
  • しかし、既に発行済みのAccess TokenのClaimは自動では変わらない。
  • Resource ServerがJWTをステートレスに検証しているだけなら、exp まで古いClaimを信じてしまう。

対策

  • Access Tokenを短命にする。
  • 重要な認可は外部認可基盤、DB、PDPに問い合わせる。
  • Token Introspectionや失効リストを使う。ただし性能・可用性コストが増える。
  • Refresh Tokenをローテーション・失効管理する。
面接での言い方：「JWTに権限を入れるのが悪い」ではありません。粗いscopeやroleを入れるのは普通にあります。問題は、変更頻度が高い・リソース単位で細かい・即時反映が必要な認可までJWTに閉じ込めることです。

Q006

RBACとReBACの違いは？

面接官の意図：OpenFGAを使った意味を理解しているか。

回答例を見る
回答例：
RBACはユーザーにロールを付与し、ロールごとに操作権限を管理する方式です。ReBACはユーザー、組織、グループ、リソースなどの関係性をもとに権限を判断する方式です。例えば「この組織の管理者は、この組織に属するリソースを更新できる」「このドキュメントの所有者は編集できる」といった関係を表現しやすいのがReBACです。単純なAPIロール制御ならRBACで十分ですが、組織階層や所有関係、共有関係が絡むとReBACが有効です。

補足知識

  • RBACは「役割」を中心に考える。例：admin、manager、operator。
  • ReBACは「関係」を中心に考える。例：user A is manager of org X、document Y belongs to org X。
  • OpenFGAはReBAC的なモデルを表現しやすい認可エンジン。

Q007

API Gatewayで認可する場合と各サービスで認可する場合の違いは？

面接官の意図：認可の責務境界と、Gatewayでできること・できないことを理解しているか。

回答例を見る
回答例：
API Gatewayで認可すると、入口で一元的に制御でき、認証済みか、Tokenが有効か、API単位のscopeやroleを満たしているか、といった粗い制御を集約できます。一方、各サービスで認可すると、DB上のリソース所有者、業務状態、テナント所属、詳細なドメインルールを見た判断がしやすくなります。実務では、Gatewayで共通的な入口制御を行い、リソース状態に依存する細かい認可はサービス側、または外部認可基盤へ問い合わせる形で分担するのが現実的です。

Gateway認可が向いているもの

  • 未認証リクエストの遮断。
  • Tokenの署名、有効期限、issuer、audience確認。
  • API単位・endpoint単位のscope/roleチェック。
  • レート制限、IP制限、共通ヘッダ付与などの横断制御。

サービス側認可が向いているもの

  • 「このユーザーはこの注文の所有者か」。
  • 「このリソースは承認済み状態なので編集不可か」。
  • 「同じ組織に所属しているが、部署単位で参照範囲が違うか」。
  • 「画面上は見えてよいが、この業務状態では更新できないか」。
精査結果：「リソースの詳細状態や業務ロジックに依存する認可はGatewayだけでは判断しづらい」という説明は概ね正しいです。ただし、Gatewayが外部認可基盤へ subject / action / resource / attributes を問い合わせる設計なら、Gateway側でも細かい認可は可能です。その場合は、Gatewayが必要なリソース属性をどう取得するか、レイテンシ、可用性、責務肥大化が設計課題になります。

Q008

Access TokenとRefresh Tokenの役割は？

面接官の意図：Access Tokenを「ユーザー情報そのもの」と誤解していないかを見る。

回答例を見る
回答例：
Access Tokenは、クライアントがResource ServerのAPIへアクセスするときに提示する資格情報です。Resource ServerはAccess Tokenを検証し、「誰の権限に基づくアクセスか」「どのクライアント向けか」「どのscopeや権限を持つか」「期限切れでないか」を確認した上で、リソースへの操作可否を判断します。Refresh Tokenは、Access Tokenの有効期限が切れたときに、認可サーバーへ提示して新しいAccess Tokenを取得するためのトークンです。Refresh TokenはResource Serverには送らず、認可サーバーにだけ提示します。

Access Tokenを提示することで何をしているのか

  • API呼び出し元が、認可サーバーから発行された有効なトークンを持っていることを示す。
  • Resource Serverに対して、ユーザー、クライアント、scope、audience、有効期限などの判断材料を渡す。
  • Resource Serverは、その情報と自分側の認可ロジックを組み合わせてアクセス可否を決める。
理解の補正：Access Tokenは「操作ユーザーがリソースに対して何をしてよいかが全部書かれているトークン」とは限りません。JWT形式ならscopeやroleなどが入ることがありますが、Tokenが不透明文字列のこともあります。また、詳細なリソース単位の認可は、Token内の情報だけでなく、DBや外部認可基盤を見て判断することも多いです。

Q009

BFF構成でCookie Sessionを使うメリットは？

面接官の意図：SPA/ブラウザセキュリティ理解を見る。

回答例を見る
回答例：
BFF構成では、ブラウザにAccess Tokenを直接持たせず、BFF側でTokenを管理し、ブラウザにはHttpOnly / Secure Cookieのセッションだけを持たせる設計にできます。これにより、XSSでJavaScriptからAccess Tokenを直接盗まれるリスクを下げられます。一方、Cookieはブラウザが自動送信するため、CSRF対策が必要です。BFFはフロントエンドに近いバックエンドとして、認証情報管理、API集約、画面向けレスポンス整形などを担います。

補足知識

  • HttpOnly CookieはJavaScriptから読み取れない。
  • Secure CookieはHTTPS通信でのみ送信される。
  • SameSite 設定はCSRFリスク低減に関係する。

Q010

権限変更を即時反映したい場合、どんな設計が考えられますか？

面接官の意図：JWTの限界と外部認可の価値を理解しているか。

回答例を見る
回答例：
JWTに権限を固定的に入れている場合、発行済みTokenの有効期限が切れるまで古い権限が残る可能性があります。即時反映したい場合は、Access Tokenを短命にする、認可判断を都度外部認可基盤やDBに問い合わせる、Token Introspectionや失効リストを使う、セッション管理で強制ログアウトする、といった設計が考えられます。ただし、即時性を高めるほど、レイテンシ、可用性、実装複雑性のコストが上がります。

面接での整理

  • 低リスクな権限：短命Access Tokenで十分なことが多い。
  • 高リスクな権限：毎回外部認可やDB確認を入れる方が安全。
  • 退職・管理者剥奪・不正検知：強制失効やセッション無効化まで必要になる。

Q011

ユーザ管理情報をKeycloak側とアプリDB側でどう分けますか？

面接官の意図：あなたの実務経験に直結。

回答例を見る
回答例：
認証に必要なユーザーID、認証情報、ログイン関連属性はKeycloak側に寄せます。一方、アプリ固有の業務属性、組織内での意味付け、ドメイン上必要な状態はアプリDBに持たせます。全部をKeycloakに寄せるとドメインモデルがIdPに依存しすぎますし、全部をアプリ側に持つと認証基盤の意味が薄れます。責務境界を明確にすることが重要です。

例

  • Keycloak側：ログインID、認証方式、パスワード、MFA、ユーザー識別子。
  • アプリDB側：事業者ID、部署、業務上の状態、アプリ固有プロフィール、利用設定。
  • 認可基盤側：ユーザーと組織・ロール・リソースの関係。

Q012

マイクロサービス間通信でユーザー文脈をどう扱いますか？

面接官の意図：Access Tokenを「ユーザー情報伝搬手段」と雑に扱っていないかを見る。

回答例を見る
回答例：
まず、その処理を「ユーザーの代理として実行するのか」「サービス自身の権限で実行するのか」を分けて考えます。ユーザーの代理として下流サービスでもユーザー単位の認可判断が必要なら、ユーザー文脈を伝える必要があります。ただし、Access Tokenを単にユーザー情報の入れ物として伝搬するのではなく、audience、scope、有効期限、下流サービスでの検証方法を設計する必要があります。サービス自身の定期処理や機械的な連携であれば、Client Credentialsなどのサービス間認証を使い、必要なユーザーIDは監査・業務データとして別に渡す方が適切な場合があります。

代表的な方式

  • Token Relay：元のAccess Tokenを下流サービスへ渡す。ユーザー文脈を維持しやすいが、Token漏洩範囲、audience不一致、権限過多に注意が必要。
  • Token Exchange / On-Behalf-Of：上流サービスが、下流サービス向けのTokenに交換する。audienceやscopeを下流用に絞りやすい。
  • Client Credentials：サービス自身の資格情報で呼び出す。ユーザー代理ではなくシステム間処理に向く。
  • 内部ヘッダでユーザーIDを渡す：GatewayやBFFで認証済みユーザーを解決し、内部向けにヘッダ付与する方式。境界内の信頼、改ざん防止、署名、mTLSなどを考える必要がある。
重要：Access Tokenは本来、APIアクセスのための資格情報です。「ユーザー情報を伝えるための便利な箱」として扱うと設計が崩れます。下流サービスに必要なのが認可判断なのか、監査用のユーザーIDなのか、画面表示用プロフィールなのかを分けるべきです。

回答例：

Q014

Controller / Service / Repository の責務は？

面接官の意図：アプリケーション層の基本設計。

回答例を見る
回答例：
ControllerはHTTPリクエスト/レスポンスの入口で、入力受付、Validation、DTO変換、ステータスコード返却を担当します。Serviceはユースケースや業務処理、トランザクション境界を担当します。RepositoryはDBアクセスを隠蔽し、永続化の詳細を扱います。Controllerに業務ロジックを書くとテストしづらく、Repositoryに業務判断を書くとDBアクセスとドメイン判断が混ざるため、層ごとの責務を分けます。

面接での一言

「Controllerは薄く、Serviceにユースケース、Repositoryに永続化」を基本線として話すと崩れにくいです。

Q015

@Transactional はどこに付けるべきですか？

面接官の意図：トランザクション境界を理解しているか。

回答例を見る
回答例：

Q022

N+1問題とは何ですか？

面接官の意図：JPAを使うなら必須。発行SQLを意識できているかを見る。

回答例を見る
回答例：
N+1問題は、親データを1回のSQLで取得した後、その親データN件それぞれに対して関連する子データを個別SQLで取得してしまい、合計で1+N回のSQLが発行される問題です。例えば注文一覧を1回取得した後、各注文の注文明細や顧客情報をループ内で遅延ロードすると、注文件数分だけ追加SQLが出ます。件数が増えるとDB往復が急増し、性能劣化します。対策は、JOIN FETCH、EntityGraph、バッチフェッチ、必要なDTOを直接取得するクエリなどです。

具体例

  1. SELECT * FROM orders LIMIT 100 で注文100件を取得する。
  2. 画面表示時に各注文の order.getItems() を呼ぶ。
  3. 遅延ロードにより SELECT * FROM order_items WHERE order_id = ? が100回発行される。
  4. 結果として、1回 + 100回 = 101回のSQLになる。

対策の使い分け

  • JOIN FETCH：必要な関連を一緒に取得する。便利だが、1対多で行数が膨らむことがある。
  • EntityGraph：JPAで取得したい関連を指定する。
  • バッチフェッチ：関連データを1件ずつではなくまとめて取得する。
  • DTOクエリ：一覧画面に必要な項目だけをSELECTする。画面/API用途では現実的。
注意：「JOINすれば全部解決」ではありません。ページングとの相性、重複行、取得しすぎ、メモリ使用量も見る必要があります。

Q024

楽観ロックと悲観ロックの違いは？

面接官の意図：並行更新の理解。違いを具体例で説明できるか。

回答例を見る
回答例：
悲観ロックは、競合が起きる前提で、更新対象の行を先にロックし、他トランザクションの更新を待たせる方式です。例えば SELECT ... FOR UPDATE のように、在庫や残高など同時更新されると困るデータで使われます。楽観ロックは、競合は少ない前提で、読み取り時にはロックせず、更新時にversion列などを条件に含め、他者が先に更新していたら更新失敗として検知する方式です。ユーザー編集画面のように競合頻度が低い場合に向きます。

悲観ロック

  • 更新前にロックを取る。
  • 競合を未然に防げる。
  • 高競合・強い整合性が必要な処理に向く。
  • 待ち時間、デッドロック、スループット低下に注意。

楽観ロック

  • 読み取り時にはロックしない。
  • 更新時に WHERE id = ? AND version = ? のように確認する。
  • 更新件数が0件なら、他者更新があったと判断してエラーや再読み込みにする。
  • 低競合でスループットを上げたい処理に向く。
違いの一言：悲観ロックは「先に止める」、楽観ロックは「後で衝突を検知する」です。

回答例：

CSRFの具体例

  1. ユーザーが銀行サイトにログイン済みで、セッションCookieを持っている。
  2. 攻撃者サイトを開く。
  3. 攻撃者サイトが銀行サイトの送金APIへフォーム送信や画像読み込み等でリクエストを誘導する。
  4. ブラウザが銀行サイトのCookieを自動送信する。
  5. 銀行サイトがCSRF対策をしていないと、本人操作と区別できず処理してしまう。

  • CSRF対策：CSRF Token、SameSite Cookie、Origin/Refererチェック、GETで状態変更しない。

整理：CSRFは「ブラウザがCookieを自動で送る」性質を悪用します。XSSは「正規サイト上で悪意あるJavaScriptが動く」ことが本質です。Cookie SessionではCSRF対策が重要で、TokenをLocalStorage等に置く構成ではXSSによるToken窃取が特に危険です。

回答例：

具体的なチェック観点

  • イメージサイズ: 不要なビルドツールやキャッシュを最終イメージに入れない。

  • セキュリティ: 可能なら非rootユーザーで実行する。秘密情報をDockerfileやイメージに含めない。

  • 起動方法: ENTRYPOINT/CMDを明確にし、アプリの終了シグナルを適切に扱えるようにする。
  • 設定: DB接続先、外部API URL、ログレベルなどは実行時設定にする。

面接で話すなら、「Mavenでビルドするステージ」と「生成されたJARをJREベースイメージで実行するステージ」を分ける、という説明が使いやすいです。さらに、Spring BootではActuatorのヘルスチェックエンドポイントをECS/ALBのヘルスチェックに使う、という運用面まで接続できます。

注意：Dockerfileはアプリを動かす手順書ではなく、配布・実行される成果物を作る定義です。ローカルで動けばよいという発想だと、サイズ肥大化、秘密情報混入、環境差異、脆弱性の問題が残ります。

  1. 業務サービスが通知要求を発行する。
  2. 通知APIがイベントを受け、notification_eventsテーブルへ保存する。

  • テンプレート: 通知本文をコードから分離し、差し替えや多言語対応をしやすくする。

面接官の意図：あなたの経験領域を、個別案件の説明ではなく汎用設計として語れるかを見る。

回答例を見る
回答例：
認証はIdP/Authorization Serverが担い、OIDCによりログイン、ID Token、Access Token発行を行います。API側はResource ServerとしてAccess Tokenを検証し、ユーザーやクライアント、Scope、Audience、有効期限を確認します。認可は、単純なAPI単位のRole/Scopeチェックで足りる場合はGatewayや各サービスで実装できます。一方、組織、リソース所有関係、業務状態に応じた細かい認可が必要な場合は、OpenFGAのような外部認可基盤、またはアプリ側の認可ロジックでsubject / resource / actionを使って判断します。設計論点は、トークン失効、権限変更の反映、管理画面、監査ログ、鍵ローテーション、JWKS配布、可用性、レイテンシ、障害時のfail-close/fail-openです。

役割分担

  • IdP / Authorization Server: ユーザー認証、トークン発行、クライアント管理を担う。

  • Resource Server: API本体。Access Tokenを検証して保護リソースを提供する。
  • PEP: Policy Enforcement Point。認可判断を強制する場所。GatewayやAPIサービスなど。
  • PDP: Policy Decision Point。認可可否を判断する場所。OpenFGAなど。
  • PAP: Policy Administration Point。権限設定やポリシーを管理する場所。

  • トークン設計: Access TokenをJWTにするか不透明トークンにするか。
  • 権限変更: 短命Access Token、外部認可問い合わせ、Introspection、失効リストをどう使うか。
  • 鍵管理: 署名鍵のローテーションとJWKS公開。
  • 監査: 誰が、いつ、どのリソースに、何をしたか。
  • 障害時: 認可基盤が落ちた場合に拒否するのか、キャッシュ済み判断を使うのか。

あなた向けの接続

あなたはKeycloak/OIDC/OpenFGA/ユーザ管理責務整理の経験があるので、「認証はKeycloak、細かい認可はOpenFGA、アプリDBはドメイン属性を管理する」という責務分担の話に接続できます。ここはかなり強い主戦場です。

注意：「認証認可基盤を作る」と言っても、ユーザーDB、ログイン画面、トークン発行、認可ポリシー、監査ログ、管理画面、SDK、運用監視まで範囲が広いです。面接では最初にスコープ確認するのが重要です。

  • 署名URLの有効期限を短くする。
  • バケットは原則Privateにし、公開はCloudFront署名URLなどを検討する。
  • ファイルサイズ、拡張子、MIME typeを検証する。
  • アップロード後にウイルススキャンやマルウェアチェックを行う。
  • S3 object keyにユーザー入力値をそのまま使わない。

運用観点

  • PENDINGのまま残ったメタデータやS3オブジェクトを定期削除する。
  • ダウンロードログや監査ログを残す。
  • 画像ならサムネイル生成などを非同期処理にする。

Kafkaの基本語彙

  • スキーマ進化: イベント形式を変更するとConsumerに影響するため、Schema Registryなどを検討する。

Q047

PUTとPATCHの違いは？

面接官の意図：HTTPメソッドの意味と、API更新設計の注意点を理解しているかを見る。

回答例を見る
回答例：
PUTは、指定したリソースをリクエストボディの内容で全体置換する用途で使われます。PATCHは、指定したリソースに対して部分的な変更を適用する用途で使われます。PUTは同じリクエストを何度送ってもサーバー上の状態が同じになるため、冪等な設計にしやすいです。PATCHは部分更新なので、リクエスト形式によって冪等にも非冪等にもなり得ます。API設計では、未指定フィールドを「変更しない」と扱うのか、「nullで上書き」と扱うのかを明確にする必要があります。また、更新競合を防ぐために、ETagやIf-Match、versionカラムを使うこともあります。

具体例

  • PUT /users/1: ユーザー1のリソース全体を、送信されたJSON全体で置き換える。
  • PATCH /users/1: ユーザー1のnameだけ、emailだけなど、一部フィールドを更新する。

PATCHで注意すること

  • 未指定フィールドは変更なしなのか。
  • nullは値を消す意味なのか、それとも未指定扱いなのか。
  • 配列は全置換なのか、追加・削除差分なのか。
  • JSON Patch、JSON Merge Patch、独自形式のどれを使うのか。

面接での言い方

「PUTは全体更新、PATCHは部分更新」で大枠は十分ですが、もう一歩踏み込むなら「冪等性」と「未指定/nullの仕様」を話してください。Web系面接ではここまで言えると実務感が出ます。

回答例：
外部キー制約のメリットは、DBレベルで参照整合性を守れることです。例えばorder_items.order_idがorders.idを参照する場合、存在しない注文IDを持つ注文明細が登録されることをDBが防げます。アプリケーションにバグがあっても、DBが最後の防衛線になります。一方、デメリットは、削除やデータ移行、バルク投入、テストデータ投入が複雑になりやすいこと、書き込み時に制約チェックのコストがあること、マイクロサービスのようにDB境界を分ける設計ではサービスをまたいだ外部キーを貼れないことです。単一サービス・単一DB内で強い整合性が必要なら外部キーは有効ですが、サービス境界をまたぐ整合性はアプリケーションやイベントで管理することになります。

メリット

  • 存在しない親レコードへの参照を防げる。

  • マイクロサービス間でDBを分ける場合、サービスをまたいだ制約は貼れない。

面接での結論

「外部キーは絶対貼るべき」「外部キーは不要」と断定しない方がよいです。単一DB内の重要な整合性はDB制約で守り、サービス境界をまたぐ整合性はAPI・イベント・バッチ検査などで守る、という整理が現実的です。

Q050

キャッシュを入れるときの注意点は？

面接官の意図：性能改善のメリットだけでなく、不整合・失効・障害時の副作用を理解しているかを見る。

回答例を見る
回答例：
キャッシュは、読み取り性能を上げ、DB負荷や外部API呼び出しを減らすために使います。ただし、キャッシュを入れると必ず不整合、失効、更新反映、キャッシュスタンピード、メモリ使用量、障害時の挙動が論点になります。基本的には、更新頻度が低く、読み取り頻度が高く、多少古いデータを許容できるものに向いています。例えばマスタデータ、設定値、公開プロフィール、URL短縮のkey→original_urlなどです。設計時には、どのキーでキャッシュするか、TTLを何秒にするか、更新時に削除するのか上書きするのか、キャッシュが落ちたときにDBへフォールバックするのかを決めます。

代表的なパターン

  • Cache Aside: アプリがまずキャッシュを見て、なければDBから取得してキャッシュへ入れる。
  • Write Through: 書き込み時にDBとキャッシュを同時に更新する。
  • Write Behind: まずキャッシュへ書き、後からDBへ反映する。高性能だが整合性リスクがある。

主な落とし穴

  • 古いデータ: DB更新後にキャッシュが残ると古い値を返す。
  • キャッシュスタンピード: 人気キーが同時に期限切れになり、大量リクエストがDBへ流れる。
  • ホットキー: 特定キーへアクセスが集中する。
  • メモリ圧迫: 無制限にキャッシュするとRedis等のメモリを使い切る。
  • 障害時: キャッシュ障害でDBに負荷が集中する。

対策例

  • TTLを設定する。
  • 更新時に該当キーを削除する。
  • 人気キーは事前に温める。
  • 期限切れにランダム性を持たせる。
  • ロックやSingle Flightで同時再計算を防ぐ。
  • 古い値を一時的に返すstale-while-revalidate的な考え方を使う。
注意：キャッシュは「速くなる魔法」ではなく、整合性を犠牲にして性能を取る設計判断です。古いデータを返してよいかを確認せずに入れると事故ります。

Q051

fail-openとfail-closeの違いは？

面接官の意図：認可基盤・外部依存が障害になったときの安全側の設計を理解しているかを見る。

回答例を見る
回答例：
fail-openは、依存先の障害や判定不能時に処理を許可する考え方です。fail-closeは、判定不能時に処理を拒否する考え方です。認証認可のようなセキュリティ境界では、基本はfail-close、つまり判断できないなら拒否する設計が安全です。例えば認可基盤に問い合わせられないときに全リクエストを許可すると、本来アクセスできないリソースへアクセスできる危険があります。一方、fail-closeにすると認可基盤障害時にサービス全体が止まる可能性があります。そのため、重要操作はfail-close、低リスクな参照は短時間のキャッシュ済み認可判断を使う、管理者操作は必ずfail-closeにするなど、リスクに応じた設計が必要です。

使い分けの考え方

  • fail-close向き: 認証、認可、決済、個人情報参照、管理者操作、権限変更。
  • fail-openを検討し得るもの: ログ送信、メトリクス送信、レコメンド表示など、止める方が影響が大きく、セキュリティリスクが低い処理。
  • 中間案: 直近の認可結果を短時間キャッシュする、読み取りだけ許可する、機能を縮退する。

認可基盤での具体例

  1. API Gatewayが認可基盤へ問い合わせる。
  2. 認可基盤がタイムアウトする。
  3. fail-openなら許可して処理を進める。
  4. fail-closeなら403/503などで拒否・停止する。
  5. キャッシュ方式なら、過去数分以内の同一subject/resource/action判断を使う可能性がある。

面接での言い方

「セキュリティはfail-closeが基本。ただし可用性への影響が大きいので、操作のリスクごとにキャッシュ、縮退、監査ログ、アラートを組み合わせる」と答えるとバランスがよいです。

注意：fail-openは便利な逃げ道に見えますが、認可で安易に使うと重大事故になります。特に個人情報・金銭・管理者権限に関わる操作では、基本的にfail-closeを前提にした方が安全です。
