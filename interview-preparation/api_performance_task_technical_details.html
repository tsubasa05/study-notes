1. 4つの指標を具体的に理解する

Google SREの代表的な考え方として、ユーザー向けシステムでは Latency、Traffic、Errors、Saturation を見る、という整理があります。

面接では、まずこの4つで全体像をつかみます。

遅い
  -> 本当に遅いのか: Latency
  -> 負荷が増えたのか: Traffic
  -> 失敗も増えているのか: Errors
  -> 何かが詰まっているのか: Saturation

1-1. Latency: レイテンシ

レイテンシは、リクエストを送ってからレスポンスが返るまでの時間です。

APIでいうと、たとえば次のような値です。

  • /accounts の平均レスポンス時間が 200ms
  • /transactions のp95が 1.2s
  • /transfer のp99が 5s

重要なのは、平均だけを見ないことです。

指標意味見え方
平均全リクエストの平均時間全体傾向は分かるが、一部の極端な遅延が隠れる
p5050%のリクエストがこの時間以内中央値。普通のユーザー体験に近い
p9595%のリクエストがこの時間以内遅いユーザーが出ていないかを見る
p9999%のリクエストがこの時間以内ごく一部の深刻な遅延を見る

ここでいう p は percentile、つまりパーセンタイルの p です。

パーセンタイルは、値を小さい順に並べたときに「何%地点にある値か」を表します。

例として、100件のAPIレスポンスタイムがあるとします。

100件のレスポンスタイムを速い順に並べる

1番目    50ms
2番目    55ms
...
50番目   120ms  -> p50
...
95番目   800ms  -> p95
...
99番目   3000ms -> p99
100番目  5000ms

この場合、p95が 800ms なら「95%のリクエストは800ms以内に返っているが、残り5%は800msより遅い」という意味です。p99が 3000ms なら「99%は3秒以内だが、残り1%は3秒より遅い」という意味です。

平均だけを見ると、極端に遅い一部のリクエストが見えにくくなります。

レスポンス時間:
  100ms, 100ms, 100ms, 100ms, 5000ms

平均:
  1080ms

実態:
  ほとんどは速いが、1件だけ極端に遅い

逆に、平均だけだと「全体が少し遅い」のか「一部だけ極端に遅い」のかが分かりません。そのため、API性能では平均、p95、p99を分けて見ます。

たとえば、平均が 200ms でも、p99が 8s なら、一部ユーザーにはかなり遅いAPIです。金融サービスでは「一部ユーザーだけ遅い」も無視しづらいです。残高照会、明細、振込、本人確認などで一部だけ極端に遅いと、問い合わせや再操作につながります。

レイテンシが悪化する原因の例:

  • DBクエリが遅い
  • N+1でSQL発行回数が増えている
  • 外部APIの応答が遅い
  • アプリ側で大量データを加工している
  • JSONレスポンスが大きすぎる
  • コネクションプール待ちが発生している
  • CPUやメモリが逼迫している
  • ロック待ちが発生している

面接での言い方:

まず平均だけではなく、p95やp99を見ます。平均が悪いなら全体的な遅さ、p99だけ悪いなら一部条件、ロック待ち、外部APIの揺らぎ、GC、コネクションプール待ちなどを疑います。

1-2. Traffic: トラフィック

トラフィックは、システムに来ている負荷量です。

APIで見る場合は、次のようなものです。

  • RPS: 1秒あたりのリクエスト数
  • QPS: 1秒あたりのクエリ数
  • 同時接続数
  • 対象APIの呼び出し数
  • パートナー別、ユーザー別、時間帯別の呼び出し数

RPSとAPI呼び出し数の違い

RPSとAPI呼び出し数はかなり近い概念ですが、見ている粒度が違います。

用語意味例
API呼び出し数ある期間にAPIが何回呼ばれたか1分間に3,000回
RPS1秒あたり何リクエスト来たか50 requests/sec
RPM1分あたり何リクエスト来たか3,000 requests/min
QPS1秒あたり何クエリ、または問い合わせが来たかDBクエリが200 queries/sec

たとえば、1分間に3,000回APIが呼ばれていれば、平均すると 50 RPS です。

3,000 requests / 60 seconds = 50 RPS

ただし、現実にはきれいに均等ではありません。

1分間で3,000リクエスト

前半30秒:
  ほぼ0リクエスト

後半30秒:
  3,000リクエスト
  -> 後半だけ見ると100 RPS

そのため、ダッシュボードでは「1分平均のRPS」「5分平均のRPS」「ピーク時RPS」のように、どの時間幅で見ているかが重要です。

面接での言い方:

API呼び出し数は一定期間の総数で、RPSはそれを1秒あたりに直した負荷指標です。1分間に3,000回なら平均50 RPSですが、実際には瞬間的なピークがあるため、平均だけでなくピークや時間帯別にも見ます。

同時接続数、同時リクエスト数とは何か

同時接続数は、ざっくり言うと「同時に処理中、または接続中の数」です。

RPSは「単位時間あたり何件来るか」、同時接続数は「今どれだけ同時に抱えているか」です。

RPS:
  1秒間に何件リクエストが来るか

同時リクエスト数:
  ある瞬間に何件のリクエストを処理中か

たとえば、同じ 100 RPS でも、1リクエストがすぐ終わるか、長くかかるかで同時処理数は変わります。

100 RPSで、1リクエストが平均100msで終わる:
  同時処理中はざっくり10件程度

100 RPSで、1リクエストが平均2秒かかる:
  同時処理中はざっくり200件程度

同じRPSでも、レイテンシが悪化すると同時処理数が増え、スレッドやDB接続が埋まりやすくなります。これがさらにレイテンシを悪化させることがあります。

面接での言い方:

RPSは入ってくるリクエストの速度で、同時接続数や同時リクエスト数は、その瞬間に処理中の数です。同じRPSでも、1件あたりの処理時間が長くなると同時処理数が増え、スレッドやDB接続が詰まりやすくなります。

基準値はあるのか

「何RPSなら多いか」はサービス設計次第です。

同じ100 RPSでも、次の条件で重さがまったく違います。

API100 RPSの重さ
静的なマスタ情報を返すAPI軽い可能性が高い
DBを1回検索するAPI普通
複数JOINして明細を返すAPI重い可能性がある
外部APIを呼ぶAPI相手先次第で重い
振込や更新系API整合性やロックも考える必要がある

見る単位としては、次のように分けるのが一般的です。

  • システム全体のRPS
  • APIエンドポイント別のRPS
  • ユーザー別、パートナー別のRPS
  • インスタンス、Pod、TaskあたりのRPS
  • DBクエリのQPS
  • 時間帯別、ピーク時のRPS

面接では、絶対値を暗記するより、基準線との比較を言える方が重要です。

RPSの絶対値だけでは重いか判断できないため、通常時のベースライン、ピーク時、対象APIの処理内容、1台あたりの負荷、DBや外部APIへの影響と合わせて見ます。

トラフィックが増えると、同じAPIでも遅くなることがあります。

通常時:
  100リクエスト/分
  DB接続に余裕あり
  平均200ms

負荷増加時:
  5,000リクエスト/分
  DB接続が埋まる
  待ち行列が発生
  平均2秒、p99 10秒

トラフィック増加が原因になりやすいパターン:

  • キャンペーンや通知でアクセスが集中する
  • 特定パートナーからのAPI呼び出しが急増する
  • バッチ処理が大量にAPIやDBへアクセスする
  • リトライが集中して、通常よりリクエスト数が増える
  • フロント側の不具合で同じAPIを何度も呼んでいる

面接での言い方:

レイテンシが上がっている場合、同じタイミングでリクエスト数も増えているかを見ます。トラフィック増加と遅延が連動していれば、どのリソースが先に詰まっているか、DB接続、CPU、外部API、スレッドなどを確認します。

1-3. Errors: エラー

エラーは、リクエストが正常に処理できなかった割合や内容です。

APIで見るエラー:

  • 4xx: クライアント側のリクエスト不正、認証認可エラーなど
  • 5xx: サーバ側の異常
  • タイムアウト
  • DB接続エラー
  • 外部APIエラー
  • 業務エラー

ただし、4xx がすべて悪いわけではありません。たとえば認証切れや入力不正は通常の業務上起こり得ます。一方で、急に 401 や 403 が増えた場合は認証認可の変更ミスかもしれません。5xx やタイムアウトが増えた場合は、サーバ側の問題である可能性が高いです。

エラーとレイテンシの組み合わせで見る例:

状況考えられること
レイテンシだけ上がり、エラーは少ない処理は成功しているが遅い。DB、外部API、CPU、プール待ちを疑う
レイテンシもエラー率も上がるタイムアウト、リソース逼迫、下流障害を疑う
エラー率だけ上がる設定ミス、認証認可、外部API仕様変更、バリデーション変更を疑う
4xxが急増クライアント変更、認可スコープ、入力仕様変更を疑う
5xxが急増アプリ例外、DB障害、外部API障害、インフラ障害を疑う

面接での言い方:

遅いだけなのか、タイムアウトや5xxも増えているのかを分けます。エラーが増えている場合は、単なる性能問題ではなく、下流障害、DB接続枯渇、設定変更、認証認可の変更も疑います。

1-4. Saturation: サチュレーション

サチュレーションは、リソースがどれだけ限界に近いか、つまり「詰まり具合」です。

代表例:

  • CPU使用率が高い
  • メモリが足りない
  • GCが頻発している
  • DBコネクションプールが枯渇している
  • Tomcatのワーカースレッドが埋まっている
  • HTTPクライアントの接続プールが埋まっている
  • DBのI/Oが詰まっている
  • ロック待ちが増えている

サチュレーションのポイントは、限界に近づくと待ち行列が発生し、レイテンシが急に悪化しやすいことです。

余裕あり:
  すぐ処理できる
  レスポンス 200ms

詰まり始め:
  DB接続待ちが少し発生
  レスポンス 800ms

限界:
  ほぼ全リクエストが待つ
  レスポンス 5s
  タイムアウト増加

よくあるサチュレーション:

詰まる場所起きること確認するもの
DB接続プールDB接続待ちでAPIが遅くなるactive、idle、pending
アプリスレッドリクエストを処理するスレッドが空かないactive threads、queue
CPU処理待ちが増えるCPU使用率、ロード、プロファイル
メモリ/GCGC停止でレスポンスが揺れるheap、GCログ、pause time
DB I/Oクエリが遅くなるIOPS、スロークエリ、実行計画
外部API接続外部API待ちで詰まるtimeout、retry、connection pool

面接での言い方:

サチュレーションは、CPUやDB接続数などのリソースが限界に近いかを見る指標です。どこかが詰まると待ち行列ができ、p95やp99が悪化しやすいので、レイテンシと合わせて見ます。

1-5. 4指標の組み合わせで原因を絞る

4つの指標は単独ではなく、組み合わせで見ます。

見え方疑うこと
Traffic増加、Latency増加、Saturation増加負荷増加でリソースが詰まっている
Trafficは同じ、Latencyだけ増加リリース、SQL変更、外部API遅延、データ増加
Errors増加、Latency増加タイムアウト、DB/外部API障害、プール枯渇
p99だけ悪い一部データ、ロック、GC、外部APIの揺らぎ、キャッシュミス
CPU低いがLatency高いDB待ち、外部API待ち、コネクションプール待ち
DB CPU高い、スロークエリ増加SQL、インデックス、取得件数、JOIN/GROUP BY/ORDER BY

2. DB周りの基本

API性能課題では、DBが原因になることがかなり多いです。ただし、面接ではDBだけに決めつけるのではなく、DBが疑わしい場合に何を見るかを説明できればよいです。

2-1. N+1クエリとは何か

N+1クエリは、最初に一覧を1回取得したあと、一覧の各行ごとに追加SQLを発行してしまう問題です。

例として、取引明細一覧を取得するAPIを考えます。

悪い例:

1回目:
  SELECT * FROM transactions WHERE user_id = ?

明細が100件ある場合:
  SELECT * FROM merchants WHERE id = ?  -- 1件目
  SELECT * FROM merchants WHERE id = ?  -- 2件目
  ...
  SELECT * FROM merchants WHERE id = ?  -- 100件目

この場合、最初の1回に加えて、100件分の追加SQLが出ます。だから N+1 です。

データが少ないと問題が見えません。

明細5件:
  SQL 6回
  そこまで遅くない

明細1,000件:
  SQL 1,001回
  急に遅くなる

N+1が起きやすい実装:

  • 一覧取得後、for文の中でDBアクセスしている
  • ORMの遅延ロードで関連テーブルを1件ずつ取りに行っている
  • DTO組み立て時に、関連情報を都度Repositoryから取得している

確認方法:

  • SQLログで、似たSQLが大量に出ていないか見る
  • 1リクエストあたりのSQL発行回数を見る
  • データ件数を増やすと処理時間が比例して悪化するか見る
  • APMやトレースでDB呼び出し回数を見る

改善策:

  • JOINで必要な関連情報をまとめて取得する
  • WHERE id IN (...) でまとめて取得する
  • ORMのfetch joinやbatch fetchを使う
  • APIで本当に必要な情報だけにDTOを絞る
  • ページングして1回の対象件数を減らす

改善策1: JOINでまとめて取得する

N+1の悪い例では、まず取引明細を取得し、その後に加盟店情報を1件ずつ取得していました。

悪い例:

SELECT id, amount, merchant_id
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC
LIMIT 100;

-- ここから100回発行される
SELECT id, name
FROM merchants
WHERE id = ?;

これをJOINでまとめると、1回のSQLで取引明細と加盟店名を取得できます。

SELECT
  t.id,
  t.amount,
  t.occurred_at,
  m.name AS merchant_name
FROM transactions t
JOIN merchants m
  ON t.merchant_id = m.id
WHERE t.account_id = ?
ORDER BY t.occurred_at DESC
LIMIT 100;

この形なら、明細100件に対して加盟店取得SQLを100回出す必要がありません。

ただし、JOINしすぎるとSQLが複雑になり、逆に重くなることもあります。必要な関連情報だけJOINします。

改善策2: IN句でまとめて取得する

JOINではなく、2回のSQLに分けてまとめて取る方法もあります。

-- 1回目: 取引明細を取得
SELECT id, amount, merchant_id
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC
LIMIT 100;

-- アプリ側でmerchant_idを重複排除する
-- merchant_id = (10, 20, 30, 40)

-- 2回目: 必要な加盟店だけまとめて取得
SELECT id, name
FROM merchants
WHERE id IN (10, 20, 30, 40);

アプリ側では、取得した加盟店一覧をMapにして、取引明細へ紐づけます。

List<Transaction> transactions =
    transactionRepository.findLatestByAccountId(accountId, limit);

Set<Long> merchantIds = transactions.stream()
    .map(Transaction::getMerchantId)
    .collect(Collectors.toSet());

Map<Long, Merchant> merchantMap =
    merchantRepository.findByIdIn(merchantIds).stream()
        .collect(Collectors.toMap(Merchant::getId, Function.identity()));

List<TransactionDto> dtoList = transactions.stream()
    .map(tx -> toDto(tx, merchantMap.get(tx.getMerchantId())))
    .toList();

SQL回数は、明細件数に関係なく基本的に2回です。

悪い例:
  明細100件 -> SQL 101回

IN句でまとめ取得:
  明細100件 -> SQL 2回

改善策3: ORMのfetch joinを使う

JPAなどのORMを使っている場合、関連エンティティの遅延ロードでN+1が起きることがあります。

イメージ:

List<Transaction> transactions =
    transactionRepository.findByAccountId(accountId);

for (Transaction tx : transactions) {
    // ここでtx.getMerchant()を呼ぶたびにSQLが出る可能性がある
    String merchantName = tx.getMerchant().getName();
}

fetch joinを使うと、関連情報を最初からまとめて取得できます。

@Query("""
    SELECT t
    FROM Transaction t
    JOIN FETCH t.merchant
    WHERE t.accountId = :accountId
    ORDER BY t.occurredAt DESC
    """)
List<Transaction> findWithMerchantByAccountId(Long accountId);

ただし、fetch joinも万能ではありません。1対多の関連を広くfetchすると、結果件数が膨らむことがあります。面接では「JOINやfetch joinでまとめる。ただし関連の多さと取得件数は確認する」と言えれば十分です。

改善策4: DTOを絞る

そもそも一覧APIで加盟店の全情報が必要ないなら、必要な項目だけ返す設計にします。

SELECT
  t.id,
  t.amount,
  t.occurred_at,
  m.name AS merchant_name
FROM transactions t
JOIN merchants m
  ON t.merchant_id = m.id
WHERE t.account_id = ?
ORDER BY t.occurred_at DESC
LIMIT 100;

一覧では merchant_name だけ返し、加盟店詳細は詳細APIで取得する、という分け方です。

改善策5: ページングする

N+1を解消しても、1回で1万件返すAPIは重くなります。ページングで1回に返す件数を制限します。

SELECT
  t.id,
  t.amount,
  t.occurred_at,
  m.name AS merchant_name
FROM transactions t
JOIN merchants m
  ON t.merchant_id = m.id
WHERE t.account_id = ?
ORDER BY t.occurred_at DESC
LIMIT 100;

ページングの詳しい説明は、2-11. ページングとは何か を参照してください。

面接での言い方:

N+1は、一覧を1回取得したあと、各レコードごとに追加SQLを発行してしまう問題です。件数が増えるほどSQL回数が増えるため、SQLログやトレースで1リクエストあたりのSQL回数を確認します。改善策としては、JOINやIN句によるまとめ取得、fetch設計、ページングを検討します。

2-2. インデックス不足とは何か

インデックスは、DBが目的の行を探しやすくするための索引です。

インデックスがない場合、DBはテーブル全体を見に行くことがあります。これをフルスキャンと呼びます。

SELECT *
FROM transactions
WHERE account_id = 'A001'
ORDER BY occurred_at DESC
LIMIT 50;

このSQLで account_id や occurred_at に適切なインデックスがないと、DBは大量の取引明細から対象口座のデータを探し、さらに並べ替える必要があります。データが少ないうちは問題にならなくても、明細が増えると遅くなります。

認識としては、ほぼ次で合っています。

インデックスなし:
  テーブルの行を順番に見る
  条件に合う行を探す
  必要なら並び替える
  件数が増えるほど遅くなりやすい

インデックスあり:
  索引を使って対象範囲へ早く到達する
  条件に合う行だけを効率よく読む
  インデックス順を使えれば並び替えも軽くなる

ただし、インデックスを貼ったら必ず使われるわけではありません。DBのオプティマイザが、テーブルの統計情報や条件を見て「このインデックスを使う方が速そう」と判断した場合に使われます。だからEXPLAINで確認します。

インデックスの中身は何か

MySQL/InnoDBで一般的に使われるインデックスは、B-tree系、より正確にはB+Tree系の構造として説明されることが多いです。

面接では、深くアルゴリズムを説明する必要は薄いですが、イメージとしては「検索しやすいようにキーが並んだ索引」です。

たとえば account_id にインデックスがあるとします。

idx_account_id

A001 -> 該当行の場所
A002 -> 該当行の場所
A003 -> 該当行の場所
...

DBはテーブル全体を上から順番に見るのではなく、インデックスをたどって account_id = 'A001' の範囲へ移動できます。

本に例えると、本文を1ページ目から全部読むのではなく、巻末索引でキーワードを探して該当ページへ行くイメージです。

重要なのは、インデックスは「値が並んだ構造」なので、等価検索だけでなく範囲検索や並び替えにも効く場合があることです。

WHERE account_id = ?
WHERE occurred_at >= ? AND occurred_at < ?
ORDER BY occurred_at DESC

このような条件で、適切なインデックスがあれば読み込む行数や並び替えコストを減らせます。

2-3. どういうインデックスを貼るのか

面接で大事なのは「検索条件、並び替え、利用頻度を見て判断する」と言えることです。

基本方針:

  • WHERE でよく使う列を候補にする
  • JOIN の結合条件に使う列を候補にする
  • ORDER BY と組み合わせて使う列を候補にする
  • 選択性が高い列を優先する
  • 複合インデックスでは左から順に効くことを意識する
  • 更新が多いテーブルでは、貼りすぎに注意する

選択性とは、値のばらつきです。

選択性が高い:
  user_id, account_id, transaction_id
  -> 値で絞ると対象行がかなり減る

選択性が低い:
  gender, statusが2種類だけ、deleted_flag
  -> 値で絞っても対象行があまり減らない

たとえば取引明細APIなら、次のような検索があり得ます。

SELECT id, amount, occurred_at, description
FROM transactions
WHERE account_id = ?
  AND occurred_at >= ?
  AND occurred_at < ?
ORDER BY occurred_at DESC
LIMIT 50;

この場合の候補:

CREATE INDEX idx_transactions_account_occurred
ON transactions (account_id, occurred_at);

理由:

  • account_id で対象口座に絞る
  • occurred_at で期間条件と並び替えに使う
  • LIMIT 50 で必要な件数だけ取る

複合インデックスはどう効くか

複合インデックスは、複数列を組み合わせたインデックスです。

CREATE INDEX idx_transactions_account_occurred
ON transactions (account_id, occurred_at);

これは、ざっくり言うと次の順で並んだ索引です。

account_id で並ぶ
  同じ account_id の中で occurred_at で並ぶ

イメージ:

(A001, 2026-06-01)
(A001, 2026-06-02)
(A001, 2026-06-03)
(A002, 2026-06-01)
(A002, 2026-06-02)

そのため、次のSQLには効きやすいです。

SELECT id, amount, occurred_at
FROM transactions
WHERE account_id = ?
  AND occurred_at >= ?
  AND occurred_at < ?
ORDER BY occurred_at DESC
LIMIT 50;

理由は、まず account_id で対象口座の範囲に移動し、その中で occurred_at の範囲を見られるからです。

一方で、同じインデックスでも次のSQLには効きにくい場合があります。

SELECT id, amount, occurred_at
FROM transactions
WHERE occurred_at >= ?
  AND occurred_at < ?
ORDER BY occurred_at DESC;

idx_transactions_account_occurred (account_id, occurred_at) は、最初に account_id で並んでいます。account_id を指定せずに occurred_at だけで検索したい場合、先頭列を使えていないため効きにくい、という考え方です。

これを「複合インデックスは左から効く」と表現します。

インデックス: (account_id, occurred_at)

効きやすい:
  WHERE account_id = ?
  WHERE account_id = ? AND occurred_at >= ?

効きにくい:
  WHERE occurred_at >= ?

インデックスとORDER BY

インデックスの並び順を使える場合、DB側のソート処理を減らせることがあります。

SELECT id, amount, occurred_at
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC
LIMIT 50;

(account_id, occurred_at) のインデックスがあれば、DBは対象口座の明細を occurred_at 順に読みやすくなります。これにより、全件を集めてから並び替えるより軽くなる可能性があります。

ただし、複雑な条件や関数を使うとインデックスが効きにくくなる場合があります。

効きにくい例:

WHERE DATE(occurred_at) = '2026-06-01'

列に関数をかけると、インデックスをそのまま使いにくくなることがあります。次のように範囲条件にする方が一般的です。

WHERE occurred_at >= '2026-06-01 00:00:00'
  AND occurred_at <  '2026-06-02 00:00:00'

ただし、インデックスは貼れば貼るほどよいわけではありません。

インデックスのデメリット:

  • INSERT/UPDATE/DELETEが遅くなる場合がある
  • ストレージを使う
  • オプティマイザが常に期待どおり使うとは限らない
  • 使われないインデックスが増えると運用が複雑になる

面接での言い方:

インデックスは、WHERE句、JOIN条件、ORDER BYでよく使われる列を見て候補を考えます。ただし貼れば常に良いわけではなく、更新性能やストレージにも影響するため、EXPLAINで実行計画を確認し、実際に使われるか、改善効果があるかを見て判断します。

2-4. SQLの基本構成

SQLの各句の意味を整理します。

SELECT account_id, COUNT(*) AS transaction_count, SUM(amount) AS total_amount
FROM transactions
WHERE occurred_at >= '2026-06-01'
  AND occurred_at < '2026-07-01'
GROUP BY account_id
HAVING SUM(amount) >= 100000
ORDER BY total_amount DESC
LIMIT 100;
句意味
SELECT返す列を指定する
FROM対象テーブルを指定する
WHERE行を絞り込む
GROUP BY指定した単位で集約する
HAVING集約後の結果を絞り込む
ORDER BY並び替える
LIMIT返す件数を制限する

性能面では、特に WHERE、JOIN、GROUP BY、ORDER BY、LIMIT を見ます。

2-5. JOINとは何か

JOINは、複数テーブルを結合して必要な情報を取得するためのものです。

例:

SELECT t.id, t.amount, m.name AS merchant_name
FROM transactions t
JOIN merchants m
  ON t.merchant_id = m.id
WHERE t.account_id = ?
ORDER BY t.occurred_at DESC
LIMIT 50;

このSQLは、取引明細 transactions と加盟店 merchants を結合しています。

JOINが重くなる原因:

  • 結合対象の件数が多い
  • JOIN条件の列にインデックスがない
  • WHEREで先に絞れていない
  • 不要なテーブルまでJOINしている
  • 1対多のJOINで結果件数が膨らんでいる

確認方法:

  • EXPLAINでJOIN順を見る
  • JOIN条件の列にインデックスがあるか見る
  • 結合前後の件数を確認する
  • 本当にそのJOINがAPIレスポンスに必要か見る

2-6. ORDER BYとは何か

ORDER BYは並び替えです。

SELECT *
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC
LIMIT 50;

並び替え対象が少なければ問題になりにくいですが、大量データを並び替えると重くなります。インデックスが使えない場合、DB側でソート処理が必要になります。

ORDER BYが重くなる原因:

  • WHEREで絞る前の対象件数が多い
  • ORDER BY列に適切なインデックスがない
  • 複数列ソートが複雑
  • ORDER BY しているのに LIMIT がない
  • ソート対象の行やカラムが大きい

改善策:

  • WHEREで先に絞る
  • 検索条件とソート条件に合う複合インデックスを検討する
  • LIMITで返す件数を制限する
  • 不要なソートをやめる

2-7. GROUP BYとは何か

GROUP BYは、データを指定した単位でまとめる処理です。

SELECT account_id, SUM(amount)
FROM transactions
WHERE occurred_at >= ?
  AND occurred_at < ?
GROUP BY account_id;

これは、口座ごとに金額を集計しています。

GROUP BYが重くなる原因:

  • 集計対象の件数が多い
  • GROUP BY列にインデックスがない
  • 集計処理をAPIリクエストのたびに実行している
  • 集計結果が大きい
  • WHEREで十分に絞れていない

改善策:

  • WHEREで期間や対象を絞る
  • 必要なら集計済みテーブルやキャッシュを検討する
  • バッチで事前集計する
  • APIで毎回重い集計をしない設計にする

金融サービスでは、残高や明細など正確性が重要なデータは、安易なキャッシュや事前集計に注意が必要です。速度のために整合性を崩すと本末転倒です。

2-8. EXPLAINで何を見るか

MySQLでは、EXPLAINでSQLの実行計画を確認できます。

見る観点:

  • 想定したインデックスが使われているか
  • フルスキャンになっていないか
  • JOIN順が不自然でないか
  • 読み込む行数が多すぎないか
  • Using filesort や Using temporary が出ていないか

ざっくりした見方:

項目見ること
key実際に使われたインデックス
rows読みそうな行数の見積もり
typeアクセス方法。ALLは全表走査の可能性
Extrafilesort、temporaryなど追加処理

面接での言い方:

EXPLAINでは、想定したインデックスが使われているか、読み込み行数が多すぎないか、フルスキャンやfilesortになっていないかを見ます。実行計画と実測を見て、インデックス追加やSQL見直しの効果を判断します。

2-9. ロックとトランザクション

DBは、データの整合性を守るためにロックを使います。たとえば同じ口座残高を複数処理が同時に更新すると不整合が起きるため、更新中の行を他の処理が勝手に変えないようにします。

ロックが問題になる例:

リクエストA:
  口座Aの残高行を更新
  長い処理をしている
  トランザクションをまだ閉じない

リクエストB:
  同じ口座Aの残高行を更新したい
  Aのロック解放待ち
  APIが遅くなる

ロック待ちが起きやすい実装:

  • トランザクション内で外部APIを呼んでいる
  • トランザクション範囲が広すぎる
  • 大量更新を1トランザクションで実行している
  • 同じ行や範囲に更新が集中している

改善策:

  • トランザクション範囲を必要最小限にする
  • 外部API呼び出しをトランザクション外に出せないか検討する
  • 更新順序を統一する
  • 長時間処理を分割する
  • ロック待ちをメトリクスやDB情報で確認する

金融サービスでは、ロックを単純に外せばよいわけではありません。整合性を守りながら、トランザクションの範囲を適切に小さくするのが重要です。

2-10. DBコネクションプール

アプリはDBへ接続するためにコネクションを使います。毎回新規接続すると重いので、通常はコネクションプールを使います。

HikariCPのようなプールでは、ざっくり次を見ます。

  • active: 使用中の接続
  • idle: 空いている接続
  • pending: 接続待ちのリクエスト
  • max pool size: 最大接続数

問題例:

max pool size = 20
active = 20
idle = 0
pending = 50

この状態は、DB接続が全部使われていて、後続リクエストが接続待ちになっています。

ただし、プールサイズを増やすだけが正解ではありません。DB側が受けられる接続数やCPUには限界があります。遅いSQLや長いトランザクションで接続を長く保持しているなら、根本原因を直す必要があります。

面接での言い方:

DBコネクションプールが枯渇している場合、単純にプールを増やすだけでなく、なぜ接続を長く保持しているかを見ます。遅いSQL、長いトランザクション、外部API待ちをトランザクション内で行っていないかを確認します。

2-11. ページングとは何か

ページングは、一覧データを一度に全部返さず、一定件数ずつ返すことです。

たとえば取引明細が10,000件あるユーザーに対して、1回のAPIで10,000件すべて返すと重くなります。

悪い例:
  取引明細10,000件を一度に取得
  DB負荷が高い
  JSON生成が重い
  レスポンスサイズが大きい
  画面側も扱いづらい

ページングする場合:

1回目:
  最新100件だけ返す

2回目:
  次の100件を返す

3回目:
  さらに次の100件を返す

代表的な方法は、LIMIT/OFFSET 方式と、カーソル方式です。

LIMIT/OFFSET方式

SELECT id, amount, occurred_at
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC
LIMIT 100 OFFSET 0;

次のページ:

SELECT id, amount, occurred_at
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC
LIMIT 100 OFFSET 100;

分かりやすい一方で、後ろのページに行くほどスキップする件数が増え、重くなることがあります。

OFFSET 0:
  最初から100件

OFFSET 10000:
  10,000件分を飛ばしてから100件
  -> 重くなりやすい

カーソル方式

前回取得した最後の値を条件にして、次のページを取得する方法です。

-- 1ページ目
SELECT id, amount, occurred_at
FROM transactions
WHERE account_id = ?
ORDER BY occurred_at DESC, id DESC
LIMIT 100;

1ページ目の最後が occurred_at = '2026-06-10 10:00:00'、id = 5000 だった場合:

-- 2ページ目
SELECT id, amount, occurred_at
FROM transactions
WHERE account_id = ?
  AND (
    occurred_at < '2026-06-10 10:00:00'
    OR (occurred_at = '2026-06-10 10:00:00' AND id < 5000)
  )
ORDER BY occurred_at DESC, id DESC
LIMIT 100;

カーソル方式は、深いページでも比較的安定しやすいです。明細やタイムラインのように「次の100件」を読むUIではよく使われます。

ページングで注意すること:

  • 並び順を安定させる
  • ORDER BY occurred_at DESC, id DESC のように同時刻の順序も決める
  • 1ページあたりの上限を決める
  • インデックスとセットで考える
  • 金融明細では、抜けや重複がないように条件を設計する

面接での言い方:

ページングは、一覧を一度に全件返さず、一定件数ずつ返すことです。大量データの一覧APIでは、DB負荷、レスポンスサイズ、JSON生成を抑えるために重要です。単純なLIMIT/OFFSETは分かりやすい一方、深いページで重くなるため、明細のようなデータではカーソル方式も検討します。

3. アプリケーション処理が遅い場合の具体例

DB以外でも、アプリケーション側の処理がAPI遅延の原因になります。

3-1. ループ内でDBや外部APIを呼ぶ

コード上では自然に見えても、件数が増えると急に遅くなる典型です。

悪いイメージ:

for (Transaction tx : transactions) {
    Merchant merchant = merchantRepository.findById(tx.getMerchantId());
    dtoList.add(toDto(tx, merchant));
}

取引が100件なら、加盟店取得SQLが100回出ます。外部APIなら100回HTTP通信が発生します。

確認方法:

  • SQLログで同じSQLが何度も出ていないか
  • 外部API呼び出し回数がデータ件数に比例していないか
  • 1件、100件、1,000件で処理時間が比例して増えていないか

改善策:

  • まとめて取得する
  • JOINやIN句を使う
  • 外部APIなら一括取得APIがないか確認する
  • キャッシュ可能なマスタ情報はキャッシュする
  • 返却項目を減らす

ソースコード側ではどう直すか

ソースコード側でも直せます。むしろ、RepositoryやSQLだけでなく、Service層の組み立て方を変えることが多いです。

悪い例:

public List<TransactionDto> getTransactions(Long accountId) {
    List<Transaction> transactions =
        transactionRepository.findLatestByAccountId(accountId);

    List<TransactionDto> result = new ArrayList<>();

    for (Transaction tx : transactions) {
        Merchant merchant = merchantRepository.findById(tx.getMerchantId());
        result.add(toDto(tx, merchant));
    }

    return result;
}

この実装は、取引明細の件数だけ merchantRepository.findById() が呼ばれます。

改善例:

public List<TransactionDto> getTransactions(Long accountId) {
    List<Transaction> transactions =
        transactionRepository.findLatestByAccountId(accountId, 100);

    Set<Long> merchantIds = transactions.stream()
        .map(Transaction::getMerchantId)
        .collect(Collectors.toSet());

    Map<Long, Merchant> merchantMap =
        merchantRepository.findByIdIn(merchantIds).stream()
            .collect(Collectors.toMap(Merchant::getId, Function.identity()));

    return transactions.stream()
        .map(tx -> toDto(tx, merchantMap.get(tx.getMerchantId())))
        .toList();
}

この改善では、アプリ側で次のことをしています。

  • まず取引明細を取得する
  • 必要な merchant_id を集める
  • 加盟店情報をまとめて取得する
  • Map化してメモリ上で紐づける

つまり、SQLだけではなく、アプリ側の処理順序を変えています。

外部APIの場合も同じ考え方です。

悪い例:

for (Transaction tx : transactions) {
    Category category = categoryClient.getCategory(tx.getCategoryId());
    result.add(toDto(tx, category));
}

改善候補:

Set<Long> categoryIds = transactions.stream()
    .map(Transaction::getCategoryId)
    .collect(Collectors.toSet());

Map<Long, Category> categoryMap =
    categoryClient.getCategories(categoryIds);

return transactions.stream()
    .map(tx -> toDto(tx, categoryMap.get(tx.getCategoryId())))
    .toList();

一括取得APIがない場合は、次のような対策を検討します。

  • 本当に外部APIから毎回取る必要があるか確認する
  • キャッシュできるマスタ情報ならキャッシュする
  • 返却項目を減らす
  • 並列化する場合は、相手先に負荷をかけすぎないよう上限を設ける
  • API仕様として一括取得を追加できないか検討する

面接での言い方:

ループ内でDBや外部APIを呼んでいる場合、RepositoryやSQLだけでなく、Service層の組み立て方を変えます。必要なIDを先に集め、IN句や一括APIでまとめて取得し、Map化してDTOへ紐づける形にすると、件数に比例した呼び出し回数を抑えられます。

3-2. JSONシリアライズ・デシリアライズが重い

シリアライズは、Java/KotlinのオブジェクトをJSON文字列に変換する処理です。デシリアライズは、JSONをオブジェクトに戻す処理です。

APIでは必ず発生します。

DB結果
  -> Entity
  -> DTO
  -> JSON
  -> HTTPレスポンス

重くなる原因:

  • レスポンス件数が多すぎる
  • DTOのネストが深い
  • 不要な項目まで返している
  • 大きい文字列や配列を返している
  • 日付や金額の変換処理が大量にある

例:

{
  "transactions": [
    {
      "id": "tx001",
      "amount": 1000,
      "merchant": {
        "id": "m001",
        "name": "sample",
        "category": {
          "id": "c001",
          "name": "food"
        }
      },
      "rawDetail": "very large text..."
    }
  ]
}

このように深いネストや大きい項目が多いと、DBは速くてもレスポンス生成が遅くなることがあります。

確認方法:

  • レスポンスサイズを見る
  • DTO変換前後の処理時間をログに出す
  • APMでController/Service/Repository以外の時間を見る
  • CPU使用率が高いか見る

改善策:

  • 必要な項目だけ返す
  • 一覧APIと詳細APIを分ける
  • ページングする
  • DTOのネストを浅くする
  • 巨大項目は必要時だけ取得する
  • 圧縮を検討する

面接での言い方:

DBが速いのにAPIが遅い場合、DTO変換やJSONシリアライズ、レスポンスサイズも見ます。一覧APIで詳細情報まで返している場合は、一覧と詳細を分ける、必要項目に絞る、ページングするなどを検討します。

3-3. 同期処理で重い処理を待っている

同期処理は、処理が終わるまでAPIが待つ形です。

例:

APIリクエスト
  -> DB更新
  -> 外部API呼び出し
  -> メール送信
  -> PDF生成
  -> レスポンス

この場合、メール送信やPDF生成が遅いとAPI全体が遅くなります。

改善策:

  • レスポンスに不要な処理は非同期化する
  • キューに積んで後続処理にする
  • 処理状態を返し、あとで確認する
  • 外部API呼び出しのタイムアウトを適切にする

ただし、金融取引の確定処理など、同期的に完了確認が必要なものは安易に非同期化できません。業務要件と整合性を確認する必要があります。

3-4. アプリ側で大量集計している

DBから大量データを取り、アプリ側で集計していると遅くなります。

悪い例:

DBから10万件取得
  -> Java側で口座ごとに集計
  -> JSONで返却

改善候補:

  • DB側で集計する
  • 事前集計テーブルを作る
  • 検索条件を絞る
  • ページングする
  • API要件を見直す

ただし、DB側でGROUP BYすれば常に速いわけではありません。集計対象が大きければDB負荷が上がります。

DB側集計とアプリ側集計の違い

DB側集計は、SQLの GROUP BY や SUM、COUNT などでDBに集計させる方法です。

SELECT account_id, SUM(amount) AS total_amount
FROM transactions
WHERE occurred_at >= ?
  AND occurred_at < ?
GROUP BY account_id;

アプリ側集計は、DBからデータを取得して、Java/Kotlin側で集計する方法です。

Map<Long, BigDecimal> totalByAccount = transactions.stream()
    .collect(Collectors.groupingBy(
        Transaction::getAccountId,
        Collectors.reducing(
            BigDecimal.ZERO,
            Transaction::getAmount,
            BigDecimal::add
        )
    ));

違い:

観点DB側集計アプリ側集計
データ転送量少なくしやすい大量データをアプリへ運びがち
DB負荷上がる相対的に下がる場合もある
アプリ負荷下がるCPU/メモリを使う
実装の単純さSQL次第複雑な業務ロジックを書きやすい
整合性DB内で一貫して扱いやすい取得後のデータ扱いに注意

DB側集計が向いているケース:

  • 集計対象が大量で、結果だけ返せばよい
  • COUNT、SUM、MAX、MIN のような単純集計
  • DBで絞り込み条件を効かせられる
  • アプリへ大量データを運びたくない

例:

SELECT category_id, COUNT(*) AS transaction_count
FROM transactions
WHERE account_id = ?
  AND occurred_at >= ?
  AND occurred_at < ?
GROUP BY category_id;

アプリ側集計が向いているケース:

  • すでに必要なデータを取得済みで、追加集計が軽い
  • 集計ロジックが複雑でSQLに押し込むと読みにくい
  • 外部API結果やアプリ内ロジックと組み合わせる必要がある
  • 対象件数が少ない

ただし、10万件、100万件をアプリに持ってきて集計するのは避けるべきです。DB、アプリ、ネットワークのどこに負荷を寄せるかを考えます。

面接での言い方:

単純な件数や合計で、集計対象が大きい場合はDB側でGROUP BYする方が、アプリへのデータ転送を減らせます。一方で、対象件数が少ない場合や、複雑な業務ロジック・外部API結果と組み合わせる場合はアプリ側集計もあり得ます。どちらが良いかは、件数、DB負荷、アプリ負荷、保守性で判断します。

3-5. ログ出力が過剰

ログも出しすぎると遅くなります。

問題例:

  • 大量ループ内でログを出す
  • リクエスト/レスポンス全文を毎回出す
  • 巨大JSONをログに出す
  • 同期的なログ出力でI/Oが詰まる

金融では監査ログが重要ですが、監査に必要なログと、デバッグログを分けて考える必要があります。

面接での言い方:

ログは調査や監査に重要ですが、大量出力や巨大JSONの出力は性能に影響します。必要な監査ログは残しつつ、デバッグログの粒度、出力条件、マスキング、非同期化を確認します。

3-6. GCやメモリが原因

Java/Kotlinでは、不要になったオブジェクトをGCが回収します。大量のオブジェクトを作るとGCが増え、場合によっては一時的に処理が止まるように見えます。

起きやすい原因:

  • 大量データを一度にメモリへ載せる
  • 巨大なListやMapを作る
  • DTO変換で大量の一時オブジェクトを作る
  • 大きいJSONを組み立てる

確認方法:

  • ヒープ使用量
  • GC回数
  • GC pause time
  • p99の悪化タイミングとGCのタイミング

改善策:

  • ページングする
  • ストリーミング処理を検討する
  • 不要な全件取得をやめる
  • レスポンス項目を減らす
  • オブジェクト生成が多い処理を見直す

4. 外部API・マイクロサービス連携

4-1. タイムアウト

タイムアウトは、相手の応答をどこまで待つかの設定です。

長すぎると、相手が遅いときに自分のAPIも長く待たされます。短すぎると、本当は成功できた処理を失敗扱いにする可能性があります。

見るポイント:

  • 接続タイムアウト
  • 読み取りタイムアウト
  • API全体のタイムアウト
  • DBクエリタイムアウト
  • リトライを含めた最大待ち時間

面接での言い方:

外部APIが遅い場合、タイムアウト値が長すぎるとアプリ側スレッドが長時間占有されます。逆に短すぎると成功できる処理まで失敗するため、相手APIの通常応答時間と業務要件を見て設定します。

4-2. リトライ、バックオフ、ジッター

リトライは、失敗した処理を再実行することです。

一時的なネットワークエラーには有効ですが、何も考えずにリトライすると障害を悪化させます。

悪い例:

外部APIが遅い
  -> 全リクエストが即リトライ
  -> 外部APIへの負荷が2倍、3倍になる
  -> さらに遅くなる

バックオフは、リトライ間隔を少しずつ長くすることです。

1回目失敗 -> 100ms待つ
2回目失敗 -> 500ms待つ
3回目失敗 -> 1s待つ

ジッターは、リトライのタイミングを少しランダムにずらすことです。全クライアントが同時に再実行して負荷が集中するのを防ぎます。

面接での言い方:

リトライは一時障害には有効ですが、同時に集中すると下流サービスへさらに負荷をかけます。そのため、最大回数、タイムアウト、バックオフ、ジッターを設計します。金融APIでは、リトライによる二重処理を防ぐため、冪等性も合わせて考えます。

4-3. 冪等性

冪等性は、同じリクエストを複数回実行しても結果が変わらない性質です。

金融APIでは特に重要です。

問題例:

振込APIを呼ぶ
  -> サーバ側では振込処理が成功
  -> しかしクライアントにはタイムアウト
  -> クライアントが再リトライ
  -> 二重振込になる可能性

対策例:

  • Idempotency-Keyを付ける
  • 取引IDを先に発行する
  • 処理状態をDBで管理する
  • 同じキーの再リクエストには同じ結果を返す
  • 状態が不明な場合に照会APIで確認できるようにする

面接での言い方:

金融APIでは、タイムアウト時に処理が成功したのか失敗したのか分からない状態が起きます。そのため、リトライ前提で冪等性キーや取引状態管理を設計し、同じ処理が二重に実行されないようにします。

4-4. サーキットブレーカーと縮退

サーキットブレーカーは、下流サービスが失敗し続けているときに、一定時間呼び出しを止める考え方です。

目的:

  • 下流サービスをさらに追い込まない
  • 自サービスのスレッドを無駄に占有しない
  • 早く失敗を返す
  • 代替レスポンスや機能制限に切り替える

縮退は、一部機能を制限してサービス全体を守ることです。

例:

  • 明細のリアルタイム更新は止めるが、過去明細は表示する
  • 外部連携先の情報は一時的に表示しない
  • 非重要な推薦・集計情報を返さない

金融サービスでは、縮退してよい機能と、正確性のために止めるべき機能を分ける必要があります。

5. 改善策の考え方を具体化する

改善策は、短期対応と恒久対応に分けます。

5-1. 短期対応

短期対応は、まず本番影響を下げるための対応です。

状況短期対応注意点
リリース直後に悪化切り戻しデータ変更がある場合は戻せるか確認
CPU/メモリ不足スケールアウト根本原因が重いSQLなら効果は限定的
特定APIだけ高負荷レート制限、一時停止影響ユーザーや業務影響を確認
外部API障害タイムアウト短縮、縮退二重処理や状態不明に注意
重いバッチと重なったバッチ停止、時間変更データ反映遅延の影響を確認
DB負荷が高い重い処理の停止、参照系制限金融業務に必要な処理を止めない

面接での言い方:

本番影響が大きい場合は、原因調査と並行して影響を下げます。直前リリースなら切り戻し、リソース逼迫なら一時的なスケール、外部APIならタイムアウトや縮退を検討します。ただし金融APIでは、整合性や二重処理に影響しないことを確認します。

5-2. 恒久対応

恒久対応は、原因に対して構造的に直す対応です。

DB起因:

  • N+1をJOIN/IN句/fetch設計で解消する
  • 検索条件に合うインデックスを追加する
  • 取得件数を制限する
  • 不要なカラムを取らない
  • 重い集計は事前集計やバッチ化を検討する
  • トランザクション範囲を小さくする

アプリ起因:

  • 一覧APIと詳細APIを分ける
  • DTOを軽くする
  • ループ内処理をまとめる
  • 不要な同期処理を非同期化する
  • キャッシュ可能なマスタ情報をキャッシュする
  • 過剰ログを見直す

外部API起因:

  • タイムアウトを適切にする
  • リトライ回数を制限する
  • バックオフとジッターを入れる
  • サーキットブレーカーを検討する
  • 冪等性キーと状態管理を入れる

インフラ起因:

  • オートスケール条件を見直す
  • CPU/メモリを増やす
  • DBスペックや接続数を見直す
  • メトリクス、ログ、トレースを追加する

5-3. 検証

改善したら、同じ条件で効果を確認します。

確認観点:

  • p50、p95、p99が改善したか
  • エラー率が増えていないか
  • DB CPU、スロークエリ、読み込み行数が減ったか
  • SQL発行回数が減ったか
  • レスポンスサイズが減ったか
  • 外部API呼び出し回数が減ったか
  • コネクションプール待ちが減ったか
  • 二重処理やデータ不整合が起きていないか

面接での言い方:

対策後は、同じデータ量、同じリクエスト条件、同じ同時実行数で比較します。平均だけでなくp95、p99、エラー率、DB負荷、外部API呼び出し回数を見ます。金融APIでは、性能改善によって二重処理や不整合が起きていないことも確認します。

7. よくある課題パターンの詳細

7-1. データ量が増えて一覧APIが遅い

よくある原因:

  • 以前は1ユーザーあたり100件だったデータが、1万件になった
  • ページングせず全件返している
  • 取引明細ごとに関連情報を取りに行っている
  • ORDER BYのソート対象が多い
  • レスポンスJSONが巨大化している

調査:

  • ユーザーごとのデータ件数を見る
  • SQLの取得件数を見る
  • 1リクエストあたりのSQL回数を見る
  • レスポンスサイズを見る
  • EXPLAINで読み込み行数を見る

改善:

  • ページング
  • LIMIT
  • 必要カラムだけ取得
  • N+1解消
  • 検索条件とソート条件に合うインデックス
  • 一覧APIと詳細APIの分離

7-2. リリース後から急に遅い

よくある原因:

  • SQLが変わった
  • 新しいJOINが増えた
  • 外部API呼び出しが追加された
  • DTOに重い項目が追加された
  • キャッシュが効かなくなった
  • ログ出力が増えた
  • 設定値が変わった

調査:

  • リリース差分を見る
  • 遅くなったAPIのコード差分を見る
  • SQL発行回数を旧版と比較する
  • レスポンスサイズを比較する
  • 外部API呼び出し回数を比較する

改善:

  • 影響が大きければ切り戻し
  • 問題処理を修正
  • 負荷試験ケースに追加
  • リリース前の監視観点に追加

7-3. p99だけ遅い

よくある原因:

  • 一部ユーザーだけデータ件数が多い
  • 一部の口座や取引だけロック待ちがある
  • 外部APIがたまに遅い
  • GCがたまに発生する
  • コネクションプールが瞬間的に枯渇する
  • キャッシュミス時だけ遅い

調査:

  • 遅いリクエストの条件を集める
  • 対象ユーザー、データ件数、時間帯を見る
  • GCやロック待ちのタイミングと突き合わせる
  • 外部APIのp95/p99を見る

改善:

  • 重いユーザー条件への対策
  • ページングや上限
  • ロック範囲見直し
  • キャッシュミス時の処理改善
  • 外部APIのタイムアウト/リトライ見直し

7-4. 外部APIが遅い

よくある原因:

  • 相手先サービスが遅い
  • ネットワークが不安定
  • タイムアウトが長すぎる
  • リトライが集中している
  • 外部APIを同期的に待っている

調査:

  • 外部APIごとの応答時間
  • タイムアウト数
  • リトライ回数
  • どのAPI呼び出しで待っているか
  • 自社APIのスレッドや接続プールが詰まっていないか

改善:

  • タイムアウト設計
  • バックオフとジッター
  • サーキットブレーカー
  • 縮退
  • 非同期化
  • 冪等性キーと状態管理

7-5. DBコネクションプールが枯渇している

よくある原因:

  • 遅いSQLが接続を長く占有している
  • トランザクションが長い
  • 外部API待ちをトランザクション内でしている
  • 負荷に対してプールサイズが小さい
  • DB側の最大接続数が足りない

調査:

  • active、idle、pending
  • 遅いSQL
  • トランザクション時間
  • DB側の接続数
  • APIの同時実行数

改善:

  • 遅いSQLを直す
  • トランザクション範囲を小さくする
  • 外部API呼び出しをトランザクション外へ出す
  • プールサイズとDB最大接続数を見直す
