X由来 / System Design S
XQ001

TinyURL / URL短縮サービスを設計してください。

面接官の意図：短い問題に見えて、ID生成、DB設計、リダイレクト、キャッシュ、ホットキー、悪用対策まで広く見られる。

回答例・補足を見る

30秒回答

URL短縮サービスは、長いURLを短いキーに対応付け、短縮URLへのアクセス時に元URLへリダイレクトするサービスです。基本構成は、短縮キー生成、URLマッピングDB、リダイレクトAPI、キャッシュ、分析ログです。面接では、まず読み取りが多いのか、書き込みが多いのか、短縮URLの有効期限やカスタムaliasの有無を確認し、キー衝突、DBインデックス、キャッシュ、悪性URL対策まで説明します。

深掘りされたときに足す観点

  • API：POST /shorten で長いURLを受け取り、短縮キーを返す。GET /{key} で元URLを取得し、301/302でリダイレクトする。
  • キー生成：DBの連番IDをBase62化する方式、ランダム文字列方式、Snowflake風ID方式などがある。衝突回避と推測困難性を分けて考える。
  • DB：short_code をユニークキーにし、original_url、created_at、expires_at、owner_id などを持つ。
  • キャッシュ：リダイレクトは読み取り頻度が高いので、short_code → original_url をRedisやCDN側でキャッシュする。
  • 運用：不正URL、スパム、削除、期限切れ、アクセス分析ログ、ホットURL対策が必要。

簡易図

Client → GET /abc123 → Redirect Service → Cache/DB → 302 Location: original_url ↓ Analytics Queue → Log Store

理解補足

XではTinyURL/URL Shortenerは初級〜中級の定番システムデザイン問題として頻繁に挙がります。重要なのは「短縮URLを作るだけ」と捉えないことです。面接官は、URLマッピングという単純な要件から、キー設計、読み取り負荷、キャッシュ、リダイレクトステータス、ログ収集、悪用対策まで広げられるかを見ています。

注意点

301は恒久的リダイレクト、302は一時的リダイレクトです。アクセス分析やリンク変更可能性を考えると、短縮URLサービスでは302の方が扱いやすいケースがあります。ここを言えると一段深く見えます。

XQ002

高トラフィックAPI向けRate Limiterを設計してください。

面接官の意図：アルゴリズム暗記ではなく、何を守るための制限か、どこで制限するか、分散環境でどう整合させるかを見る。

回答例・補足を見る

30秒回答

Rate Limiterは、一定期間に許可するリクエスト数を制限し、バックエンドを過負荷・不正利用・ブルートフォース攻撃から守る仕組みです。方式にはFixed Window、Sliding Window、Token Bucket、Leaky Bucketがあります。API GatewayやLoad Balancerの手前で粗く制限し、アプリ側でユーザー単位・API単位の細かい制限を行う設計が現実的です。

深掘りされたときに足す観点

  • Fixed Window：1分100回のように固定区間で数える。簡単だが、境界時刻にリクエストが集中すると実質2倍近く通ることがある。
  • Sliding Window：直近60秒など移動窓で数える。精度は高いが、実装・ストレージ負荷は上がる。Redis Sorted Setで実装しやすい。
  • Token Bucket：時間経過でトークンが補充され、リクエストごとに消費する。平均レートを守りつつ短時間のバーストを許容できる。
  • Leaky Bucket：キューから一定速度で流す。出力レートを平滑化できるが、待ち時間やキューあふれを考える必要がある。
  • 返却：制限超過時は一般に 429 Too Many Requests と Retry-After を返す。

簡易図

Client → API Gateway Rate Limit → App Rate Limit → Business Logic ↓ ↓ coarse limit Redis counters / tokens

理解補足

Xでよく出るRate Limiter問題は、単に「Redisでカウントします」と答えると弱いです。誰を単位にするのか、IP単位かユーザー単位かAPIキー単位か、グローバル制限かテナント制限か、分散インスタンス間でカウンタをどう共有するかまで話す必要があります。AWS API Gatewayでも、リクエストがレート/バースト制限を超えると429を返すようなスロットリングが提供されています。

注意点

完全に正確な分散Rate Limiterは難しいです。面接では、厳密性よりも「多少の誤差を許すのか」「Redis集中でよいのか」「リージョン分散時にローカル制限でよいのか」を明示するとよいです。

XQ003

有名人が5,000万人のフォロワーに投稿したら、フィードをどう配信しますか？

面接官の意図：fan-out on write/read、ホットユーザー、非同期処理、キャッシュ、最終整合性を理解しているかを見る。

回答例・補足を見る

30秒回答

フィードは、ユーザーに表示する投稿一覧・タイムラインです。通常ユーザーは投稿時にフォロワーのフィードへ事前配信するfan-out on writeが有効ですが、有名人のようにフォロワーが極端に多い場合は書き込みが爆発します。そのため、有名人投稿はfan-out on readに寄せる、またはハイブリッド方式で扱います。

深掘りされたときに足す観点

  • fan-out on write：投稿時にフォロワー全員のtimelineに投稿IDを配る。読み取りは速いが、巨大フォロワーを持つユーザーの投稿で書き込みが爆発する。
  • fan-out on read：読み取り時にフォロー中ユーザーの投稿を集めてランキングする。書き込みは軽いが、読み取りが重くなりやすい。
  • ハイブリッド：一般ユーザーはwrite fan-out、有名人はread時合成、または一部キャッシュを使う。
  • 非同期：投稿イベントをQueueに入れ、Fanout Workerが分散処理する。
  • 整合性：フィードは多少遅れても許容されやすく、結果整合性で設計しやすい。

簡易図

Post Service → Post Event → Queue → Fanout Workers → Timeline Store ↑ ↓ Celebrity posts may be merged at read time ← Read API

理解補足

Xでよく見かける「celebrity with 50M followers」系の問題です。狙いは、単純なフィード設計からホットキー/ホットユーザー問題に気づけるかです。全員に即時配信しようとすると、1投稿で5,000万件のtimeline書き込みが発生します。これは非同期化しても巨大なので、ユーザー種別ごとに方式を変えるのが現実的です。

注意点

「全部Kafkaに流せば解決」とは言わない方がいいです。Queueは処理を平滑化できますが、総処理量そのものは消えません。5,000万件の書き込みを本当に行う必要があるのかを先に疑うべきです。

XQ004

1億ユーザー向け通知システムを設計してください。

面接官の意図：通知種別、配信チャネル、優先度、再試行、重複排除、ユーザー設定、外部サービス連携を整理できるかを見る。

回答例・補足を見る

30秒回答

通知システムは、イベントを受け取り、ユーザー設定に応じてPush、Email、SMS、アプリ内通知などに配信する仕組みです。同期処理で直接送るのではなく、イベント受付、通知生成、Queue、チャネル別Worker、外部Provider、配信ログの構成にします。重要なのは、重複送信防止、再試行、優先度、ユーザーの通知設定、Provider障害時の扱いです。

深掘りされたときに足す観点

  • 入力：注文完了、コメント、セキュリティ通知などのイベントを受け取る。
  • Preference：ユーザーがEmailを許可しているか、夜間通知を止めるかなどを確認する。
  • Template：通知文面をテンプレート化し、多言語・変数埋め込みを扱う。
  • Queue：チャネル別・優先度別に分ける。緊急通知と広告通知を同じキューで扱わない。
  • Delivery Log：送信済み、失敗、再試行回数、Providerレスポンスを記録する。

簡易図

Event → Notification Service → Preference / Template → Priority Queue → Channel Workers → APNs/FCM/Email/SMS → Delivery Log / Retry / DLQ

理解補足

Xで「Design a notification system for 100M users」のような問題が頻出です。重要なのは、「通知を送るAPI」ではなく、「通知のライフサイクル」を設計することです。通知は外部Provider依存が強いため、失敗・遅延・重複を前提にします。

注意点

通知は完全なexactly-once配信が難しい領域です。面接では「少なくとも1回送信＋冪等キーで重複抑制」「Provider側の重複仕様」「ユーザー体験として許容できる重複/遅延」を話せると強いです。

XQ005

Amazonの注文管理システムを設計してください。

面接官の意図：注文、在庫、決済、配送、キャンセル、整合性をどう分けるかを見る。

回答例・補足を見る

30秒回答

注文管理は、カートから注文を確定し、在庫引当、決済、配送、キャンセル、返金まで状態遷移を管理するシステムです。Order Serviceを中心に、Inventory、Payment、Shippingと連携します。同期一発で全て成功させるのではなく、注文状態を持ち、失敗時は補償処理を行うSaga的な考え方が重要です。

深掘りされたときに足す観点

  • 注文状態：CREATED → INVENTORY_RESERVED → PAID → SHIPPED → COMPLETED のように状態を明示する。
  • 在庫：注文確定前に在庫を予約する。過剰販売を避けるため、在庫引当は強い整合性寄りにする。
  • 決済：外部決済APIはタイムアウト・二重送信があるため、Idempotency-Keyを使う。
  • 補償：決済成功後に配送作成失敗なら、返金や再試行などの補償が必要。
  • イベント：注文状態変更をイベントとして発行し、通知・分析・配送連携へ流す。

簡易図

Order Service ──reserve──> Inventory │ └──fail: cancel order ├──charge───> Payment │ └──timeout: retry with idempotency key └──ship────> Shipping

理解補足

Xで見かけるSenior/Staff向け注文管理問題は、「CRUD設計」ではありません。複数サービスにまたがる状態遷移と失敗処理を見ています。特に、在庫と決済は業務上の不整合が大きな問題になるため、どこを強い整合性にするか、どこをイベント駆動で遅延許容にするかを切り分けます。

注意点

いきなり「マイクロサービスに分けます」と言うより、まず注文の状態遷移と不整合時の業務影響を説明する方がよいです。技術分割より業務の不変条件が先です。

XQ006

Google Mapsのような地図・位置検索システムをどう設計しますか？

面接官の意図：地理空間データ、近傍検索、タイル、キャッシュ、読み取り負荷を大まかに整理できるかを見る。

回答例・補足を見る

30秒回答

地図系システムは、地図表示、経路検索、近くの店舗検索、位置更新など複数の問題に分かれます。面接では、対象を絞るのが最初です。近傍検索なら緯度経度を空間インデックスに変換し、GeoHashやQuadTree、R-tree系の考え方で候補を絞り、詳細距離計算で並べます。地図表示ならタイルを事前生成し、CDNに乗せて配信します。

深掘りされたときに足す観点

  • 要件分解：地図表示、検索、ルート、リアルタイム位置共有は別問題として扱う。
  • 近傍検索：緯度経度そのものを全件比較せず、空間インデックスで候補を絞る。
  • タイル：地図はズームレベルごとにタイル画像/ベクトルタイルとして分割し、CDNで配信する。
  • 更新頻度：地図データは比較的低頻度更新、ユーザー位置は高頻度更新。ストレージを分ける。
  • キャッシュ：人気エリア・タイル・検索結果はキャッシュ対象になりやすい。

簡易図

Map Display: Client → CDN → Tile Store Nearby Search: Client → Search API → Geo Index → Candidate POIs → Distance Sort

理解補足

XでGoogle Maps設計のような大きい問題が出ることがありますが、全部を設計しようとすると破綻します。最初に「今日は近くの店舗検索に絞ってよいですか」「地図タイル配信に絞りますか」とスコープを確認するのが正解です。

注意点

知らない空間インデックス名を無理に出すより、「全件検索は無理なので、緯度経度をグリッド化して候補範囲を絞る」と説明できれば十分です。

XQ007

Queueが詰まり、DBが遅く、Cacheが古く、ログも足りない。どう調査しますか？

面接官の意図：障害時に闇雲に直さず、影響範囲・時系列・ボトルネック・暫定対処を切り分けられるかを見る。

回答例・補足を見る

30秒回答

まずユーザー影響とSLOを確認し、いつから・どのAPI・どのテナント・どのリージョンで起きているかを切り分けます。次にメトリクスでQueue backlog、DB latency、Cache hit rate、エラー率、CPU/メモリを見ます。暫定対処として、重い機能の制限、ワーカー増強、リトライ停止、キャッシュTTL調整、レート制限を検討します。恒久対策として、ログ・メトリクス・トレース整備、負荷試験、バックプレッシャー設計を入れます。

深掘りされたときに足す観点

  • 影響確認：全ユーザーか、一部APIか、一部テナントか。売上・注文・ログインなど重要機能への影響を優先。
  • 時系列：直前のデプロイ、バッチ、キャンペーン、外部API障害、トラフィック急増を確認する。
  • Queue：流入量、消費量、滞留時間、DLQ件数を見る。
  • DB：スロークエリ、ロック、接続数、インデックス、I/O待ちを見る。
  • Cache：Hit率、stale data、hot key、TTL、再生成負荷を見る。

簡易図

Impact → Timeline → Metrics → Hypothesis → Mitigation → Root Cause → Prevention

理解補足

Xで見かける「全部悪い状態」の障害質問は、正解の技術を聞いているのではなく、調査順序を見ています。良い回答は「まず観測」「次に影響制御」「最後に恒久対策」です。いきなりDBチューニングやサーバー増強に飛ぶと、障害対応経験が浅く見えます。

注意点

ログが足りない場合でも、ALB/API Gateway/DB/Queue/CloudWatchなど周辺のメトリクスから推定できます。「ログがないので分かりません」ではなく、「今ある観測点で絞る」と答えるべきです。

XQ008

なぜ最初からDBをシャーディングしない方がよいのですか？

面接官の意図：スケール手段の副作用を理解し、単純な解決策に飛びつかないかを見る。

回答例・補足を見る

30秒回答

シャーディングは、データを複数DBに分割して水平スケールする方法ですが、最初から導入すると運用・クエリ・トランザクション・再配置が複雑になります。まずはクエリ改善、適切なインデックス、読み取りレプリカ、キャッシュ、アーカイブ、パーティショニングで対応できないかを検討します。シャーディングは、単一DBの限界が明確で、アクセスキーやデータ分布が設計できる段階で導入するのが現実的です。

深掘りされたときに足す観点

  • メリット：データ量・書き込み負荷を複数ノードに分散できる。
  • デメリット：JOIN、集計、トランザクション、バックアップ、再シャーディングが難しくなる。
  • シャードキー：ユーザーID、テナントID、地域など。偏りがあるとホットシャードが発生する。
  • 代替：インデックス、クエリ改善、レプリカ、キャッシュ、アーカイブ、パーティショニング。
  • 判断基準：データ量、書き込みQPS、単一ノード限界、成長予測、運用体制。

簡易図

Before sharding: App → Single Primary DB + Read Replicas + Cache After sharding : App → Shard Router → DB shard 1 / 2 / 3 ...

理解補足

Xでは「shard from day one」は面接での罠としてよく語られます。シニア面接では、派手な分散設計よりも、まず単純な構成でどこまで持つか、どの時点で複雑化するかを説明できる方が評価されやすいです。

注意点

シャーディングは「DBが遅いから入れる魔法」ではありません。遅さの原因がインデックス不足やN+1なら、シャーディングしても本質的には直りません。

XQ009

Cache-Aside、Read-Through、Write-Through、Write-Behindの違いは？

面接官の意図：キャッシュを「速くする箱」ではなく、読み書き経路と整合性の問題として捉えられるかを見る。

回答例・補足を見る

30秒回答

Cache-Asideはアプリがキャッシュを明示的に読む方式で、ミスしたらDBから読み、キャッシュに戻します。Read-Throughはキャッシュ層がDB読み込みまで担います。Write-Throughは書き込み時にDBとキャッシュを同期更新します。Write-Behindは先にキャッシュ/キューへ書き、後でDBへ反映します。読み取り性能、整合性、実装複雑性、障害時の安全性で選びます。

深掘りされたときに足す観点

  • Cache-Aside：最も一般的。アプリがcache miss時のDB読み込みとcache populateを制御する。
  • Read-Through：アプリから見るとキャッシュがデータソースのように振る舞う。キャッシュ製品側の機能に依存。
  • Write-Through：書き込み時にDBとキャッシュを同時更新。読み取りは新しくなりやすいが書き込み遅延が増える。
  • Write-Behind：書き込みを一旦キャッシュ/キューに受け、非同期でDBへ反映。高スループットだがデータ損失対策が必要。
  • TTL：古いデータをいつまで許すか。TTLが短いとDB負荷、長いとstale dataが増える。

簡易図

Cache-Aside Read: App → Cache miss → DB → Cache set → Response Write-Through: App → Cache/DB update together → Response

理解補足

XではRedis、TTL、cache invalidation、read-through/write-throughが頻出です。面接では方式名より「このデータは古くてもよいか」「書き込み後すぐ読まれるか」「キャッシュ削除とDB更新の順序はどうするか」を説明する方が重要です。

注意点

キャッシュは障害要因にもなります。stale data、hot key、cache stampede、DBとの不整合、削除漏れを必ずセットで話してください。

XQ010

Hot Key / Thundering Herd / Cache Stampedeをどう防ぎますか？

面接官の意図：キャッシュ導入後に起きる高負荷パターンを理解しているかを見る。

回答例・補足を見る

30秒回答

Hot Keyは特定キーにアクセスが集中する問題です。Thundering HerdやCache Stampedeは、キャッシュ失効時に大量リクエストが同時にDBへ流れる問題です。対策は、TTLにランダム性を持たせる、単一リクエストだけが再生成するロックを使う、古い値を一時的に返す、事前更新する、キーを分散する、CDNや多層キャッシュを使うことです。

深掘りされたときに足す観点

  • Hot Key：有名人のプロフィール、人気商品の在庫、トップページなど特定キーに集中する。
  • Stampede：人気キーが同時にexpireし、全リクエストがDBへ向かう。
  • TTL jitter：TTLにランダム値を足して、同時失効を避ける。
  • Single Flight：同じキーの再生成を1リクエストにまとめ、他リクエストは待つ/古い値を返す。
  • Stale-while-revalidate：古い値を返しつつ裏で更新する。

簡易図

Cache expires 1000 clients → Cache miss → DB overload ← Stampede Mitigation: lock / stale value / TTL jitter / pre-warm

理解補足

Xのシステム設計系アカウントでは、キャッシュはほぼ必ず登場します。ただし、キャッシュを入れるだけの回答は弱いです。キャッシュが壊れた時、古い時、集中した時、DBに戻った時の設計まで話せると実務感が出ます。

注意点

在庫や残高のような強整合が必要な値は、安易にキャッシュだけで判断しない方がよいです。キャッシュ可能なデータと、最終確認をDBで行うデータを分けて説明してください。

注意点

Cursor方式では、単にIDだけをcursorにするとソート順と矛盾することがあります。created_at desc, id desc のように安定ソートキーを組み合わせるのが実務的です。

XQ012

決済・注文APIでIdempotency Keyをどう使いますか？

面接官の意図：ネットワーク失敗時のリトライと二重実行問題を理解しているかを見る。

回答例・補足を見る

30秒回答

Idempotency Keyは、同じ操作の再送をサーバー側で同一リクエストとして扱うためのキーです。決済や注文作成では、クライアントがUUIDなどの一意キーを送り、サーバーはそのキーと結果を保存します。同じキーの再送が来たら、処理をもう一度実行せず、保存済み結果を返します。これにより、タイムアウト後のリトライで二重決済・二重注文を防ぎます。

深掘りされたときに足す観点

  • 対象：決済、注文作成、送金、在庫予約など「二重実行が危険」な操作。
  • 保存：idempotency_key、request hash、status、response、created_atをDB/Redisに保存する。
  • 同じキー・同じ内容：保存済み結果を返す。
  • 同じキー・異なる内容：クライアントバグの可能性があるため409などで拒否する。
  • 有効期限：キーを永久保存せず、業務に応じたTTLを決める。

簡易図

Client --POST /payments + Idempotency-Key--> API API checks key: not exists → execute payment → save result exists → return saved result

理解補足

StripeのAPIでもIdempotency-Keyは有名です。Stripeは同じidempotency keyの初回結果を保存し、後続リクエストに同じ結果を返す仕組みを説明しています。面接では、ただ「冪等にします」ではなく、キー保存、リクエスト内容比較、同時実行制御まで話す必要があります。

注意点

HTTPのGET/PUTの冪等性と、業務操作のIdempotency Keyは関連しますが同じではありません。POST /payments のような本来非冪等なAPIでも、Idempotency Keyにより「同じ操作の再送」を安全にできます。

注意点

Queueを使う設計で、再試行・DLQ・Backpressureをどう考えますか？

面接官の意図：非同期化を万能視せず、失敗時に安全に処理を収束させられるかを見る。

回答例・補足を見る

30秒回答

Queueは処理を非同期化し、ピーク負荷を平滑化します。ただし、失敗時の再試行、重複処理、順序性、DLQ、滞留監視が必要です。下流が遅いときは無限にQueueへ積むのではなく、Backpressureとして受付制限、優先度制御、リトライ間隔調整、ワーカー数制御を行います。

深掘りされたときに足す観点

  • Retry：一時的障害には指数バックオフで再試行する。すぐ再試行すると障害を悪化させる。
  • DLQ：一定回数失敗したメッセージを隔離し、調査・再投入できるようにする。
  • 冪等性：同じメッセージが複数回処理されても結果が壊れないようにする。
  • Backpressure：下流の処理能力を超えたら、入口で制限・遅延・拒否する。
  • 監視：queue depth、oldest message age、processing rate、DLQ countを見る。

簡易図

Producer → Queue → Worker → DB / External API ↓ Retry Policy → DLQ ↑ Monitoring / Backpressure

理解補足

Xでは「queue is full」という障害シナリオがよく出ます。Queueを入れると同期APIは軽く見えますが、処理が消えるわけではありません。滞留が増えると、ユーザーには遅延や未反映として見えます。

注意点

Queueは「後でやる」仕組みです。後でやることが許されない処理、たとえば在庫の最終確定や残高更新は、同期・強整合寄りにする必要があります。

  • Execution Plan：Index ScanかSeq Scanか、想定行数と実行行数のズレを見る。
  • N+1：一覧取得後に1件ずつ関連データを取りに行っていないか。
  • Lock：更新系でロック待ちが発生していないか。
  • Connection Pool：DB接続枯渇や過剰接続がないか。

簡易図

App latency ↑ ├ API CPU? ├ DB slow query? → EXPLAIN → index/query fix ├ Lock wait? └ Connection pool exhausted?

理解補足

Xでは「missing index can cost more than infrastructure upgrade」のような趣旨の投稿が多いです。これは真実です。DB性能問題は、まずクエリとアクセスパターンを見るべきです。

注意点

インデックスも追加すればよいわけではありません。書き込み性能、ストレージ、更新コスト、選択度、複合インデックスの順序を考える必要があります。

XQ022

Cache Invalidationはなぜ難しいのですか？

面接官の意図：キャッシュ更新とDB更新のタイミング差、不整合、分散環境の難しさを理解しているかを見る。

回答例・補足を見る

30秒回答

Cache Invalidationが難しい理由は、DB更新とキャッシュ削除/更新が別操作になり、途中失敗や競合で古い値が残るからです。代表的には、DB更新後にキャッシュ削除する、キャッシュ更新ではなく削除する、TTLを併用する、イベントで非同期無効化する、バージョンを持たせるなどの対策があります。

深掘りされたときに足す観点

  • DB→Cache削除：DB更新後にキャッシュを消す。次回読み取りでDBから新値を入れる。一般的。
  • Cache更新：DB更新時にキャッシュも更新するが、複雑な集計や複数キーでは漏れやすい。
  • TTL：削除漏れの保険。ただし古い値を一定時間許容する。
  • 非同期無効化：DB更新イベントを流して関連キャッシュを消す。遅延がある。
  • Race：削除と再生成が競合し、古い値を再キャッシュすることがある。

簡易図

Update flow: 1. Update DB 2. Delete cache key 3. Next read repopulates cache Failure between 1 and 2 → stale cache risk

理解補足

Xでcache invalidationは定番です。キャッシュは読み取り性能を上げますが、更新系と組み合わせた瞬間に難しくなります。面接では「TTLを入れます」だけでなく、DB更新順序と失敗時の扱いまで説明してください。

注意点

金額・権限・在庫のように古い値が危険なデータでは、キャッシュ利用範囲を限定し、最終判断はDBや認可基盤で行う方が安全です。

XQ023

Consistent Hashingは何のために使いますか？

面接官の意図：ノード追加・削除時の再配置コストを理解しているかを見る。

回答例・補足を見る

30秒回答

Consistent Hashingは、キャッシュノードやシャードノードを増減するときに、キーの再配置を最小限にするための方式です。通常のhash(key) % NではNが変わると多くのキーの配置先が変わります。Consistent Hashingではハッシュリング上にキーとノードを配置し、ノード増減時に影響範囲を限定します。

深掘りされたときに足す観点

  • 用途：分散キャッシュ、シャーディング、ロード分散。
  • 問題：単純Modulo方式はノード数が変わると大量のキーが移動する。
  • リング：キーのハッシュ値からリング上で次のノードに割り当てる。
  • 仮想ノード：ノードごとの偏りを減らすため、1物理ノードを複数点として配置する。
  • 限界：データの偏り、ホットキー、再配置中の整合性は別途考える必要がある。

簡易図

hash(key) % N: N changes → many keys move Consistent Hash Ring: node added/removed → nearby range moves

理解補足

XのSystem Designロードマップでは、scalabilityやconsistent hashingがよく挙がります。面接で名前だけ出すと危険ですが、「ノード数変更時に再配置を減らすため」と説明できればまず十分です。

注意点

Consistent Hashingはホットキー問題を自動解決しません。1つのキーにアクセスが集中する場合、どのノードに置いてもそこが熱くなります。

XQ024

Batch ProcessingとStream Processingの違いは？

面接官の意図：データ処理を、遅延・整合性・コスト・ユースケースで分けられるかを見る。

回答例・補足を見る

30秒回答

Batch Processingは、一定量のデータをまとめて処理する方式です。日次集計、請求、レポートに向きます。Stream Processingは、イベントをほぼリアルタイムに処理する方式で、通知、不正検知、リアルタイム分析に向きます。Batchは単純で再実行しやすい一方、遅延があります。Streamは低遅延ですが、順序、重複、状態管理、再処理が難しくなります。

深掘りされたときに足す観点

  • Batch：日次・時間単位でまとめて処理。失敗時に再実行しやすい。
  • Stream：イベント到着ごとに処理。低遅延だが状態管理が難しい。
  • ユースケース：請求集計はBatch、セキュリティ通知はStream寄り。
  • 重複：Streamではat-least-once配信による重複処理を想定する。
  • ハイブリッド：リアルタイム概算をStream、正確な最終集計をBatchで行うこともある。

簡易図

Batch : Data Lake/DB → Scheduled Job → Report Stream: Event → Broker → Stream Processor → Real-time Store

理解補足

X上のSystem Designリストでは、Batch/Streamがよく並びます。大事なのは、どちらが上位かではなく、遅延許容と再処理容易性で選ぶことです。

注意点

リアルタイムに見せたいからといって、全てStreamにすると運用が難しくなります。正確性が重要な集計ではBatchで最終確定する設計も現実的です。

Leaderboardを設計してください。

注意点

スコアが同点の場合の並び順、期間切り替え、ユーザー退会時の扱いなど、仕様の細部が重要です。最初に要件確認してください。

Webhook配信システムを設計してください。

面接官の意図：外部顧客向けイベント配信で、再試行・署名・順序・重複・観測性を設計できるかを見る。

回答例・補足を見る

30秒回答

Webhookは、自社システム内のイベントを外部顧客のHTTPエンドポイントへ通知する仕組みです。イベント発生時にQueueへ積み、Delivery Workerが顧客URLへPOSTします。署名ヘッダで改ざん検知し、失敗時は指数バックオフで再試行し、一定回数失敗したら停止・DLQ・管理画面で確認できるようにします。

深掘りされたときに足す観点

  • Event：注文作成、決済成功、ユーザー更新などをイベント化する。
  • Subscription：顧客ごとに通知先URL、購読イベント、secretを管理する。
  • 署名：payload + timestampをHMAC署名し、受信側が検証する。
  • Retry：5xx/timeoutは再試行、4xxは設定ミスとして扱うなど方針を分ける。
  • 冪等性：event_idを付与し、受信側が重複処理できるようにする。

簡易図

Internal Event → Webhook Queue → Delivery Worker → Customer Endpoint ↓ ↓ Retry / DLQ Delivery Log

理解補足

注意点

Webhookは顧客側のエンドポイント品質に依存します。自社システムを守るため、タイムアウト、同時接続数、再試行回数、顧客単位のレート制限を必ず設けます。

XQ027

分散環境で定期ジョブを安全に実行するには？

面接官の意図：複数インスタンスで同じジョブが二重実行される問題に気づけるかを見る。

回答例・補足を見る

30秒回答

分散環境では、複数アプリインスタンスが同じCronを持つと二重実行される可能性があります。対策は、専用Schedulerを1つ置く、DB/Redisの分散ロックを使う、ジョブテーブルで状態管理する、Kubernetes CronJobやEventBridge Schedulerのような外部スケジューラを使うことです。ジョブ自体も冪等に設計します。

深掘りされたときに足す観点

  • 問題：アプリ3台すべてで同じ定期処理が走ると、二重請求・二重通知などが起きる。
  • Lock：DB row lock、Redis SET NX、リース期限付きロックを使う。
  • Job Table：pending/running/succeeded/failed状態をDBで管理する。
  • 外部Scheduler：クラウドのスケジューラから1回だけイベントを送る。
  • 冪等性：ロックが破れても壊れないよう、処理側も二重実行に耐える。

簡易図

Scheduler → Job Table/Queue → Worker ↑ distributed lock / status transition

理解補足

Xでは直接Schedulerという名前でなくても、queue、batch、retry、idempotencyの文脈で出ます。実務では「定期処理をSpringで書きました」だけだと、スケール時に二重実行問題が出ます。

注意点

分散ロックはロックの期限、処理時間超過、プロセスクラッシュ時の解放が難しいです。ロックだけに依存せず、処理を冪等にしてください。

リージョン障害対策は、まずRTOとRPOを確認します。短時間停止が許されるならバックアップ復旧やWarm Standbyでよく、ほぼ無停止が必要ならActive-ActiveやMulti-Region構成を検討します。ただし、データ整合性、コスト、運用複雑性が大きく上がるため、全システムに最初から入れるものではありません。

深掘りされたときに足す観点

  • RTO：どれくらいの時間で復旧すべきか。

簡易図

注意点

マルチリージョンは万能ではありません。DBの書き込み整合性、ユーザーセッション、外部API、DNS切替、運用訓練まで必要です。
