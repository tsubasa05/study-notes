面接 APIパフォーマンス課題 対策資料 2026-06-18

0. この資料の目的

過去に出た面接の次の質問に対する対策資料です。

  • 画面共有による技術課題: パフォーマンス問題の原因調査と対策
  • APIのパフォーマンス劣化の調査方法と改善策を考える簡単な課題

この2つは、ほぼ同じ評価観点だと考えてよいです。

面接官が見ているのは、正解を一発で当てる力ではなく、次の力です。

  • 推測で対策に飛びつかず、事実を確認できるか
  • レイヤーを分けて原因を切り分けられるか
  • API、アプリ、DB、外部連携、インフラをつなげて考えられるか
  • 金融サービスとして、速さだけでなく正しさ、二重処理防止、監査性を考えられるか
  • 自分の経験範囲を盛らず、分かることと分からないことを切り分けて話せるか

用語やDB・SQL・アプリ処理の意味を補う詳細資料は、次を参照してください。

---

1. 面接での基本方針

最初に言うこと

まず推測でインデックス追加やキャッシュ導入に飛びつかず、どのAPIが、いつから、どの条件で、どの指標として遅くなっているのかを確認します。

そのうえで、API全体の時間を、アプリケーション処理、DB、外部API、インフラ・リソースに分けて、どこで時間を使っているかを切り分けます。

面接中に絶対に避けること

  • 最初から「インデックスを貼ります」と言う
  • 最初から「キャッシュします」と言う
  • 平均レスポンスだけを見る
  • DBだけ、アプリだけで原因を決めつける
  • 金融APIなのに、リトライやキャッシュによる二重処理・不整合を考えない
  • 経験していないパフォーマンスチューニングを経験済みのように話す

見せるべき姿勢

主担当として大規模な性能改善をやり切った経験は厚くありませんが、ログ、リクエスト、ソースコード、シーケンスから原因を切り分け、関係チームと修正・再デプロイ・動作確認まで進めた経験があります。

パフォーマンス課題でも、まず事実を集め、レイヤーごとに分解し、再現条件と改善効果を確認しながら進めたいです。

---

2. 最初に確認する質問

画面共有課題では、すぐ答えを出すより、まず前提確認をします。

症状

  • どのAPIが遅いのか
  • いつから遅いのか
  • 常に遅いのか、特定条件だけか
  • 平均が遅いのか、p95/p99のようなテールレイテンシが悪いのか
  • エラー率も上がっているのか
  • レスポンスは遅いが成功しているのか、タイムアウトや5xxも出ているのか

影響範囲

  • 全ユーザーか、一部ユーザーか
  • 全APIか、特定APIか
  • 特定のパートナー、支店、端末、地域、時間帯だけか
  • 本番だけか、検証環境でも再現するか

直前の変更

  • アプリケーションのリリース
  • DBスキーマ変更
  • インデックス変更
  • 外部API仕様変更
  • データ量の増加
  • インフラ設定変更
  • トラフィック増加
  • バッチや集計処理の追加

見たい情報

  • APIのレスポンスタイム分布
  • リクエスト数
  • エラー率
  • CPU、メモリ、GC
  • スレッドプール
  • DBコネクションプール
  • DBのスロークエリ
  • 実行計画
  • 外部APIの応答時間
  • タイムアウト・リトライ回数
  • ログ、トレースID、分散トレース

---

3. 調査の型

3-1. まず4つの指標で見る

Google SREでは、ユーザー向けシステムで見るべき代表指標として、Latency、Traffic、Errors、Saturationの4つが整理されています。

面接では、次の言い方で十分です。

まず、レイテンシ、リクエスト数、エラー率、リソース逼迫を見ます。平均値だけでなく、p95やp99を見て、一部のリクエストだけ極端に遅いのか、全体的に遅いのかを分けます。

指標見るもの例
Latencyレスポンスタイム平均、p95、p99、成功時と失敗時の時間
Traffic負荷RPS、同時接続数、対象APIの呼び出し数
Errors失敗4xx、5xx、タイムアウト、業務エラー
Saturation逼迫CPU、メモリ、GC、DB接続数、スレッド、I/O

詳細:

  • 4つの指標の具体説明

3-2. API処理時間を分解する

まず、1リクエストの処理時間を次に分けます。

クライアント
  -> LB / API Gateway
  -> Controller
  -> Service
  -> Repository / DB
  -> 外部API / 他マイクロサービス
  -> レスポンス生成

面接での言い方:

API全体が遅いと言っても、Controllerに入る前が遅いのか、アプリ内処理が遅いのか、DBが遅いのか、外部APIが遅いのかで対策が変わります。可能であればトレースIDやログで、処理区間ごとの時間を分けて見ます。

3-3. 変更履歴と突き合わせる

性能劣化は、次のどれかで起きやすいです。

  • リリース後から遅くなった
  • データ量が増えて遅くなった
  • トラフィックが増えて遅くなった
  • 外部APIの応答が遅くなった
  • バッチや集計処理と時間帯が重なった
  • DBインデックスやクエリが変わった

面接での言い方:

いつから遅くなったかを確認し、直前のリリース、DB変更、データ量増加、外部API変更、トラフィック増加と突き合わせます。性能劣化のタイミングと変更点が一致すれば、調査の優先順位を付けやすくなります。

---

4. 原因候補と確認方法

この章は面接中に見る要約です。N+1、インデックス、JOIN、ORDER BY、GROUP BY、JSONシリアライズなどの詳しい意味は、次を参照してください。

  • DB周りの基本
  • アプリケーション処理が遅い場合の具体例
  • 外部API・マイクロサービス連携

4-1. DBが遅い

代表的な原因:

  • N+1クエリ
  • インデックス不足
  • 検索条件やJOINが重い
  • ORDER BY、GROUP BYが重い
  • 取得件数が多すぎる
  • 不要なカラムを取りすぎている
  • トランザクションが長い
  • ロック競合
  • コネクションプール枯渇
  • DBのCPU、メモリ、I/O逼迫

確認すること:

  • スロークエリログ
  • 実行計画
  • 実行回数
  • 取得件数
  • インデックス利用状況
  • ロック待ち
  • DB接続数

面接での回答:

DBが疑わしい場合は、まずスロークエリと実行計画を確認します。MySQLであればEXPLAINを使い、想定したインデックスが使われているか、フルスキャンになっていないか、JOIN順や取得件数が適切かを見ます。

ただし、インデックスは追加すれば常に良いわけではなく、更新性能やストレージにも影響します。そのため、どの検索条件で頻繁に使われるか、実行計画上の効果があるかを確認して判断します。

4-2. アプリケーション処理が遅い

代表的な原因:

  • ループ内でDBや外部APIを呼んでいる
  • JSONのシリアライズ・デシリアライズが重い
  • 大きすぎるレスポンスを返している
  • 同期処理で重い処理を待っている
  • 不要な変換や集計をアプリ側で行っている
  • キャッシュできるマスタ情報を毎回取得している
  • ログ出力が過剰
  • メモリ使用量が多くGCが増えている

確認すること:

  • 区間ごとの処理時間ログ
  • CPU使用率
  • メモリ使用量
  • GCログ
  • プロファイラ
  • レスポンスサイズ
  • ループ内処理

面接での回答:

アプリケーション側では、処理区間ごとのログを見て、どの処理で時間を使っているかを確認します。ループ内でDBや外部APIを呼んでいないか、大量データを一度に取得していないか、レスポンスサイズが大きすぎないか、GCやCPUが逼迫していないかを見ます。

4-3. 外部API・他マイクロサービスが遅い

代表的な原因:

  • 外部APIの応答遅延
  • タイムアウト設定が長すぎる
  • リトライが多すぎる
  • リトライが同時に集中して負荷を増やしている
  • 下流サービスの障害
  • ネットワーク遅延

確認すること:

  • 外部APIごとの応答時間
  • タイムアウト回数
  • リトライ回数
  • リトライ間隔
  • 成功率
  • 呼び出し先のエラー
  • サーキットブレーカーの有無

面接での回答:

外部APIが疑わしい場合は、外部呼び出しの開始・終了ログ、タイムアウト、リトライ回数を確認します。リトライは有効な場合もありますが、AWS Builders Libraryでも説明されているように、タイムアウト、リトライ、バックオフ、ジッターを適切に設計しないと、障害時に下流へさらに負荷をかける可能性があります。

金融APIでは、リトライで二重振替や二重処理が起きないように、冪等性キーや処理状態の管理も重要です。

4-4. スレッド・コネクションプールが詰まっている

代表的な原因:

  • DBコネクションプールの枯渇
  • HTTPクライアントの接続プール不足
  • Tomcatのワーカースレッド枯渇
  • 遅い外部API待ちでスレッドが占有される
  • DB待ちでアプリ側スレッドが詰まる

確認すること:

  • アクティブスレッド数
  • キュー待ち
  • HikariCPなどのactive、idle、pending
  • HTTPクライアント接続数
  • タイムアウト設定

面接での回答:

CPUやDBが極端に高くないのにAPIが詰まる場合、スレッドやコネクションプールの枯渇も疑います。DBや外部APIの応答待ちでスレッドが長時間占有されると、後続リクエストが待たされます。Spring Boot ActuatorやMicrometerで、HTTP、JVM、DBコネクションプールのメトリクスを確認できる構成にしておくと切り分けやすいです。

4-5. インフラ・リソースが逼迫している

代表的な原因:

  • CPU使用率が高い
  • メモリ不足、GC増加
  • ディスクI/O
  • ネットワーク帯域
  • Pod / Task / Instance数不足
  • オートスケールの遅れ
  • ロードバランサの設定

確認すること:

  • CPU、メモリ、I/O
  • コンテナ数、タスク数
  • オートスケール履歴
  • ALB / API Gatewayのレイテンシ
  • アプリに入る前の遅延

面接での回答:

インフラ側では、CPU、メモリ、GC、I/O、ネットワーク、コンテナ数、オートスケール履歴を確認します。アプリに入る前のLBやAPI Gatewayの時間と、アプリ内部の時間を分けて見ることで、アプリ起因かインフラ起因かを切り分けます。

---

5. 改善策の考え方

短期対応

まず影響を下げます。

  • 問題のあるリリースを切り戻す
  • 特定機能を一時停止する
  • タイムアウト値を調整する
  • 負荷の高い処理を一時的に止める
  • スケールアウトする
  • 重いバッチの時間をずらす
  • 外部API障害なら縮退する

面接での言い方:

本番影響がある場合は、原因調査と並行して、影響を下げる短期対応を考えます。直前リリースが原因なら切り戻し、リソース逼迫なら一時的なスケールアウト、外部APIが原因ならタイムアウトや縮退を検討します。

恒久対応

原因に応じて直します。

原因恒久対応
N+1JOIN、fetch設計、バッチ取得、クエリ見直し
インデックス不足検索条件・ソート条件に合うインデックス追加
取得件数過多ページング、上限、必要カラムだけ取得
外部API遅延タイムアウト、バックオフ、ジッター、サーキットブレーカー
二重処理リスク冪等性キー、状態管理、重複排除
プール枯渇プールサイズ、タイムアウト、処理時間短縮
大量同期処理非同期化、キュー化、分割処理
レスポンス肥大DTO見直し、圧縮、必要情報だけ返却
リソース不足スケール、負荷試験、キャパシティ設計
原因が見えないログ、メトリクス、トレース追加

詳細:

  • 改善策の考え方を具体化する

検証

対策後は、必ず同じ条件で確認します。

  • 同じデータ量
  • 同じリクエスト条件
  • 同じ同時実行数
  • p50、p95、p99
  • エラー率
  • DB負荷
  • CPU、メモリ、GC
  • 外部API呼び出し回数
  • 金融APIなら二重処理がないこと

6. 画面共有課題での話し方

この章は面接中の発言メモです。グラフ、ログ、コード、SQLを見たときの詳しい確認観点は次を参照してください。

  • 画面共有課題で見るものを詳細化する

6-1. 最初の30秒

まず、どのAPIがどの条件で遅いのかを確認します。平均だけでなくp95やp99、エラー率、リクエスト数、CPU・DB接続数なども見たいです。

そのうえで、処理時間をアプリ、DB、外部API、インフラに分け、どこで時間を使っているかを切り分けます。

6-2. 画面にログやグラフが出ている場合

見る順番:

  1. 時系列でいつから悪くなったか
  2. レイテンシとリクエスト数が同時に上がっているか
  3. エラー率も上がっているか
  4. CPU、メモリ、DB接続数、スロークエリが同時に変化しているか
  5. 直前リリースがあるか
  6. 特定API、特定DBクエリ、特定外部APIに偏っているか

発言例:

このグラフだと、レイテンシが上がったタイミングとリクエスト数の増加が重なっています。まず負荷増加に対して、どのリソースが飽和しているかを確認したいです。CPU、DB接続数、スロークエリ、外部APIの応答時間を見ます。

6-3. コードが出ている場合

見る順番:

  1. ループ内DBアクセス
  2. ループ内外部APIアクセス
  3. 不要な全件取得
  4. ページング不足
  5. Transaction範囲が広すぎる
  6. 同期処理が長すぎる
  7. キャッシュ可能なマスタ情報を毎回取得している
  8. レスポンスDTOが重すぎる

発言例:

このコードだと、一覧取得後に各レコードごとに追加情報をDBから取得しているため、件数が増えるとN+1になりそうです。まず実際のSQL発行回数をログで確認し、問題であればJOINやIN句でのまとめ取得、必要に応じてDTO設計の見直しを検討します。

6-4. SQLが出ている場合

見る順番:

  1. WHERE条件
  2. JOIN条件
  3. ORDER BY
  4. GROUP BY
  5. LIMITの有無
  6. インデックス候補
  7. 実行計画
  8. 取得件数

発言例:

このSQLは検索条件とソート条件に対してインデックスが効いているかを確認したいです。EXPLAINでフルスキャンになっていないか、JOIN順が妥当か、取得件数が多すぎないかを見ます。インデックス追加は更新性能にも影響するため、利用頻度と実行計画を見て判断します。

7. よくある課題パターンと回答

この章は回答の型です。各パターンの背景、調査、改善の詳細は次を参照してください。

  • よくある課題パターンの詳細

パターン1: データ量が増えて一覧APIが遅い

原因候補:

  • 全件取得
  • ページングなし
  • N+1
  • インデックス不足
  • ソートが重い
  • レスポンスサイズが大きい

回答:

まずデータ量増加前後で、APIの処理時間、SQL、取得件数、レスポンスサイズを比較します。DB側ではスロークエリとEXPLAINを確認し、インデックス、JOIN、ORDER BY、取得件数を見ます。アプリ側ではN+1や不要な全件取得、DTOの肥大化を確認します。

改善策としては、ページング、必要カラムのみ取得、N+1解消、適切なインデックス、検索条件の見直しを検討します。

パターン2: リリース後から急に遅い

原因候補:

  • 新しい処理追加
  • SQL変更
  • 外部API呼び出し追加
  • キャッシュ無効化
  • 設定変更
  • ログ増加

回答:

リリース前後で変わった処理を確認します。APMやログで、旧バージョンと新バージョンの処理時間を区間別に比較し、DBクエリ、外部API呼び出し回数、レスポンスサイズが増えていないかを見ます。

本番影響が大きければ、原因調査と並行して切り戻しを検討します。

パターン3: 平均は悪くないがp99だけ遅い

原因候補:

  • 一部の重いデータ
  • ロック待ち
  • GC
  • コネクションプール待ち
  • 外部APIの一部遅延
  • キャッシュミス

回答:

平均だけでは一部の遅いリクエストが隠れるため、p95、p99、遅いリクエストの条件を見ます。遅いリクエストのユーザー、データ件数、外部API呼び出し、DBロック、GC、コネクションプール待ちを確認します。金融サービスでは一部ユーザーだけでも影響が大きいため、テールレイテンシを重視します。

パターン4: 外部APIが遅い

原因候補:

  • 外部API自体の遅延
  • タイムアウトが長い
  • リトライが集中
  • 同期呼び出しで待っている
  • 障害時の縮退がない

回答:

外部API呼び出し時間、タイムアウト、リトライ回数を確認します。短期的にはタイムアウトや縮退、必要なら機能制限を検討します。恒久的には、バックオフとジッター、サーキットブレーカー、非同期化、結果状態の管理を検討します。金融APIでは、リトライによる二重処理を防ぐため、冪等性キーや取引状態管理が重要です。

パターン5: DBコネクションプールが枯渇している

原因候補:

  • 遅いSQLで接続を長く保持
  • トランザクションが長い
  • 外部API呼び出しまでトランザクション内に入っている
  • プールサイズが負荷に合っていない

回答:

DBコネクションプールのactive、idle、pendingを見ます。枯渇している場合、単純にプールを増やすだけでなく、接続を長く保持している原因を見ます。遅いSQL、長いトランザクション、外部API呼び出しをトランザクション内で待っていないかを確認します。

9. 暗記する回答

30秒版

まず、どのAPIが、いつから、どの条件で遅くなっているかを確認します。平均だけでなくp95、p99、エラー率、リクエスト数も見ます。

そのうえで、API全体の時間を、アプリケーション処理、DB、外部API、インフラに分け、どこで時間を使っているかを切り分けます。

DBであればスロークエリと実行計画、アプリであればN+1や重い同期処理、外部APIであればタイムアウトやリトライ、インフラであればCPU、メモリ、コネクションプールを確認します。金融APIでは、改善時に二重処理や不整合が起きないことも重視します。

60秒版

まず推測で対策に飛びつかず、どのAPIが、いつから、どの条件で遅いのかを確認します。平均値だけでなく、p95やp99、エラー率、リクエスト数、CPUやDB接続数なども確認します。

次に、処理時間をアプリケーション、DB、外部API、インフラに分けます。DBが原因であれば、スロークエリとEXPLAINで、インデックス、JOIN、取得件数、ロックを見ます。アプリ側であれば、N+1、不要な全件取得、重い同期処理、レスポンスサイズ、GC、スレッドやコネクションプールを確認します。外部APIであれば、タイムアウト、リトライ、バックオフ、二重処理防止を見ます。

本番影響が大きければ、切り戻しやスケールアウトなどの短期対応も並行します。恒久対応後は、同じ条件で負荷試験を行い、p95、p99、エラー率、DB負荷、二重処理がないことを確認します。

10. 逆質問に転用できる質問

  1. APIやDBのパフォーマンス問題が起きた際、開発チームはログ、メトリクス、トレース、DB実行計画をどの程度自分たちで見て改善まで担当していますか。
  1. BaaSや外部パートナー連携のAPIでは、タイムアウト、リトライ、冪等性、監査ログなどをチーム共通でどのように設計されていますか。
  1. 金融サービスとして、単にAPIを高速化するだけでなく、整合性や二重処理防止を守るために、性能改善時に特に注意している観点はありますか。
  1. Spring BootやMySQLまわりで、開発者が日常的に確認しているメトリクスや、性能劣化時の標準的な調査手順はありますか。
