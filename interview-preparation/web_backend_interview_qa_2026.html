Amazon系 / CS基礎 S
OQ001

配列・ListとHashMapをどう使い分けますか？

面接官の意図：データ構造を、名前の暗記ではなくアクセスパターンから選べるかを見る。

回答例・補足を見る

30秒回答

配列・List・HashMapは「どのアクセスを速くしたいか」で選びます。Listは順序を持つ集合を前から処理したり、インデックスで取り出したい場合に向きます。HashMapはキーから値を高速に引く場合に向きます。LinkedHashMapや「双方向リスト＋HashMap」は、順序とキー検索を両方扱えますが、メモリ消費や実装複雑性が増えるので、Listを常に上回る万能構造ではありません。

深掘りされたときに足す観点

  • ArrayList：内部的には配列で要素を保持する。挿入順を維持し、インデックスアクセスは速いが、途中挿入・途中削除は後続要素のずらしが発生しやすい。
  • LinkedList：各要素が前後のノード参照を持つ。ノード位置が分かっていれば挿入・削除はしやすいが、インデックス指定でたどる場合は先頭/末尾から順に探索するため遅い。
  • HashMap：キーから値を引く構造。順序や重複要素の扱いより、ID・コード・キーによる検索を高速化する用途に向く。
  • LinkedHashMap：HashMapに順序付きのリンク構造を足したもの。キー検索と挿入順/アクセス順の反復を両立できる。
  • 双方向リスト＋HashMap：LRUキャッシュのように「キーで即座に探し、順序も更新し、末尾を即座に消す」用途で有効。
理解補足：JavaのListはインターフェースです。代表実装のArrayListは内部配列で順序を保持し、list.get(3)のようなインデックスアクセスが得意です。一方、HashMapはmap.get(userId)のようにキーで引く構造です。あなたの言う「リンクとHashMapを使えば順列＋キー探索ができる」は正しく、これはLinkedHashMapやLRUキャッシュの発想に近いです。ただし、キーが一意である前提が必要で、要素重複やインデックスアクセス、単純な順次処理ではListの方が自然です。
用途で見る選び方 入力順に全件処理したい → List / ArrayList 3番目・10番目を取りたい → ArrayList userId から User を引きたい → HashMap<UserId, User> キー検索 + 挿入順に走査したい → LinkedHashMap キー検索 + 最近使った順に並べたい → LinkedHashMap(accessOrder=true) / 双方向リスト + HashMap
注意点：HashMap系は平均O(1)で便利ですが、メモリを追加で使います。また、キー設計、equals/hashCode、順序要件、重複要件を誤るとバグになります。面接では「HashMapの方が速い」ではなく、「アクセスパターンに応じて選ぶ」と答えるのが正確です。

OQ002

計算量 O(n) と O(log n) の違いを説明してください。

面接官の意図：処理時間が入力サイズに対してどう増えるかを説明できるかを見る。

回答例・補足を見る

30秒回答

O(n)は入力件数に比例して処理量が増える計算量です。O(log n)は、探索範囲を半分ずつ削るような処理で、入力件数が増えても処理量の増え方が緩やかです。線形探索はO(n)、ソート済み配列に対する二分探索はO(log n)です。

深掘りされたときに足す観点

  • 1,000件ならO(n)でも問題にならないことは多いが、1,000万件や高頻度APIでは差が大きくなる。
  • O(log n)は典型的には二分探索、B-treeインデックス、平衡木などで出てくる。
  • 実務では、計算量だけでなくI/O、DBアクセス、ネットワーク、ロック待ちも支配的になる。
理解補足：コーディング面接では、まず素朴解を出してから「この実装だとO(n^2)なので、HashSetを使ってO(n)にできます」のように改善できると評価されやすいです。Webバックエンドでは、アプリ内の計算量よりDBクエリや外部API呼び出しの方が重いことも多いので、どこがボトルネックかを見る視点も重要です。
注意点：Big-Oは上限の増え方を表すもので、実測性能そのものではありません。小さい入力ではO(n)の単純実装の方が速いこともあります。
Amazon系 / Coding A
OQ003

2つの配列に共通する要素を求めるならどう実装しますか？

面接官の意図：素朴解から改善解へ進めるか、仕様確認を入れられるかを見る。

回答例・補足を見る

30秒回答

まず、共通要素を一意にするか、重複数を残すか、結果順をどうするか確認します。素朴に二重ループで比較するとO(n*m)です。改善案として、片方の配列をHashSetに入れる処理にO(n)、もう片方の配列を走査してSetに含まれるか確認する処理にO(m)かかるため、合計で平均O(n+m)になります。

深掘りされたときに足す観点

  • 「片方がすでにHashSetとして存在している」前提なら、もう片方を走査するだけなのでO(m)と説明してよい。
  • ただし、配列からHashSetを作るところも含めて実装するなら、Set構築分のO(n)が必要になる。
  • 重複を考慮するならMap<値, 出現回数>を使い、片方の出現回数を減らしながら結果に追加する。
  • メモリを追加で使いたくない場合、両方をソートして2ポインタで走査する案もある。ただしソートにO(n log n + m log m)かかる。
理解補足：O(n+m)の理由は「Setを作るコスト」と「検索するコスト」を足しているからです。例えば配列Aがn件、配列Bがm件なら、Aを全部HashSetへ入れるのにn回、Bを全部確認するのにm回処理します。HashSetのcontainsは平均O(1)なので、B側の確認はO(m)です。合計でO(n+m)です。
例：AとBの共通要素を求める 1. AをHashSetに入れる → n件処理 = O(n) 2. Bを1件ずつ見る → m件処理 = O(m) 3. set.contains(b) は平均O(1) 合計：O(n) + O(m) = O(n+m) ※ すでにAがHashSetなら、1は不要なのでO(m)
注意点：HashSetのO(1)は平均計算量です。また、結果の順序や重複をどう扱うかで実装が変わります。面接では、計算量だけでなく「仕様確認」が重要です。

OQ004

オブジェクト指向設計で、継承よりコンポジションを選ぶ理由は？

面接官の意図：変更に強い設計、責務分離、テスト容易性を理解しているかを見る。

回答例・補足を見る

30秒回答

コンポジションは、あるオブジェクトが別のオブジェクトを部品として持ち、その処理を委譲する設計です。Javaで言うと、クラスのフィールドに別クラスやインターフェース型の依存を持たせる形です。継承はis-a関係、コンポジションはhas-a関係です。継承は親クラスの変更が子クラスへ波及しやすいため、振る舞いを差し替えたい場合は、インターフェース＋実装クラスを注入するコンポジションの方が変更に強くなります。

深掘りされたときに足す観点

  • 継承：class Dog extends Animal のように「DogはAnimalである」という分類関係を表す。
  • コンポジション：OrderService が PaymentClient をフィールドとして持つように、「AはBを使う」という関係を表す。
  • インターフェース：コンポジションそのものではないが、差し替え可能な部品の契約としてよく使う。
  • Springでは、private final NotificationSender sender; のようにインターフェース型をコンストラクタ注入する形が典型。
Javaでのイメージ interface NotificationSender { void send(String message); } class EmailNotificationSender implements NotificationSender { public void send(String message) { /* メール送信 */ } } class OrderService { private final NotificationSender sender; // コンポジション：部品として持つ OrderService(NotificationSender sender) { this.sender = sender; } void completeOrder() { // 注文確定処理 sender.send("注文が完了しました"); // 振る舞いを委譲 } }
理解補足：「コンポジション＝インターフェース」ではありません。コンポジションは、あるクラスが別のオブジェクトを保持して使う構造です。インターフェースは、その保持する部品を差し替えやすくするためによく使う道具です。Java/Springでは、ServiceがRepositoryや外部APIクライアントをフィールドとして持つ設計もコンポジションです。
注意点：継承が常に悪いわけではありません。ただし「コード再利用したいから継承する」は危険です。親クラスの実装詳細に子クラスが依存しやすく、変更に弱くなるためです。

OQ005

トランザクションのACIDを説明してください。

面接官の意図：DBの基本的な信頼性モデルを理解しているかを見る。

回答例・補足を見る

30秒回答

ACIDは、Atomicity、Consistency、Isolation、Durabilityの頭文字です。Atomicityは全部成功か全部失敗、Consistencyは制約を満たす一貫した状態を保つこと、Isolationは並行実行されるトランザクション同士の干渉を制御すること、Durabilityはコミット後の結果が永続化されることです。

深掘りされたときに足す観点

  • 注文作成と在庫減算は、片方だけ成功すると不整合になるためAtomicityが重要。
  • 外部キー、一意制約、チェック制約などを満たす状態を維持するのがConsistency。
  • Isolationが弱いと、ダーティリード、ノンリピータブルリード、ファントムリードなどが問題になる。
  • Durabilityは、障害後もコミット済みデータが失われないこと。
理解補足：Web系面接では、ACIDを言えるだけでは不十分です。「どこを1トランザクションにするか」が重要です。例えば、注文作成、在庫確保、決済API呼び出しを全部DBトランザクションに入れると、外部API待ちでロック時間が伸びます。外部APIを含む業務では、DBトランザクションと業務上の補償処理を分けて考える必要があります。
注意点：外部API呼び出しはDBトランザクションではロールバックできません。決済や外部連携では、ステータス管理、リトライ、補償処理、冪等性が必要になります。
Amazon系 / DB A
OQ006

SQLとNoSQLをどう使い分けますか？

面接官の意図：データモデル、整合性、アクセスパターンからストレージを選べるかを見る。

回答例・補足を見る

30秒回答

SQL/RDBは、テーブル構造、制約、JOIN、トランザクション、一貫性が重要なデータに向きます。NoSQLは、スキーマ変更の柔軟性、大量データ、高スループット、特定アクセスパターンに最適化したい場合に向きます。注文、請求、権限などの中核データはRDBを優先し、ログ、イベント、キャッシュ、セッション、検索用データなどはNoSQLや専用ストアを検討します。

深掘りされたときに足す観点

  • スキーマ：データの構造定義のこと。RDBではテーブル、カラム、型、制約、リレーションなどを指す。
  • RDBは、外部キー、一意制約、トランザクション、JOINに強い。
  • NoSQLは、Key-Value、Document、Wide-column、Graphなど種類によって適性が違う。
  • NoSQLは「スキーマが完全にない」ではなく、アプリケーション側が構造と整合性を管理する比率が上がることが多い。
  • アクセスパターンが明確で、JOINよりキー検索・ドキュメント単位取得が中心ならNoSQLが合う場合がある。
理解補足：スキーマは、雑に言えば「このデータはどういう形で保存されるか」という設計図です。RDBなら、usersテーブルにid、email、created_atがあり、emailは一意で、orders.user_idはusers.idを参照する、というような定義がスキーマです。NoSQLでも暗黙のデータ構造は存在するため、スキーマレスという言葉を「何も設計しなくてよい」と解釈すると危険です。
RDBスキーマの例 users - id: bigint, primary key - email: varchar, unique - name: varchar orders - id: bigint, primary key - user_id: bigint, foreign key -> users.id - status: varchar - created_at: timestamp
注意点：「NoSQLは速いから使う」は浅い回答です。RDBでも適切なインデックス設計で十分速いケースは多いです。まず整合性要求、クエリの複雑さ、データ量、更新頻度、スケール要件から選ぶべきです。

OQ007

分散システムで障害を前提に設計するとはどういうことですか？

面接官の意図：ネットワーク・依存サービス・部分障害を前提に考えられるかを見る。

回答例・補足を見る

30秒回答

分散システムでは、単一プロセス内の例外だけでなく、タイムアウト、部分障害、依存APIの遅延、リトライによる重複、メッセージ欠落、ネットワーク分断を前提にします。失敗しない設計ではなく、失敗しても影響を限定し、検知し、復旧できる設計を目指します。

深掘りされたときに足す観点

  • タイムアウトを設定しないと、依存先の遅延が呼び出し元へ波及する。
  • リトライには冪等性が必要。重複実行されても壊れない設計にする。
  • 依存先障害に対して、サーキットブレーカー、キュー、フォールバック、縮退運転を検討する。
  • 障害検知にはログ、メトリクス、トレース、アラートが必要。
理解補足：あなたのマイクロサービス経験に接続するなら、認証認可基盤や外部APIが落ちた場合にAPIをfail-closeにするか、キャッシュした権限で一時的に通すか、といった設計判断になります。セキュリティ系は基本fail-closeですが、可用性とのトレードオフがあります。
注意点：「リトライすればよい」は浅いです。リトライは依存先をさらに圧迫することがあります。バックオフ、最大回数、タイムアウト、冪等性、DLQまでセットで考える必要があります。

OQ008

LRUキャッシュを設計するとしたら、どんなデータ構造を使いますか？

面接官の意図：複数のデータ構造を組み合わせて要件を満たせるかを見る。

回答例・補足を見る

30秒回答

LRUキャッシュは、Least Recently Used、つまり「最近使われていないデータから追い出す」キャッシュです。容量がいっぱいになったとき、最も長い間アクセスされていない要素を削除します。O(1)でget/put/削除を実現するには、キー検索用のHashMapと、利用順を管理する双方向リストを組み合わせます。JavaならLinkedHashMapのaccessOrderを使うと近い実装ができます。

深掘りされたときに足す観点

  • HashMap：key → リストノードを保持し、対象要素をO(1)で見つける。
  • 双方向リスト：先頭を最近使った要素、末尾を最近使っていない要素として管理する。
  • get(key)したら、そのノードを先頭へ移動する。
  • put(key, value)で容量超過したら、末尾ノードを削除し、HashMapからも消す。
LRUのイメージ 最近使った ←----------------------→ 最近使っていない HEAD [C] <-> [A] <-> [B] <-> [D] TAIL get(B) すると B を先頭へ移動 HEAD [B] <-> [C] <-> [A] <-> [D] TAIL 容量超過時は TAIL の D を削除
理解補足：単なるListだけだと、keyから対象要素を探すのにO(n)かかります。単なるHashMapだけだと、どれが「最近使われていないか」を管理しづらいです。だから、HashMapで場所を即座に見つけ、双方向リストで利用順を管理します。あなたがOQ001で言っていた「リンク＋HashMap」は、このLRUキャッシュで典型的に使われる組み合わせです。
注意点：実務の分散キャッシュでは、単一JVM内のLRUだけでは不十分なことがあります。Redis、TTL、キャッシュ無効化、キャッシュスタンピード、複数サーバー間の整合性も論点になります。

面接官の意図：基本アルゴリズムの境界条件を正確に扱えるかを見る。

回答例・補足を見る

30秒回答

二分探索では、配列がソート済みであること、探索範囲の境界を一貫して扱うこと、無限ループにならないことに注意します。典型的には、両端を含む[left, right]方式ならwhile (left <= right)、右端を含まない[left, right)方式ならwhile (left < right)を使います。どちらかに統一することが重要です。

深掘りされたときに足す観点

  • 両端含む方式：存在確認に向く。初期値はleft=0, right=n-1。
  • 半開区間方式：lower_bound、つまり「target以上の最初の位置」を探すときに扱いやすい。初期値はleft=0, right=n。
  • mid = left + (right - left) / 2と書くと、巨大な値でのオーバーフローを避けやすい。
  • 更新時にleft = midやright = midを誤ると、範囲が縮まず無限ループになる。
両端含む [left, right]：targetが存在するか探す left = 0, right = n - 1 while left <= right: mid = left + (right - left) / 2 if a[mid] == target: return mid if a[mid] < target: left = mid + 1 else: right = mid - 1 return -1 半開区間 [left, right)：target以上の最初の位置を探す left = 0, right = n while left < right: mid = left + (right - left) / 2 if a[mid] < target: left = mid + 1 else: right = mid return left
理解補足：二分探索で混乱しやすいのは、「rightが有効な添字なのか」「rightは範囲外の境界なのか」が曖昧になるからです。[left, right]はrightも探索対象です。[left, right)はright自体は探索対象ではなく、範囲の外側の境界です。慣れるまでは、存在確認は両端含む方式、挿入位置探索は半開区間方式、と使い分けると整理しやすいです。
注意点：配列が未ソートなら二分探索は使えません。ソートしてから二分探索する場合、ソートコストO(n log n)も含めて考える必要があります。

OQ028

レートリミッターを設計してください。

面接官の意図：API保護、アルゴリズム、分散環境での制御を理解しているかを見る。

回答例・補足を見る

30秒回答

レートリミッターは、ユーザー、IP、APIキーなどを単位に、一定時間あたりのリクエスト数を制限する仕組みです。代表方式にはFixed Window、Sliding Window、Token Bucket、Leaky Bucketがあります。単一アプリのメモリではなく、複数インスタンスで共有できるRedisやAPI Gatewayで実装することが多いです。

方式ごとの違い

  • Fixed Window：「1分あたり100回」のように固定時間枠でカウントする。簡単だが、分境界の前後で200回近く通ることがある。
  • Sliding Window Log：各リクエスト時刻を記録し、直近60秒の件数を見る。正確だがメモリを使う。
  • Sliding Window Counter：現在ウィンドウと前ウィンドウを重みづけして概算する。Fixed Windowより滑らかで、Logより軽い。
  • Token Bucket：一定速度でトークンが補充され、リクエストごとに1トークン消費する。バケット容量分の短期バーストを許容できる。
  • Leaky Bucket：リクエストをキューに入れ、一定速度で処理する。出力を平滑化しやすいが、キューが溢れると拒否する。
Token Bucket のイメージ bucket capacity = 10 refill rate = 1 token/sec リクエスト到着 ├─ tokenあり → tokenを1つ消費して許可 └─ tokenなし → 429 Too Many Requests 特徴：通常は1秒1回程度に制限しつつ、最大10回までの短期バーストは許容できる
理解補足：Fixed Windowは「13:00:00〜13:00:59に100回」のように数えるため、12:59:59に100回、13:00:00に100回のような境界バーストが起きます。Token Bucketは「普段は一定速度、ただし貯まっている分だけ一時的に多めに通す」方式なので、API利用制限でよく使われます。
注意点：アプリインスタンスごとのメモリで制限すると、10台にスケールした瞬間に実質10倍リクエストを許可してしまいます。分散環境では共有ストアかGatewayレイヤで制御する必要があります。

  • 完全一致：where email = ?。B-treeインデックスで高速化しやすい。

  • ファセット検索：カテゴリ、価格帯、ステータスなどで絞り込む検索。
  • オートコンプリート：入力途中で候補を出す。前方一致、N-gram、専用インデックスを検討する。
検索設計の選択肢 小規模・完全一致中心 → RDB + Index 管理画面の簡単な部分一致 → RDB LIKE でも可。ただしデータ量に注意 大量データ・全文検索・ランキング → OpenSearch / Elasticsearch DB更新と検索インデックス DB更新 → Event/CDC → Indexer → Search Index更新

ページングAPIを設計してください。OFFSET方式とCursor方式の違いは？

面接官の意図：大量データ取得、安定したページング、性能劣化を理解しているかを見る。

回答例・補足を見る

30秒回答

OFFSET方式は、limitとoffsetで「何件飛ばして何件取るか」を指定するページングです。実装は簡単ですが、offsetが大きいとDBが多くの行を読み飛ばすため遅くなります。Cursor方式は、最後に取得したレコードのIDや時刻を次回リクエストに渡し、「その続き」を取得する方式です。大量データや無限スクロールではCursor方式が向きます。

深掘りされたときに足す観点

  • OFFSET：page=3&limit=20なら、40件飛ばして20件取る。ページ番号UIに向く。
  • Cursor：前回最後のcreated_atやidを基準に次の20件を取る。無限スクロールや時系列一覧に向く。
  • OFFSETは、途中でデータが追加・削除されると重複や抜けが起きやすい。
  • Cursorには安定したソートキーが必要。created_at + idのように一意性を持たせる。
  • Cursorは内部キーをそのまま出さず、Base64等でopaque tokenにすることが多い。
OFFSET方式 GET /notifications?limit=20&offset=40 SQL: order by created_at desc limit 20 offset 40 意味：上位40件を読み飛ばして、次の20件を返す 問題：offset=100000 などが遅い Cursor方式 GET /notifications?limit=20&cursor=2026-06-01T10:00:00Z_12345 SQL: where (created_at, id) < (:createdAt, :id) order by created_at desc, id desc limit 20 意味：前回最後に見たレコードより後続の20件を返す
理解補足：OFFSETは人間に分かりやすい「ページ番号」に向いていますが、DB内部では指定件数を読み飛ばす必要があります。Cursorは「最後に見た地点から次を取る」ため、大量データでも性能が安定しやすいです。ただし、任意のページ番号へジャンプする用途には向きません。
注意点：created_atだけをCursorにすると、同時刻のデータが複数ある場合に順序が不安定になります。必ずcreated_atとidのように一意に並ぶキーを使います。

強い整合性と結果整合性の違いは？

面接官の意図：分散システムで可用性・性能・整合性のトレードオフを理解しているかを見る。

回答例・補足を見る

30秒回答

強い整合性は、書き込みが完了した後の読み取りで常に最新状態が見える性質です。結果整合性は、一時的には古い値や未反映の値が見える可能性があるものの、時間が経てば最終的に整合する性質です。残高、注文確定、認可判定のような誤りが重大な領域は強い整合性を重視し、通知数、検索インデックス、分析集計のように多少遅れても許容できる領域は結果整合性を検討します。

メリット・デメリット

  • 強い整合性のメリット：利用者や開発者にとって分かりやすい。二重在庫引当、残高不整合、権限ミスを防ぎやすい。
  • 強い整合性のデメリット：ロック、同期通信、合意形成が必要になり、レイテンシや可用性、スケールにコストがかかる。
  • 結果整合性のメリット：非同期処理や複製を使いやすく、可用性・スループット・スケールを上げやすい。
  • 結果整合性のデメリット：一時的に古い情報が見える。ユーザー体験、補正処理、再試行、監視が必要になる。
例 強い整合性が欲しい - 決済残高 - 注文確定 - 在庫引当 - 認可判定 結果整合性を許容しやすい - 検索インデックス反映 - 通知未読件数 - アクセス分析 - ランキング集計
理解補足：結果整合性は「不整合でよい」という意味ではありません。「短時間の遅れやズレは許容し、最終的に正しい状態へ収束させる」という意味です。例えば、DB更新後に検索インデックスへ非同期反映する場合、数秒間は検索結果に出ないことがありますが、業務上許容できるなら結果整合性でよい判断になります。
注意点：認証認可、決済、在庫、契約状態のような領域で安易に結果整合性を採用すると、セキュリティ事故や金銭事故につながります。どの不整合を何秒まで許すのか、具体的に言える必要があります。

メッセージキューで、at-least-once と exactly-once の違いは？

面接官の意図：非同期処理の重複・欠落・冪等性を理解しているかを見る。

回答例・補足を見る

30秒回答

at-least-onceは、メッセージが少なくとも1回は処理される保証です。失敗時に再配信されるため欠落しにくい一方、同じメッセージが複数回処理される可能性があります。exactly-onceは、論理的に1回だけ処理されたように見せる性質ですが、実現は難しく、ブローカーだけでなくDB更新や外部API呼び出しまで含めて考える必要があります。実務ではat-least-once前提でコンシューマを冪等にすることが多いです。

深掘りされたときに足す観点

  • at-most-once：最大1回。重複はしにくいが、失敗時に欠落する可能性がある。
  • at-least-once：少なくとも1回。欠落しにくいが、重複処理が起きる可能性がある。
  • exactly-once：論理的に1回だけ処理された状態を目指す。ただし、外部副作用を含むと難しい。
  • 冪等性：同じメッセージを2回処理しても結果が壊れないようにする。
  • 重複対策：message_id、idempotency_key、処理済みテーブル、一意制約を使う。
at-least-onceで重複が起きる例 Queue → Worker が message M を取得 Worker → DB更新成功 Worker → ack送信前にクラッシュ Queue → ack未受信なので message M を再配信 Worker → message M をもう一度処理 対策：message_id を処理済みテーブルに記録し、2回目はスキップする
理解補足：exactly-onceという言葉は誤解しやすいです。Kafkaなど一部の基盤は特定条件下でexactly-onceに近い処理をサポートしますが、メール送信、決済API、外部システム登録のような外部副作用まで完全に1回にするのは別問題です。そのため、アプリケーション側で冪等キーや状態遷移を設計する必要があります。
注意点：「exactly-onceを使えば重複しません」と言い切るのは危険です。面接では、「基盤の保証範囲」と「アプリケーション全体での副作用」を分けて説明する方が正確です。

面接官の意図：責務分担の基本を理解しているかを見る。

回答例・補足を見る

30秒回答

深掘りされたときに足す観点

理解補足：あなたが得意な「責務境界の整理」と接続しやすい概念です。ユーザ管理機能で、Keycloakが持つ責務とアプリDBが持つ責務を分けた話も、高凝集・低結合の文脈で説明できます。
注意点：低結合を目指しすぎて抽象化を増やしすぎると、逆に読みにくくなります。変更可能性がある場所に絞って抽象化するのが現実的です。

回答例・補足を見る

URL短縮サービスは、長いURLを短いキーに対応付け、短縮URLへのアクセス時に元URLへリダイレクトするサービスです。基本構成は、短縮キー生成、URLマッピングDB、リダイレクトAPI、キャッシュ、分析ログです。面接では、まず読み取りが多いのか、書き込みが多いのか、短縮URLの有効期限やカスタムaliasの有無を確認し、キー衝突、DBインデックス、キャッシュ、悪性URL対策まで説明します。

深掘りされたときに足す観点

  • API：POST /shorten で長いURLを受け取り、短縮キーを返す。GET /{key} で元URLを取得し、301/302でリダイレクトする。
  • キー生成：DBの連番IDをBase62化する方式、ランダム文字列方式、Snowflake風ID方式などがある。衝突回避と推測困難性を分けて考える。

  • キャッシュ：リダイレクトは読み取り頻度が高いので、short_code → original_url をRedisやCDN側でキャッシュする。

注意点

XQ002

高トラフィックAPI向けRate Limiterを設計してください。

面接官の意図：アルゴリズム暗記ではなく、何を守るための制限か、どこで制限するか、分散環境でどう整合させるかを見る。

回答例・補足を見る

30秒回答

Rate Limiterは、一定期間に許可するリクエスト数を制限し、バックエンドを過負荷・不正利用・ブルートフォース攻撃から守る仕組みです。方式にはFixed Window、Sliding Window、Token Bucket、Leaky Bucketがあります。API GatewayやLoad Balancerの手前で粗く制限し、アプリ側でユーザー単位・API単位の細かい制限を行う設計が現実的です。

深掘りされたときに足す観点

  • Fixed Window：1分100回のように固定区間で数える。簡単だが、境界時刻にリクエストが集中すると実質2倍近く通ることがある。
  • Sliding Window：直近60秒など移動窓で数える。精度は高いが、実装・ストレージ負荷は上がる。Redis Sorted Setで実装しやすい。
  • Token Bucket：時間経過でトークンが補充され、リクエストごとに消費する。平均レートを守りつつ短時間のバーストを許容できる。
  • Leaky Bucket：キューから一定速度で流す。出力レートを平滑化できるが、待ち時間やキューあふれを考える必要がある。

Cache-Aside、Read-Through、Write-Through、Write-Behindの違いは？

回答例・補足を見る

30秒回答

Cache-Asideはアプリがキャッシュを明示的に読む方式で、ミスしたらDBから読み、キャッシュに戻します。Read-Throughはキャッシュ層がDB読み込みまで担います。Write-Throughは書き込み時にDBとキャッシュを同期更新します。Write-Behindは先にキャッシュ/キューへ書き、後でDBへ反映します。読み取り性能、整合性、実装複雑性、障害時の安全性で選びます。

深掘りされたときに足す観点

  • Cache-Aside：最も一般的。アプリがcache miss時のDB読み込みとcache populateを制御する。
  • Read-Through：アプリから見るとキャッシュがデータソースのように振る舞う。キャッシュ製品側の機能に依存。
  • Write-Through：書き込み時にDBとキャッシュを同時更新。読み取りは新しくなりやすいが書き込み遅延が増える。
  • Write-Behind：書き込みを一旦キャッシュ/キューに受け、非同期でDBへ反映。高スループットだがデータ損失対策が必要。
  • TTL：古いデータをいつまで許すか。TTLが短いとDB負荷、長いとstale dataが増える。

簡易図

Cache-Aside Read: App → Cache miss → DB → Cache set → Response Write-Through: App → Cache/DB update together → Response

理解補足

XではRedis、TTL、cache invalidation、read-through/write-throughが頻出です。面接では方式名より「このデータは古くてもよいか」「書き込み後すぐ読まれるか」「キャッシュ削除とDB更新の順序はどうするか」を説明する方が重要です。

注意点

キャッシュは障害要因にもなります。stale data、hot key、cache stampede、DBとの不整合、削除漏れを必ずセットで話してください。

決済・注文APIでIdempotency Keyをどう使いますか？

面接官の意図：ネットワーク失敗時のリトライと二重実行問題を理解しているかを見る。

回答例・補足を見る

30秒回答

Idempotency Keyは、同じ操作の再送をサーバー側で同一リクエストとして扱うためのキーです。決済や注文作成では、クライアントがUUIDなどの一意キーを送り、サーバーはそのキーと結果を保存します。同じキーの再送が来たら、処理をもう一度実行せず、保存済み結果を返します。これにより、タイムアウト後のリトライで二重決済・二重注文を防ぎます。

深掘りされたときに足す観点

  • 対象：決済、注文作成、送金、在庫予約など「二重実行が危険」な操作。
  • 保存：idempotency_key、request hash、status、response、created_atをDB/Redisに保存する。
  • 同じキー・同じ内容：保存済み結果を返す。
  • 同じキー・異なる内容：クライアントバグの可能性があるため409などで拒否する。
  • 有効期限：キーを永久保存せず、業務に応じたTTLを決める。

簡易図

Cache Invalidationはなぜ難しいのですか？

面接官の意図：キャッシュ更新とDB更新のタイミング差、不整合、分散環境の難しさを理解しているかを見る。

回答例・補足を見る

30秒回答

Cache Invalidationが難しい理由は、DB更新とキャッシュ削除/更新が別操作になり、途中失敗や競合で古い値が残るからです。代表的には、DB更新後にキャッシュ削除する、キャッシュ更新ではなく削除する、TTLを併用する、イベントで非同期無効化する、バージョンを持たせるなどの対策があります。

深掘りされたときに足す観点

  • DB→Cache削除：DB更新後にキャッシュを消す。次回読み取りでDBから新値を入れる。一般的。
  • Cache更新：DB更新時にキャッシュも更新するが、複雑な集計や複数キーでは漏れやすい。
  • TTL：削除漏れの保険。ただし古い値を一定時間許容する。
  • 非同期無効化：DB更新イベントを流して関連キャッシュを消す。遅延がある。
  • Race：削除と再生成が競合し、古い値を再キャッシュすることがある。

簡易図

Update flow: 1. Update DB 2. Delete cache key 3. Next read repopulates cache Failure between 1 and 2 → stale cache risk

理解補足

Xでcache invalidationは定番です。キャッシュは読み取り性能を上げますが、更新系と組み合わせた瞬間に難しくなります。面接では「TTLを入れます」だけでなく、DB更新順序と失敗時の扱いまで説明してください。

注意点

金額・権限・在庫のように古い値が危険なデータでは、キャッシュ利用範囲を限定し、最終判断はDBや認可基盤で行う方が安全です。

Consistent Hashingは何のために使いますか？

面接官の意図：ノード追加・削除時の再配置コストを理解しているかを見る。

回答例・補足を見る

30秒回答

Consistent Hashingは、キャッシュノードやシャードノードを増減するときに、キーの再配置を最小限にするための方式です。通常のhash(key) % NではNが変わると多くのキーの配置先が変わります。Consistent Hashingではハッシュリング上にキーとノードを配置し、ノード増減時に影響範囲を限定します。

深掘りされたときに足す観点

  • 用途：分散キャッシュ、シャーディング、ロード分散。
  • 問題：単純Modulo方式はノード数が変わると大量のキーが移動する。
  • リング：キーのハッシュ値からリング上で次のノードに割り当てる。
  • 仮想ノード：ノードごとの偏りを減らすため、1物理ノードを複数点として配置する。
  • 限界：データの偏り、ホットキー、再配置中の整合性は別途考える必要がある。

簡易図

hash(key) % N: N changes → many keys move Consistent Hash Ring: node added/removed → nearby range moves

理解補足

XのSystem Designロードマップでは、scalabilityやconsistent hashingがよく挙がります。面接で名前だけ出すと危険ですが、「ノード数変更時に再配置を減らすため」と説明できればまず十分です。

注意点

Consistent Hashingはホットキー問題を自動解決しません。1つのキーにアクセスが集中する場合、どのノードに置いてもそこが熱くなります。

回答例・補足を見る

30秒回答

Batch Processingは、一定量のデータをまとめて処理する方式です。日次集計、請求、レポートに向きます。Stream Processingは、イベントをほぼリアルタイムに処理する方式で、通知、不正検知、リアルタイム分析に向きます。Batchは単純で再実行しやすい一方、遅延があります。Streamは低遅延ですが、順序、重複、状態管理、再処理が難しくなります。

深掘りされたときに足す観点

  • Batch：日次・時間単位でまとめて処理。失敗時に再実行しやすい。
  • Stream：イベント到着ごとに処理。低遅延だが状態管理が難しい。
  • ユースケース：請求集計はBatch、セキュリティ通知はStream寄り。
  • 重複：Streamではat-least-once配信による重複処理を想定する。
  • ハイブリッド：リアルタイム概算をStream、正確な最終集計をBatchで行うこともある。

簡易図

Batch : Data Lake/DB → Scheduled Job → Report Stream: Event → Broker → Stream Processor → Real-time Store

理解補足

X上のSystem Designリストでは、Batch/Streamがよく並びます。大事なのは、どちらが上位かではなく、遅延許容と再処理容易性で選ぶことです。

注意点

リアルタイムに見せたいからといって、全てStreamにすると運用が難しくなります。正確性が重要な集計ではBatchで最終確定する設計も現実的です。

Authorization Code Flowを使う理由は？

面接官の意図：ブラウザ/バックエンド構成の安全性と、フローの役割を理解しているか。

回答例を見る
回答例：
Authorization Code Flowは、認可サーバーがブラウザ経由で直接トークンを返すのではなく、まず短命の認可コードをクライアントへ返し、その後クライアントがトークンエンドポイントで認可コードをAccess TokenやID Tokenに交換する方式です。メリットは、トークンをURLフラグメントなどのフロントチャネルに直接露出させずに済むこと、サーバー側クライアントであればクライアント認証を組み合わせられること、PKCEによって認可コード横取りへの耐性を上げられることです。BFF構成では、この流れを使ってトークンをBFF側に保持し、ブラウザにはHttpOnly Cookieのセッションだけを持たせる設計にできます。

端的な役割

  1. ユーザーを認可サーバーへリダイレクトする。
  2. 認可サーバーでユーザー認証・同意・認可判断を行う。
  3. 認可サーバーがブラウザ経由で 認可コード をクライアントに返す。
  4. クライアントがバックチャネルでトークンエンドポイントに認可コードを提示し、Access Tokenなどを取得する。

なぜ安全性が上がるのか

  • Access Tokenそのものを認可エンドポイントからブラウザ経由で返さないため、URL、履歴、Referer、フロント側ログなどに露出するリスクを下げられる。
  • 認可コードは短命・一回限りで、盗まれてもPKCEやクライアント認証があれば悪用しにくい。
  • トークン発行をトークンエンドポイントに集約でき、クライアント認証、リダイレクトURI検証、PKCE検証などを組み合わせられる。
重要な補正：Authorization Code Flow自体が常に「ブラウザにAccess Tokenを保持させない」わけではありません。SPAでAuthorization Code Flow + PKCEを使う場合、最終的にブラウザ側がTokenを扱う構成もあります。ブラウザにAccess Tokenを持たせないのは、BFFやサーバーサイドWebアプリとして設計した場合のメリットです。

JWTのメリットとデメリットは？

面接官の意図：トークンベース認証の仕組みとトレードオフを説明できるか。

回答例を見る
回答例：
JWTは、ヘッダー・ペイロード・署名からなるトークン形式で、ペイロードにユーザー識別子、発行者、有効期限、スコープなどのClaimを入れられます。メリットは、Resource Server側で署名、有効期限、発行者、Audienceなどを検証すれば、毎回認可サーバーやセッションストアへ問い合わせずにトークンの正当性を確認できる点です。デメリットは、発行後の即時失効や権限変更の即時反映が難しいこと、漏洩時に有効期限まで悪用されること、Claimを詰め込みすぎると肥大化すること、署名付きJWTは暗号化ではないため中身を読めることです。

署名検証では何を検証しているのか

  • JWTは一般に Base64Url(header).Base64Url(payload).signature という形です。
  • 署名検証では、受け取った header.payload に対して、発行者の公開鍵、または共有秘密鍵を使って署名を検証します。
  • これにより「信頼している発行者が発行したこと」と「途中でペイロードが改ざんされていないこと」を確認します。

  • 署名付きJWT = 改ざん検知できる。中身を隠すものではない。暗号化したいならJWEなど別の仕組みが必要。

  • Access TokenとしてのJWTは、Resource ServerがAPIアクセス可否を判断する材料になる。ただし詳細な認可判断を全部JWTだけで完結させる必要はない。

  • ユーザーが「管理者」ロールを持つAccess Tokenを受け取る。

回答例を見る

Q022

N+1問題とは何ですか？

面接官の意図：JPAを使うなら必須。発行SQLを意識できているかを見る。

回答例を見る
回答例：
N+1問題は、親データを1回のSQLで取得した後、その親データN件それぞれに対して関連する子データを個別SQLで取得してしまい、合計で1+N回のSQLが発行される問題です。例えば注文一覧を1回取得した後、各注文の注文明細や顧客情報をループ内で遅延ロードすると、注文件数分だけ追加SQLが出ます。件数が増えるとDB往復が急増し、性能劣化します。対策は、JOIN FETCH、EntityGraph、バッチフェッチ、必要なDTOを直接取得するクエリなどです。

具体例

  1. SELECT * FROM orders LIMIT 100 で注文100件を取得する。
  2. 画面表示時に各注文の order.getItems() を呼ぶ。
  3. 遅延ロードにより SELECT * FROM order_items WHERE order_id = ? が100回発行される。
  4. 結果として、1回 + 100回 = 101回のSQLになる。

対策の使い分け

  • JOIN FETCH：必要な関連を一緒に取得する。便利だが、1対多で行数が膨らむことがある。
  • EntityGraph：JPAで取得したい関連を指定する。
  • バッチフェッチ：関連データを1件ずつではなくまとめて取得する。
  • DTOクエリ：一覧画面に必要な項目だけをSELECTする。画面/API用途では現実的。
注意：「JOINすれば全部解決」ではありません。ページングとの相性、重複行、取得しすぎ、メモリ使用量も見る必要があります。

ざっくりした正規形

  • 第1正規形：1つのカラムに複数値を詰め込まない。値を原子的にする。
  • 第2正規形：複合主キーの一部だけに依存する項目を分離する。
  • 第3正規形：主キー以外の項目に依存する項目を分離する。推移的依存を避ける。

Q024

楽観ロックと悲観ロックの違いは？

面接官の意図：並行更新の理解。違いを具体例で説明できるか。

回答例を見る
回答例：
悲観ロックは、競合が起きる前提で、更新対象の行を先にロックし、他トランザクションの更新を待たせる方式です。例えば SELECT ... FOR UPDATE のように、在庫や残高など同時更新されると困るデータで使われます。楽観ロックは、競合は少ない前提で、読み取り時にはロックせず、更新時にversion列などを条件に含め、他者が先に更新していたら更新失敗として検知する方式です。ユーザー編集画面のように競合頻度が低い場合に向きます。

悲観ロック

  • 更新前にロックを取る。
  • 競合を未然に防げる。
  • 高競合・強い整合性が必要な処理に向く。
  • 待ち時間、デッドロック、スループット低下に注意。

楽観ロック

  • 読み取り時にはロックしない。
  • 更新時に WHERE id = ? AND version = ? のように確認する。
  • 更新件数が0件なら、他者更新があったと判断してエラーや再読み込みにする。
  • 低競合でスループットを上げたい処理に向く。
違いの一言：悲観ロックは「先に止める」、楽観ロックは「後で衝突を検知する」です。

Q030

CSRFとXSSの違いは？

面接官の意図：Cookie Sessionを語るなら必須。攻撃の成立条件を説明できるか。

回答例を見る
回答例：
CSRFは、ログイン済みユーザーのブラウザに、別サイトから意図しないリクエストを送らせる攻撃です。Cookieは対象サイトへのリクエスト時にブラウザが自動送信するため、対策がないと攻撃者がユーザー本人の操作に見せかけて状態変更リクエストを送れる場合があります。XSSは、攻撃者のスクリプトを正規サイト上で実行させる攻撃です。XSSが成立すると、画面改ざん、任意リクエスト送信、LocalStorage上のToken窃取などが可能になります。

CSRFの具体例

  1. ユーザーが銀行サイトにログイン済みで、セッションCookieを持っている。
  2. 攻撃者サイトを開く。
  3. 攻撃者サイトが銀行サイトの送金APIへフォーム送信や画像読み込み等でリクエストを誘導する。
  4. ブラウザが銀行サイトのCookieを自動送信する。
  5. 銀行サイトがCSRF対策をしていないと、本人操作と区別できず処理してしまう。

主な対策

  • CSRF対策：CSRF Token、SameSite Cookie、Origin/Refererチェック、GETで状態変更しない。
  • XSS対策：出力エスケープ、入力値の適切な検証、CSP、危険なHTML挿入の回避、CookieのHttpOnly化。
整理：CSRFは「ブラウザがCookieを自動で送る」性質を悪用します。XSSは「正規サイト上で悪意あるJavaScriptが動く」ことが本質です。Cookie SessionではCSRF対策が重要で、TokenをLocalStorage等に置く構成ではXSSによるToken窃取が特に危険です。

回答例：

  1. ユーザーがドメインへアクセスする。

  • タスクのポートとターゲットグループのポートが噛み合っていない。

Kafkaの基本語彙

  • Producer: イベントをTopicへ書き込むアプリケーション。
  • Topic: イベントの論理的な保存先。
  • Partition: Topicを分割した単位。並列処理やスケール、順序性に関係する。
  • Consumer: Topicからイベントを読み取って処理するアプリケーション。
  • Consumer Group: 複数ConsumerでPartitionを分担して処理する単位。
  • Offset: Consumerがどこまで読んだかを示す位置情報。

設計で見るべき論点

  • 順序性: KafkaはPartition内では順序を保てるが、Topic全体で完全順序を保つわけではない。
  • 重複: 再送やリトライで同じイベントを複数回処理する可能性があるため、Consumer側の冪等性が重要。
  • 再処理: Offsetを戻して再処理できるが、外部副作用のある処理では注意が必要。
  • スキーマ進化: イベント形式を変更するとConsumerに影響するため、Schema Registryなどを検討する。
  • 整合性: 同期APIより疎結合になる一方、結果整合になるため業務要件に合うか確認する。

「触ったことはありません」で終わらせないこと。関連経験、技術の目的、設計観点、検証手順まで言えば、未経験でも思考力は伝わります。

回答例を見る
回答例：
GraphQLの実務経験はありませんが、GraphQLはAPI向けのクエリ言語とサーバーサイドランタイムで、クライアントが必要なフィールドを指定してデータを取得できる方式だと理解しています。RESTでは、リソース単位のエンドポイントを設計しやすい一方、画面が必要とするデータに対して過不足が出ることがあります。GraphQLでは、1つのQueryで複数の関連データを取得しやすく、クライアントごとの柔軟性が高いです。一方で、Resolver設計を誤るとN+1が起きやすく、認可をフィールド単位・リソース単位でどう適用するか、キャッシュをどう効かせるか、複雑すぎるクエリをどう制限するかが重要になります。

GraphQLの基本要素

  • Schema: APIで扱える型と関係を定義する。
  • Query: データ取得。
  • Mutation: データ変更。
  • Resolver: 各フィールドのデータ取得処理。
  • Subscription: リアルタイム通知用途で使われることがある。

RESTとの比較

  • REST: HTTPメソッドとリソースURIでシンプルに設計しやすい。HTTPキャッシュとも相性がよい。
  • GraphQL: クライアントが必要データを指定しやすい。BFF的な用途に向く場合がある。
  • 注意: GraphQLにしても、DB設計、認可、性能、キャッシュ、エラー設計は消えない。

よくある落とし穴

  • ネストしたQueryが大量のDBアクセスを生み、N+1になる。
  • HTTPステータスだけでエラー分類しづらく、エラーレスポンス設計が必要。
  • 柔軟すぎるQueryでサーバー負荷が読みにくくなる。
  • 認可チェックをResolverごとにどう適用するかが複雑になる。

Q045

Beanとは何ですか？

面接官の意図：SpringのDI/Iocを語るための基本語彙を理解しているかを見る。

回答例を見る
回答例：
Beanとは、Springコンテナによって生成・管理されるオブジェクトです。通常のJavaでは必要なクラスをnewして使いますが、Springでは@Component、@Service、@Repository、@Controller、@BeanなどでBeanとして登録し、依存関係をDIで注入します。これにより、オブジェクトの生成、ライフサイクル、依存関係、設定、AOP、トランザクション、例外変換などをSpringの仕組みに乗せられます。重要なのは、Beanとして管理されているからこそ、@Transactionalなどの機能がプロキシを通じて適用される点です。逆に、自分でnewしたオブジェクトはSpringコンテナの管理外なので、DIやAOPの対象になりません。

Beanを理解するポイント

  • IoC: オブジェクト生成や依存解決をアプリ側ではなくコンテナ側に任せる考え方。
  • DI: 必要な依存オブジェクトを外から注入する仕組み。
  • Scope: デフォルトはSingleton。リクエストごと、セッションごとなどのScopeもある。
  • Lifecycle: 初期化・破棄のタイミングをSpringが管理できる。

面接での具体例

ServiceクラスがRepositoryをコンストラクタで受け取る場合、ServiceもRepositoryもBeanとして登録されていれば、Springが依存関係を解決してインスタンスを生成します。これにより、テスト時にはRepositoryをモックへ差し替えるなどの設計もしやすくなります。

Q046

@Component、@Service、@Repositoryの違いは？

面接官の意図：責務ごとにSpringのステレオタイプを使い分けられるかを見る。

回答例を見る
回答例：
@Component、@Service、@RepositoryはいずれもSpringのComponent Scan対象になり、Beanとして登録されるステレオタイプアノテーションです。@Componentは汎用的なコンポーネント、@Serviceは業務処理やユースケースを表すサービス層、@RepositoryはDBアクセスなどの永続化層を表すために使います。実装上は同じようにBean登録されますが、コードを読む人に責務を伝える意味があります。また@Repositoryには、永続化層の例外をSpringのDataAccessException体系に変換する対象として扱う意味もあります。したがって、単に動くから何でも@Componentにするのではなく、責務に応じて使い分けるべきです。

使い分け

  • @Component: 特定レイヤーに限定されない汎用部品。
  • @Service: 業務ルール、ユースケース、トランザクション境界を持ちやすい層。
  • @Repository: DBや外部ストレージへのアクセスを担当する層。
  • @Controller / @RestController: HTTPリクエストを受けるプレゼンテーション層。

面接での補足

「全部Bean登録されるので同じです」だけだと浅いです。「実行上の共通点」と「設計上の意味づけ」を分けて説明するとよいです。特に@Serviceには業務処理、@Repositoryには永続化層という責務を示す意味があります。

Q047

PUTとPATCHの違いは？

面接官の意図：HTTPメソッドの意味と、API更新設計の注意点を理解しているかを見る。

回答例を見る
回答例：
PUTは、指定したリソースをリクエストボディの内容で全体置換する用途で使われます。PATCHは、指定したリソースに対して部分的な変更を適用する用途で使われます。PUTは同じリクエストを何度送ってもサーバー上の状態が同じになるため、冪等な設計にしやすいです。PATCHは部分更新なので、リクエスト形式によって冪等にも非冪等にもなり得ます。API設計では、未指定フィールドを「変更しない」と扱うのか、「nullで上書き」と扱うのかを明確にする必要があります。また、更新競合を防ぐために、ETagやIf-Match、versionカラムを使うこともあります。

具体例

  • PUT /users/1: ユーザー1のリソース全体を、送信されたJSON全体で置き換える。
  • PATCH /users/1: ユーザー1のnameだけ、emailだけなど、一部フィールドを更新する。

PATCHで注意すること

  • 未指定フィールドは変更なしなのか。
  • nullは値を消す意味なのか、それとも未指定扱いなのか。
  • 配列は全置換なのか、追加・削除差分なのか。
  • JSON Patch、JSON Merge Patch、独自形式のどれを使うのか。

面接での言い方

「PUTは全体更新、PATCHは部分更新」で大枠は十分ですが、もう一歩踏み込むなら「冪等性」と「未指定/nullの仕様」を話してください。Web系面接ではここまで言えると実務感が出ます。

Q048

外部キー制約を貼るメリット・デメリットは？

面接官の意図：DB制約で守るべき整合性と、アプリ側で守るべき境界を理解しているかを見る。

回答例を見る
回答例：
外部キー制約のメリットは、DBレベルで参照整合性を守れることです。例えばorder_items.order_idがorders.idを参照する場合、存在しない注文IDを持つ注文明細が登録されることをDBが防げます。アプリケーションにバグがあっても、DBが最後の防衛線になります。一方、デメリットは、削除やデータ移行、バルク投入、テストデータ投入が複雑になりやすいこと、書き込み時に制約チェックのコストがあること、マイクロサービスのようにDB境界を分ける設計ではサービスをまたいだ外部キーを貼れないことです。単一サービス・単一DB内で強い整合性が必要なら外部キーは有効ですが、サービス境界をまたぐ整合性はアプリケーションやイベントで管理することになります。

メリット

  • 存在しない親レコードへの参照を防げる。
  • DBが参照整合性を保証するため、アプリの実装漏れに強い。
  • データモデル上の関係が明示される。
  • JOINやER図の理解にもつながる。

デメリット

  • 削除時に参照先が残っていると削除できない。Cascade設計が必要。
  • 大量データ投入や移行時に制約順序を意識する必要がある。
  • 論理削除と相性が悪い場合がある。
  • マイクロサービス間でDBを分ける場合、サービスをまたいだ制約は貼れない。

面接での結論

「外部キーは絶対貼るべき」「外部キーは不要」と断定しない方がよいです。単一DB内の重要な整合性はDB制約で守り、サービス境界をまたぐ整合性はAPI・イベント・バッチ検査などで守る、という整理が現実的です。

Q049

ヘルスチェックはなぜ必要ですか？

面接官の意図：アプリが起動していることと、利用可能であることの違いを理解しているかを見る。

回答例を見る
回答例：
ヘルスチェックは、ロードバランサやECSが正常なタスクだけに通信を流し、異常なタスクを切り離すために必要です。プロセスが起動していても、DB接続ができない、依存サービスが落ちている、アプリ初期化が終わっていない、スレッドが詰まっているといった状態では、ユーザーリクエストを処理できません。ALBはターゲットグループのヘルスチェックでHealthyなターゲットへだけルーティングします。ECSもヘルスチェックに失敗したタスクを置き換えることがあります。設計では、単なる/healthだけでなく、起動確認、受付可能性、依存サービス確認をどう分けるかが重要です。Spring BootならActuatorのhealth endpointを使う選択肢もあります。

ヘルスチェックの種類

  • Liveness: プロセスが生きているか。落ちているなら再起動対象。
  • Readiness: リクエストを受け付けてよい状態か。初期化中や依存先障害なら外す。
  • Startup: 起動完了まで時間がかかるアプリで、起動中に誤って落とされないようにする。

設計上の注意

  • DBや外部APIまで毎回深く確認すると、ヘルスチェック自体が重くなる。
  • 依存サービス障害時に全タスクがUnhealthyになり、かえって復旧を難しくすることがある。
  • ALB向けには軽量で安定したエンドポイントを用意し、詳細診断は別の内部用エンドポイントに分けることもある。
  • DB必須のアプリなら、ReadinessではDB接続を確認する価値がある。

面接での言い方

「ヘルスチェックは死活監視です」だけだと浅いです。「プロセスが生きていること」と「ユーザーリクエストを処理できること」は違う、と説明できると運用視点が出ます。

Q050

キャッシュを入れるときの注意点は？

面接官の意図：性能改善のメリットだけでなく、不整合・失効・障害時の副作用を理解しているかを見る。

回答例を見る
回答例：
キャッシュは、読み取り性能を上げ、DB負荷や外部API呼び出しを減らすために使います。ただし、キャッシュを入れると必ず不整合、失効、更新反映、キャッシュスタンピード、メモリ使用量、障害時の挙動が論点になります。基本的には、更新頻度が低く、読み取り頻度が高く、多少古いデータを許容できるものに向いています。例えばマスタデータ、設定値、公開プロフィール、URL短縮のkey→original_urlなどです。設計時には、どのキーでキャッシュするか、TTLを何秒にするか、更新時に削除するのか上書きするのか、キャッシュが落ちたときにDBへフォールバックするのかを決めます。

代表的なパターン

  • Cache Aside: アプリがまずキャッシュを見て、なければDBから取得してキャッシュへ入れる。
  • Write Through: 書き込み時にDBとキャッシュを同時に更新する。
  • Write Behind: まずキャッシュへ書き、後からDBへ反映する。高性能だが整合性リスクがある。

主な落とし穴

  • 古いデータ: DB更新後にキャッシュが残ると古い値を返す。
  • キャッシュスタンピード: 人気キーが同時に期限切れになり、大量リクエストがDBへ流れる。
  • ホットキー: 特定キーへアクセスが集中する。
  • メモリ圧迫: 無制限にキャッシュするとRedis等のメモリを使い切る。
  • 障害時: キャッシュ障害でDBに負荷が集中する。

対策例

  • TTLを設定する。
  • 更新時に該当キーを削除する。
  • 人気キーは事前に温める。
  • 期限切れにランダム性を持たせる。
  • ロックやSingle Flightで同時再計算を防ぐ。
  • 古い値を一時的に返すstale-while-revalidate的な考え方を使う。
注意：キャッシュは「速くなる魔法」ではなく、整合性を犠牲にして性能を取る設計判断です。古いデータを返してよいかを確認せずに入れると事故ります。

Q051

fail-openとfail-closeの違いは？

面接官の意図：認可基盤・外部依存が障害になったときの安全側の設計を理解しているかを見る。

回答例を見る
回答例：
fail-openは、依存先の障害や判定不能時に処理を許可する考え方です。fail-closeは、判定不能時に処理を拒否する考え方です。認証認可のようなセキュリティ境界では、基本はfail-close、つまり判断できないなら拒否する設計が安全です。例えば認可基盤に問い合わせられないときに全リクエストを許可すると、本来アクセスできないリソースへアクセスできる危険があります。一方、fail-closeにすると認可基盤障害時にサービス全体が止まる可能性があります。そのため、重要操作はfail-close、低リスクな参照は短時間のキャッシュ済み認可判断を使う、管理者操作は必ずfail-closeにするなど、リスクに応じた設計が必要です。

使い分けの考え方

  • fail-close向き: 認証、認可、決済、個人情報参照、管理者操作、権限変更。
  • fail-openを検討し得るもの: ログ送信、メトリクス送信、レコメンド表示など、止める方が影響が大きく、セキュリティリスクが低い処理。
  • 中間案: 直近の認可結果を短時間キャッシュする、読み取りだけ許可する、機能を縮退する。

認可基盤での具体例

  1. API Gatewayが認可基盤へ問い合わせる。
  2. 認可基盤がタイムアウトする。
  3. fail-openなら許可して処理を進める。
  4. fail-closeなら403/503などで拒否・停止する。
  5. キャッシュ方式なら、過去数分以内の同一subject/resource/action判断を使う可能性がある。

面接での言い方

「セキュリティはfail-closeが基本。ただし可用性への影響が大きいので、操作のリスクごとにキャッシュ、縮退、監査ログ、アラートを組み合わせる」と答えるとバランスがよいです。

注意：fail-openは便利な逃げ道に見えますが、認可で安易に使うと重大事故になります。特に個人情報・金銭・管理者権限に関わる操作では、基本的にfail-closeを前提にした方が安全です。
