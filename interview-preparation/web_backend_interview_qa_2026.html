4. 海外面接型 Q&A

Amazon/Meta/Monzo/Stripe系の面接観点を、日本語の質問・回答例として再構成したものです。

Amazon系 / CS基礎 S
Q001

配列とHashMapをどう使い分けますか？

面接官の意図：データ構造の基本選択を説明できるか。

回答例を見る
配列はインデックスで順序を持つデータを連続的に扱うときに向きます。HashMapはキーから値を高速に取り出したいときに向きます。面接では、検索・挿入・削除・順序保持・重複許可・メモリ使用量を比較して選びます。例えばユーザーIDからユーザー情報を引くならMap、時系列に並んだログを順に処理するならList/配列を選びます。
Amazon系 / CS基礎 S
Q002

計算量 O(n) と O(log n) の違いを説明してください。

面接官の意図：アルゴリズムの効率を言語化できるか。

回答例を見る
O(n)は入力件数に比例して処理量が増えます。O(log n)は探索範囲を半分ずつ削るような処理で、入力が増えても処理増加が緩やかです。例えば線形探索はO(n)、ソート済み配列の二分探索はO(log n)です。実務では数千件なら差が小さくても、数百万件・高頻度APIになると設計判断に影響します。
Amazon系 / Coding A
Q003

2つの配列に共通する要素を求めるならどう実装しますか？

面接官の意図：素朴解から改善案に進めるか。

回答例を見る
まず二重ループだとO(n*m)です。片方の配列をHashSetに入れ、もう片方を走査すれば平均O(n+m)で求められます。重複を残すか、一意にするか、結果順を入力順にするかで実装は変わります。面接では最初に仕様確認を入れます。
Amazon系 / OOD A
Q004

オブジェクト指向設計で、継承よりコンポジションを選ぶ理由は？

面接官の意図：設計変更に強い構造を理解しているか。

回答例を見る
継承はis-a関係が明確で、基底クラスの契約が安定している場合に有効です。ただし継承階層が深くなると、親の変更が子に波及しやすくなります。コンポジションは必要な振る舞いを部品として持つため、差し替えやテストがしやすい。実務では、機能追加や仕様変更が見込まれる箇所ほどコンポジションを優先します。
Amazon系 / DB S
Q005

トランザクションのACIDを説明してください。

面接官の意図：DBの基本的な信頼性モデルを理解しているか。

回答例を見る
Atomicityは全部成功か全部失敗、Consistencyは制約を満たした状態を維持、Isolationは並行実行時の干渉を制御、Durabilityはコミット後に永続化される性質です。例えば注文作成と在庫減算は、途中失敗で片方だけ反映されると不整合になるため、トランザクション境界を明確にします。
Amazon系 / Distributed Systems A
Q006

分散システムで障害を前提に設計するとはどういうことですか？

面接官の意図：ネットワーク・依存サービス失敗を前提に考えられるか。

回答例を見る
単一プロセス内の例外だけでなく、タイムアウト、部分障害、リトライによる重複、依存APIの遅延、メッセージ欠落、ネットワーク分断を前提にします。対策は、タイムアウト、リトライ、冪等性、サーキットブレーカー、キュー、監視、フォールバックなどです。重要なのは『失敗しない設計』ではなく『失敗しても被害を限定し復旧できる設計』です。
Meta系 / Coding S
Q007

コーディング面接で、実装前に何を確認しますか？

面接官の意図：要件確認・仮定明示・協調的な進め方ができるか。

回答例を見る
入力の型、サイズ、null/空、重複、ソート済みか、返却形式、エラー時の扱いを確認します。その後、例を1〜2個作り、素朴解と改善解を比較し、計算量を説明してから実装します。黙って書き始めるより、前提を揃えてから進めた方が評価されやすいです。
Meta系 / System Design A
Q008

チャットサービスを設計してください。最初に何から話しますか？

面接官の意図：広い問題を分解できるか。

回答例を見る
まずスコープを絞ります。1対1かグループか、既読・未読、履歴保存、リアルタイム性、通知、添付ファイル、想定ユーザー数を確認します。次にAPI、データモデル、メッセージ配送、WebSocket/ポーリング、DB、キャッシュ、キュー、障害時再送、順序保証を順に設計します。最初から詳細に飛び込まず、機能要件と非機能要件を切り分けます。
Meta系 / Behavioral A
Q009

技術的に意見が対立した経験を説明してください。

面接官の意図：協調性・技術判断・説得の仕方を見る。

回答例を見る
回答はSTARで構成します。状況、対立した論点、判断基準、結果、学びです。自分の正しさだけを主張せず、性能・保守性・納期・運用負荷などの評価軸を明示し、合意形成のために小さな検証や比較表を使った、と話すと技術職として自然です。
Meta系 / Design A
Q010

高トラフィックなフィード機能をどう設計しますか？

面接官の意図：読み書き比率、キャッシュ、非同期処理を考えられるか。

回答例を見る
投稿、フォロー関係、フィード取得の読み書き比率を確認します。読み取りが多いなら、投稿時にフォロワーのフィードを事前生成するfan-out on write、または取得時に集約するfan-out on readを比較します。ユーザー規模、投稿頻度、有名人ユーザー、整合性要求によってハイブリッドにします。キャッシュ、キュー、ランキング、重複排除も論点です。
Monzo系 / Project Deep Dive S
Q011

最近のプロジェクトで、一番難しかった技術課題は何ですか？

面接官の意図：経験の深さ、技術判断、振り返りを見る。

回答例を見る
あなたの場合は『共通認証・認可基盤をマイクロサービス全体に適用する設計』を話すのが強いです。Keycloak/OIDC/OpenFGA、ユーザ管理の責務分担、どのタイミングで認可確認するか、各社の実装前提を揃える必要があった点を説明します。単なる調整ではなく、認証・認可・責務境界の技術的整理だったと明確に話します。
Monzo系 / Project Deep Dive S
Q012

その設計で、どんなトレードオフがありましたか？

面接官の意図：設計判断の比較軸を持っているか。

回答例を見る
例として『認可情報を各サービスに持たせるか、OpenFGAのような外部認可に寄せるか』を話せます。各サービス内で完結すればシンプルでレイテンシも抑えやすい一方、権限ロジックが分散します。外部認可は一元化・変更容易性がある一方、依存先障害やネットワーク遅延を考慮する必要があります。
Monzo系 / Pair Coding A
Q013

実務的な小プログラムをペアで作る場合、どう進めますか？

面接官の意図：実装力だけでなく、対話しながら進められるか。

回答例を見る
最初に入力・出力・異常系・サンプルを確認し、最小実装を作ります。その後、テストケースを追加し、境界値や例外を潰します。実装中は『いま何を確認しているか』『次に何を改善するか』を短く説明します。完璧な設計より、動くものを作って段階的に改善する姿勢が重要です。
Monzo系 / Reliability A
Q014

顧客向けAPIで障害が起きた場合、どう調査しますか？

面接官の意図：運用・可観測性・優先順位を理解しているか。

回答例を見る
まず影響範囲、発生時刻、エラー率、特定ユーザー/全体、直近リリース有無を確認します。ログ、メトリクス、トレース、ALB/ECS/アプリログ、DB接続状況を見ます。一次対応として切り戻し・機能停止・リトライ制御を検討し、恒久対応では再発防止、監視、アラート、テスト追加まで落とします。
Stripe系 / Bug Bash A
Q015

既存コードにバグがあると言われたら、どう調査しますか？

面接官の意図：闇雲に修正せず、仮説検証できるか。

回答例を見る
まず再現条件を固定します。期待値と実際の差分を確認し、入力、状態、外部依存、境界値を切り分けます。ログやテストを追加して、どこで値が崩れるかを追います。修正は最小差分で行い、回帰テストを追加します。『なんとなく直す』のではなく、再現→仮説→検証→修正→再発防止の順に進めます。
Stripe系 / Integration A
Q016

外部API連携を実装するとき、何に注意しますか？

面接官の意図：実務的なAPI連携の失敗パターンを理解しているか。

回答例を見る
認証方式、タイムアウト、リトライ、冪等性、レート制限、エラーコード、レスポンス互換性、監査ログ、秘密情報管理を確認します。決済や注文系なら重複実行が致命的なので、idempotency keyや状態遷移を明確にします。外部APIは落ちる前提で、失敗時のユーザー体験と再処理方法まで設計します。
Stripe系 / Practical Coding A
Q017

標準的なLeetCodeではなく、業務ロジック実装問題が出たらどう戦いますか？

面接官の意図：抽象アルゴリズムではなく、仕様理解と堅実な実装を見る。

回答例を見る
まず仕様を小さく分けます。正常系、異常系、境界値、状態遷移、入出力形式を確認し、シンプルなデータ構造で実装します。最初から過剰な抽象化をせず、テストケースを増やしながら堅くします。業務ロジック問題では、最短コードより仕様漏れの少なさが重要です。
Stripe系 / Communication A
Q018

制約が途中で変わった場合、どう対応しますか？

面接官の意図：変更への耐性、設計の柔軟性を見る。

回答例を見る
まず変更内容が機能要件・非機能要件・データモデル・API契約のどこに影響するかを切り分けます。既存設計との差分、壊れる前提、移行手段、互換性維持の必要性を確認します。その上で短期対応と恒久対応を分け、必要なら段階リリースやfeature flagでリスクを下げます。

5. あなた向け優先 Q&A

あなたの職務経験と志向を踏まえると、面接で一番守るべき領域は 認証認可・API・Java/Spring・DB・AWS/ECS/CI/CD・システム設計 です。特に認証認可とAWSは経験として話せる分、深掘りに耐えられないと逆に弱点に見えます。

S: 経験深掘り・認証認可 S
Q001

認証と認可の違いを説明してください。

面接官の意図：あなたの主戦場。ここで浅いと経験全体が弱く見える。

回答例を見る
認証は『誰であるか』を確認すること、認可は『その人が何をしてよいか』を判断することです。例えばログインしてユーザーIDを確認するのが認証で、そのユーザーが特定APIを実行できるか確認するのが認可です。実務ではKeycloakのようなIdPで認証し、API実行可否はロールやOpenFGAのような認可基盤で判断する構成にできます。
S: 経験深掘り・認証認可 S
Q002

OAuth 2.0 と OIDC の違いは何ですか？

面接官の意図：OIDC経験の本質理解を見る。

回答例を見る
OAuth 2.0は主に認可のフレームワークで、クライアントがリソースへアクセスするためのアクセストークンを取得する仕組みです。OIDCはOAuth 2.0の上に認証レイヤーを追加したもので、ID Tokenによってユーザーの認証結果をクライアントが確認できます。つまり、OAuth 2.0はアクセス権限、OIDCはユーザー認証情報も扱う、という整理です。
S: 経験深掘り・認証認可 S
Q003

Authorization Code Flowを使う理由は？

面接官の意図：ブラウザ/バックエンド構成の安全性を理解しているか。

回答例を見る
ブラウザにアクセストークンを直接返すより、認可コードを一度バックエンドやクライアントに返し、トークンエンドポイントで交換する方が安全に扱いやすいからです。特に機密クライアントやBFF構成では、トークンをサーバ側で扱い、ブラウザにはセッションCookieを持たせる設計にできます。PKCEを併用すると、認可コード横取りリスクも下げられます。
S: 経験深掘り・認証認可 S
Q004

JWTのメリットとデメリットは？

面接官の意図：トークンベース認証のトレードオフを説明できるか。

回答例を見る
メリットは、署名検証により各サービスが認証結果を確認でき、セッションストアへの問い合わせを減らせることです。デメリットは、発行後の即時失効が難しい、トークン肥大化、漏洩時の影響、権限変更の即時反映が難しい点です。したがって、短い有効期限、Refresh Token管理、必要以上の権限情報を詰めない設計が重要です。
S: 経験深掘り・認証認可 S
Q005

認可情報をJWTに全部入れない理由は？

面接官の意図：OpenFGA利用理由を語れるか。

回答例を見る
JWTに権限を詰め込むと、トークンが肥大化し、権限変更の即時反映が難しくなります。また、複雑な組織・リソース単位の権限をトークン内で表現すると管理が難しくなります。外部認可基盤に問い合わせる設計なら、認可ロジックを一元化し、リソース・関係性ベースの判断を柔軟に変更できます。ただし、認可基盤への依存、レイテンシ、障害時挙動は設計課題です。
S: 経験深掘り・認証認可 S
Q006

RBACとReBACの違いは？

面接官の意図：OpenFGAを使った意味を理解しているか。

回答例を見る
RBACはユーザーにロールを付与し、ロールごとに操作権限を管理する方式です。ReBACはユーザー、組織、リソースなどの関係性をもとに権限を判断します。例えば『この組織の管理者は、この組織配下のリソースを操作できる』という関係を表現しやすいのがReBACです。単純なAPIロール制御ならRBACで十分ですが、組織階層や所有関係が絡むとReBACが強くなります。
S: 経験深掘り・認証認可 S
Q007

API Gatewayで認可する場合と各サービスで認可する場合の違いは？

面接官の意図：境界設計を語れるか。

回答例を見る

現時点ではJava/Spring Bootが中心で、Goの実務経験は限定的です。ただ、深めたいのはバックエンドにおけるAPI設計、認証認可、DB、運用を見据えた実装です。Goを使う場合は、標準ライブラリ、goroutine/channel、エラーハンドリング、context、interface、テストの書き方を重点的にキャッチアップします。言語差分はありますが、バックエンド設計の基本は転用できます。
S: 未経験技術への回答型 A
Q043

Kubernetesの経験はありますか？

面接官の意図：ECS経験との比較を見られる。

回答例を見る
Kubernetesを本格運用した経験はありません。実務ではECS Fargateでコンテナ環境を構築しました。KubernetesはPod、Deployment、Service、Ingress、ConfigMap/Secretなどでコンテナを柔軟に管理できる一方、クラスタ運用の学習・管理コストが高いと理解しています。要件がECSで満たせるなら運用負荷を下げられますが、マルチクラウドや高度な制御が必要ならKubernetesが候補になります。
S: 未経験技術への回答型 A
Q044

GraphQLの経験はありますか？

面接官の意図：REST以外のAPIへの理解。

回答例を見る
実務経験はありませんが、GraphQLはクライアントが必要なフィールドを指定でき、複数リソースの取得を柔軟に扱えるAPI方式と理解しています。RESTはリソース単位でシンプルに設計しやすい一方、画面要件により過不足が出ることがあります。GraphQLは柔軟ですが、認可、N+1、キャッシュ、スキーマ設計、クエリ複雑性制限が重要になります。
A: Java / Spring Boot B
Q045

Beanとは何ですか？

面接官の意図：Springコンテナの基本語彙。

回答例を見る
Springコンテナによって生成・管理されるオブジェクトです。@Component、@Service、@Repository、@Controller、@Beanなどで登録され、DIにより他のBeanへ注入できます。単にnewするのではなくコンテナに管理させることで、ライフサイクル、設定、AOP、トランザクションなどの仕組みと連携できます。
A: Java / Spring Boot B
Q046

@Component、@Service、@Repositoryの違いは？

面接官の意図：責務を意識して使えているか。

回答例を見る
いずれもBean登録されますが、意味づけが違います。@Componentは汎用コンポーネント、@Serviceは業務処理・ユースケース層、@Repositoryは永続化層です。@RepositoryにはDB例外変換などSpringの永続化サポートの意味もあります。実装上同じように動いても、読み手に責務を伝えるために使い分けます。
A: API / Web / Security B
Q047

PUTとPATCHの違いは？

面接官の意図：HTTPメソッドの使い分け。

回答例を見る
PUTはリソース全体の置換、PATCHは一部更新として使うことが多いです。PUTは同じリクエストを何度実行しても同じ結果になる冪等な設計にしやすいです。PATCHは部分更新なので、フィールド未指定を『変更しない』と解釈するなど、API仕様を明確にする必要があります。
A: DB / SQL B
Q048

外部キー制約を貼るメリット・デメリットは？

面接官の意図：DB制約とアプリ制御の比較。

回答例を見る
メリットはDBレベルで参照整合性を守れることです。アプリのバグがあっても不正な参照を防ぎやすくなります。デメリットは削除や移行、バルク投入、マイクロサービス分割時に制約が重くなることです。単一DB内で強い整合性が必要なら有効ですが、境界をまたぐ場合はアプリ側やイベントで整合性を管理することもあります。
A: AWS / Docker / CI/CD B
Q049

ヘルスチェックはなぜ必要ですか？

面接官の意図：運用視点。

回答例を見る
ロードバランサやECSが正常なタスクだけに通信を流すためです。プロセスが起動していてもDB接続や依存サービスが死んでいる場合があるため、単なるポート確認では不十分な場合があります。/healthの設計では、軽量性と実際の利用可能性のバランスを取ります。
S: システム設計 B
Q050

キャッシュを入れるときの注意点は？

面接官の意図：性能改善の副作用を理解しているか。

回答例を見る
キャッシュは読み取り性能を上げ、DB負荷を下げますが、データ不整合、期限切れ、無効化、キャッシュスタンピード、メモリ使用量が課題になります。更新頻度が低く読み取りが多いデータに向きます。どのキーでキャッシュし、いつ消すか、古いデータを許容できるかを先に決めます。
S: 経験深掘り・認証認可 B
Q051

fail-openとfail-closeの違いは？

面接官の意図：認可基盤障害時の設計。

回答例を見る
fail-openは依存先障害時に処理を許可する考え方、fail-closeは拒否する考え方です。認可では安全側に倒すならfail-closeが基本ですが、可用性要求が高い内部処理では一部キャッシュ許可などもありえます。セキュリティリスクとサービス停止リスクを比較して決めます。

6. 未経験技術を聞かれたときの回答テンプレート

知らない技術を聞かれたときに黙る必要はありません。ただし、知ったかぶりは最悪です。次の型で答えます。

実務で直接扱った経験はありません。 ただ、関連する経験としては〇〇があります。 私の理解では、この技術は□□を解決するためのものです。 設計時に見るべき観点は、A、B、Cだと思います。 実務で扱うなら、まず公式ドキュメントと既存実装を確認し、小さく検証して、性能・運用・障害時の挙動を確認します。

この型の狙い

面接官は、未経験技術を知っているかだけを見ているわけではありません。未知の技術に対して、既存知識から構造化し、検証計画を立てられるかを見ています。あなたは「深く理解した領域では強い」タイプなので、未経験技術でもこの型を使えば沈黙を避けられます。

7. 面接前の学習順序

  1. 1日目: 自分の経験深掘り — 認証認可、AWS、CI/CD、ユーザ管理責務分担を30秒/2分で話す。
  2. 2日目: Java/Spring — DI、Controller/Service/Repository、@Transactional、例外処理、JPA/MyBatis。
  3. 3日目: DB/API — インデックス、N+1、トランザクション、REST、HTTPステータス、冪等性。
  4. 4日目: AWS/Docker/CI/CD — ALB→ECS、ECR、CodePipeline、CloudFormation、ヘルスチェック。
  5. 5日目: システム設計 — URL短縮、通知、認証基盤、EC注文、ファイルアップロード。
  6. 6日目以降: 口頭練習 — このHTMLのdetailsを閉じた状態で、回答を声に出してから開いて確認。

厳しめの現実

あなたが落ちるとしたら、未知技術を知らないからではなく、経験済みの認証認可・AWS・API設計を面接レベルで説明し切れない場合です。まずは「自分がやったこと」を技術質問に耐える形へ変換してください。
