User API Search and JPA Basics

日付: 2026-05-21

対象アプリ: apps/user-app

1. JPA Entity で final を避ける理由

JPA の Entity は、データベースから取得した値をもとにインスタンスを復元する。 そのため、Entity のフィールドは通常 final にしない。 final を付けると Java のルール上、コンストラクタで必ず初期化する必要がある。 しかし @GeneratedValue の id は DB 側で採番されるため、コンストラクタで値を入れられずコンパイルエラーになる。 

@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;

final を付けないことは「自由に変更してよい」という意味ではない。 アプリケーションコードから変更させたくない場合は、setter を作らず、必要な変更だけを業務メソッドとして用意する。 

public void changeName(String name) {
    this.name = name;
}

2. JPA Entity に引数なしコンストラクタが必要な理由

JPA は DB の行から Entity を復元するとき、まず空のインスタンスを生成し、その後でフィールドへ値を設定する。 そのため Entity には引数なしコンストラクタが必要になる。 

protected User() {
}

public にするとアプリケーションコードから不完全な User を作れてしまうため、JPA 用には protected がよく使われる。

3. enum の valueOf() と name()

Role が enum の場合、リクエスト DTO では文字列、Entity では enum として扱うことがある。 その場合、保存前に String から Role へ変換し、レスポンス作成時に Role から String へ戻す。 

Role role = Role.valueOf(userCreateRequest.role());

valueOf() は文字列を enum に変換する。例えば "MEMBER" は Role.MEMBER になる。 ただし完全一致が必要で、"member" や "Member" はそのままだと失敗する。 

String roleName = saved.getRole().name();

name() は enum を文字列に変換する。例えば Role.MEMBER.name() は "MEMBER" になる。

注意: IDE の自動 import により、誤って org.springframework.context.support.BeanDefinitionDsl.Role を import すると、 自作の ts.com.user_app.entity.Role と型が合わずコンパイルエラーになる。 
// 正しい import
import ts.com.user_app.entity.Role;

4. 検索 API の設計

今回の検索 API は、戻り値をすべて List<UserResponse> に統一する方針にした。 id、loginId、name は結果が 0 件または 1 件になりやすいが、API としては配列で返すことで、複数値検索にも対応できる。

リクエスト例 意味
GET /user/search?id=1&id=2 複数 ID で検索
GET /user/search?loginId=tanaka&loginId=suzuki 複数 loginId で検索
GET /user/search?name=田中&name=鈴木 複数 name で検索
GET /user/search?role=MEMBER role で検索

検索条件は「1種類だけ」許可する。ただし、その条件の値は複数指定できる。 

// 許可
GET /user/search?loginId=tanaka&loginId=suzuki

// 禁止
GET /user/search?loginId=tanaka&name=田中

5. Controller で行う制御

「検索条件が 1 種類だけ指定されているか」は HTTP リクエストの形の妥当性なので、Controller 側で制御するのが自然。 Service は、すでに意味が確定した検索処理を受け持つ。 

@GetMapping("/search")
public ResponseEntity<List<UserResponse>> searchUsers(
        @RequestParam(required = false) List<Long> id,
        @RequestParam(required = false) List<String> loginId,
        @RequestParam(required = false) List<String> name,
        @RequestParam(required = false) List<String> role
) {
    int conditionCount = 0;

    if (id != null && !id.isEmpty()) conditionCount++;
    if (loginId != null && !loginId.isEmpty()) conditionCount++;
    if (name != null && !name.isEmpty()) conditionCount++;
    if (role != null && !role.isEmpty()) conditionCount++;

    if (conditionCount == 0) {
        throw new IllegalArgumentException("検索条件を1種類指定してください。");
    }

    if (conditionCount > 1) {
        throw new IllegalArgumentException("検索条件は1種類だけ指定してください。");
    }

    if (id != null && !id.isEmpty()) {
        return ResponseEntity.ok(userService.searchUserById(id));
    }

    if (loginId != null && !loginId.isEmpty()) {
        return ResponseEntity.ok(userService.searchUserByLoginId(loginId));
    }

    if (name != null && !name.isEmpty()) {
        return ResponseEntity.ok(userService.searchUserByName(name));
    }

    return ResponseEntity.ok(userService.searchUserByRole(role));
}
ポイント: /user/search は検索条件ありの API として扱う。 全件取得は別 API の /user/all に分けるなら、検索条件なしの /user/search は 400 Bad Request 相当にする。

6. Repository の標準メソッドと独自検索メソッド

JpaRepository が最初から持っているメソッドは、Repository に書かなくても使える。

メソッド Repository への記述 理由
findAll() 不要 JpaRepository 標準
findById(Long id) 不要 JpaRepository 標準
findAllById(Iterable<Long> ids) 不要 JpaRepository 標準
save(User user) 不要 JpaRepository 標準
deleteById(Long id) 不要 JpaRepository 標準
findByLoginIdIn(...) 必要 User 固有のフィールド検索

loginId は User Entity 固有のフィールドなので、複数 loginId 検索をしたい場合は Repository に明示的に宣言する。 

public interface UserRepository extends JpaRepository<User, Long> {

    List<User> findByLoginIdIn(List<String> loginIds);

    List<User> findByNameIn(List<String> names);

    List<User> findByRoleIn(List<Role> roles);
}

findByLoginIdIn は Spring Data JPA がメソッド名を解析し、実行時に次のような SQL 相当のクエリを生成する。 

where login_id in (?, ?, ?)

7. Service 実装の例

複数 loginId 検索では、Repository の findByLoginIdIn を呼び出し、取得した User を UserResponse に変換する。 

@Override
public List<UserResponse> searchUserByLoginId(List<String> loginId) {
    return userRepository.findByLoginIdIn(loginId)
            .stream()
            .map(user -> new UserResponse(
                    user.getId(),
                    user.getLoginId(),
                    user.getName(),
                    user.getRole().name(),
                    user.getPassword()
            ))
            .toList();
}

8. 今回見つかった典型的なエラー

  • return がない: 戻り値があるメソッドでは、すべての分岐で値を返す必要がある。
  • Interface と実装クラスの不一致: UserService に定義したメソッドは UserServiceImpl で実装する必要がある。
  • 不要な @Override: Interface にないメソッドへ @Override を付けるとエラーになる。
  • 誤った import: 自作の Role ではなく Spring 側の Role を import すると型不一致になる。
  • Repository メソッド未定義: findByLoginIdIn などの固有検索は Repository に宣言が必要。
  • Repository 呼び出しミス: userRepository(updatedUser) ではなく userRepository.save(updatedUser)。

9. 今後の改善ポイント

  • Role.valueOf(...) に不正な値が来た場合、400 Bad Request として返す例外ハンドリングを追加する。
  • name が本当に一意なら、DB 側にも unique 制約を付ける。
  • UserResponse に password を含めるかは再検討する。通常、API レスポンスでは password を返さない。
  • Entity 名 User は DB の予約語と衝突する可能性があるため、@Table(name = "users") などを検討する。
