1. 全体像

このテーマで最も重要なのは、仕様・実装・実行環境・アプリケーションコードを分けて考えることです。 混乱の多くは、「コンテナ」「サーバ」「API」という言葉が複数のレイヤーで使われることから起きます。

核心 Webアプリは、あなたが書いたJavaコードだけで動いているわけではありません。 あなたのコード + フレームワークのコード + コンテナ/サーバのコード + JVMが一緒に動いています。
大まかなレイヤー
OS
  ↓
JVM
  ↓
Javaで動くサーバ/フレームワーク
  ├─ Jakarta EEアプリケーションサーバ
  │   ├─ Webコンテナ
  │   ├─ CDIコンテナ
  │   ├─ EJBコンテナ
  │   └─ 各種サービス
  │
  └─ Spring Bootアプリ
      ├─ 組み込みTomcat
      ├─ Spring ApplicationContext
      ├─ Spring MVC
      ├─ Spring Security
      └─ Spring Transaction
  ↓
あなたのアプリケーションコード

疑問Jakarta EEアプリケーションサーバはJVMなのか？

解説違います。JVMはJavaバイトコードを実行する土台です。Jakarta EEアプリケーションサーバは、JVM上で動くサーバ製品です。Servlet、CDI、EJB、トランザクション管理などはJVM自体が提供するのではなく、サーバ製品やライブラリが提供します。

2. 用語の位置づけ

用語 分類 意味
JVM 実行基盤 Javaのバイトコードを実行するランタイム。ServletやSpring Beanを直接提供するものではない。
Jakarta EE 標準仕様群 Javaでエンタープライズアプリケーションを作るための標準仕様。Servlet、CDI、EJB、JPA、JTA、Validationなどを含む。
Jakarta EE API プログラミングAPI Jakarta EE仕様として定義されたJavaのクラス、インターフェース、アノテーション群。
Jakarta EEアプリケーションサーバ サーバ製品 Jakarta EE仕様を実装したサーバ。WildFly、Payara、Open Liberty、WebLogicなど。
Jakarta EEコンテナ 実行機構 アプリケーションサーバ内部で、Servlet、CDI Bean、EJBなどを生成・管理・呼び出しする仕組み。
Webコンテナ / Servletコンテナ HTTP/Servlet実行環境 HTTPリクエストを受け、Servletを実行する環境。Tomcatが代表例。
CDIコンテナ Jakarta EE側のDIコンテナ CDI Beanの生成、スコープ管理、依存性注入を行う。
EJBコンテナ Enterprise Beans実行環境 EJBのライフサイクル、トランザクション、セキュリティ、プーリングなどを管理する。
Tomcat Servletコンテナ Servlet/JSP周辺を実装するWebコンテナ。フルJakarta EEアプリケーションサーバではない。
Spring Framework Javaフレームワーク DI、AOP、MVC、Transactionなどを提供するSpring独自のフレームワーク。
Spring Boot Springアプリ構築基盤 Springアプリを簡単に構成・起動する仕組み。組み込みTomcatを含めて実行可能JARとして動かす構成が多い。
Spring DIコンテナ Spring側のDIコンテナ Spring Beanを生成・注入・管理するApplicationContextのこと。
注意 「コンテナ」という言葉は1種類ではありません。Servletコンテナ、CDIコンテナ、EJBコンテナ、Spring DIコンテナ、Dockerコンテナは、それぞれ別の文脈の言葉です。

3. Jakarta EE仕様とAPI

Jakarta EEは、単なる参考資料ではなく、Javaでエンタープライズアプリケーションを作るための標準仕様群です。 仕様とは、「どのようなAPIを提供し、どのように振る舞うべきか」を定義するルールです。

仕様・API・実装・アプリの関係
Jakarta EE仕様
  ├─ 仕様書
  ├─ Java API
  │   ├─ class
  │   ├─ interface
  │   └─ annotation
  └─ 準拠確認用のテスト/ルール

実装製品
  ├─ Tomcat
  ├─ Jetty
  ├─ WildFly
  ├─ Payara
  └─ Open Liberty

アプリ開発者
  └─ Jakarta EE APIを使ってアプリを書く

3.1 Jakarta EE APIはインターフェースだけなのか

インターフェースだけではありません。クラス、抽象クラス、アノテーション、例外クラスなども含みます。

種類 例 説明
interface HttpServletRequest, HttpServletResponse 実装クラスを隠し、アプリコードとコンテナの接点になる。
class / abstract class HttpServlet 開発者が継承して使う基底クラス。
annotation @Inject, @Entity, @NotNull コンテナや実装ライブラリに意味を伝えるメタデータ。
exception / utility 各種例外クラスなど API利用時の例外表現や補助機能。

疑問Jakarta EE APIはインターフェースで、実装クラスはTomcat / Jetty / WildFlyが実装する理解でよいか？

解説大筋では合っています。ただしAPIはインターフェースだけではありません。Jakarta EE APIにはクラスやアノテーションも含まれます。実際の処理は、Tomcat、Jetty、WildFly、Payara、Hibernateなどの実装製品が仕様に沿って提供します。

3.2 「契約」と表現する理由

Jakarta EE APIは「Javaのクラス・インターフェース・アノテーション群」と言っても間違いではありません。 ただし、それだけだと「そのAPIライブラリを入れれば実行機能まで入る」と誤解しやすいです。

ここでいう「契約」とは、アプリ開発者とサーバ/ライブラリ実装者の双方が従う標準ルールという意味です。

アプリ開発者側の契約

このAPIを使って、この形式でコードを書けば、コンテナが適切に呼び出してくれる。

実装製品側の契約

このAPIに対応する実装を用意し、仕様で定められた振る舞いを満たす。

疑問Javaのすべてのライブラリも契約ではないのか？

解説広い意味ではその通りです。すべてのAPIは利用者との契約です。ただしJakarta EE APIは、単一ライブラリのAPIというより、複数の実装製品が準拠するための標準仕様としての契約です。そのため特に「契約」と表現しました。

4. Jakarta EEアプリケーションサーバとコンテナ

4.1 Jakarta EEアプリケーションサーバとは

Jakarta EEアプリケーションサーバは、Jakarta EE仕様を実装したサーバ製品です。 JVM上で動作し、内部にWebコンテナ、CDIコンテナ、EJBコンテナ、トランザクション管理、セキュリティ、リソース管理などを持ちます。

Jakarta EEアプリケーションサーバのイメージ
Jakarta EEアプリケーションサーバ
  ├─ Webコンテナ
  │   └─ Servlet / JSP / RESTリソースなどを実行
  ├─ CDIコンテナ
  │   └─ CDI Beanの生成・DI・スコープ管理
  ├─ EJBコンテナ
  │   └─ EJBのライフサイクル・トランザクション・セキュリティ管理
  ├─ JTA Transaction Manager
  ├─ DataSource / 接続プール
  ├─ Security
  └─ その他エンタープライズ機能

4.2 Jakarta EEアプリケーションとは

Jakarta EEアプリケーションは、開発者が作る業務アプリケーションです。 典型的にはWARやEARとしてパッケージングされ、Jakarta EEアプリケーションサーバにデプロイされます。 

myapp.war / myapp.ear
  ├─ Servlet / RESTリソース
  ├─ CDI Bean
  ├─ EJB
  ├─ JPA Entity
  ├─ 設定ファイル
  └─ 業務ロジック

疑問Jakarta EEアプリケーションとJakarta EEコンテナのイメージが湧かない。

解説Jakarta EEアプリケーションは「開発者が作る業務アプリ」です。コンテナは、そのアプリを読み込み、インスタンス生成、DI、リクエスト時のメソッド呼び出し、トランザクション、セキュリティ、リソース管理を行う実行環境です。アプリは単独で全部を動かすのではなく、コンテナに管理される部品群として作られます。

4.3 コンテナは具体的に何をしているのか

コンテナの本質は、あなたのアプリケーションコードを、適切なタイミング・依存関係・実行条件で呼び出すことです。

WAR/EARを読む
クラスをスキャン
アノテーション解析
インスタンス生成
DI
URL対応付け
リクエスト処理
トランザクション制御
HTTPリクエスト
  ↓
アプリケーションサーバ
  ↓
Webコンテナ
  ↓
対象のServlet / RESTリソースを特定
  ↓
必要なBeanを取得・DI
  ↓
認証・認可、トランザクションなどを制御
  ↓
あなたの業務メソッドを呼び出す
  ↓
HTTPレスポンスを返す

疑問なぜJavaコードで実現していないのか？

解説実際にはJavaコードで実現されています。ただし、それはあなたのアプリケーションコードではなく、Tomcat、WildFly、Payaraなどのサーバ製品側のコードです。HTTP処理、DI、トランザクション、リソース管理などを毎回アプリ開発者が実装するのは非現実的なため、サーバやフレームワークが共通基盤として提供します。

5. Tomcatの位置づけ

Tomcatは、主にServletコンテナ / Webコンテナです。 Servlet/JSPなどを実行するHTTP系の実行基盤であり、フル機能のJakarta EEアプリケーションサーバではありません。

製品 位置づけ 主な役割
Tomcat Servletコンテナ HTTPを受ける、Servletを動かす、JSPを処理する。
Jetty Servletコンテナ 軽量なServlet/Web実行基盤。
WildFly Jakarta EEアプリケーションサーバ Servlet、CDI、EJB、JTA、JPA、JMSなど広範囲を提供。
Payara Jakarta EEアプリケーションサーバ Jakarta EE仕様群を広く実装。

疑問Tomcatは結局どこに位置するのか？

解説TomcatはServlet仕様を実装するWebコンテナです。Spring Bootでは組み込みTomcatとしてアプリ内に入り、HTTPリクエストを受ける役割を担います。ただし、SpringのDIを担当するのはTomcatではなくSpring ApplicationContextです。

Spring Bootアプリ
  ├─ 組み込みTomcat
  │   └─ HTTP受付 / Servlet実行
  │
  └─ Spring ApplicationContext
      └─ Controller / Service / Repositoryの生成・DI・管理

6. Spring / Spring Bootとの違い

6.1 SpringはJakarta EEアプリケーションサーバではない

SpringはJakarta EEアプリケーションサーバの実装ではありません。 Springは、Javaアプリケーションを作るための独自フレームワークです。

ただし、SpringはJakarta EE由来のAPIを一部利用・統合しています。たとえばServlet、JPA、Bean Validationなどです。

領域 Springでの位置づけ
DI Spring独自のApplicationContextが中心。
HTTP / Web Spring MVCはServlet上で動く。HTTP受付はTomcatなどが担当。
JPA Jakarta Persistence APIとHibernateなどをSpringが統合する。
Validation Jakarta Bean ValidationのアノテーションをSpringが利用する。
Security Spring Securityが中心。
Transaction Spring Transactionが中心。

疑問SpringはJakarta EE仕様を参照しているわけではなく、独自仕様のフレームワークなのか？

解説Springの中核モデルは独自です。DI、AOP、Bean管理、Spring MVC、Spring TransactionなどはSpringのプログラミングモデルです。ただしServlet、JPA、Validation、InjectなどJakarta EE系のAPIも利用・サポートします。完全に無関係でも、Jakarta EEそのものでもありません。

6.2 Spring DIとCDIの違い

項目 Spring DI CDI
所属 Spring Framework Jakarta EE
管理主体 Spring ApplicationContext CDIコンテナ
代表アノテーション @Component, @Service, @Autowired @Inject, @ApplicationScoped
使う場面 Spring / Spring Bootアプリ Jakarta EEアプリ
方針 Spring Bootを使うなら、基本はSpring DIを使います。CDIはJakarta EE側のDI標準として理解しておけば十分です。

7. 外部サーバ方式と組み込みサーバ方式

Webアプリには、HTTPリクエストを受ける実行基盤が必要です。 その実行基盤をアプリの外に置くか、アプリの中に同梱するかで構成が変わります。

外部サーバ方式 
OS
 ↓
JVM
 ↓
Tomcat / WildFly
 ↓
myapp.war

先にサーバを起動し、そこにWAR/EARをデプロイする。

組み込みサーバ方式 
OS
 ↓
JVM
 ↓
myapp.jar
  ├─ Spring Boot
  ├─ Tomcat
  └─ アプリコード

java -jarでアプリを起動すると、内包されたTomcatも起動する。

観点 外部サーバ方式 組み込みサーバ方式
起動の主役 Tomcat / WildFlyなどの外部サーバ アプリケーション自身
成果物 WAR / EARが多い 実行可能JARが多い
Tomcatの位置 アプリの外 アプリの中
起動方法 サーバ起動 → アプリをデプロイ java -jar app.jar
アプリの性質 サーバに載せる部品 自分でサーバ機能を持って起動するアプリ

疑問外部サーバにデプロイすることと、アプリ内にサーバを内包することは何が違うのか？

解説HTTP実行基盤をどこに置くかが違います。外部サーバ方式では、TomcatやWildFlyが先に存在し、アプリはそこに配置されます。組み込みサーバ方式では、Spring Bootアプリ自身がTomcatを依存ライブラリとして持ち、アプリ起動時にTomcatも一緒に起動します。

8. WAR / EAR / JARの違い

WAR、EAR、JARの違いは、何をまとめたファイルかとどこで実行される前提かです。

形式 正式名 中身 実行前提 代表用途
JAR Java Archive Javaクラス、リソース、依存ライブラリなど JVM。実行可能JARなら単体起動可能。 ライブラリ、Spring Bootアプリ
WAR Web Application Archive Webアプリ、WEB-INF、クラス、ライブラリ、静的ファイルなど Tomcat/WildFlyなどのWebコンテナ Servlet/JSP/Spring MVCなどのWebアプリ
EAR Enterprise Application Archive WAR、EJB JAR、共通JARなど複数モジュール Jakarta EEアプリケーションサーバ 大規模Jakarta EEアプリ
成果物ごとのイメージ
実行可能JAR
  = java -jar app.jar で起動する
  = Spring BootではTomcatも中に含むことが多い

WAR
  = Tomcat / WildFlyなどにデプロイするWebアプリ部品

EAR
  = WARやEJB JARなどをまとめたJakarta EEアプリ一式

疑問WAR/EARのデプロイとJARのデプロイの違いは？

解説WAR/EARは外部サーバに載せる前提の成果物です。一方、Spring Bootの実行可能JARはアプリ自身が起動の主体で、組み込みTomcatなども含めてjava -jarで起動できます。

9. REST APIとプログラミングAPI

「API」という言葉は、REST APIだけを指すわけではありません。 APIの本来の意味は、あるプログラムが別のプログラムや機能を使うための接点です。

種類 REST API Jakarta EE API / Spring API
分類 ネットワークAPI プログラミングAPI
利用方法 HTTPで呼ぶ Javaコードでimportして使う
利用者 ブラウザ、フロントエンド、外部システム Javaアプリ開発者
例 GET /users/1 HttpServletRequest, @Entity, @Service
接点 外部システム ↔ アプリ アプリコード ↔ ライブラリ/コンテナ/フレームワーク

9.1 Jakarta EE APIの具体例

// Servlet API
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

// CDI API
import jakarta.inject.Inject;
import jakarta.enterprise.context.ApplicationScoped;

// JPA API
import jakarta.persistence.Entity;
import jakarta.persistence.Id;

// Bean Validation API
import jakarta.validation.NotNull;
import jakarta.validation.Size;

9.2 Spring APIの具体例

import org.springframework.stereotype.Service;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.transaction.annotation.Transactional;

疑問REST APIは理解しているが、ここでいうAPIは別物ではないか？

解説その認識は正しいです。ここでいうJakarta EE APIは、HTTPで外部から呼ばれるAPIではなく、Javaコード内で利用するクラス、インターフェース、アノテーション群です。REST APIは外部システム向けの入口、Jakarta EE APIはJavaアプリがコンテナや標準機能を使うための入口です。

整理 @RestControllerや@GetMappingはSpringのプログラミングAPIです。 その結果として外部に公開されるGET /users/1がREST APIです。

10. 最終理解モデル

これまでの理解を、Spring Boot中心のバックエンド開発者向けに最終整理すると以下です。

JVM

Javaプログラムを実行する土台。ServletやSpring Beanを直接提供するものではない。

Jakarta EE

エンタープライズJavaの標準仕様群。APIと振る舞いのルールを定義する。

Jakarta EEアプリケーションサーバ

Jakarta EE仕様を実装したサーバ製品。アプリを読み込み、コンテナ機能で管理・実行する。

Tomcat

Servletコンテナ。HTTPを受け、Servletを動かす。Spring Bootでは組み込みTomcatとして使われる。

Spring Framework

Spring独自のDI、AOP、MVC、Transactionなどを提供するJavaフレームワーク。

Spring Boot

Springアプリを簡単に構成・起動する仕組み。実行可能JARとして組み込みTomcatと一緒に起動する構成が多い。

Spring Bootでのリクエスト処理の理解

HTTPクライアント
  ↓
組み込みTomcat
  ↓
DispatcherServlet
  ↓
Spring MVC
  ↓
Controller Bean
  ↓
Service Bean
  ↓
Repository Bean
  ↓
DB

役割分担

対象 役割
Tomcat HTTP受付、Servlet実行。
DispatcherServlet Spring MVCの入口。
Spring ApplicationContext Controller、Service、RepositoryなどのBean生成・DI・管理。
Spring Security 認証・認可。
Spring Transaction トランザクション境界制御。
あなたのアプリコード 業務ロジック、API設計、DBアクセス、例外処理など。
最重要結論 Jakarta EEでもSpring Bootでも、根本は「自分のコードを自分で全部呼び出す」のではなく、 コンテナ/フレームワークに管理され、適切なタイミングで呼び出されるという考え方です。

学習優先順位

Spring Bootでバックエンド実装力を上げる目的なら、Jakarta EE/CDI/EJBを深掘りするより、まず以下を優先するとよいです。

  1. Spring DIコンテナ：Bean、DI、スコープ、ライフサイクル
  2. Spring MVC：DispatcherServlet、Controller、リクエストマッピング
  3. Tomcat：HTTPを受けるServletコンテナとしての役割
  4. Spring Transaction：@Transactionalとプロキシ
  5. Spring Data JPA / MyBatis：DBアクセス層
  6. Spring Security：認証・認可
率直な補足 今回混乱した根本原因は、「API」「コンテナ」「サーバ」という言葉をすべて同じ粒度で捉えようとしたことです。 技術用語はレイヤーごとに意味が変わります。今後は、各用語を必ず「仕様なのか、実装なのか、実行環境なのか、アプリコードなのか」に分類してから理解すると崩れにくくなります。
