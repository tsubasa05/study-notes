Task API Entity / JPA 学習記録

日付: 2026-05-19

対象: Spring Boot / Task API / DTO / Entity / Repository / Spring Data JPA

今回学習したこと

  • DTOを controller や service と同じ階層に置く構成は、初期のCRUD練習では問題ない。
  • 今回は TaskCreateRequest、TaskUpdateRequest、TaskResponse があればまず十分。
  • 永続化層は、シンプルに entity と repository パッケージを作る方針で進める。
  • 今回のCRUD練習では MyBatis より Spring Data JPA の方が始めやすい。
  • JPA Entityでは @Entity、@Table、@Id、@GeneratedValue を使う。
  • enumをDBに保存する場合は @Enumerated(EnumType.STRING) を使う。

採用するパッケージ構成

今回の練習では、説明しやすく再現しやすい層別パッケージを使う。

ts.com.taskapi
  controller
    TaskController.java
  service
    TaskService.java
    TaskServiceImpl.java
  dto
    TaskCreateRequest.java
    TaskUpdateRequest.java
    TaskResponse.java
  entity
    Task.java
    TaskStatus.java
  repository
    TaskRepository.java
面接での説明例: ControllerはHTTPの受け口、Serviceは業務処理、RepositoryはDBアクセス、EntityはDBテーブルに対応するクラスとして分けています。

DTOの考え方

DTOは、外部との入出力に使うデータの形を表すクラス。EntityをそのままAPIのリクエストやレスポンスに使わないために用意する。

DTO 役割
TaskCreateRequest タスク作成時にクライアントから受け取る値を表す。
TaskUpdateRequest タスク更新時にクライアントから受け取る値を表す。
TaskResponse タスク情報をクライアントへ返す形を表す。

今後、検索条件やページングが増えたら TaskSearchRequest や TaskListResponse を追加する可能性がある。ただし、最初のCRUD練習ではまだ不要。

JPAを選んだ理由

Spring Data JPAを使うと、基本的なCRUD操作をRepositoryインターフェースだけで扱える。

選択肢 特徴 今回の判断
Spring Data JPA findAll、findById、save、delete などをすぐ使える。 CRUD練習に向いているため採用。
MyBatis SQLを自分で書ける。複雑なSQL制御には向いている。 最初のCRUD練習では覚えることが増えるため、今回は優先度を下げる。

Entityの役割

Entityは、DBテーブルに対応するJavaクラス。今回の Task は、タスクテーブルの1行を表す。

アノテーション 意味
@Entity このクラスがJPA Entityであることを示す。
@Table(name = "tasks") このEntityが対応するDBテーブル名を明示する。今回は Task クラスを tasks テーブルに対応させる。
@Id 主キーのフィールドであることを示す。
@GeneratedValue(strategy = GenerationType.IDENTITY) IDをDB側で自動採番する。
@Enumerated(EnumType.STRING) enumを文字列としてDBに保存する。

Task Entityの基本形

今回作成したEntityの基本方針は以下。

@Entity
@Table(name = "tasks")
public class Task {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    private String description;

    @Enumerated(EnumType.STRING)
    private TaskStatus taskStatus;

    private LocalDateTime dueTime;

    private LocalDateTime createdAt;

    private LocalDateTime updatedAt;

    protected Task() {
    }

    public Task(String title, String description, TaskStatus taskStatus, LocalDateTime dueTime) {
        this.title = title;
        this.description = description;
        this.taskStatus = taskStatus;
        this.dueTime = dueTime;
    }
}

覚えるポイント

  • @Table(name = "tasks") を付けると、Javaクラス名とDBテーブル名の対応を明示できる。
  • protected Task() {} はJPAがEntityを生成するために必要。
  • id はDBが自動採番するため、作成用コンストラクタでは受け取らない。
  • createdAt と updatedAt はリクエストから受け取るより、Service側やJPAの仕組みで設定する方が自然。
  • Entityは通常の class で作る。DTOのように record にしない。

TaskStatus enum

タスクの状態は TaskStatus enum として表す。

public enum TaskStatus {
    TODO,
    IN_PROGRESS,
    DONE
}

boolean completed よりも状態を増やしやすい。たとえば「未着手」「進行中」「完了」のような状態を表現できる。

@Enumeratedの理解

@Enumerated は、JavaのenumをDBにどう保存するかをJPAに教えるアノテーション。

@Enumerated(EnumType.STRING)
private TaskStatus taskStatus;

この場合、DBには以下のように文字列で保存される。

TaskStatus.TODO        -> "TODO"
TaskStatus.IN_PROGRESS -> "IN_PROGRESS"
TaskStatus.DONE        -> "DONE"

EnumType.STRING と EnumType.ORDINAL

指定 保存される値 評価
EnumType.STRING TODO、IN_PROGRESS、DONE 読みやすく、enumの並び順変更に強い。基本はこちらを使う。
EnumType.ORDINAL 0、1、2 enumの順番を変えると意味が壊れる可能性があるため避ける。
面接での説明例: JavaのenumをDBに保存するために @Enumerated を使います。 EnumType.STRING を指定するとenum名が文字列で保存されるため、enumの並び順変更に強く、DB上でも意味を読み取りやすいです。

次に作るRepository

次のステップでは repository パッケージに TaskRepository を作る。

public interface TaskRepository extends JpaRepository<Task, Long> {
}

Spring Data JPAではRepositoryの実装クラスを自分で作らない。インターフェースを作ると、Springが実行時に実装を用意してくれる。

メソッド 用途
findAll() 全件取得。
findById(id) IDで1件取得。戻り値は Optional<Task>。
save(task) 新規作成または更新。
delete(task) Entityを削除。
existsById(id) IDの存在確認。

TaskRepository作成後に学習したこと

TaskRepository は空のインターフェースでも、JpaRepository を継承するだけで基本的なDB操作を使える。

package ts.com.taskapi.repository;

import org.springframework.data.jpa.repository.JpaRepository;

import ts.com.taskapi.entity.Task;

public interface TaskRepository extends JpaRepository<Task, Long> {
}

JpaRepository<Task, Long> の意味

指定 意味
Task このRepositoryが扱うEntity。
Long Task の主キー id の型。
面接での説明例: Spring Data JPAでは JpaRepository を継承したRepositoryインターフェースを作ることで、基本的なCRUD処理をSpringが用意してくれます。 そのため、単純なCRUDではRepositoryの実装クラスを自分で書きません。

ServiceからRepositoryを使う流れ

TaskServiceImpl では、Repositoryをコンストラクタで受け取って使う。

private final TaskRepository taskRepository;

public TaskServiceImpl(TaskRepository taskRepository) {
    this.taskRepository = taskRepository;
}

この形をコンストラクタインジェクションという。@Service が付いたクラスにコンストラクタが1つだけある場合、Springが自動で TaskRepository を渡してくれる。

save() の理解

taskRepository.save(task) は、Entity全体をDBに保存する処理。

Task saved = taskRepository.save(task);

task の中に入っている値がDBに保存される。たとえば以下の値をEntityに詰めてから save() する。

  • title
  • description
  • taskStatus
  • dueTime
  • createdAt
  • updatedAt

id は @GeneratedValue により、DB登録時に自動採番される。save() の戻り値には、採番後の id が入る。

Task saved = taskRepository.save(task);
Long id = saved.getId();

作成処理の値の出どころ

項目 値の作られ方
id DB登録時に自動採番される。
title TaskCreateRequest から受け取る。
description TaskCreateRequest から受け取る。
taskStatus 新規作成時はService側で TaskStatus.TODO を設定する。
dueTime TaskCreateRequest から受け取る。自動生成ではない。
createdAt Service側で LocalDateTime.now() を設定する。
updatedAt 作成時は createdAt と同じ値を入れると扱いやすい。

createdAt と updatedAt

作成日時・更新日時は、Entityを作っただけでは自動で入らない。最初の練習ではService側で明示的に設定する。

LocalDateTime now = LocalDateTime.now();
task.setCreatedAt(now);
task.setUpdatedAt(now);

updatedAt を null にしても、saved.getUpdatedAt() を呼ぶだけならエラーにはならない。getterは単にフィールドの値を返すため、値がなければ null が返る。

ただし、APIレスポンスやDB上で扱いやすくするなら、作成時点では createdAt と updatedAt を同じ値にしておく方が自然。

覚え方: createdAt は作成された日時。 updatedAt は最後に更新された日時。 新規作成直後は、最後に更新された日時も作成日時と同じと考える。

record DTOの値の取り出し方

TaskCreateRequest が record の場合、値はフィールドアクセスではなくメソッド呼び出しで取り出す。

// NG
taskCreateRequest.title
taskCreateRequest.dueTime

// OK
taskCreateRequest.title()
taskCreateRequest.dueTime()

record は、定義した要素名と同じ名前のアクセサメソッドを自動で持つ。

削除処理の理解

IDで削除したい場合は deleteById(id) を使う。

taskRepository.deleteById(id);

delete(id) ではなく deleteById(id) と覚える。

メソッド 渡すもの 用途
delete(task) Task Entity 取得済みのEntityを削除する。
deleteById(id) ID IDを指定して削除する。

実務的には、削除前に existsById(id) や findById(id) で存在確認し、存在しない場合は404用の例外を投げる。

if (!taskRepository.existsById(id)) {
    // 後でNotFoundExceptionを投げる
}

taskRepository.deleteById(id);
