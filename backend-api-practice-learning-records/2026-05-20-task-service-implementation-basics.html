Task API Service 実装 学習記録

作成日: 2026-05-20 / 対象: TaskService, TaskServiceImpl, DTO, Entity, Repository

今回学んだ結論

  • TaskService をインターフェースにし、TaskServiceImpl で実装する形は標準的。
  • Controller は TaskService に依存し、実装クラスには直接依存しない。
  • Service は DTO と Entity の変換、Repository 呼び出し、存在チェック、業務ルールを担当する。
  • Entity の id は JPA が管理するため、通常は setId() を用意しない。
  • findById() は存在しない可能性があるため Optional<Task> を返す。

Service インターフェースの考え方

TaskService は Controller から見た Service の入口を定義する。 Java のインターフェースメソッドは自動的に public なので、通常は public を省略する。 

public interface TaskService {
    List<TaskResponse> findAll();
    TaskResponse findById(Long id);
    TaskResponse create(TaskCreateRequest taskCreateRequest);
    TaskResponse update(Long id, TaskUpdateRequest taskUpdateRequest);
    void delete(Long id);
}
面接での説明: Controller は HTTP の受け口に集中させ、Service に処理を委譲します。 Controller はインターフェースである TaskService に依存させ、実装は @Service を付けた TaskServiceImpl に任せます。

VS Code で実装クラスを作る操作

  1. TaskServiceImpl.java を作る。
  2. public class TaskServiceImpl implements TaskService と書く。
  3. 赤線が出たら Ctrl + . を押す。
  4. Add unimplemented methods を選ぶ。
  5. Spring に Service として認識させるため @Service を付ける。
@Service
public class TaskServiceImpl implements TaskService {
    private final TaskRepository taskRepository;

    public TaskServiceImpl(TaskRepository taskRepository) {
        this.taskRepository = taskRepository;
    }
}

create() の基本パターン

create() は Request DTO から Entity を作り、Repository で保存し、Response DTO に変換して返す。 作成時のステータスをアプリ側で TODO に固定するなら、Request DTO から taskStatus を受け取らない設計が自然。 

@Override
public TaskResponse create(TaskCreateRequest taskCreateRequest) {
    LocalDateTime now = LocalDateTime.now();

    Task task = new Task(
        taskCreateRequest.title(),
        taskCreateRequest.description(),
        TaskStatus.TODO,
        taskCreateRequest.dueTime()
    );

    task.setCreatedAt(now);
    task.setUpdatedAt(now);

    Task savedTask = taskRepository.save(task);

    return new TaskResponse(
        savedTask.getId(),
        savedTask.getTitle(),
        savedTask.getDescription(),
        savedTask.getTaskStatus(),
        savedTask.getDueTime(),
        savedTask.getCreatedAt(),
        savedTask.getUpdatedAt()
    );
}

入力値の考え方

項目 扱い 理由
title @NotBlank 必須項目。空文字も許可しない。
description @Size(max = 500) 任意項目なら null や空文字を許可してよい。
dueTime @NotNull 期限を必須にするなら null を許可しない。
taskStatus Service で TaskStatus.TODO を設定 作成時は必ず TODO から始める業務ルールにするため。

delete() の2つのパターン

パターン1: 204 No Content

削除成功時にレスポンスボディを返さない。REST API では標準的で、面接でも説明しやすい。 

@DeleteMapping("/{id}")
public ResponseEntity<Void> delete(@PathVariable Long id) {
    taskService.delete(id);
    return ResponseEntity.noContent().build();
}
@Override
public void delete(Long id) {
    Task task = taskRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("Task not found"));

    taskRepository.delete(task);
}

パターン2: 削除したデータを返す

削除した内容をクライアントに返したい場合は 200 OK と TaskResponse を返す。 204 No Content と body は組み合わせない。 

@DeleteMapping("/{id}")
public ResponseEntity<TaskResponse> delete(@PathVariable Long id) {
    return ResponseEntity.ok(taskService.delete(id));
}
注意: ResponseEntity.status(HttpStatus.NO_CONTENT).body(taskResponse) は不自然。 204 は「本文なし」を意味するため、body を返すなら 200 にする。

findAll() と stream()

findAll() は List<Task> を List<TaskResponse> に変換する処理。 現代的な書き方としては stream().map().toList() がよく使われる。 

@Override
public List<TaskResponse> findAll() {
    return taskRepository.findAll().stream()
        .map(task -> new TaskResponse(
            task.getId(),
            task.getTitle(),
            task.getDescription(),
            task.getTaskStatus(),
            task.getDueTime(),
            task.getCreatedAt(),
            task.getUpdatedAt()
        ))
        .toList();
}

ラムダ式の return

task -> new TaskResponse(...) は省略形なので、return は書かない。 return を使う場合はブロック { } が必要。 

// 標準的な省略形
.map(task -> new TaskResponse(...))

// return を使うならブロックが必要
.map(task -> {
    return new TaskResponse(...);
})
これは構文エラー: .map(task -> return new TaskResponse(...))

findById() と Optional

taskRepository.findById(id) は、ID に対応するデータが存在しない可能性があるため Optional<Task> を返す。 Optional<Task> 自体には getId() や getTitle() はない。 先に中の Task を取り出す必要がある。 

@Override
public TaskResponse findById(Long id) {
    Task task = taskRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("Task not found"));

    return new TaskResponse(
        task.getId(),
        task.getTitle(),
        task.getDescription(),
        task.getTaskStatus(),
        task.getDueTime(),
        task.getCreatedAt(),
        task.getUpdatedAt()
    );
}
面接での説明: findById は指定 ID のデータが存在しない可能性があるため、null ではなく Optional で返されます。 Service 側では orElseThrow を使い、存在しない場合は例外に変換します。

update() では setId() しない

更新処理では、新しい Entity を作って setId(id) するより、既存 Entity を取得して中身を書き換える方が標準的。 id は JPA が管理する値なので、通常は Entity に setId() を用意しない。 

@Override
public TaskResponse update(Long id, TaskUpdateRequest taskUpdateRequest) {
    Task task = taskRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("Task not found"));

    task.setTitle(taskUpdateRequest.title());
    task.setDescription(taskUpdateRequest.description());
    task.setTaskStatus(taskUpdateRequest.taskStatus());
    task.setDueTime(taskUpdateRequest.dueTime());
    task.setUpdatedAt(LocalDateTime.now());

    Task updatedTask = taskRepository.save(task);

    return new TaskResponse(
        updatedTask.getId(),
        updatedTask.getTitle(),
        updatedTask.getDescription(),
        updatedTask.getTaskStatus(),
        updatedTask.getDueTime(),
        updatedTask.getCreatedAt(),
        updatedTask.getUpdatedAt()
    );
}

この形のメリット

  • createdAt が消えない。
  • 存在しない ID の更新を検知できる。
  • JPA が管理する id をアプリ側で無理に変更しない。
  • 面接で説明しやすい標準的な更新処理になる。

今後の改善ポイント

  • RuntimeException を TaskNotFoundException に置き換える。
  • @RestControllerAdvice で 404 エラーを共通処理する。
  • TaskResponse への変換処理が重複する場合、private メソッドに切り出す。
  • createdAt / updatedAt は慣れてきたら @PrePersist / @PreUpdate も検討する。
  • deleteById(id) は存在確認なしだと 404 を返しにくいため、まず findById で存在確認する形にする。

面接で説明できるようにすること

  • Controller / Service / Repository の責務分担。
  • DTO と Entity を分ける理由。
  • Optional を使う理由。
  • DELETE で 204 No Content を返す理由。
  • 更新時に setId() しない理由。
  • stream().map().toList() の意味。

振り返りメモ

手が止まった箇所 Optional<Task> から直接 getId() しようとした点。orElseThrow で Task を取り出す必要がある。
覚えるべき文法 record の値取得は request.title()。ラムダ式の省略形では return を書かない。
覚えるべき Spring / JPA の考え方 JpaRepository#save は保存後の Entity を返す。findById は Optional を返す。@Service を付けると Spring の DI 対象になる。
