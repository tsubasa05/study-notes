Task API Controller 学習記録

日付: 2026-05-18

対象: Spring Boot / Task API / Controller の基本

今回学習したこと

  • Controller は HTTP リクエストの入口を担当する。
  • Controller には DB 操作や業務ロジックを書きすぎない。
  • VS Code では Java の機能を使うと package 文を自動生成できる。
  • Spring アノテーションは Quick Fix や Organize Imports で import を整理できる。
  • @PathVariable と @RequestParam は用途が違う。

Controller の役割

Controller は、HTTP の受け口として以下を担当する。

  • URL の定義
  • HTTP メソッドの定義
  • リクエスト DTO の受け取り
  • パス変数やクエリパラメータの受け取り
  • Service の呼び出し
  • HTTP ステータスコードの指定
面接での説明例: Controller は HTTP リクエストを受け取り、必要な値を取り出して Service に処理を渡す層です。 業務ロジックや DB アクセスは Controller には書かず、Service や Repository に分けます。

VS Code でクラスを作る方法

  1. src/main/java 配下の対象パッケージを開く。
  2. ts/com/taskapi の下に controller フォルダを作る。
  3. controller フォルダを右クリックする。
  4. New Java Class を選ぶ。
  5. クラス名に TaskController を入力する。

この方法で作成すると、以下のような package 文が自動で入る。

package ts.com.taskapi.controller;

public class TaskController {

}

Spring アノテーションの import 方法

  1. 赤い波線が出ているアノテーション名にカーソルを置く。
  2. Ctrl + . を押す。
  3. Import 'RestController' のような候補を選ぶ。

Controller でよく使うアノテーション

アノテーション 用途 使用例
@RestController REST API の Controller であることを示す。 @RestController
@RequestMapping Controller 全体の共通 URL を指定する。 @RequestMapping("/tasks")
@GetMapping GET API を作る。取得処理で使う。 @GetMapping("/{id}")
@PostMapping POST API を作る。新規作成で使う。 @PostMapping
@PutMapping PUT API を作る。更新で使う。 @PutMapping("/{id}")
@DeleteMapping DELETE API を作る。削除で使う。 @DeleteMapping("/{id}")
@RequestBody JSON リクエストを DTO として受け取る。 @RequestBody TaskCreateRequest request
@PathVariable URL パスの一部を受け取る。 @PathVariable Long id
@RequestParam クエリパラメータを受け取る。 @RequestParam String status
@Valid Request DTO の Bean Validation を有効にする。 @Valid @RequestBody TaskCreateRequest request

最小実装例

Controller のイメージをつかむための最小例。

package ts.com.taskapi.controller;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/tasks")
public class TaskController {

    @GetMapping("/{id}")
    public String findById(@PathVariable Long id) {
        return "task id = " + id;
    }
}

実際の練習では、戻り値は String ではなく TaskResponse などの DTO にして、処理は Service に渡す。

@PathVariable と @RequestParam の違い

@PathVariable

URL のパスの一部を受け取る。主に ID のように、特定のリソースを指定する値に使う。

GET /tasks/10
@GetMapping("/{id}")
public TaskResponse findById(@PathVariable Long id) {
    return taskService.findById(id);
}

この場合、10 が id に入る。

@RequestParam

? の後ろにあるクエリパラメータを受け取る。主に検索条件、絞り込み条件、ページ番号などに使う。

GET /tasks?status=DONE
@GetMapping
public List<TaskResponse> findAll(@RequestParam String status) {
    return taskService.findAll(status);
}

この場合、DONE が status に入る。

観点 @PathVariable @RequestParam
URL例 /tasks/1 /tasks?status=DONE
用途 特定の1件を指定する 一覧の検索・絞り込みをする
よく使う値 ID status, keyword, page, size など
面接での説明例: @PathVariable は URL パスの一部を受け取るために使い、主に ID などリソースを特定する値に使います。 @RequestParam はクエリパラメータを受け取るために使い、検索条件や絞り込み条件に使います。

ResponseEntity の使い方

ResponseEntity は、レスポンスボディだけでなく HTTP ステータスコードも明示したいときに使う。

書き方 意味 よく使う場面
ResponseEntity.ok(body) 200 OK とレスポンスボディを返す。 取得、更新後の結果返却
ResponseEntity.status(HttpStatus.CREATED).body(body) 201 Created とレスポンスボディを返す。 新規作成
ResponseEntity.noContent().build() 204 No Content を返す。ボディは返さない。 削除成功
ResponseEntity.badRequest().body(body) 400 Bad Request とエラー内容を返す。 入力不正。ただし通常は例外ハンドラで共通化する。

よくある間違い

ResponseEntity はジェネリクスに変数名を書いて返すものではない。

// NG
return ResponseEntity<taskResponse>;

// OK
return ResponseEntity.ok(taskResponse);

Task API の HTTP ステータス

今回のような基本的な CRUD API では、まず以下を使い分ければよい。

API 成功時ステータス 理由
GET /tasks 200 OK 一覧取得に成功したため。
GET /tasks/{id} 200 OK 指定した1件の取得に成功したため。
POST /tasks 201 Created 新しいリソースを作成したため。
PUT /tasks/{id} 200 OK 更新後の内容をレスポンスボディで返すため。
DELETE /tasks/{id} 204 No Content 削除成功だけを伝え、レスポンスボディを返さないため。
面接での説明例: GET と PUT は成功時に 200 OK、POST は新規作成なので 201 Created、 DELETE は返すボディが不要なので 204 No Content にしています。

URL 設計の基本

URL には create や delete のような動詞を入れすぎず、HTTP メソッドで操作を表す。

避けたい例 推奨例 意味
GET /tasks/all GET /tasks タスク一覧を取得する。
POST /tasks/create POST /tasks タスクを新規作成する。
PUT /tasks/update/{id} PUT /tasks/{id} 指定したタスクを更新する。
DELETE /tasks/delete/{id} DELETE /tasks/{id} 指定したタスクを削除する。

主要アノテーションを使ったControllerサンプル

Controller でよく使うアノテーションを一通り確認するためのサンプル。Service の中身は別で実装する。

package ts.com.taskapi.controller;

import java.util.List;

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.DeleteMapping;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.PutMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

import jakarta.validation.Valid;
import ts.com.taskapi.dto.TaskCreateRequest;
import ts.com.taskapi.dto.TaskResponse;
import ts.com.taskapi.dto.TaskUpdateRequest;
import ts.com.taskapi.service.TaskService;

@RestController
@RequestMapping("/tasks")
public class TaskController {

    private final TaskService taskService;

    public TaskController(TaskService taskService) {
        this.taskService = taskService;
    }

    @GetMapping
    public ResponseEntity<List<TaskResponse>> findAll(
            @RequestParam(required = false) String status) {
        List<TaskResponse> responses = taskService.findAll(status);
        return ResponseEntity.ok(responses);
    }

    @GetMapping("/{id}")
    public ResponseEntity<TaskResponse> findById(@PathVariable Long id) {
        TaskResponse response = taskService.findById(id);
        return ResponseEntity.ok(response);
    }

    @PostMapping
    public ResponseEntity<TaskResponse> create(
            @Valid @RequestBody TaskCreateRequest request) {
        TaskResponse response = taskService.create(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }

    @PutMapping("/{id}")
    public ResponseEntity<TaskResponse> update(
            @PathVariable Long id,
            @Valid @RequestBody TaskUpdateRequest request) {
        TaskResponse response = taskService.update(id, request);
        return ResponseEntity.ok(response);
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        taskService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
注意: 上のサンプルは Controller の書き方を確認するための例。 taskService.findAll(status) や taskService.update(id, request) を使う場合は、 Service 側のメソッド定義も同じ引数にそろえる。

DTO 層の基本

DTO は API の入出力専用の形を表すクラス。Entity をそのままリクエストやレスポンスに使わず、Request DTO と Response DTO に分ける。

クラス 役割 入れる項目の考え方
TaskCreateRequest 作成APIの入力 クライアントが作成時に入力する値だけ。通常 id, createdAt, updatedAt は入れない。
TaskUpdateRequest 更新APIの入力 更新可能な値だけ。更新対象の id は URL の @PathVariable で受け取る。
TaskResponse APIの出力 クライアントに返したい値。id, createdAt, updatedAt を含めてもよい。
面接での説明例: Request DTO にはクライアントが入力する項目だけを持たせ、id や作成日時・更新日時のようなサーバー管理項目は基本的に入れません。 Response DTO には、APIとしてクライアントへ返したい項目を定義します。

record を DTO に使う理由

record は、値のまとまりを簡潔に表すための Java の構文。DTO は値の受け渡しが主目的なので、record と相性がよい。

public record TaskResponse(
        Long id,
        String title
) {
}

上のように書くと、Java が以下を自動生成する。

  • private final フィールド
  • 全項目を受け取るコンストラクタ
  • id(), title() のようなアクセサ
  • equals()
  • hashCode()
  • toString()

値を取り出すときは、getter ではなくコンポーネント名のメソッドを使う。

TaskResponse response = new TaskResponse(1L, "Javaを勉強する");

Long id = response.id();
String title = response.title();

record が変更できない理由

record のコンポーネントは自動的に private final フィールドになる。作成後にフィールドへ再代入できず、setter を作っても final フィールドには代入できない。

public record TaskResponse(Long id, String title) {

    // NG: title は final フィールドなので再代入できない
    // public void setTitle(String title) {
    //     this.title = title;
    // }
}
注意: record はフィールドの再代入を防ぐ。ただし、List のような可変オブジェクトを持つ場合、その中身まで完全に不変になるわけではない。

Bean Validation の基本

Bean Validation は、Java オブジェクトの入力チェックをアノテーションで書くための仕組み。Spring Boot では spring-boot-starter-validation を入れると使える。

Controller の引数に @Valid を付けると、Request DTO に付けた制約が実行される。

@PostMapping
public ResponseEntity<TaskResponse> create(
        @Valid @RequestBody TaskCreateRequest request) {
    TaskResponse response = taskService.create(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(response);
}
アノテーション 用途 注意点
@NotBlank null, 空文字, 空白だけの文字列を禁止する。 String 用。LocalDateTime には使わない。
@NotNull null を禁止する。 日時、Enum、数値などを必須にしたいときに使う。
@Size 文字列やリストの長さを制限する。 @Size(max = 100) のように使う。
@Min, @Max 数値の最小値・最大値を制限する。 優先度や数量などに使う。
@Email メールアドレス形式をチェックする。 ユーザー登録系でよく使う。
@Pattern 正規表現で文字列をチェックする。 単純な状態値は Enum で表す方が分かりやすい場合もある。
@Future, @Past 日時が未来または過去かをチェックする。 期限日などに使える。

Task DTO の設計メモ

今回の Task API では、まず以下の分け方を基本にする。

TaskCreateRequest
- title
- description
- taskStatus
- dueTime

TaskUpdateRequest
- title
- description
- taskStatus
- dueTime

TaskResponse
- id
- title
- description
- taskStatus
- dueTime
- createdAt
- updatedAt

作成時の taskStatus は、設計として次のどちらかを選ぶ。

方針 DTO 説明
クライアントが指定する TaskCreateRequest に taskStatus を入れる。 入力値として @NotNull を付ける候補になる。
必ず初期値にする TaskCreateRequest から taskStatus を外す。 Service 側で TODO などの初期値を設定する。

今回のレビューで出た注意点

  • taskstatus や taskStatu ではなく、Java の命名として taskStatus にする。
  • LocalDateTime に @NotBlank は使わない。必須なら @NotNull を使う。
  • record の最後の項目に余計なカンマを残さない。
  • Request DTO に id, createdAt, updatedAt を入れすぎない。
  • PUT の全体更新なら、title を @NotBlank にする方針でよい。
