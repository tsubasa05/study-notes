User API Security, Validation, and Error Handling

日付: 2026-05-21

対象アプリ: apps/user-app

1. 今回の改善全体像

今回の修正では、User API を「動く CRUD」から「入力、保存、エラー、テストまで最低限安全に扱える API」へ近づけた。 特に重要なのは、パスワードをレスポンスに返さないこと、DB に平文で保存しないこと、更新時に既存データを壊さないこと、 例外発生時にクライアントへ一貫した JSON を返すことである。

領域 主な修正 目的
Security PasswordEncoder と BCryptPasswordEncoder を導入 パスワードを平文保存しない
DTO UserResponse から password を削除 API レスポンスで秘密情報を漏らさない
Update PUT /user/{id} を部分更新に変更 未指定項目を null で上書きしない
Validation @Valid, @NotBlank, @Size, @Pattern を利用 Controller 入り口で不正な入力を止める
Error Handling @RestControllerAdvice と独自例外を追加 400, 404, 409 を統一 JSON で返す
Test MockMvc による API 統合テストを追加 正常系と異常系の API 挙動を固定する
設計のポイント: Controller は HTTP の入り口、Service は業務処理、Repository は DB アクセス、Entity は永続化対象、DTO は API 入出力という役割に分けている。 今回の改善も、この責務分離を崩さないように実装している。

2. パスワードをレスポンスから消した理由

以前の UserResponse は password を含んでいたため、ユーザー作成、一覧取得、検索、更新などのレスポンスで パスワードが外部へ返ってしまう状態だった。これはパスワードが平文かハッシュかに関係なく避けるべきである。 ハッシュ値であっても、攻撃者に渡れば総当たり攻撃や辞書攻撃の材料になる。

修正後の UserResponse

public record UserResponse(
    Long id,
    String loginId,
    String name,
    String role
) {

}

ここでは Java の record を使っている。record は Java 16 以降で正式化された構文で、 値を運ぶだけのクラスを短く書ける。上記の定義だけで、次のものが自動生成される。

  • 全フィールドを受け取るコンストラクタ
  • id(), loginId(), name(), role() のアクセサメソッド
  • equals(), hashCode(), toString()
注意: record は不変データを表現するのに向いている。 API レスポンス DTO のように「生成後に値を書き換える必要がない」用途と相性がよい。 一方で、JPA Entity には通常使わない。JPA は引数なしコンストラクタやフィールド更新を必要とするためである。

3. BCrypt によるパスワードハッシュ化

パスワードは DB に保存する前に PasswordEncoder でハッシュ化するようにした。 実装クラスとしては BCryptPasswordEncoder を使っている。

PasswordConfig

@Configuration
public class PasswordConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

@Configuration は「このクラスは Spring の設定クラスです」という意味である。 @Bean が付いたメソッドの戻り値は、Spring の DI コンテナに登録される。 その結果、Service では次のようにコンストラクタ引数として PasswordEncoder を受け取れる。 

private final PasswordEncoder passwordEncoder;

public UserServiceImpl(UserRepository userRepository, PasswordEncoder passwordEncoder) {
    this.userRepository = userRepository;
    this.passwordEncoder = passwordEncoder;
}

これはコンストラクタインジェクションと呼ばれる。フィールドに直接 @Autowired を付ける方法よりも、 必須依存がコンストラクタで明確になるため、テストしやすく、設計上も扱いやすい。

保存時の encode

User user = new User(
    userCreateRequest.loginId(),
    userCreateRequest.name(),
    parseRole(userCreateRequest.role()),
    passwordEncoder.encode(userCreateRequest.password())
);

passwordEncoder.encode(...) は、入力されたパスワードを BCrypt 形式のハッシュ文字列に変換する。 BCrypt は同じパスワードでも毎回違うハッシュを作る。これはソルトと呼ばれるランダム要素が含まれるためである。 そのため、テストではハッシュ文字列を直接比較せず、matches を使う。 

assertNotEquals("secret", saved.getPassword());
assertTrue(passwordEncoder.matches("secret", saved.getPassword()));

matches(平文, ハッシュ) は、平文パスワードがそのハッシュに対応しているかを検証する。 ログイン機能を作る場合も、DB のハッシュを復号するのではなく、同じく matches で確認する。

4. Entity の DB 制約と更新用メソッド

User Entity には @Column を追加し、DB レベルの制約を明示した。 DTO のバリデーションだけでは、Service やテストから直接 Entity を保存する経路まで完全には守れない。 DB 制約を持たせることで、永続化の最終防衛線を作っている。 

@Column(nullable = false, unique = true, length = 15)
private String loginId;

@Column(nullable = false, length = 15)
private String name;

@Enumerated(EnumType.STRING)
@Column(nullable = false, length = 20)
private Role role;

@Column(nullable = false, length = 100)
private String password;
指定 意味
nullable = false DB カラムに null を保存できない
unique = true 同じ loginId を複数ユーザーで使えない
length = 15 文字列カラムの最大長を指定する
@Enumerated(EnumType.STRING) Enum を数値ではなく文字列として保存する

setId をやめた理由

以前の更新処理では、新しい User インスタンスを作り、setId(id) で ID を入れて save() していた。 この方法は、存在しない ID でも新規登録に近い挙動になったり、未指定項目を null で保存したりする危険がある。

修正後は、まず既存ユーザーを DB から取得し、その Entity に対して必要な項目だけを変更する。 Entity には汎用 setter ではなく、意味のある更新用メソッドを用意した。 

public void updateLoginId(String loginId) {
    this.loginId = loginId;
}

public void updateName(String name) {
    this.name = name;
}

public void updateRole(Role role) {
    this.role = role;
}

public void updatePassword(String password) {
    this.password = password;
}
設計上の意図: setter を大量に公開すると、どこからでも Entity の内部状態を変更できる。 今回は学習用としてシンプルな updateXxx メソッドにしているが、実務では「メールアドレス変更」「退会処理」など業務用語に近いメソッド名にすることも多い。

5. 部分更新の実装

PUT /user/{id} は、リクエストに含まれる項目だけを更新する仕様にした。 例えば {"name": "New Name"} だけを送った場合、loginId, role, password は変更されない。

UserUpdateRequest

public record UserUpdateRequest(
    @Size(max = 15)
    @Pattern(regexp = ".*\\S.*", message = "loginId must not be blank")
    String loginId,

    @Size(max = 15)
    @Pattern(regexp = ".*\\S.*", message = "name must not be blank")
    String name,

    @Pattern(regexp = "MANAGER|MEMBER", message = "role must be MANAGER or MEMBER")
    String role,

    @Size(max = 20)
    @Pattern(regexp = ".*\\S.*", message = "password must not be blank")
    String password
) {

    public boolean hasAnyValue() {
        return loginId != null || name != null || role != null || password != null;
    }
}

更新 DTO では @NotBlank を外している。理由は、部分更新では未指定を null として許可する必要があるためである。 ただし、指定された場合に空文字は許可したくない。そのため @Pattern(regexp = ".*\\S.*") を使っている。

値 扱い 理由
項目なし、または null 更新しない 部分更新の未指定扱い
"" 400 Bad Request 空文字で既存値を壊さない
" " 400 Bad Request 空白だけの値も実質空文字だから
"New Name" 更新する 有効な値だから

Service 側の更新処理

if (!userUpdateRequest.hasAnyValue()) {
    throw new BadRequestException("at least one field must be specified");
}

User user = userRepository.findById(id)
    .orElseThrow(() -> new ResourceNotFoundException("user not found: " + id));

if (userUpdateRequest.name() != null) {
    user.updateName(userUpdateRequest.name());
}

findById(id) の戻り値は Optional<User> である。 Optional は「値があるかもしれないし、ないかもしれない」ことを型で表す Java のクラスである。 orElseThrow を使うと、値があれば取り出し、なければ例外を投げられる。

() -> new ResourceNotFoundException(...) はラムダ式である。 ラムダ式は「後で実行される処理」を短く書く構文で、ここでは Optional が空だった場合だけ例外インスタンスを作る。

6. Role の扱いと Enum

Role は Java の enum として定義されている。 Enum は決められた候補だけを値として持てる型で、今回なら MANAGER と MEMBER だけが有効である。 

public enum Role {
    MANAGER,
    MEMBER
}

API では role を文字列として受け取るため、Service で Role.valueOf(role) を使って Enum に変換している。 ただし Role.valueOf("ADMIN") のような不正値は IllegalArgumentException になるため、 独自の BadRequestException に変換して 400 として返している。 

private Role parseRole(String role) {
    try {
        return Role.valueOf(role);
    } catch (IllegalArgumentException exception) {
        throw new BadRequestException("role must be MANAGER or MEMBER");
    }
}

try-catch は例外処理の構文である。try ブロック内で例外が発生した場合、 対応する catch ブロックに処理が移る。ここでは Java 標準の例外を API 用の例外に置き換えている。

7. Bean Validation と @Valid

DTO に @NotBlank, @Size, @Pattern を付けただけでは、 Controller のリクエストボディ検証は実行されない。Controller 引数に @Valid を付けることで、 Spring がリクエスト JSON を DTO に変換した後、Bean Validation を実行する。 

@PostMapping("/create")
public ResponseEntity<UserResponse> createUser(
    @Valid @RequestBody UserCreateRequest userCreateRequest
) {
    return ResponseEntity.status(HttpStatus.CREATED).body(userService.createUser(userCreateRequest));
}
アノテーション 役割
@RequestBody HTTP リクエストボディの JSON を Java オブジェクトへ変換する
@Valid DTO に付いている Validation アノテーションを実行する
@NotBlank null, 空文字, 空白だけの文字列を拒否する
@Size(max = 15) 文字数の上限を指定する
@Pattern 正規表現に一致する文字列だけを許可する

8. Controller の役割と検索条件チェック

UserController は HTTP リクエストを受け取り、Service を呼び出して、HTTP レスポンスを返す層である。 今回、検索 API では「検索条件は 1 種類だけ指定できる」というルールを Controller でチェックしている。 

@GetMapping("/search")
public ResponseEntity<List<UserResponse>> searchUsers(
    @RequestParam(required = false) List<Long> id,
    @RequestParam(required = false) List<String> loginId,
    @RequestParam(required = false) List<String> name,
    @RequestParam(required = false) List<String> role
) {
    int conditionCount = 0;

    if (hasValues(id)) conditionCount++;
    if (hasValues(loginId)) conditionCount++;
    if (hasValues(name)) conditionCount++;
    if (hasValues(role)) conditionCount++;

    if (conditionCount == 0) {
        throw new BadRequestException("specify one search condition");
    }

    if (conditionCount > 1) {
        throw new BadRequestException("specify only one search condition");
    }

    ...
}

@RequestParam(required = false) は、クエリパラメータが必須ではないことを表す。 例えば /user/search?loginId=member1 の loginId を Java の引数に受け取る。 型を List<String> にしているため、?loginId=a&loginId=b のような複数指定にも対応できる。 

private boolean hasValues(List<?> values) {
    return values != null && !values.isEmpty();
}

List<?> の ? はワイルドカードである。 「要素の型は何でもよい List」を表す。List<Long>, List<String> のどちらも受け取れるため、 共通の空チェックメソッドとして使える。

9. HTTP ステータスと ResponseEntity

Controller では ResponseEntity を使って、レスポンスボディだけでなく HTTP ステータスも明示している。 

return ResponseEntity.status(HttpStatus.CREATED).body(userService.createUser(userCreateRequest));

作成 API は 201 Created、更新 API は既存方針に合わせて 202 Accepted、削除 API は 204 No Content を返すようにした。 

@DeleteMapping("/delete/{id}")
public ResponseEntity<Void> deleteUser(@PathVariable Long id) {
    userService.deleteUserById(id);
    return ResponseEntity.noContent().build();
}

ResponseEntity<Void> は、レスポンスボディがないことを型で表している。 noContent() は 204 を作るための便利メソッドである。

10. 独自例外と統一 JSON エラー

API では、エラー時にも一定の形式でレスポンスを返すことが重要である。 クライアントは形式が揃っていれば、画面表示やログ出力を共通処理にできる。

エラーレスポンス DTO

public record ApiErrorResponse(
    Instant timestamp,
    int status,
    String error,
    String message,
    String path
) {
}

Instant は Java 標準の日時型で、UTC 基準の時刻を表す。 API エラーの発生時刻として利用している。

独自例外

public class ResourceNotFoundException extends RuntimeException {

    public ResourceNotFoundException(String message) {
        super(message);
    }
}

RuntimeException を継承した独自例外を作ることで、 Service から「これは 404 にしたい」「これは 409 にしたい」という意図を表現しやすくなる。 super(message) は親クラスのコンストラクタを呼び出す Java の構文である。

GlobalExceptionHandler

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiErrorResponse> handleNotFound(
        ResourceNotFoundException exception,
        HttpServletRequest request
    ) {
        return buildResponse(HttpStatus.NOT_FOUND, exception.getMessage(), request);
    }
}

@RestControllerAdvice は、複数 Controller にまたがって例外処理を共通化するための Spring アノテーションである。 Controller や Service で例外が投げられたとき、該当する @ExceptionHandler メソッドが呼ばれる。

例外 HTTP ステータス 代表例
BadRequestException 400 Bad Request 検索条件なし、空更新、不正 role
ResourceNotFoundException 404 Not Found 存在しない ID の更新、削除
ConflictException 409 Conflict loginId 重複
MethodArgumentNotValidException 400 Bad Request Bean Validation エラー
HttpMessageNotReadableException 400 Bad Request 壊れた JSON
DataIntegrityViolationException 409 Conflict DB 制約違反

11. Repository の派生クエリメソッド

UserRepository は JpaRepository<User, Long> を継承している。 これにより、findAll(), findById(), save(), delete() などの基本操作が自動で使える。 

public interface UserRepository extends JpaRepository<User, Long> {

    List<User> findByLoginIdIn(List<String> loginIds);

    List<User> findByNameIn(List<String> names);

    List<User> findByRoleIn(List<Role> roles);

    boolean existsByLoginId(String loginId);

    boolean existsByLoginIdAndIdNot(String loginId, Long id);
}

Spring Data JPA はメソッド名を解析して SQL 相当のクエリを自動生成する。 例えば existsByLoginIdAndIdNot は「指定された loginId を持ち、かつ ID が指定 ID ではないユーザーが存在するか」を調べる。 更新時に自分自身の loginId は許可しつつ、他ユーザーとの重複を拒否するために使っている。

12. @Transactional の意味

Service の各メソッドには @Transactional を付けた。 トランザクションは、DB 操作をひとまとまりとして扱う仕組みである。 途中で例外が発生した場合は、原則としてその処理内の DB 変更がロールバックされる。 

@Override
@Transactional
public UserResponse updateUser(Long id, UserUpdateRequest userUpdateRequest) {
    ...
}

検索や一覧取得のように読み取りだけの処理には @Transactional(readOnly = true) を付けている。 これは「この処理では DB を更新しない」という意図を Spring や JPA に伝えるためである。 

@Override
@Transactional(readOnly = true)
public List<UserResponse> getUserAll() {
    return userRepository.findAll().stream().map(this::toResponse).toList();
}

stream().map(this::toResponse).toList() は Java Stream API の書き方である。 findAll() で取得した List<User> の各要素を UserResponse に変換し、 新しいリストとして返している。 this::toResponse はメソッド参照で、user -> this.toResponse(user) とほぼ同じ意味である。

13. テストの追加内容

以前のテストは Spring の起動確認だけだったが、今回は API の重要な挙動を MockMvc で検証するようにした。 MockMvc は実際の HTTP サーバーを起動せずに、Controller へ HTTP リクエスト相当のテストを実行できる仕組みである。 

@SpringBootTest
@AutoConfigureMockMvc
class UserAppApplicationTests {

    @Autowired
    private MockMvc mockMvc;
}
テスト 確認していること
作成 API レスポンスに password がなく、DB には BCrypt ハッシュが保存される
部分更新 指定した項目だけが変わり、未指定項目は維持される
パスワード更新 新しいパスワードがハッシュ化され、レスポンスには返らない
404 存在しない ID の更新、削除が 404 JSON を返す
400 不正 role、空文字、空更新が 400 JSON を返す
検索 条件なし、複数条件は 400。id/loginId/name/role 検索は成功

MockMvc の基本形

mockMvc.perform(post("/user/create")
        .contentType(MediaType.APPLICATION_JSON)
        .content("""
            {
              "loginId": "member1",
              "name": "Member One",
              "role": "MEMBER",
              "password": "secret"
            }
            """))
    .andExpect(status().isCreated())
    .andExpect(jsonPath("$.password").doesNotExist());

""" ... """ は Java のテキストブロックである。 複数行文字列を読みやすく書けるため、JSON をテストに埋め込むときに便利である。

jsonPath("$.password").doesNotExist() は、レスポンス JSON に password という項目が存在しないことを検証している。 セキュリティ上重要な仕様なので、テストで明示的に固定している。

14. リクエストからレスポンスまでの流れ

例としてユーザー作成 API の流れを追う。

  1. クライアントが POST /user/create に JSON を送る。
  2. Spring が JSON を UserCreateRequest record に変換する。
  3. @Valid により @NotBlank, @Size, @Pattern が実行される。
  4. Controller が userService.createUser(...) を呼び出す。
  5. Service が loginId 重複を確認する。
  6. Service が role 文字列を Role enum に変換する。
  7. Service が passwordEncoder.encode(...) でパスワードをハッシュ化する。
  8. Repository が Entity を DB に保存する。
  9. Service が Entity を UserResponse に変換する。
  10. Controller が 201 Created と JSON を返す。

どこかで例外が発生した場合は、GlobalExceptionHandler が捕捉し、 timestamp/status/error/message/path の JSON に変換する。

15. 今回の設計判断まとめ

  • パスワードは「受け取るが返さない」。保存時は BCrypt でハッシュ化する。
  • レスポンス DTO は公開してよい情報だけを持つ。
  • 更新は既存 Entity を取得してから、指定された項目だけを変更する。
  • Controller では HTTP 入力の形と検索条件ルールを確認する。
  • Service では重複確認、存在確認、ハッシュ化、Entity 更新を行う。
  • Repository は Spring Data JPA の派生クエリで DB アクセスを簡潔に書く。
  • 例外は独自例外と @RestControllerAdvice で HTTP レスポンスへ変換する。
  • テストではセキュリティ仕様と異常系を明示的に検証する。
確認済み: 実装後に mvn clean test を実行し、7 件のテストがすべて成功した。
